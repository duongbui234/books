In generic terms, modules are the bricks for structuring non-trivial applications. Modules allow you to divide the codebase into small units that can be developed and tested independently. Modules are also the main mechanism to enforce information hiding by keeping private all the functions and variables that are not explicitly marked to be exported.

> Nói một cách tổng quát, module là những viên gạch để cấu trúc nên các ứng dụng phi tầm thường. Module cho phép bạn chia codebase thành các đơn vị nhỏ có thể được phát triển và kiểm thử một cách độc lập. Module cũng là cơ chế chính để thực thi việc che giấu thông tin (information hiding) bằng cách giữ riêng tư tất cả các hàm và biến không được đánh dấu xuất ra một cách tường minh.

If you come from other languages, you have probably seen similar concepts being referred to with different names: **package** (Java, Go, PHP, Rust, or Dart), assembly (.NET), library (Ruby), or unit (Pascal dialects). The terminology is not perfectly interchangeable because every language or ecosystem comes with its own unique characteristics, but there is a significant overlap between these concepts.

> Nếu bạn đến từ các ngôn ngữ khác, có lẽ bạn đã từng thấy những khái niệm tương tự được gọi bằng những cái tên khác nhau: package (Java, Go, PHP, Rust hay Dart), assembly (.NET), library (Ruby) hay unit (các phương ngữ Pascal). Thuật ngữ không hoàn toàn thay thế cho nhau được vì mỗi ngôn ngữ hay hệ sinh thái đều có những đặc trưng riêng, nhưng có sự chồng lấp đáng kể giữa các khái niệm này.

Interestingly enough, Node.js currently comes with two different **module** systems: **CommonJS** (CJS) and ECMAScript modules (**ESM** or ES modules). In this chapter, we will discuss why there are two alternatives, we will learn about their pros and cons, and, finally, we will analyze several common patterns that are relevant when using or writing Node.js modules. By the end of this chapter, you should be able to make pragmatic choices about how to use modules effectively and how to write your own custom modules.

> Điều thú vị là Node.js hiện tại đi kèm với hai hệ thống module khác nhau: CommonJS (CJS) và ECMAScript modules (ESM hay ES modules). Trong chương này, chúng ta sẽ bàn về lý do tại sao lại có hai lựa chọn, chúng ta sẽ tìm hiểu ưu và nhược điểm của chúng, và cuối cùng, chúng ta sẽ phân tích một số mẫu phổ biến có liên quan khi sử dụng hoặc viết module Node.js. ## The need for modules — Nhu cầu về module

A good **module system** should help with addressing some fundamental needs of software engineering:

> Một hệ thống module tốt nên giúp giải quyết một số nhu cầu cơ bản của kỹ thuật phần mềm:

- Having a way to split the codebase into multiple files. This helps with keeping the code more organized, making it easier to understand but also helps with developing and testing various pieces of functionality independently from each other.
- Allowing code reuse across different projects. A module can, in fact, implement a generic feature that can be useful for different projects. Organizing such functionality within a module can make it easier to bring it into the different projects that may want to use it.
- **Encapsulation** (or information hiding). It is generally a good idea to hide implementation complexity and only expose simple interfaces with clear responsibilities. Most module systems allow to selectively keep the private part of the code hidden, while exposing a public interface, such as functions, classes, or objects that are meant to be used by the consumers of the module.
- Managing dependencies. A good module system should make it easy for module developers to build on top of existing modules, including third-party ones. A module system should also make it easy for module users to import the chain of dependencies that are necessary for a given module to run (transient dependencies).

> - Có một cách để chia codebase thành nhiều file. Điều này giúp giữ cho code có tổ chức hơn, dễ hiểu hơn nhưng cũng giúp phát triển và kiểm thử các phần chức năng khác nhau một cách độc lập với nhau.
> - Cho phép tái sử dụng code giữa các dự án khác nhau. Thực tế, một module có thể hiện thực một tính năng tổng quát hữu ích cho nhiều dự án khác nhau. Tổ chức chức năng như vậy bên trong một module có thể giúp đưa nó vào các dự án khác nhau muốn sử dụng nó dễ dàng hơn.
> - Đóng gói (encapsulation) (hay che giấu thông tin). Nhìn chung, ẩn đi sự phức tạp của phần hiện thực và chỉ phơi bày các interface đơn giản với trách nhiệm rõ ràng là một ý tưởng tốt. Hầu hết các hệ thống module cho phép giữ ẩn phần riêng tư của code một cách có chọn lọc, đồng thời phơi bày một interface công khai, chẳng hạn như các hàm, lớp hay đối tượng dành cho những bên tiêu thụ module sử dụng.
> - Quản lý các phụ thuộc (dependency). Một hệ thống module tốt nên giúp các nhà phát triển module dễ dàng xây dựng dựa trên các module có sẵn, bao gồm cả các module của bên thứ ba. Một hệ thống module cũng nên giúp người dùng module dễ dàng import chuỗi các phụ thuộc cần thiết để một module nhất định có thể chạy (các phụ thuộc bắc cầu - transient dependencies).

It is important to clarify the distinction between a module and a module system. We can define a module as the actual unit of software, while a module system is the syntax and the tooling that allows us to define modules and to use them within our projects.

> Điều quan trọng là phải làm rõ sự phân biệt giữa module và hệ thống module. Chúng ta có thể định nghĩa module là đơn vị phần mềm thực tế, trong khi hệ thống module là cú pháp và bộ công cụ cho phép chúng ta định nghĩa các module và sử dụng chúng trong các dự án của mình.

## Module systems in JavaScript and Node.js — Hệ thống module trong JavaScript và Node.js

Not all programming languages come with a built-in module system, and JavaScript had been lacking this feature for a long time.

> Không phải mọi ngôn ngữ lập trình đều đi kèm với một hệ thống module dựng sẵn, và JavaScript đã thiếu tính năng này trong một thời gian dài.

In the browser landscape, it is possible to split the codebase into multiple files and then import them by using different `<script>` tags. For many years, this approach was good enough to build simple interactive websites, and JavaScript developers managed to get things done without having a fully-fledged module system.

> Trong bối cảnh trình duyệt, có thể chia codebase thành nhiều file rồi import chúng bằng cách dùng nhiều thẻ `<script>` khác nhau. Trong nhiều năm, cách tiếp cận này đủ tốt để xây dựng các website tương tác đơn giản, và các nhà phát triển JavaScript đã xoay xở hoàn thành công việc mà không cần đến một hệ thống module đầy đủ.

Only when JavaScript browser applications became more complicated and frameworks like jQuery, Backbone, and AngularJS took over the ecosystem, the JavaScript community came up with several initiatives aimed at defining a module system that could be effectively adopted within JavaScript projects. The most successful ones were asynchronous module definition (AMD), popularized by RequireJS (`nodejsdp.link/requirejs`), and later Universal Module Definition (UMD – `nodejsdp.link/umd`).

> Chỉ khi các ứng dụng JavaScript trên trình duyệt trở nên phức tạp hơn và các framework như jQuery, Backbone và AngularJS chiếm lĩnh hệ sinh thái, cộng đồng JavaScript mới đưa ra một số sáng kiến nhằm định nghĩa một hệ thống module có thể được áp dụng hiệu quả trong các dự án JavaScript. Những sáng kiến thành công nhất là asynchronous module definition (AMD), được phổ biến bởi RequireJS (`nodejsdp.link/requirejs`), và sau đó là Universal Module Definition (UMD – `nodejsdp.link/umd`).

When Node.js was created, it was conceived as a server **runtime** for JavaScript with direct access to the underlying filesystem so there was a unique opportunity to introduce a different way to manage modules. The idea was not to rely on HTML `<script>` tags and resources accessible through URLs. Instead, the choice was to rely purely on JavaScript files available on the local filesystem. For its module system, Node.js came up with an implementation of the CommonJS specification (sometimes also referred to as CJS, `nodejsdp.link/commonjs`), which was designed to provide a module system for JavaScript in browserless environments.

> Khi Node.js được tạo ra, nó được hình dung như một runtime phía máy chủ cho JavaScript với quyền truy cập trực tiếp vào hệ thống tập tin bên dưới, do đó có một cơ hội độc đáo để giới thiệu một cách quản lý module khác. Ý tưởng là không dựa vào các thẻ `<script>` của HTML và các tài nguyên có thể truy cập thông qua URL. Thay vào đó, lựa chọn là dựa hoàn toàn vào các file JavaScript có sẵn trên hệ thống tập tin cục bộ. Đối với hệ thống module của mình, Node.js đã đưa ra một hiện thực của đặc tả CommonJS (đôi khi cũng được gọi là CJS, `nodejsdp.link/commonjs`), vốn được thiết kế để cung cấp một hệ thống module cho JavaScript trong các môi trường không có trình duyệt.

CommonJS has been the dominant module system in Node.js since its inception and it has become very prominent also in the browser landscape thanks to module bundlers like Browserify (`nodejsdp.link/browserify`) and webpack (`nodejsdp.link/` `webpack`).

> CommonJS đã là hệ thống module thống trị trong Node.js kể từ khi nó ra đời và cũng đã trở nên rất nổi bật trong bối cảnh trình duyệt nhờ các module bundler như Browserify (`nodejsdp.link/browserify`) và webpack (`nodejsdp.link/` `webpack`).

In 2015, with the release of ECMAScript 6 (also called ECMAScript 2015 or ES2015), there was finally an official proposal for a standard module system: ESM or ECMAScript modules. ESM brings a lot of innovation in the JavaScript ecosystem and, among other things, it tries to bridge the gap between how modules are managed on browsers and servers.

> Năm 2015, với sự ra mắt của ECMAScript 6 (còn gọi là ECMAScript 2015 hay ES2015), cuối cùng đã có một đề xuất chính thức cho một hệ thống module chuẩn: ESM hay ECMAScript modules. ESM mang lại rất nhiều đổi mới trong hệ sinh thái JavaScript và, cùng với những điều khác, nó cố gắng thu hẹp khoảng cách giữa cách module được quản lý trên trình duyệt và máy chủ.

ECMAScript 6 defined only the formal specification for ESM in terms of syntax and semantics, but it didn't provide any implementation details. It took different browser companies and the Node.js community several years to come up with solid implementations of the specification. Node.js ships with stable support for ESM starting from version 13.2. At the time of writing, the general feeling is that ESM is going to become the de facto way to manage JavaScript modules in both the browser and the server landscape. The reality today, though, is that the majority of projects are still heavily relying on CommonJS and it will take some time for ESM to catch up and eventually become the dominant standard.

> ECMAScript 6 chỉ định nghĩa đặc tả chính thức cho ESM về mặt cú pháp và ngữ nghĩa, nhưng nó không cung cấp bất kỳ chi tiết hiện thực nào. Phải mất vài năm để các công ty trình duyệt khác nhau và cộng đồng Node.js đưa ra được những hiện thực vững chắc cho đặc tả này. Node.js đi kèm hỗ trợ ổn định cho ESM bắt đầu từ phiên bản 13.2. Tại thời điểm viết sách, cảm nhận chung là ESM sẽ trở thành cách thực tế (de facto) để quản lý module JavaScript trong cả bối cảnh trình duyệt lẫn máy chủ. Tuy nhiên, thực tế ngày nay là phần lớn các dự án vẫn phụ thuộc nặng nề vào CommonJS và sẽ mất một thời gian để ESM bắt kịp và cuối cùng trở thành chuẩn thống trị.

To provide a comprehensive overview of module-related patterns in Node.js, in the first part of this chapter, we will discuss them in the context of CommonJS, and then, in the second part of the chapter, we will revisit our learnings using ESM.

> Để cung cấp một cái nhìn tổng quan toàn diện về các mẫu liên quan đến module trong Node.js, trong phần đầu của chương này, chúng ta sẽ bàn về chúng trong bối cảnh CommonJS, và sau đó, trong phần thứ hai của chương, chúng ta sẽ xem lại những gì đã học bằng ESM.

The goal of this chapter is to make you comfortable with both module systems, but in the rest of the book, we will only be using ESM for our code examples. The idea is to encourage you to leverage ESM as much as possible so that your code will be more future-proof.

> Mục tiêu của chương này là giúp bạn cảm thấy thoải mái với cả hai hệ thống module, nhưng trong phần còn lại của cuốn sách, chúng ta sẽ chỉ sử dụng ESM cho các ví dụ code. Ý tưởng là khuyến khích bạn tận dụng ESM càng nhiều càng tốt để code của bạn được chuẩn bị tốt hơn cho tương lai.

If you are reading this chapter a few years after its publication, you are probably not too worried about CommonJS, and you might want to jump straight into the ESM part. This is probably fine, but we still encourage you to go through the entire chapter, because understanding CommonJS and its characteristics will certainly be beneficial in helping you to understand ESM and its strengths in much more depth.

> Nếu bạn đang đọc chương này vài năm sau khi nó được xuất bản, có lẽ bạn không quá lo lắng về CommonJS, và bạn có thể muốn nhảy thẳng vào phần ESM. Điều này có lẽ ổn, nhưng chúng tôi vẫn khuyến khích bạn đọc qua toàn bộ chương, bởi vì việc hiểu CommonJS và các đặc điểm của nó chắc chắn sẽ có ích trong việc giúp bạn hiểu ESM và những điểm mạnh của nó một cách sâu sắc hơn nhiều.

## The module system and its patterns — Hệ thống module và các mẫu của nó

As we said, modules are the bricks for structuring non-trivial applications and the main mechanism to enforce information hiding by keeping private all the functions and variables that are not explicitly marked to be exported.

> Như đã nói, module là những viên gạch để cấu trúc nên các ứng dụng phi tầm thường và là cơ chế chính để thực thi việc che giấu thông tin bằng cách giữ riêng tư tất cả các hàm và biến không được đánh dấu xuất ra một cách tường minh.

Before getting into the specifics of CommonJS, let's discuss a generic pattern that helps with information hiding and that we will be using for building a simple module system, which is the revealing module pattern.

> Trước khi đi vào cụ thể của CommonJS, hãy bàn về một mẫu tổng quát giúp ích cho việc che giấu thông tin và mà chúng ta sẽ dùng để xây dựng một hệ thống module đơn giản, đó là revealing module pattern.

### The revealing module pattern — Revealing module pattern

One of the bigger problems with JavaScript in the browser is the lack of namespacing. Every script runs in the global **scope**; therefore, internal application code or third-party dependencies can pollute the scope while exposing their own pieces of functionality. This can be extremely harmful. Imagine, for instance, that a third-party library instantiates a global variable called `utils`. If any other library, or the application code itself, accidentally overrides or alters `utils`, the code that relies on it will likely crash in some unpredictable way. Unpredictable side effects can also happen if other libraries or the application code accidentally invoke a function of another library meant for internal use only. In short, relying on the global scope is a very risky business, especially as your application grows and you have to rely more and more on functionality implemented by other individuals.

> Một trong những vấn đề lớn hơn với JavaScript trong trình duyệt là thiếu cơ chế không gian tên (namespacing). Mọi script đều chạy trong phạm vi toàn cục; do đó, code nội bộ của ứng dụng hoặc các phụ thuộc của bên thứ ba có thể làm ô nhiễm phạm vi này trong khi phơi bày các phần chức năng của riêng chúng. Điều này có thể cực kỳ tai hại. Ví dụ, hãy tưởng tượng rằng một thư viện của bên thứ ba khởi tạo một biến toàn cục tên là `utils`. Nếu bất kỳ thư viện nào khác, hoặc chính code của ứng dụng, vô tình ghi đè hoặc thay đổi `utils`, thì code dựa vào nó có thể sẽ sập theo một cách khó lường nào đó. Các tác dụng phụ khó lường cũng có thể xảy ra nếu các thư viện khác hoặc code của ứng dụng vô tình gọi một hàm của một thư viện khác vốn chỉ dành cho mục đích sử dụng nội bộ. Tóm lại, dựa vào phạm vi toàn cục là một việc rất rủi ro, đặc biệt là khi ứng dụng của bạn lớn dần và bạn phải ngày càng dựa vào các chức năng được hiện thực bởi những cá nhân khác.

A popular technique to solve this class of problems is called the revealing module pattern, and it looks like this:

> Một kỹ thuật phổ biến để giải quyết lớp vấn đề này được gọi là revealing module pattern, và nó trông như thế này:

```js
const myModule = (() => {
  const privateFoo = () => {}
  const privateBar = []
  const exported = {
    publicFoo: () => {},
    publicBar: () => {}
  }
  return exported
})() // once the parenthesis here are parsed, the function
     // will be invoked
console.log(myModule)
console.log(myModule.privateFoo, myModule.privateBar)
```

This pattern leverages a self-invoking function. This type of function is sometimes also referred to as Immediately Invoked Function Expression (IIFE) and it is used to create a private scope, exporting only the parts that are meant to be public.

> Mẫu này tận dụng một hàm tự gọi (self-invoking function). Loại hàm này đôi khi cũng được gọi là Immediately Invoked Function Expression (IIFE) và nó được dùng để tạo ra một phạm vi riêng tư, chỉ xuất ra những phần được dành để công khai.

In JavaScript, variables created inside a function are not accessible from the outer scope (outside the function). Functions can use the `return` statement to selectively propagate information to the outer scope.

> Trong JavaScript, các biến được tạo bên trong một hàm thì không thể truy cập được từ phạm vi bên ngoài (bên ngoài hàm). Các hàm có thể dùng câu lệnh `return` để truyền thông tin một cách có chọn lọc ra phạm vi bên ngoài.

This pattern is essentially exploiting these properties to keep the private information hidden and export only a public-facing **API**.

> Mẫu này về cơ bản đang khai thác các thuộc tính này để giữ ẩn thông tin riêng tư và chỉ xuất ra một API hướng ra công khai.

In the preceding code, the `myModule` variable contains only the exported API, while the rest of the module content is practically inaccessible from outside.

> Trong đoạn code trên, biến `myModule` chỉ chứa API được xuất ra, trong khi phần còn lại của nội dung module trên thực tế không thể truy cập được từ bên ngoài.

The `log` statement is going to print something like this:

> Câu lệnh `log` sẽ in ra một thứ gì đó như thế này:

```js
{ publicFoo: [Function: publicFoo],
  publicBar: [Function: publicBar] }
undefined undefined
```

This demonstrates that only the `exported` properties are directly accessible from `myModule`.

> Điều này chứng minh rằng chỉ các thuộc tính của `exported` mới có thể truy cập trực tiếp từ `myModule`.

As we will see in a moment, the idea behind this pattern is used as a base for the CommonJS module system.

> Như chúng ta sẽ thấy ngay sau đây, ý tưởng đằng sau mẫu này được dùng làm nền tảng cho hệ thống module CommonJS.

## CommonJS modules — Module CommonJS

CommonJS is the first module system originally built into Node.js. Node.js' CommonJS implementation respects the CommonJS specification, with the addition of some custom extensions.

> CommonJS là hệ thống module đầu tiên vốn được dựng sẵn trong Node.js. Hiện thực CommonJS của Node.js tuân theo đặc tả CommonJS, có bổ sung thêm một số phần mở rộng tùy chỉnh.

Let's summarize two of the main concepts of the CommonJS specification:

> Hãy tóm tắt hai trong số các khái niệm chính của đặc tả CommonJS:

- `require` is a function that allows you to import a module from the local filesystem
- `exports` and `module.exports` are special variables that can be used to export public functionality from the current module

> - `require` là một hàm cho phép bạn import một module từ hệ thống tập tin cục bộ
> - `exports` và `module.exports` là các biến đặc biệt có thể được dùng để xuất các chức năng công khai từ module hiện tại

This information is sufficient for now; we will learn more details and some of the nuances of the CommonJS specification in the next few sections.

> Thông tin này hiện tại là đủ; chúng ta sẽ tìm hiểu thêm chi tiết và một số sắc thái của đặc tả CommonJS trong vài phần tiếp theo.

### A homemade module loader — Một bộ nạp module tự chế

To understand how CommonJS works in Node.js, let's build a similar system from scratch. The code that follows creates a function that mimics a subset of the functionality of the original `require()` function of Node.js.

> Để hiểu cách CommonJS hoạt động trong Node.js, hãy xây dựng một hệ thống tương tự từ đầu. Đoạn code sau đây tạo ra một hàm mô phỏng một tập con chức năng của hàm `require()` gốc của Node.js.

Let's start by creating a function that loads the content of a module, wraps it into a private scope, and evaluates it:

> Hãy bắt đầu bằng cách tạo một hàm nạp nội dung của một module, gói nó vào một phạm vi riêng tư, rồi đánh giá (evaluate) nó:

```js
function loadModule (filename, module, require) {
  const wrappedSrc =
    `(function (module, exports, require) {
      ${fs.readFileSync(filename, 'utf8')}
    })(module, module.exports, require)`
  eval(wrappedSrc)
}
```

The source code of a module is essentially wrapped into a function, as it was for the revealing module pattern. The difference here is that we pass a list of variables to the module, in particular, `module`, `exports`, and `require`. Make a note of how the `exports` argument of the wrapping function is initialized with the content of `module.exports`, as we will talk about this later. Another important detail to mention is that we are using `readFileSync` to read the module's content. While it is generally not recommended to use the synchronous version of the filesystem APIs, here it makes sense to do so. The reason for that is that loading modules in CommonJS are deliberately synchronous operations. This approach makes sure that, if we are importing multiple modules, they (and their dependencies) are loaded in the right order. We will talk more about this aspect later in the chapter.

> Mã nguồn của một module về cơ bản được gói vào trong một hàm, giống như đối với revealing module pattern. Điểm khác biệt ở đây là chúng ta truyền một danh sách các biến vào module, cụ thể là `module`, `exports` và `require`. Hãy lưu ý cách tham số `exports` của hàm bao bọc được khởi tạo bằng nội dung của `module.exports`, vì chúng ta sẽ bàn về điều này sau. Một chi tiết quan trọng khác cần đề cập là chúng ta đang dùng `readFileSync` để đọc nội dung của module. Mặc dù nhìn chung không nên dùng phiên bản đồng bộ của các API hệ thống tập tin, nhưng ở đây làm vậy là hợp lý. Lý do là việc nạp các module trong CommonJS là các thao tác đồng bộ một cách có chủ đích. Cách tiếp cận này đảm bảo rằng, nếu chúng ta đang import nhiều module, thì chúng (và các phụ thuộc của chúng) được nạp theo đúng thứ tự. Chúng ta sẽ bàn thêm về khía cạnh này ở phần sau của chương.

Bear in mind that this is only an example, and you will rarely need to evaluate some source code in a real application. Features such as `eval()` or the functions of the `vm` module (`nodejsdp.link/vm`) can be easily used in the wrong way or with the wrong input, thus opening a system to code injection attacks. They should always be used with extreme care or avoided altogether.

> Hãy nhớ rằng đây chỉ là một ví dụ, và bạn sẽ hiếm khi cần phải đánh giá một đoạn mã nguồn nào đó trong một ứng dụng thực tế. Các tính năng như `eval()` hay các hàm của module `vm` (`nodejsdp.link/vm`) rất dễ bị dùng sai cách hoặc với đầu vào sai, do đó mở ra cho hệ thống các cuộc tấn công tiêm code (code injection). Chúng luôn phải được dùng với sự cẩn trọng cực độ hoặc tránh dùng hoàn toàn.

Let's now implement the `require()` function:

> Bây giờ hãy hiện thực hàm `require()`:

```js
function require (moduleName) {
  console.log(`Require invoked for module: ${moduleName}`)
  const id = require.resolve(moduleName)                   // (1)
  if (require.cache[id]) {                                 // (2)
    return require.cache[id].exports
  }
  // module metadata
  const module = {                                         // (3)
    exports: {},
    id
  }
  // Update the cache
  require.cache[id] = module                               // (4)
  // load the module
  loadModule(id, module, require)                          // (5)
  // return exported variables
  return module.exports                                    // (6)
}
require.cache = {}
require.resolve = (moduleName) => {
  /* resolve a full module id from the moduleName */
}
```

The previous function simulates the behavior of the original `require()` function of Node.js, which is used to load a module. Of course, this is just for educational purposes and does not accurately or completely reflect the internal behavior of the real `require()` function, but it's great to understand the internals of the Node.js module system, including how a module is defined and loaded.

> Hàm trước đó mô phỏng hành vi của hàm `require()` gốc của Node.js, vốn được dùng để nạp một module. Tất nhiên, điều này chỉ nhằm mục đích giáo dục và không phản ánh một cách chính xác hay đầy đủ hành vi nội bộ của hàm `require()` thực sự, nhưng nó rất hữu ích để hiểu cơ chế nội bộ của hệ thống module Node.js, bao gồm cách một module được định nghĩa và nạp.

What our homemade module system does is explained as follows:

> Những gì hệ thống module tự chế của chúng ta làm được giải thích như sau:

1. A module name is accepted as input, and the very first thing that we do is resolve the full path of the module, which we call `id`. This task is delegated to `require.resolve()`, which implements a specific resolving algorithm (we will talk about it later).
2. If the module has already been loaded in the past, it should be available in the cache. If this is the case, we just return it immediately.
3. If the module has never been loaded before, we set up the environment for the first load. In particular, we create a `module` object that contains an `exports` property initialized with an empty object literal. This object will be populated by the code of the module to export its public API.
4. After the first load, the `module` object is cached.
5. The module source code is read from its file and the code is evaluated, as we saw before. We provide the module with the `module` object that we just created, and a reference to the `require()` function. The module exports its public API by manipulating or replacing the `module.exports` object.
6. Finally, the content of `module.exports`, which represents the public API of the module, is returned to the caller.

> 1. Một tên module được nhận làm đầu vào, và điều đầu tiên chúng ta làm là phân giải (resolve) đường dẫn đầy đủ của module, mà chúng ta gọi là `id`. Nhiệm vụ này được ủy thác cho `require.resolve()`, vốn hiện thực một thuật toán phân giải cụ thể (chúng ta sẽ bàn về nó sau).
> 2. Nếu module đã được nạp trước đó, nó sẽ có sẵn trong cache. Nếu đúng như vậy, chúng ta chỉ cần trả về nó ngay lập tức.
> 3. Nếu module chưa bao giờ được nạp trước đó, chúng ta thiết lập môi trường cho lần nạp đầu tiên. Cụ thể, chúng ta tạo một đối tượng `module` chứa một thuộc tính `exports` được khởi tạo bằng một object literal rỗng. Đối tượng này sẽ được code của module điền vào để xuất ra API công khai của nó.
> 4. Sau lần nạp đầu tiên, đối tượng `module` được lưu vào cache.
> 5. Mã nguồn của module được đọc từ file của nó và code được đánh giá, như chúng ta đã thấy trước đó. Chúng ta cung cấp cho module đối tượng `module` mà chúng ta vừa tạo, và một tham chiếu đến hàm `require()`. Module xuất ra API công khai của nó bằng cách thao tác hoặc thay thế đối tượng `module.exports`.
> 6. Cuối cùng, nội dung của `module.exports`, vốn đại diện cho API công khai của module, được trả về cho bên gọi.

As we can see, there is nothing magical behind the workings of the Node.js module system. The trick is all in the wrapper we create around a module's source code and the artificial environment in which we run it.

> Như chúng ta có thể thấy, không có gì kỳ diệu đằng sau cách hoạt động của hệ thống module Node.js. Mẹo nằm hết ở lớp bao bọc mà chúng ta tạo quanh mã nguồn của module và môi trường nhân tạo mà chúng ta chạy nó trong đó.

By looking at how our custom `require()` function works, we should now be able to understand how to define a module. The following code gives us an example:

> Bằng cách xem cách hàm `require()` tùy chỉnh của chúng ta hoạt động, giờ đây chúng ta sẽ có thể hiểu cách định nghĩa một module. Đoạn code sau đây cho chúng ta một ví dụ:

### module.exports versus exports — module.exports so với exports

For many developers who are not yet familiar with Node.js, a common source of confusion is the difference between using `exports` and `module.exports` to expose a public API. The code of our custom `require()` function should again clear any doubt. The `exports` variable is just a reference to the initial value of `module.exports`. We have seen that such a value is essentially a simple object literal created before the module is loaded.

> Đối với nhiều nhà phát triển chưa quen với Node.js, một nguồn gây nhầm lẫn phổ biến là sự khác biệt giữa việc dùng `exports` và `module.exports` để phơi bày một API công khai. Code của hàm `require()` tùy chỉnh của chúng ta một lần nữa sẽ xua tan mọi nghi ngờ. Biến `exports` chỉ là một tham chiếu đến giá trị ban đầu của `module.exports`. Chúng ta đã thấy rằng giá trị như vậy về cơ bản là một object literal đơn giản được tạo trước khi module được nạp.

This means that we can only attach new properties to the object referenced by the `exports` variable, as shown in the following code:

> Điều này có nghĩa là chúng ta chỉ có thể gắn thêm các thuộc tính mới vào đối tượng được tham chiếu bởi biến `exports`, như được minh họa trong đoạn code sau:

```js
exports.hello = () => {
  console.log('Hello')
}
```

Reassigning the `exports` variable doesn't have any effect, because it doesn't change the content of `module.exports`. It will only reassign the variable itself. The following code is therefore wrong:

> Việc gán lại biến `exports` không có tác dụng gì, bởi vì nó không làm thay đổi nội dung của `module.exports`. Nó chỉ gán lại chính biến đó. Do đó, đoạn code sau là sai:

```js
exports = () => {
  console.log('Hello')
}
```

If we want to export something other than an object literal, such as a function, an instance, or even a string, we have to reassign `module.exports` as follows:

> Nếu chúng ta muốn xuất ra một thứ gì đó khác ngoài một object literal, chẳng hạn như một hàm, một instance, hay thậm chí một chuỗi, chúng ta phải gán lại `module.exports` như sau:

```js
module.exports = () => {
  console.log('Hello')
}
```

### The require function is synchronous — Hàm require là đồng bộ

A very important detail that we should take into account is that our homemade `require()` function is synchronous. In fact, it returns the module contents using a simple direct style, and no **callback** is required. This is true for the original Node.js `require()` function too. As a consequence, any assignment to `module.` `exports` must be synchronous as well. For example, the following code is incorrect and it will cause trouble:

> Một chi tiết rất quan trọng mà chúng ta nên lưu ý là hàm `require()` tự chế của chúng ta là đồng bộ. Thực tế, nó trả về nội dung của module bằng một phong cách trực tiếp đơn giản, và không cần callback nào cả. Điều này cũng đúng với hàm `require()` gốc của Node.js. Hệ quả là, bất kỳ phép gán nào cho `module.` `exports` cũng phải đồng bộ. Ví dụ, đoạn code sau là không đúng và nó sẽ gây ra rắc rối:

```js
setTimeout(() => {
  module.exports = function() {...}
}, 100)
```

The synchronous nature of `require()` has important repercussions on the way we define modules, as it limits us to mostly using synchronous code during the definition of a module. This is one of the most important reasons why the core Node.js libraries offer synchronous APIs as an alternative to most of the asynchronous ones.

> Bản chất đồng bộ của `require()` có những hệ quả quan trọng đối với cách chúng ta định nghĩa module, vì nó giới hạn chúng ta chủ yếu phải dùng code đồng bộ trong quá trình định nghĩa một module. Đây là một trong những lý do quan trọng nhất khiến các thư viện lõi của Node.js cung cấp các API đồng bộ như một lựa chọn thay thế cho hầu hết các API bất đồng bộ.

If we need some asynchronous initialization steps for a module, we can always define and export an uninitialized module that is initialized asynchronously at a later time. The problem with this approach, though, is that loading such a module using `require()` does not guarantee that it's ready to be used. In Chapter 11, Advanced Recipes, we will analyze this problem in detail and present some patterns to solve this issue elegantly.

> Nếu chúng ta cần một số bước khởi tạo bất đồng bộ cho một module, chúng ta luôn có thể định nghĩa và xuất ra một module chưa được khởi tạo, sau đó được khởi tạo một cách bất đồng bộ vào lúc muộn hơn. Tuy nhiên, vấn đề với cách tiếp cận này là việc nạp một module như vậy bằng `require()` không đảm bảo rằng nó đã sẵn sàng để sử dụng. Trong Chương 11, Các công thức nâng cao, chúng ta sẽ phân tích vấn đề này một cách chi tiết và trình bày một số mẫu để giải quyết vấn đề này một cách thanh lịch.

For the sake of curiosity, you might want to know that in its early days, Node.js used to have an asynchronous version of `require()`, but it was soon removed because it was overcomplicating a functionality that was actually only meant to be used at initialization time and where asynchronous I/O brings more complexities than advantages.

> Vì sự tò mò, có thể bạn muốn biết rằng trong những ngày đầu, Node.js từng có một phiên bản bất đồng bộ của `require()`, nhưng nó sớm bị loại bỏ vì nó làm phức tạp hóa quá mức một chức năng vốn thực ra chỉ nhằm được dùng vào lúc khởi tạo và là nơi mà I/O bất đồng bộ mang lại nhiều phức tạp hơn là lợi ích.

### The resolving algorithm — Thuật toán phân giải (resolving)

The term **dependency** hell describes a situation whereby two or more dependencies of a program in turn depend on a shared dependency, but require different incompatible versions. Node.js solves this problem elegantly by loading a different version of a module depending on where the module is loaded from. All the merits of this feature go to the way Node.js package managers (such as npm or yarn) organize the dependencies of the application, and also to the resolving algorithm used in the `require()` function. Let's now give a quick overview of this algorithm. As we saw, the `resolve()` function takes a module name (which we will call `moduleName`) as input and it returns the full path of the module. This path is then used to load its code and also to identify the module uniquely. The resolving algorithm can be divided into the following three major branches:

> Thuật ngữ dependency hell mô tả tình huống mà trong đó hai hay nhiều phụ thuộc của một chương trình lại phụ thuộc vào một phụ thuộc chung, nhưng đòi hỏi các phiên bản khác nhau và không tương thích với nhau. Node.js giải quyết vấn đề này một cách thanh lịch bằng cách nạp một phiên bản module khác nhau tùy thuộc vào nơi module được nạp. Mọi công lao của tính năng này thuộc về cách mà các trình quản lý gói của Node.js (như npm hay yarn) tổ chức các phụ thuộc của ứng dụng, và cũng thuộc về thuật toán phân giải được dùng trong hàm `require()`. Bây giờ hãy điểm qua nhanh về thuật toán này. Như chúng ta đã thấy, hàm `resolve()` nhận một tên module (mà chúng ta sẽ gọi là `moduleName`) làm đầu vào và nó trả về đường dẫn đầy đủ của module. Đường dẫn này sau đó được dùng để nạp code của nó và cũng để định danh module một cách duy nhất. Thuật toán phân giải có thể được chia thành ba nhánh chính sau đây:

- File modules: If `moduleName` starts with `/`, it is already considered an absolute path to the module and it's returned as it is. If it starts with `./`, then `moduleName` is considered a relative path, which is calculated starting from the directory of the requiring module.
- Core modules: If `moduleName` is not prefixed with `/` or `./`, the algorithm will first try to search within the core Node.js modules.
- Package modules: If no core module is found matching `moduleName`, then the search continues by looking for a matching module in the first `node_` `modules` directory that is found navigating up in the directory structure starting from the requiring module. The algorithm continues to search for a match by looking into the next `node_modules` directory up in the directory tree, until it reaches the root of the filesystem.

> - File modules: Nếu `moduleName` bắt đầu bằng `/`, nó đã được coi là một đường dẫn tuyệt đối đến module và được trả về nguyên trạng. Nếu nó bắt đầu bằng `./`, thì `moduleName` được coi là một đường dẫn tương đối, được tính bắt đầu từ thư mục của module đang yêu cầu.
> - Core modules: Nếu `moduleName` không có tiền tố `/` hay `./`, thuật toán trước tiên sẽ thử tìm kiếm trong các module lõi của Node.js.
> - Package modules: Nếu không tìm thấy module lõi nào khớp với `moduleName`, thì việc tìm kiếm tiếp tục bằng cách tìm một module khớp trong thư mục `node_` `modules` đầu tiên được tìm thấy khi điều hướng đi lên trong cấu trúc thư mục bắt đầu từ module đang yêu cầu. Thuật toán tiếp tục tìm kiếm một kết quả khớp bằng cách xem xét thư mục `node_modules` tiếp theo đi lên trong cây thư mục, cho đến khi nó đạt đến gốc của hệ thống tập tin.

For file and package modules, both files and directories can match `moduleName`. In particular, the algorithm will try to match the following:

> Đối với file modules và package modules, cả file lẫn thư mục đều có thể khớp với `moduleName`. Cụ thể, thuật toán sẽ thử khớp những điều sau:

- `<moduleName>.js`
- `<moduleName>/index.js`
- The directory/file specified in the `main` property of `<moduleName>/package.json`

> - `<moduleName>.js`
> - `<moduleName>/index.js`
> - Thư mục/file được chỉ định trong thuộc tính `main` của `<moduleName>/package.json`

The complete, formal documentation of the resolving algorithm can be found at `nodejsdp.link/resolve`.

> Tài liệu chính thức, đầy đủ về thuật toán phân giải có thể được tìm thấy tại `nodejsdp.link/resolve`.

The `node_modules` directory is actually where the package managers install the dependencies of each package. This means that, based on the algorithm we just described, each package can have its own private dependencies. For example, consider the following directory structure:

> Thư mục `node_modules` thực ra là nơi các trình quản lý gói cài đặt các phụ thuộc của mỗi gói. Điều này có nghĩa là, dựa trên thuật toán chúng ta vừa mô tả, mỗi gói có thể có các phụ thuộc riêng tư của riêng nó. Ví dụ, hãy xem xét cấu trúc thư mục sau:

```js
myApp
├── foo.js
└── node_modules
    ├── depA
    │   └── index.js
    ├── depB
    │   ├── bar.js
    │   └── node_modules
    │       └── depA
    │           └── index.js
    └── depC
        ├── foobar.js
        └── node_modules
            └── depA
                └── index.js
```

In the previous example, `myApp`, `depB`, and `depC` all depend on `depA`. However, they all have their own private version of the dependency! Following the rules of the resolving algorithm, using `require('depA')` will load a different file depending on the module that requires it, for example:

> Trong ví dụ trước, `myApp`, `depB` và `depC` đều phụ thuộc vào `depA`. Tuy nhiên, tất cả chúng đều có phiên bản riêng tư của riêng mình của phụ thuộc đó! Tuân theo các quy tắc của thuật toán phân giải, việc dùng `require('depA')` sẽ nạp một file khác nhau tùy thuộc vào module yêu cầu nó, ví dụ:

- Calling `require('depA')` from `/myApp/foo.js` will load `/myApp/node_modules/depA/index.js`
- Calling `require('depA')` from `/myApp/node_modules/depB/bar.js` will load `/myApp/node_modules/depB/node_modules/depA/index.js`
- Calling `require('depA')` from `/myApp/node_modules/depC/foobar.js` will load `/myApp/node_modules/depC/node_modules/depA/index.js`

> - Gọi `require('depA')` từ `/myApp/foo.js` sẽ nạp `/myApp/node_modules/depA/index.js`
> - Gọi `require('depA')` từ `/myApp/node_modules/depB/bar.js` sẽ nạp `/myApp/node_modules/depB/node_modules/depA/index.js`
> - Gọi `require('depA')` từ `/myApp/node_modules/depC/foobar.js` sẽ nạp `/myApp/node_modules/depC/node_modules/depA/index.js`

The resolving algorithm is the core part behind the robustness of the Node.js dependency management, and it makes it possible to have hundreds or even thousands of packages in an application without having collisions or problems of version compatibility.

> Thuật toán phân giải là phần lõi đằng sau tính vững chắc của việc quản lý phụ thuộc của Node.js, và nó giúp có thể có hàng trăm hoặc thậm chí hàng nghìn gói trong một ứng dụng mà không xảy ra xung đột hay vấn đề về tương thích phiên bản.

The resolving algorithm is applied transparently for us when we invoke `require()`. However, if needed, it can still be used directly by any module by simply invoking `require.resolve()`.

> Thuật toán phân giải được áp dụng một cách trong suốt cho chúng ta khi chúng ta gọi `require()`. Tuy nhiên, nếu cần, nó vẫn có thể được dùng trực tiếp bởi bất kỳ module nào chỉ bằng cách gọi `require.resolve()`.

### The module cache — Module cache

Each module is only loaded and evaluated the first time it is required, since any subsequent call of `require()` will simply return the cached version. This should be clear by looking at the code of our homemade `require()` function. Caching is crucial for performance, but it also has some important functional implications:

> Mỗi module chỉ được nạp và đánh giá vào lần đầu tiên nó được require, vì bất kỳ lần gọi `require()` nào sau đó cũng sẽ chỉ trả về phiên bản đã được cache. Điều này sẽ rõ ràng khi xem code của hàm `require()` tự chế của chúng ta. Việc cache rất quan trọng đối với hiệu năng, nhưng nó cũng có một số hệ quả chức năng quan trọng:

- It makes it possible to have cycles within module dependencies
- It guarantees, to some extent, that the same instance is always returned when requiring the same module from within a given package

> - Nó giúp có thể có các vòng (cycle) bên trong các phụ thuộc của module
> - Nó đảm bảo, ở một mức độ nào đó, rằng cùng một instance luôn được trả về khi require cùng một module từ bên trong một gói nhất định

The module cache is exposed via the `require.cache` variable, so it is possible to directly access it if needed. A common use case is to invalidate any cached module by deleting the relative key in the `require.cache` variable, a practice that can be useful during testing but very dangerous if applied in normal circumstances.

> Module cache được phơi bày thông qua biến `require.cache`, nên có thể truy cập trực tiếp vào nó nếu cần. Một trường hợp sử dụng phổ biến là làm mất hiệu lực bất kỳ module đã cache nào bằng cách xóa key tương ứng trong biến `require.cache`, một thực hành có thể hữu ích trong quá trình kiểm thử nhưng rất nguy hiểm nếu áp dụng trong các tình huống bình thường.

### Circular dependencies — Phụ thuộc vòng (Circular dependencies)

Many consider circular dependencies an intrinsic design issue, but it is something that might actually happen in a real project, so it's useful for us to know at least how this works with CommonJS. If we look again at our homemade `require()` function, we immediately get a glimpse of how this might work and what its caveats are.

> Nhiều người coi phụ thuộc vòng (circular dependency) là một vấn đề thiết kế cố hữu, nhưng nó là điều thực sự có thể xảy ra trong một dự án thực tế, nên việc biết ít nhất là cách nó hoạt động với CommonJS là hữu ích cho chúng ta. Nếu chúng ta nhìn lại hàm `require()` tự chế của mình, chúng ta sẽ ngay lập tức thấy được thoáng qua về cách điều này có thể hoạt động và những điều cần lưu ý của nó là gì.

But let's walk together through an example to see how CommonJS behaves when dealing with circular dependencies. Let's suppose we have the scenario represented in Figure 2.1:

> Nhưng hãy cùng đi qua một ví dụ để xem CommonJS hành xử thế nào khi xử lý phụ thuộc vòng. Giả sử chúng ta có kịch bản được biểu diễn trong Hình 2.1:

![](images/ch02-58_1.png)

Figure 2.1: An example of **circular dependency**

> Hình 2.1: Một ví dụ về phụ thuộc vòng

A module called `main.js` requires `a.js` and `b.js`. In turn, `a.js` requires `b.js`. But `b.js` relies on `a.js` as well! It's obvious that we have a circular dependency here as module `a.js` requires module `b.js` and module `b.js` requires module `a.js`. Let's have a look at the code of these two modules:

> Một module tên là `main.js` require `a.js` và `b.js`. Đến lượt nó, `a.js` require `b.js`. Nhưng `b.js` cũng dựa vào `a.js`! Rõ ràng là ở đây chúng ta có một phụ thuộc vòng vì module `a.js` require module `b.js` và module `b.js` require module `a.js`. Hãy xem code của hai module này:

- Module `a.js`:

> - Module `a.js`:

```js
exports.loaded = false
const b = require('./b')
module.exports = {
  b,
  loaded: true // overrides the previous export
}
```

- Module `b.js`:

> - Module `b.js`:

```js
exports.loaded = false
const a = require('./a')
module.exports = {
  a,
  loaded: true
}
```

Now, let's see how these modules are required by `main.js`:

> Bây giờ, hãy xem các module này được require bởi `main.js` như thế nào:

```js
const a = require('./a')
const b = require('./b')
console.log('a ->', JSON.stringify(a, null, 2))
console.log('b ->', JSON.stringify(b, null, 2))
```

If we run `main.js`, we will see the following output:

> Nếu chúng ta chạy `main.js`, chúng ta sẽ thấy kết quả sau:

```js
a -> {
  "b": {
    "a": {
      "loaded": false
    },
    "loaded": true
  },
  "loaded": true
}
b -> {
  "a": {
    "loaded": false
  },
  "loaded": true
}
```

This result reveals the caveats of circular dependencies with CommonJS, that is, different parts of our application will have a different view of what is exported by module `a.js` and module `b.js`, depending on the order in which those dependencies are loaded. While both the modules are completely initialized as soon as they are required from the module `main.js`, the `a.js` module will be incomplete when it is loaded from `b.js`. In particular, its **state** will be the one that it reached the moment `b.js` was required.

> Kết quả này tiết lộ những điều cần lưu ý của phụ thuộc vòng với CommonJS, đó là, các phần khác nhau của ứng dụng của chúng ta sẽ có một góc nhìn khác nhau về những gì được xuất ra bởi module `a.js` và module `b.js`, tùy thuộc vào thứ tự mà các phụ thuộc đó được nạp. Trong khi cả hai module đều được khởi tạo hoàn toàn ngay khi chúng được require từ module `main.js`, thì module `a.js` sẽ không đầy đủ khi nó được nạp từ `b.js`. Cụ thể, trạng thái của nó sẽ là trạng thái mà nó đạt được vào thời điểm `b.js` được require.

In order to understand in more detail what happens behind the scenes, let's analyze step by step how the different modules are interpreted and how their local scope changes along the way:

> Để hiểu chi tiết hơn về những gì xảy ra đằng sau hậu trường, hãy phân tích từng bước cách các module khác nhau được diễn giải và cách phạm vi cục bộ của chúng thay đổi dọc theo quá trình:

![](images/ch02-60_1.png)

Figure 2.2: A visual representation of how a dependency loop is managed in Node.js The steps are as follows:

> Hình 2.2: Một biểu diễn trực quan về cách một vòng lặp phụ thuộc được quản lý trong Node.js. Các bước như sau:

1. The processing starts in `main.js`, which immediately requires `a.js`
2. The first thing that module `a.js` does is set an exported value called `loaded` to `false`
3. At this point, module `a.js` requires module `b.js`
4. Like `a.js`, the first thing that module `b.js` does is set an exported value called `loaded` to `false`
5. Now, `b.js` requires `a.js` (cycle)
6. Since `a.js` has already been traversed, its currently exported value is immediately copied into the scope of module `b.js`
7. Module `b.js` finally changes the `loaded` value to `true`
8. Now that `b.js` has been fully executed, the control returns to `a.js,` which now holds a copy of the current state of module `b.js` in its own scope
9. The last step of module `a.js` is to set its `loaded` value to `true`
10. Module `a.js` is now completely executed and the control returns to `main.js`, which now has a copy of the current state of module `a.js` in its internal scope
11. `main.js` requires `b.js`, which is immediately loaded from cache
12. The current state of module `b.js` is copied into the scope of module `main.js` where we can finally see the complete picture of what the state of every module is

> 1. Quá trình xử lý bắt đầu trong `main.js`, vốn ngay lập tức require `a.js`
> 2. Điều đầu tiên mà module `a.js` làm là đặt một giá trị được xuất ra tên là `loaded` thành `false`
> 3. Tại thời điểm này, module `a.js` require module `b.js`
> 4. Giống như `a.js`, điều đầu tiên mà module `b.js` làm là đặt một giá trị được xuất ra tên là `loaded` thành `false`
> 5. Bây giờ, `b.js` require `a.js` (vòng lặp)
> 6. Vì `a.js` đã được duyệt qua, giá trị hiện đang được xuất ra của nó ngay lập tức được sao chép vào phạm vi của module `b.js`
> 7. Module `b.js` cuối cùng thay đổi giá trị `loaded` thành `true`
> 8. Giờ đây `b.js` đã được thực thi hoàn toàn, quyền điều khiển trả về cho `a.js`, vốn giờ đây giữ một bản sao của trạng thái hiện tại của module `b.js` trong phạm vi của riêng nó
> 9. Bước cuối cùng của module `a.js` là đặt giá trị `loaded` của nó thành `true`
> 10. Module `a.js` giờ đây đã được thực thi hoàn toàn và quyền điều khiển trả về cho `main.js`, vốn giờ đây có một bản sao của trạng thái hiện tại của module `a.js` trong phạm vi nội bộ của nó
> 11. `main.js` require `b.js`, vốn được nạp ngay lập tức từ cache
> 12. Trạng thái hiện tại của module `b.js` được sao chép vào phạm vi của module `main.js`, nơi cuối cùng chúng ta có thể thấy bức tranh đầy đủ về trạng thái của mọi module

As we said, the issue here is that module `b.js` has a partial view of module `a.js`, and this partial view gets propagated over when `b.js` is required in `main.js`. This behavior should spark an intuition which can be confirmed if we swap the order in which the two modules are required in `main.js`. If you actually try this, you will see that this time it will be the `a.js` module that will receive an incomplete version of `b.js`.

> Như đã nói, vấn đề ở đây là module `b.js` có một góc nhìn không đầy đủ về module `a.js`, và góc nhìn không đầy đủ này được lan truyền tiếp khi `b.js` được require trong `main.js`. Hành vi này hẳn sẽ khơi gợi một trực giác mà có thể được xác nhận nếu chúng ta hoán đổi thứ tự mà hai module được require trong `main.js`. Nếu bạn thực sự thử điều này, bạn sẽ thấy rằng lần này sẽ là module `a.js` nhận được một phiên bản không đầy đủ của `b.js`.

We understand now that this can become quite a fuzzy business if we lose control of which module is loaded first, which can happen quite easily if the project is big enough.

> Bây giờ chúng ta hiểu rằng đây có thể trở thành một việc khá mơ hồ nếu chúng ta mất kiểm soát về module nào được nạp trước, điều có thể xảy ra khá dễ dàng nếu dự án đủ lớn.

Later in this chapter, we will see how ESM can deal with circular dependencies in a more effective way. Meanwhile, if you are using CommonJS, be very careful about this behavior and the way it can affect your application.

> Ở phần sau của chương này, chúng ta sẽ thấy ESM có thể xử lý phụ thuộc vòng một cách hiệu quả hơn như thế nào. Trong khi đó, nếu bạn đang dùng CommonJS, hãy hết sức cẩn thận về hành vi này và cách nó có thể ảnh hưởng đến ứng dụng của bạn.

In the next section, we will discuss some patterns to define modules in Node.js.

> Trong phần tiếp theo, chúng ta sẽ bàn về một số mẫu để định nghĩa module trong Node.js.

## Module definition patterns — Các mẫu định nghĩa module

The module system, besides being a mechanism for loading dependencies, is also a tool for defining APIs. Like any other problem related to API design, the main factor to consider is the balance between private and public functionality. The aim is to maximize information hiding and API usability, while balancing these with other software qualities, such as extensibility and code reuse.

> Hệ thống module, ngoài việc là một cơ chế để nạp các phụ thuộc, còn là một công cụ để định nghĩa các API. Giống như bất kỳ vấn đề nào khác liên quan đến thiết kế API, yếu tố chính cần cân nhắc là sự cân bằng giữa chức năng riêng tư và chức năng công khai. Mục tiêu là tối đa hóa việc che giấu thông tin và khả năng sử dụng của API, đồng thời cân bằng những điều này với các phẩm chất phần mềm khác, chẳng hạn như khả năng mở rộng và tái sử dụng code.

In this section, we will analyze some of the most popular patterns for defining modules in Node.js, such as named exports, exporting functions, classes and instances, and monkey patching. Each one has its own balance of information hiding, extensibility, and code reuse.

> Trong phần này, chúng ta sẽ phân tích một số mẫu phổ biến nhất để định nghĩa module trong Node.js, chẳng hạn như named exports, exporting functions, classes và instances, và monkey patching. Mỗi mẫu có sự cân bằng riêng giữa che giấu thông tin, khả năng mở rộng và tái sử dụng code.

### Named exports — Named exports

The most basic method for exposing a public API is using named exports, which involves assigning the values we want to make public to properties of the object referenced by `exports` (or `module.exports`). In this way, the resulting exported object becomes a container or namespace for a set of related functionalities.

> Phương pháp cơ bản nhất để phơi bày một API công khai là dùng named exports, vốn liên quan đến việc gán các giá trị mà chúng ta muốn công khai vào các thuộc tính của đối tượng được tham chiếu bởi `exports` (hay `module.exports`). Theo cách này, đối tượng được xuất ra trở thành một bộ chứa hoặc không gian tên cho một tập các chức năng liên quan.

The following code shows a module implementing this pattern:

> Đoạn code sau cho thấy một module hiện thực mẫu này:

```js
// file logger.js
exports.info = (message) => {
  console.log(`info: ${message}`)
}
exports.verbose = (message) => {
  console.log(`verbose: ${message}`)
}
```

The exported functions are then available as properties of the loaded module, as shown in the following code:

> Các hàm được xuất ra sau đó có sẵn dưới dạng các thuộc tính của module đã nạp, như được thể hiện trong đoạn code sau:

```js
// file main.js
const logger = require('./logger')
logger.info('This is an informational message')
logger.verbose('This is a verbose message')
```

Most of the Node.js core modules use this pattern. However, the CommonJS specification only allows the use of the `exports` variable to expose public members. Therefore, the named exports pattern is the only one that is really compatible with the CommonJS specification. The use of `module.exports` is an extension provided by Node.js to support a broader range of module definition patterns, which we are going to see next.

> Hầu hết các module lõi của Node.js đều dùng mẫu này. Tuy nhiên, đặc tả CommonJS chỉ cho phép dùng biến `exports` để phơi bày các thành viên công khai. Do đó, mẫu named exports là mẫu duy nhất thực sự tương thích với đặc tả CommonJS. Việc dùng `module.exports` là một phần mở rộng do Node.js cung cấp để hỗ trợ một phạm vi rộng hơn các mẫu định nghĩa module, mà chúng ta sẽ thấy tiếp theo.

### Exporting a function — Xuất ra một hàm

One of the most popular module definition patterns consists of reassigning the whole `module.exports` variable to a function. The main strength of this pattern is the fact that it allows you to expose only a single functionality, which provides a clear entry point for the module, making it simpler to understand and use; it also honors the principle of small surface area very well. This way of defining modules is also known in the community as the substack pattern, after one of its most prolific adopters, James Halliday (nickname substack – `https://github.com/substack`). Have a look at this pattern in the following example:

> Một trong những mẫu định nghĩa module phổ biến nhất bao gồm việc gán lại toàn bộ biến `module.exports` cho một hàm. Điểm mạnh chính của mẫu này là việc nó cho phép bạn chỉ phơi bày một chức năng duy nhất, vốn cung cấp một điểm vào (entry point) rõ ràng cho module, làm cho nó đơn giản hơn để hiểu và sử dụng; nó cũng tôn trọng rất tốt nguyên tắc về bề mặt nhỏ (small surface area). Cách định nghĩa module này cũng được biết đến trong cộng đồng với tên gọi substack pattern, theo tên một trong những người áp dụng nó nhiều nhất, James Halliday (biệt danh substack – `https://github.com/substack`). Hãy xem mẫu này trong ví dụ sau:

```js
// file logger.js
module.exports = (message) => {
  console.log(`info: ${message}`)
}
```

A possible extension of this pattern is using the exported function as a namespace for other public APIs. This is a very powerful combination because it still gives the module the clarity of a single entry point (the main exported function) and at the same time it allows us to expose other functionalities that have secondary or more advanced use cases. The following code shows us how to extend the module we defined previously by using the exported function as a namespace:

> Một phần mở rộng có thể có của mẫu này là dùng hàm được xuất ra làm một không gian tên cho các API công khai khác. Đây là một sự kết hợp rất mạnh mẽ vì nó vẫn cho module sự rõ ràng của một điểm vào duy nhất (hàm chính được xuất ra) và đồng thời nó cho phép chúng ta phơi bày các chức năng khác có các trường hợp sử dụng thứ yếu hoặc nâng cao hơn. Đoạn code sau cho chúng ta thấy cách mở rộng module mà chúng ta đã định nghĩa trước đó bằng cách dùng hàm được xuất ra làm một không gian tên:

```js
module.exports.verbose = (message) => {
  console.log(`verbose: ${message}`)
}
```

This code demonstrates how to use the module that we just defined:

> Đoạn code này minh họa cách dùng module mà chúng ta vừa định nghĩa:

```js
// file main.js
const logger = require('./logger')
logger('This is an informational message')
logger.verbose('This is a verbose message')
```

Even though exporting just a function might seem like a limitation, in reality, it's a perfect way to put the emphasis on a single functionality, the most important one for the module, while giving less visibility to secondary or internal aspects, which are instead exposed as properties of the exported function itself. The modularity of Node.js heavily encourages the adoption of the single-responsibility principle (SRP): every module should have responsibility over a single functionality and that responsibility should be entirely encapsulated by the module.

> Mặc dù việc chỉ xuất ra một hàm có vẻ là một sự giới hạn, nhưng trên thực tế, đó là một cách hoàn hảo để đặt sự nhấn mạnh vào một chức năng duy nhất, chức năng quan trọng nhất đối với module, đồng thời cho ít hiển thị hơn đối với các khía cạnh thứ yếu hoặc nội bộ, vốn thay vào đó được phơi bày dưới dạng các thuộc tính của chính hàm được xuất ra. Tính module hóa của Node.js khuyến khích mạnh mẽ việc áp dụng nguyên tắc trách nhiệm đơn (single-responsibility principle - SRP): mỗi module nên có trách nhiệm đối với một chức năng duy nhất và trách nhiệm đó nên được đóng gói hoàn toàn bởi module.

### Exporting a class — Xuất ra một lớp (class)

A module that exports a class is a specialization of a module that exports a function. The difference is that with this new pattern we allow the user to create new instances using the constructor, but we also give them the ability to extend its prototype and forge new classes. The following is an example of this pattern:

> Một module xuất ra một lớp là một sự chuyên biệt hóa của một module xuất ra một hàm. Sự khác biệt là với mẫu mới này, chúng ta cho phép người dùng tạo các instance mới bằng cách dùng hàm dựng (constructor), nhưng chúng ta cũng cho họ khả năng mở rộng prototype của nó và tạo ra các lớp mới. Sau đây là một ví dụ về mẫu này:

```js
class Logger {
  constructor (name) {
    this.name = name
  }
  log (message) {
    console.log(`[${this.name}] ${message}`)
  }
  info (message) {
    this.log(`info: ${message}`)
  }
  verbose (message) {
    this.log(`verbose: ${message}`)
  }
}
module.exports = Logger
```

And, we can use the preceding module as follows:

> Và, chúng ta có thể dùng module trên như sau:

```js
// file main.js
const Logger = require('./logger')
const dbLogger = new Logger('DB')
dbLogger.info('This is an informational message')
const accessLogger = new Logger('ACCESS')
accessLogger.verbose('This is a verbose message')
```

Exporting a class still provides a single entry point for the module, but compared to the substack pattern, it exposes a lot more of the module internals. On the other hand, it allows much more power when it comes to extending its functionality.

> Việc xuất ra một lớp vẫn cung cấp một điểm vào duy nhất cho module, nhưng so với substack pattern, nó phơi bày nhiều hơn rất nhiều phần nội bộ của module. Mặt khác, nó cho phép nhiều sức mạnh hơn khi nói đến việc mở rộng chức năng của nó.

### Exporting an instance — Xuất ra một instance

We can leverage the caching mechanism of `require()` to easily define stateful instances created from a constructor or a **factory**, which can be shared across different modules. The following code shows an example of this pattern:

> Chúng ta có thể tận dụng cơ chế caching của `require()` để dễ dàng định nghĩa các instance có trạng thái (stateful) được tạo từ một hàm dựng hoặc một factory, vốn có thể được chia sẻ giữa các module khác nhau. Đoạn code sau cho thấy một ví dụ về mẫu này:

```js
// file logger.js
class Logger {
  constructor (name) {
    this.count = 0
    this.name = name
  }
  log (message) {
    this.count++
    console.log('[' + this.name + '] ' + message)
  }
}
module.exports = new Logger('DEFAULT')
```

This newly defined module can then be used as follows:

> Module mới được định nghĩa này sau đó có thể được dùng như sau:

```js
// main.js
const logger = require('./logger')
logger.log('This is an informational message')
```

Because the module is cached, every module that requires the `logger` module will actually always retrieve the same instance of the object, thus sharing its state. This pattern is very much like creating a **singleton**. However, it does not guarantee the uniqueness of the instance across the entire application, as it happens in the traditional singleton pattern. When analyzing the resolving algorithm, we have seen that a module might be installed multiple times inside the dependency tree of an application. This results in multiple instances of the same logical module, all running in the context of the same Node.js application. We will analyze the Singleton pattern and its caveats in more detail in Chapter 7, Creational Design Patterns.

> Bởi vì module được cache, mọi module require module `logger` thực ra sẽ luôn lấy được cùng một instance của đối tượng, do đó chia sẻ trạng thái của nó. Mẫu này rất giống với việc tạo một Singleton. Tuy nhiên, nó không đảm bảo tính duy nhất của instance trên toàn bộ ứng dụng, như điều xảy ra trong mẫu Singleton truyền thống. Khi phân tích thuật toán phân giải, chúng ta đã thấy rằng một module có thể được cài đặt nhiều lần bên trong cây phụ thuộc của một ứng dụng. Điều này dẫn đến nhiều instance của cùng một module về mặt logic, tất cả đều chạy trong bối cảnh của cùng một ứng dụng Node.js. Chúng ta sẽ phân tích mẫu Singleton và những điều cần lưu ý của nó chi tiết hơn trong Chương 7, Các mẫu thiết kế khởi tạo.

One interesting detail of this pattern is that it does not preclude the opportunity to create new instances, even if we are not explicitly exporting the class. In fact, we can rely on the `constructor` property of the exported instance to construct a new instance of the same type:

> Một chi tiết thú vị của mẫu này là nó không loại trừ cơ hội tạo các instance mới, ngay cả khi chúng ta không xuất ra lớp một cách tường minh. Thực tế, chúng ta có thể dựa vào thuộc tính `constructor` của instance được xuất ra để dựng một instance mới cùng kiểu:

```js
const customLogger = new logger.constructor('CUSTOM')
customLogger.log('This is an informational message')
```

As you can see, by using `logger.constructor()`, we can instantiate new `Logger` objects. Note that this technique must be used with caution or avoided altogether. Consider that, if the module author decided not to export the class explicitly, they probably wanted to keep this class private.

> Như bạn có thể thấy, bằng cách dùng `logger.constructor()`, chúng ta có thể khởi tạo các đối tượng `Logger` mới. Lưu ý rằng kỹ thuật này phải được dùng một cách thận trọng hoặc tránh dùng hoàn toàn. Hãy cân nhắc rằng, nếu tác giả module đã quyết định không xuất ra lớp một cách tường minh, thì có lẽ họ muốn giữ lớp này riêng tư.

### Modifying other modules or the global scope — Sửa đổi các module khác hoặc phạm vi toàn cục

A module can even export nothing. This can seem a bit out of place; however, we should not forget that a module can modify the global scope and any object in it, including other modules in the cache. Please note that these are in general considered bad practices, but since this pattern can be useful and safe under some circumstances (for example, for testing) and it's sometimes used in real-life projects, it's worth knowing.

> Một module thậm chí có thể không xuất ra gì cả. Điều này có vẻ hơi lạc lõng; tuy nhiên, chúng ta không nên quên rằng một module có thể sửa đổi phạm vi toàn cục và bất kỳ đối tượng nào trong đó, bao gồm cả các module khác trong cache. Xin lưu ý rằng những điều này nhìn chung được coi là các thực hành xấu, nhưng vì mẫu này có thể hữu ích và an toàn trong một số trường hợp (ví dụ, để kiểm thử) và nó đôi khi được dùng trong các dự án thực tế, nên việc biết về nó là đáng giá.

We said that a module can modify other modules or objects in the global scope; well, this is called monkey patching. It generally refers to the practice of modifying the existing objects at runtime to change or extend their behavior or to apply temporary fixes.

> Chúng ta đã nói rằng một module có thể sửa đổi các module khác hoặc các đối tượng trong phạm vi toàn cục; vâng, điều này được gọi là monkey patching. Nó thường ám chỉ việc thực hành sửa đổi các đối tượng hiện có vào lúc chạy (runtime) để thay đổi hoặc mở rộng hành vi của chúng hoặc để áp dụng các bản vá tạm thời.

The following example shows us how we can add a new function to another module:

> Ví dụ sau đây cho chúng ta thấy cách chúng ta có thể thêm một hàm mới vào một module khác:

```js
// file patcher.js
// ./logger is another module
require('./logger').customMessage = function () {
  console.log('This is a new functionality')
}
```

Using our new `patcher` module is as easy as writing the following code:

> Việc dùng module `patcher` mới của chúng ta dễ dàng như việc viết đoạn code sau:

```js
// file main.js
require('./patcher')
const logger = require('./logger')
logger.customMessage()
```

The technique described here can be very dangerous to use. The main concern is that having a module that modifies the global namespace or other modules is an operation with side effects. In other words, it affects the state of entities outside their scope, which can have consequences that aren't easily predictable, especially when multiple modules interact with the same entities. Imagine having two different modules trying to set the same global variable, or modifying the same property of the same module. The effects can be unpredictable (which module wins?), but most importantly it would have repercussions on the entire application. So, again use this technique with care and make sure you understand all the possible side effects while doing so.

> Kỹ thuật được mô tả ở đây có thể rất nguy hiểm khi sử dụng. Mối quan tâm chính là việc có một module sửa đổi không gian tên toàn cục hoặc các module khác là một thao tác có tác dụng phụ (side effect). Nói cách khác, nó ảnh hưởng đến trạng thái của các thực thể bên ngoài phạm vi của chúng, điều có thể gây ra những hậu quả không dễ lường trước, đặc biệt là khi nhiều module tương tác với cùng các thực thể đó. Hãy tưởng tượng có hai module khác nhau cùng cố đặt cùng một biến toàn cục, hoặc sửa đổi cùng một thuộc tính của cùng một module. Các hệ quả có thể khó lường (module nào thắng?), nhưng quan trọng nhất là nó sẽ có ảnh hưởng đến toàn bộ ứng dụng. Vậy nên, một lần nữa, hãy dùng kỹ thuật này một cách cẩn thận và đảm bảo rằng bạn hiểu tất cả các tác dụng phụ có thể có khi làm vậy.

If you want a real-life example of how this can be useful, have a look at `nock` (`nodejsdp.link/nock`), a module that allows you to mock HTTP responses in your tests. The way `nock` works is by monkey patching the Node.js `http` module and by changing its behavior so that it will provide the mocked response rather than issuing a real HTTP **request**. This allows our unit test to run without hitting the actual production HTTP endpoints, something that's very convenient when writing tests for code that relies on third-party APIs.

> Nếu bạn muốn một ví dụ thực tế về cách điều này có thể hữu ích, hãy xem `nock` (`nodejsdp.link/nock`), một module cho phép bạn giả lập (mock) các phản hồi HTTP trong các bài kiểm thử của mình. Cách `nock` hoạt động là bằng cách monkey patching module `http` của Node.js và thay đổi hành vi của nó để nó cung cấp phản hồi đã được giả lập thay vì phát đi một yêu cầu HTTP thực sự. Điều này cho phép unit test của chúng ta chạy mà không cần truy cập đến các endpoint HTTP production thực tế, một điều rất tiện lợi khi viết các bài kiểm thử cho code dựa vào các API của bên thứ ba.

At this point, we should have a quite complete understanding of CommonJS and some of the patterns that are generally used with it. In the next section, we will explore ECMAScript modules, also known as ESM.

> Tại thời điểm này, chúng ta hẳn đã có một sự hiểu biết khá đầy đủ về CommonJS và một số mẫu thường được dùng với nó. Trong phần tiếp theo, chúng ta sẽ khám phá ECMAScript modules, còn được gọi là ESM.

## ESM: ECMAScript modules — ESM: ECMAScript modules

ECMAScript modules (also known as ES modules or ESM) were introduced as part of the ECMAScript 2015 specification with the goal to give JavaScript an official module system suitable for different execution environments. The ESM specification tries to retain some good ideas from previous existing module systems like CommonJS and AMD. The syntax is very simple and compact. There is support for cyclic dependencies and the possibility to load modules asynchronously.

> ECMAScript modules (còn được gọi là ES modules hay ESM) được giới thiệu như một phần của đặc tả ECMAScript 2015 với mục tiêu mang lại cho JavaScript một hệ thống module chính thức phù hợp với các môi trường thực thi khác nhau. Đặc tả ESM cố gắng giữ lại một số ý tưởng tốt từ các hệ thống module đã tồn tại trước đó như CommonJS và AMD. Cú pháp rất đơn giản và gọn gàng. Có hỗ trợ cho phụ thuộc vòng và khả năng nạp module một cách bất đồng bộ.

The most important differentiator between ESM and CommonJS is that ES modules are static, which means that imports are described at the top level of every module and outside any **control flow** statement. Also, the name of the imported modules cannot be dynamically generated at runtime using expressions, only constant strings are allowed.

> Điểm khác biệt quan trọng nhất giữa ESM và CommonJS là ES modules có tính tĩnh (static), nghĩa là các import được mô tả ở cấp cao nhất (top level) của mỗi module và bên ngoài bất kỳ câu lệnh điều khiển luồng nào. Ngoài ra, tên của các module được import không thể được tạo ra một cách động vào lúc chạy bằng các biểu thức, chỉ cho phép các chuỗi hằng (constant string).

For instance, the following code wouldn't be valid when using ES modules:

> Ví dụ, đoạn code sau sẽ không hợp lệ khi dùng ES modules:

```js
if (condition) {
  import module1 from 'module1'
} else {
  import module2 from 'module2'
}
```

While in CommonJS, it is perfectly fine to write something like this:

> Trong khi đó với CommonJS, viết một thứ gì đó như thế này là hoàn toàn ổn:

```js
let module = null
if (condition) {
  module = require('module1')
} else {
  module = require('module2')
}
```

At a first glance, this characteristic of ESM might seem an unnecessary limitation, but in reality, having static imports opens up a number of interesting scenarios that are not practical with the dynamic nature of CommonJS. For instance, static imports allow the static analysis of the dependency tree, which allows optimizations such as dead code elimination (tree shaking) and more.

> Thoạt nhìn, đặc điểm này của ESM có vẻ là một sự giới hạn không cần thiết, nhưng trên thực tế, việc có các import tĩnh mở ra một số kịch bản thú vị vốn không khả thi với bản chất động của CommonJS. Ví dụ, các import tĩnh cho phép phân tích tĩnh cây phụ thuộc, vốn cho phép các tối ưu hóa như loại bỏ code chết (tree shaking) và hơn thế nữa.

### Using ESM in Node.js — Dùng ESM trong Node.js

Node.js will consider every `.js` file to be written using the CommonJS syntax by default; therefore, if we use the ESM syntax inside a `.js` file, the interpreter will simply throw an error.

> Node.js sẽ coi mọi file `.js` là được viết bằng cú pháp CommonJS theo mặc định; do đó, nếu chúng ta dùng cú pháp ESM bên trong một file `.js`, trình thông dịch sẽ chỉ đơn giản ném ra một lỗi.

There are several ways to tell the Node.js interpreter to consider a given module as an ES module rather than a CommonJS module:

> Có một số cách để báo cho trình thông dịch Node.js coi một module nhất định là một ES module thay vì một module CommonJS:

- Give the module file the extension `.mjs`
- Add to the nearest parent `package.json` a field called "`type"` with a value of `"module"`

> - Đặt cho file module phần mở rộng `.mjs`
> - Thêm vào file `package.json` cha gần nhất một trường gọi là "`type"` với giá trị là `"module"`

Throughout the rest of this book and in the code examples provided, we will keep using the `.js` extension to keep the code more easily accessible to most text editors, so if you are copying and pasting examples straight from the book, make sure that you also create a `package.json` file with the `"type":"module"` entry.

> Xuyên suốt phần còn lại của cuốn sách này và trong các ví dụ code được cung cấp, chúng tôi sẽ tiếp tục dùng phần mở rộng `.js` để giữ cho code dễ tiếp cận hơn đối với hầu hết các trình soạn thảo văn bản, nên nếu bạn đang sao chép và dán các ví dụ thẳng từ sách, hãy đảm bảo rằng bạn cũng tạo một file `package.json` với mục `"type":"module"`.

Let's now have a look at the ESM syntax.

> Bây giờ hãy xem qua cú pháp ESM.

### Named exports and imports — Named exports và imports

ESM allows us to export functionality from a module through the `export` keyword.

> ESM cho phép chúng ta xuất ra chức năng từ một module thông qua từ khóa `export`.

Note that ESM uses the singular word `export` as opposed to the plural (`exports` and `module.exports`) used by CommonJS. In an ES module, everything is private by default and only exported entities are publicly accessible from other modules.

> Lưu ý rằng ESM dùng từ ở dạng số ít `export` trái với dạng số nhiều (`exports` và `module.exports`) được dùng bởi CommonJS. Trong một ES module, mọi thứ đều là riêng tư theo mặc định và chỉ các thực thể được xuất ra mới có thể truy cập công khai từ các module khác.

The `export` keyword can be used in front of the entities that we want to make available to the module users. Let's see an example:

> Từ khóa `export` có thể được dùng đặt phía trước các thực thể mà chúng ta muốn làm cho có sẵn cho người dùng module. Hãy xem một ví dụ:

```js
// logger.js
// exports a function as `log`
export function log (message) {
  console.log(message)
}
// exports a constant as `DEFAULT_LEVEL`
export const DEFAULT_LEVEL = 'info'
// exports an object as `LEVELS`
export const LEVELS = {
  error: 0,
  debug: 1,
  warn: 2,
  data: 3,
  info: 4,
  verbose: 5
}
// exports a class as `Logger`
export class Logger {
  constructor (name) {
    this.name = name
  }
  log (message) {
    console.log(`[${this.name}] ${message}`)
  }
}
```

If we want to import entities from a module we can use the `import` keyword. The syntax is quite flexible, and it allows us to import one or more entities and even to rename imports. Let's see some examples:

> Nếu chúng ta muốn import các thực thể từ một module, chúng ta có thể dùng từ khóa `import`. Cú pháp khá linh hoạt, và nó cho phép chúng ta import một hoặc nhiều thực thể và thậm chí đổi tên các import. Hãy xem một số ví dụ:

```js
import * as loggerModule from './logger.js'
console.log(loggerModule)
```

In this example, we are using the `*` syntax (also called namespace import) to import all the members of the module and assign them to the local `loggerModule` variable. This example will output something like this:

> Trong ví dụ này, chúng ta đang dùng cú pháp `*` (còn gọi là namespace import) để import tất cả các thành viên của module và gán chúng cho biến cục bộ `loggerModule`. Ví dụ này sẽ xuất ra một thứ gì đó như thế này:

```js
[Module] {
  DEFAULT_LEVEL: 'info',
  LEVELS: { error: 0, debug: 1, warn: 2, data: 3, info: 4,
    verbose: 5 },
  Logger: [Function: Logger],
  log: [Function: log]
}
```

As we can see, all the entities exported in our module are now accessible in the `loggerModule` namespace. For instance, we could refer to the `log()` function through `loggerModule.log`.

> Như chúng ta có thể thấy, tất cả các thực thể được xuất ra trong module của chúng ta giờ đây có thể truy cập trong không gian tên `loggerModule`. Ví dụ, chúng ta có thể tham chiếu đến hàm `log()` thông qua `loggerModule.log`.

It's very important to note that, as opposed to CommonJS, with ESM we have to specify the file extension of the imported modules. With CommonJS we can use either `./logger` or `./logger.js`, with ESM we are forced to use `./logger.js`.

> Điều rất quan trọng cần lưu ý là, trái với CommonJS, với ESM chúng ta phải chỉ định phần mở rộng file của các module được import. Với CommonJS chúng ta có thể dùng `./logger` hoặc `./logger.js`, còn với ESM chúng ta buộc phải dùng `./logger.js`.

If we are using a large module, most often we don't want to import all of its functionality, but only one or few entities from it:

> Nếu chúng ta đang dùng một module lớn, thường thì chúng ta không muốn import tất cả chức năng của nó, mà chỉ một hoặc vài thực thể từ nó:

```js
import { log } from './logger.js'
log('Hello World')
```

If we want to import more than one entity, this is how we would do that:

> Nếu chúng ta muốn import nhiều hơn một thực thể, đây là cách chúng ta sẽ làm điều đó:

```js
import { log, Logger } from './logger.js'
log('Hello World')
const logger = new Logger('DEFAULT')
logger.log('Hello world')
```

When we use this type of `import` statement, the entities are imported into the current scope, so there is a risk of a name clash. The following code, for example, would not work:

> Khi chúng ta dùng loại câu lệnh `import` này, các thực thể được import vào phạm vi hiện tại, nên có nguy cơ xảy ra xung đột tên. Ví dụ, đoạn code sau sẽ không hoạt động:

```js
import { log } from './logger.js'
const log = console.log
```

If we try to execute the preceding snippet, the interpreter fails with the following error:

> Nếu chúng ta thử thực thi đoạn trên, trình thông dịch thất bại với lỗi sau:

```js
SyntaxError: Identifier 'log' has already been declared
```

In situations like this one, we can resolve the clash by renaming the imported entity with the `as` keyword:

> Trong những tình huống như thế này, chúng ta có thể giải quyết xung đột bằng cách đổi tên thực thể được import với từ khóa `as`:

```js
import { log as log2 } from './logger.js'
const log = console.log
log('message from log')
log2('message from log2')
```

This approach can be particularly useful when the clash is generated by importing two entities with the same name from different modules, and therefore changing the original names is outside the consumer's control.

> Cách tiếp cận này có thể đặc biệt hữu ích khi xung đột được tạo ra bởi việc import hai thực thể cùng tên từ các module khác nhau, và do đó việc thay đổi tên gốc nằm ngoài tầm kiểm soát của bên tiêu thụ.

### Default exports and imports — Default exports và imports

One widely used feature of CommonJS is the ability to export a single unnamed entity through the assignment of `module.exports`. We saw that this is very convenient as it encourages module developers to follow the single-responsibility principle and expose only one clear interface. With ESM, we can do something similar through what's called a default export. A default export makes use of the `export default` keywords and it looks like this:

> Một tính năng được dùng rộng rãi của CommonJS là khả năng xuất ra một thực thể không tên duy nhất thông qua việc gán cho `module.exports`. Chúng ta đã thấy rằng điều này rất tiện lợi vì nó khuyến khích các nhà phát triển module tuân theo nguyên tắc trách nhiệm đơn và chỉ phơi bày một interface rõ ràng. Với ESM, chúng ta có thể làm một điều tương tự thông qua cái được gọi là default export. Một default export sử dụng các từ khóa `export default` và nó trông như thế này:

```js
// logger.js
export default class Logger {
  constructor (name) {
    this.name = name
  }
  log (message) {
    console.log(`[${this.name}] ${message}`)
  }
}
```

In this case, the name `Logger` is ignored, and the entity exported is registered under the name `default`. This exported name is handled in a special way, and it can be imported as follows:

> Trong trường hợp này, cái tên `Logger` bị bỏ qua, và thực thể được xuất ra được đăng ký dưới cái tên `default`. Tên được xuất ra này được xử lý theo một cách đặc biệt, và nó có thể được import như sau:

```js
// main.js
import MyLogger from './logger.js'
const logger = new MyLogger('info')
logger.log('Hello World')
```

The difference with named ESM imports is that here, since the default export is considered unnamed, we can import it and at the same time assign it a local name of our choice. In this example, we can replace `MyLogger` with anything else that makes sense in our context. This is very similar to what we do with CommonJS modules. Note also that we don't have to wrap the import name around brackets or use the `as` keyword when renaming.

> Sự khác biệt với named ESM imports là ở đây, vì default export được coi là không tên, chúng ta có thể import nó và đồng thời gán cho nó một tên cục bộ do chúng ta chọn. Trong ví dụ này, chúng ta có thể thay thế `MyLogger` bằng bất cứ thứ gì khác có ý nghĩa trong bối cảnh của chúng ta. Điều này rất giống với những gì chúng ta làm với các module CommonJS. Cũng lưu ý rằng chúng ta không phải bọc tên import trong dấu ngoặc nhọn hoặc dùng từ khóa `as` khi đổi tên.

Internally, a default export is equivalent to a named export with `default` as the name. We can easily verify this statement by running the following snippet of code:

> Về mặt nội bộ, một default export tương đương với một named export với tên là `default`. Chúng ta có thể dễ dàng xác minh phát biểu này bằng cách chạy đoạn code sau:

```js
// showDefault.js
import * as loggerModule from './logger.js'
console.log(loggerModule)
```

When executed, the previous code will print something like this:

> Khi được thực thi, đoạn code trên sẽ in ra một thứ gì đó như thế này:

```js
[Module] { default: [Function: Logger] }
```

One thing that we cannot do, though, is import the default entity explicitly. In fact, something like the following will fail:

> Tuy nhiên, một điều mà chúng ta không thể làm là import thực thể default một cách tường minh. Thực tế, một thứ gì đó như sau sẽ thất bại:

```js
import { default } from './logger.js'
```

The execution will fail with a `SyntaxError: Unexpected reserved word` error. This happens because the `default` keyword cannot be used as a variable name. It is valid as an object attribute, so in the previous example, it is okay to use `loggerModule.` `default`, but we can't have a variable named `default` directly in the scope.

> Việc thực thi sẽ thất bại với lỗi `SyntaxError: Unexpected reserved word`. Điều này xảy ra vì từ khóa `default` không thể được dùng làm tên biến. Nó hợp lệ khi làm thuộc tính của đối tượng, nên trong ví dụ trước, dùng `loggerModule.` `default` là ổn, nhưng chúng ta không thể có một biến tên là `default` trực tiếp trong phạm vi.

### Mixed exports — Mixed exports

It is possible to mix named exports and a default export within an ES module. Let's have a look at an example:

> Có thể trộn lẫn named exports và một default export trong một ES module. Hãy xem một ví dụ:

```js
// logger.js
export default function log (message) {
  console.log(message)
}
export function info (message) {
  log(`info: ${message}`)
}
```

The preceding code is exporting the `log()` function as a default export and a named export for a function called `info()`. Note that `info()` can reference `log()` internally. It would not be possible to replace the call to `log()` with `default()` to do that, as it would be a syntax error (Unexpected token `default`).

> Đoạn code trên đang xuất ra hàm `log()` dưới dạng một default export và một named export cho một hàm tên là `info()`. Lưu ý rằng `info()` có thể tham chiếu đến `log()` một cách nội bộ. Sẽ không thể thay thế lời gọi đến `log()` bằng `default()` để làm điều đó, vì đó sẽ là một lỗi cú pháp (Unexpected token `default`).

If we want to import both the default export and one or more named exports, we can do it using the following format:

> Nếu chúng ta muốn import cả default export và một hoặc nhiều named exports, chúng ta có thể làm điều đó bằng định dạng sau:

```js
import mylog, { info } from './logger.js'
```

In the preceding example, we are importing the default export from `logger.js` as `mylog` and also the named export `info`.

> Trong ví dụ trên, chúng ta đang import default export từ `logger.js` dưới dạng `mylog` và cũng cả named export `info`.

Let's now discuss some key details and differences between the default export and named exports:

> Bây giờ hãy bàn về một số chi tiết và khác biệt then chốt giữa default export và named exports:

- Named exports are explicit. Having predetermined names allows IDEs to support the developer with automatic imports, autocomplete, and refactoring tools. For instance, if we type `writeFileSync`, the editor might automatically add `import { writeFileSync } from 'fs'` at the beginning of the current file. Default exports, on the contrary, make all these things more complicated as a given functionality could have different names in different files, so it's harder to make inferences on which module might provide a given functionality based only on a given name.
- The default export is a convenient mechanism to communicate what is the single most important functionality for a module. Also, from the perspective of the user, it can be easier to import the obvious piece of functionality without having to know the exact name of the binding.
- In some circumstances, default exports might make it harder to apply dead code elimination (tree shaking). For example, a module could provide only a default export, which is an object where all the functionality is exposed as properties of such an object. When we import this default object, most module bundlers will consider the entire object being used and they won't be able to eliminate any unused code from the exported functionality.

> - Named exports là tường minh. Việc có các tên định sẵn cho phép các IDE hỗ trợ nhà phát triển với các import tự động, autocomplete và các công cụ tái cấu trúc. Ví dụ, nếu chúng ta gõ `writeFileSync`, trình soạn thảo có thể tự động thêm `import { writeFileSync } from 'fs'` vào đầu file hiện tại. Ngược lại, default exports làm cho tất cả những điều này phức tạp hơn vì một chức năng nhất định có thể có những cái tên khác nhau trong các file khác nhau, nên khó hơn để suy luận xem module nào có thể cung cấp một chức năng nhất định chỉ dựa trên một cái tên nhất định.
> - Default export là một cơ chế tiện lợi để truyền đạt đâu là chức năng quan trọng nhất duy nhất của một module. Ngoài ra, từ góc nhìn của người dùng, có thể dễ dàng hơn để import phần chức năng hiển nhiên mà không cần phải biết tên chính xác của binding.
> - Trong một số trường hợp, default exports có thể làm cho việc loại bỏ code chết (tree shaking) khó áp dụng hơn. Ví dụ, một module có thể chỉ cung cấp một default export, vốn là một đối tượng nơi tất cả chức năng được phơi bày dưới dạng các thuộc tính của đối tượng đó. Khi chúng ta import đối tượng default này, hầu hết các module bundler sẽ coi toàn bộ đối tượng là đang được sử dụng và chúng sẽ không thể loại bỏ bất kỳ code không dùng đến nào khỏi chức năng được xuất ra.

For these reasons, it is generally considered good practice to stick with named exports, especially when you want to expose more than one functionality, and only use default exports if it's one clear functionality you want to export.

> Vì những lý do này, nhìn chung được coi là thực hành tốt khi gắn bó với named exports, đặc biệt khi bạn muốn phơi bày nhiều hơn một chức năng, và chỉ dùng default exports nếu đó là một chức năng rõ ràng duy nhất mà bạn muốn xuất ra.

This is not a hard rule and there are notable exceptions to this suggestion. For instance, all Node.js core modules have both a default export and a number of named exports. Also, React (`nodejsdp.link/react`) uses mixed exports. Consider carefully what the best approach for your specific module is and what you want the developer experience to be for the users of your module.

> Đây không phải là một quy tắc cứng và có những ngoại lệ đáng chú ý đối với gợi ý này. Ví dụ, tất cả các module lõi của Node.js đều có cả một default export và một số named exports. Ngoài ra, React (`nodejsdp.link/react`) dùng mixed exports. Hãy cân nhắc kỹ đâu là cách tiếp cận tốt nhất cho module cụ thể của bạn và bạn muốn trải nghiệm của nhà phát triển dành cho người dùng module của bạn là gì.

### Module identifiers — Module identifiers

Module identifiers (also called module specifiers) are the different types of values that we can use in our `import` statements to specify the location of the module we want to load.

> Module identifiers (còn gọi là module specifiers) là các loại giá trị khác nhau mà chúng ta có thể dùng trong các câu lệnh `import` để chỉ định vị trí của module mà chúng ta muốn nạp.

So far, we have seen only relative paths, but there are several other possibilities and some nuances to keep in mind. Let's list all the possibilities:

> Cho đến giờ, chúng ta mới chỉ thấy các đường dẫn tương đối, nhưng còn một số khả năng khác và một số sắc thái cần lưu ý. Hãy liệt kê tất cả các khả năng:

- Relative specifiers like `./logger.js` or `../logger.js`. They are used to refer to a path relative to the location of the importing file.
- Absolute specifiers like `file:///opt/nodejs/config.js`. They refer directly and explicitly to a full path. Note that this is the only way with ESM to refer to an absolute path for a module, using a `/` or a `//` prefix won't work. This is a significant difference with CommonJS.
- Bare specifiers are identifiers like `fastify` or `http`, and they represent modules available in the `node_modules` folder and generally installed through a package manager (such as npm) or available as core Node.js modules.
- Deep import specifiers like `fastify/lib/logger.js`, which refer to a path within a package in `node_modules` (`fastify`, in this case).

> - Relative specifiers như `./logger.js` hay `../logger.js`. Chúng được dùng để tham chiếu đến một đường dẫn tương đối với vị trí của file đang import.
> - Absolute specifiers như `file:///opt/nodejs/config.js`. Chúng tham chiếu trực tiếp và tường minh đến một đường dẫn đầy đủ. Lưu ý rằng đây là cách duy nhất với ESM để tham chiếu đến một đường dẫn tuyệt đối cho một module, dùng tiền tố `/` hay `//` sẽ không hoạt động. Đây là một khác biệt đáng kể so với CommonJS.
> - Bare specifiers là các định danh như `fastify` hay `http`, và chúng đại diện cho các module có sẵn trong thư mục `node_modules` và nhìn chung được cài đặt thông qua một trình quản lý gói (như npm) hoặc có sẵn dưới dạng các module lõi của Node.js.
> - Deep import specifiers như `fastify/lib/logger.js`, vốn tham chiếu đến một đường dẫn bên trong một gói trong `node_modules` (`fastify`, trong trường hợp này).

In browser environments, it is possible to import modules directly by specifying the module URL, for instance, `https://unpkg.com/lodash`. This feature is not supported by Node.js.

> Trong các môi trường trình duyệt, có thể import các module trực tiếp bằng cách chỉ định URL của module, ví dụ, `https://unpkg.com/lodash`. Tính năng này không được hỗ trợ bởi Node.js.

### Async imports — Async imports

As we have seen in the previous section, the `import` statement is static and therefore subject to two important limitations:

> Như chúng ta đã thấy trong phần trước, câu lệnh `import` là tĩnh và do đó chịu hai giới hạn quan trọng:

- A module identifier cannot be constructed at runtime
- Module imports are declared at the top level of every file and they cannot be nested within control flow statements

> - Một module identifier không thể được dựng vào lúc chạy
> - Các module import được khai báo ở cấp cao nhất của mỗi file và chúng không thể được lồng bên trong các câu lệnh điều khiển luồng

There are some use cases when these limitations can become a little bit too restrictive. Imagine, for instance, if we have to import a specific translation module for the current user language, or a variation of a module that depends on the user's operating system. Also, what if we want to load a given module, which might be particularly heavy, only if the user is accessing the piece of functionality that requires that module?

> Có một số trường hợp sử dụng mà các giới hạn này có thể trở nên hơi quá hạn chế. Ví dụ, hãy tưởng tượng nếu chúng ta phải import một module dịch thuật cụ thể cho ngôn ngữ hiện tại của người dùng, hoặc một biến thể của một module phụ thuộc vào hệ điều hành của người dùng. Ngoài ra, sẽ thế nào nếu chúng ta muốn nạp một module nhất định, vốn có thể đặc biệt nặng, chỉ khi người dùng đang truy cập vào phần chức năng đòi hỏi module đó?

To allow us to overcome these limitations ES modules provides **async** imports (also called dynamic imports).

> Để cho phép chúng ta vượt qua các giới hạn này, ES modules cung cấp async imports (còn gọi là dynamic imports).

Async imports can be performed at runtime using the special `import()` operator.

> Async imports có thể được thực hiện vào lúc chạy bằng toán tử đặc biệt `import()`.

The `import()` operator is syntactically equivalent to a function that takes a module identifier as an argument and it returns a **promise** that resolves to a module object.

> Toán tử `import()` về mặt cú pháp tương đương với một hàm nhận một module identifier làm đối số và nó trả về một promise phân giải (resolve) thành một đối tượng module.

We will learn more about promises in Chapter 5, Asynchronous Control Flow Patterns with Promises and Async/**Await**, so don't worry too much about understanding all the nuances of the specific promise syntax for now.

> Chúng ta sẽ tìm hiểu thêm về promise trong Chương 5, Các mẫu điều khiển luồng bất đồng bộ với Promise và Async/Await, nên đừng quá lo lắng về việc hiểu hết các sắc thái của cú pháp promise cụ thể vào lúc này.

The module identifier can be any module identifier supported by static imports as discussed in the previous section. Now, let's see how to use dynamic imports with a simple example.

> Module identifier có thể là bất kỳ module identifier nào được hỗ trợ bởi static imports như đã bàn trong phần trước. Bây giờ, hãy xem cách dùng dynamic imports với một ví dụ đơn giản.

We want to build a command line application that can print "Hello World" in different languages. In the future, we will probably want to support many more phrases and languages, so it makes sense to have one file with the translations of all the user-facing strings for each supported language.

> Chúng ta muốn xây dựng một ứng dụng dòng lệnh có thể in "Hello World" bằng các ngôn ngữ khác nhau. Trong tương lai, chúng ta có lẽ sẽ muốn hỗ trợ nhiều cụm từ và ngôn ngữ hơn nữa, nên việc có một file với các bản dịch của tất cả các chuỗi hướng tới người dùng cho mỗi ngôn ngữ được hỗ trợ là hợp lý.

Let's create some example modules for some of the languages we want to support:

> Hãy tạo một số module ví dụ cho một vài ngôn ngữ mà chúng ta muốn hỗ trợ:

```js
// strings-el.js
export const HELLO = 'Γεια σου κόσμε'
// strings-en.js
export const HELLO = 'Hello World'
// strings-es.js
export const HELLO = 'Hola mundo'
// strings-it.js
export const HELLO = 'Ciao mondo'
// strings-pl.js
export const HELLO = 'Witaj świecie'
```

Now let's create the main script that takes a language code from the command line and prints "Hello World" in the selected language:

> Bây giờ hãy tạo script chính nhận một mã ngôn ngữ từ dòng lệnh và in "Hello World" bằng ngôn ngữ được chọn:

```js
// main.js
const SUPPORTED_LANGUAGES = ['el', 'en', 'es', 'it', 'pl']   // (1)
const selectedLanguage = process.argv[2]                     // (2)
if (!SUPPORTED_LANGUAGES.includes(selectedLanguage)) {       // (3)
  console.error('The specified language is not supported')
  process.exit(1)
}
const translationModule = `./strings-${selectedLanguage}.js` // (4)
import(translationModule)                                    // (5)
  .then((strings) => {                                       // (6)
    console.log(strings.HELLO)
  })
```

The first part of the script is quite simple. What we do there is:

> Phần đầu của script khá đơn giản. Những gì chúng ta làm ở đó là:

1. Define a list of supported languages.
2. Read the selected language from the first argument passed in the command line.
3. Finally, we handle the case where the selected language is not supported.

> 1. Định nghĩa một danh sách các ngôn ngữ được hỗ trợ.
> 2. Đọc ngôn ngữ được chọn từ đối số đầu tiên được truyền vào trong dòng lệnh.
> 3. Cuối cùng, chúng ta xử lý trường hợp ngôn ngữ được chọn không được hỗ trợ.

The second part of the code is where we actually use dynamic imports:

> Phần thứ hai của code là nơi chúng ta thực sự dùng dynamic imports:

4. First of all, we dynamically build the name of the module we want to import based on the selected language. Note that the module name needs to be a relative path to the module file, that's why we are prepending `./` to the filename.
5. We use the `import()` operator to trigger the dynamic import of the module.
6. The dynamic import happens asynchronously, so we can use the `.then()` hook on the returned promise to get notified when the module is ready to be used. The function passed to `then()` will be executed when the module is fully loaded and `strings` will be the module namespace imported dynamically. After that, we can access `strings.HELLO` and print its value to the console.

> 4. Trước tiên, chúng ta dựng một cách động tên của module mà chúng ta muốn import dựa trên ngôn ngữ được chọn. Lưu ý rằng tên module cần phải là một đường dẫn tương đối đến file module, đó là lý do tại sao chúng ta đặt `./` ở đầu tên file.
> 5. Chúng ta dùng toán tử `import()` để kích hoạt việc import động của module.
> 6. Việc import động xảy ra một cách bất đồng bộ, nên chúng ta có thể dùng hook `.then()` trên promise được trả về để được thông báo khi module sẵn sàng để sử dụng. Hàm được truyền vào `then()` sẽ được thực thi khi module được nạp hoàn toàn và `strings` sẽ là namespace của module được import động. Sau đó, chúng ta có thể truy cập `strings.HELLO` và in giá trị của nó ra console.

Now we can execute this script like this:

> Bây giờ chúng ta có thể thực thi script này như thế này:

```js
node main.js it
```

And we should see Ciao mondo being printed to our console.

> Và chúng ta sẽ thấy Ciao mondo được in ra console của mình.

### Module loading in depth — Nạp module một cách chuyên sâu

To understand how ESM actually works and how it can deal effectively with circular dependencies, we have to deep dive a little bit more into how JavaScript code is parsed and evaluated when using ES modules.

> Để hiểu ESM thực sự hoạt động như thế nào và cách nó có thể xử lý phụ thuộc vòng một cách hiệu quả, chúng ta phải đi sâu hơn một chút vào cách code JavaScript được phân tích cú pháp và đánh giá khi dùng ES modules.

In this section, we will learn how ECMAScript modules are loaded, we will present the idea of read-only live bindings, and, finally, we will discuss an example with circular dependencies.

> Trong phần này, chúng ta sẽ tìm hiểu cách các ECMAScript module được nạp, chúng ta sẽ trình bày ý tưởng về các read-only live binding, và cuối cùng, chúng ta sẽ bàn về một ví dụ với phụ thuộc vòng.

#### Loading phases — Các pha nạp (Loading phases)

The goal of the interpreter is to build a graph of all the necessary modules (a dependency graph).

> Mục tiêu của trình thông dịch là xây dựng một đồ thị (graph) của tất cả các module cần thiết (một đồ thị phụ thuộc - dependency graph).

In generic terms, a dependency graph can be defined as a directed graph (`nodejsdp.link/directed-graph`) representing the dependencies of a group of objects. In the context of this section, when we refer to a dependency graph, we want to indicate the dependency relationship between ECMAScript modules. As we will see, using a dependency graph allows us to determine the order in which all the necessary modules should be loaded in a given project.

> Nói một cách tổng quát, một đồ thị phụ thuộc có thể được định nghĩa là một đồ thị có hướng (`nodejsdp.link/directed-graph`) biểu diễn các phụ thuộc của một nhóm đối tượng. Trong bối cảnh của phần này, khi chúng ta đề cập đến một đồ thị phụ thuộc, chúng ta muốn chỉ mối quan hệ phụ thuộc giữa các ECMAScript module. Như chúng ta sẽ thấy, dùng một đồ thị phụ thuộc cho phép chúng ta xác định thứ tự mà tất cả các module cần thiết nên được nạp trong một dự án nhất định.

Essentially, the dependency graph is needed by the interpreter to figure out how modules depend on each other and in what order the code needs to be executed. When the `node` interpreter is launched, it gets passed some code to execute, generally in the form of a JavaScript file. This file is the starting point for the dependency resolution, and it is called the entry point. From the entry point, the interpreter will find and follow all the `import` statements recursively in a depth-first fashion, until all the necessary code is explored and then evaluated.

> Về cơ bản, đồ thị phụ thuộc được trình thông dịch cần đến để tìm ra cách các module phụ thuộc vào nhau và theo thứ tự nào code cần được thực thi. Khi trình thông dịch `node` được khởi chạy, nó được truyền vào một số code để thực thi, nhìn chung là dưới dạng một file JavaScript. File này là điểm khởi đầu cho việc phân giải phụ thuộc, và nó được gọi là điểm vào (entry point). Từ điểm vào, trình thông dịch sẽ tìm và đi theo tất cả các câu lệnh `import` một cách đệ quy theo kiểu duyệt theo chiều sâu (depth-first), cho đến khi tất cả code cần thiết được khám phá và sau đó được đánh giá.

More specifically, this process happens in three separate phases:

> Cụ thể hơn, quá trình này xảy ra trong ba pha riêng biệt:

- Phase 1 - Construction (or parsing): Find all the imports and recursively load the content of every module from the respective file.
- Phase 2 - Instantiation: For every exported entity, keep a named reference in memory, but don't assign any value just yet. Also, references are created for all the `import` and `export` statements tracking the dependency relationship between them (linking). No JavaScript code has been executed at this stage.
- Phase 3 - Evaluation: Node.js finally executes the code so that all the previously instantiated entities can get an actual value. Now running the code from the entry point is possible because all the blanks have been filled.

> - Pha 1 - Construction (hay parsing): Tìm tất cả các import và nạp đệ quy nội dung của mỗi module từ file tương ứng.
> - Pha 2 - Instantiation: Đối với mỗi thực thể được xuất ra, giữ một tham chiếu có tên trong bộ nhớ, nhưng chưa gán bất kỳ giá trị nào. Ngoài ra, các tham chiếu được tạo cho tất cả các câu lệnh `import` và `export` theo dõi mối quan hệ phụ thuộc giữa chúng (linking). Chưa có code JavaScript nào được thực thi ở giai đoạn này.
> - Pha 3 - Evaluation: Node.js cuối cùng thực thi code để tất cả các thực thể đã được instantiate trước đó có thể nhận một giá trị thực sự. Bây giờ việc chạy code từ điểm vào là khả thi vì tất cả các chỗ trống đã được điền vào.

In simple terms, we could say that Phase 1 is about finding all the dots, Phase 2 connects those creating paths, and, finally, Phase 3 walks through the paths in the right order.

> Nói một cách đơn giản, chúng ta có thể nói rằng Pha 1 là về việc tìm tất cả các điểm, Pha 2 nối các điểm đó để tạo ra các đường đi, và cuối cùng, Pha 3 đi qua các đường đi theo đúng thứ tự.

At first glance, this approach doesn't seem very different from what CommonJS does, but there's a fundamental difference. Due to its dynamic nature, CommonJS will execute all the files while the dependency graph is explored. We have seen that every time a new `require` statement is found, all the previous code has already been executed. This is why you can use `require` even within `if` statements or loops, and construct module identifiers from variables.

> Thoạt nhìn, cách tiếp cận này có vẻ không khác lắm so với những gì CommonJS làm, nhưng có một sự khác biệt cơ bản. Do bản chất động của nó, CommonJS sẽ thực thi tất cả các file trong khi đồ thị phụ thuộc được khám phá. Chúng ta đã thấy rằng mỗi khi một câu lệnh `require` mới được tìm thấy, tất cả code trước đó đã được thực thi rồi. Đây là lý do tại sao bạn có thể dùng `require` ngay cả bên trong các câu lệnh `if` hoặc vòng lặp, và dựng các module identifier từ các biến.

In ESM, these three phases are totally separate from each other, no code can be executed until the dependency graph has been fully built, and therefore module imports and exports have to be static.

> Trong ESM, ba pha này hoàn toàn tách biệt với nhau, không code nào có thể được thực thi cho đến khi đồ thị phụ thuộc được xây dựng hoàn toàn, và do đó các module import và export phải là tĩnh.

#### Read-only live bindings — Read-only live binding

Another fundamental characteristic of ES modules, which helps with cyclic dependencies, is the idea that imported modules are effectively read-only live bindings to their exported values.

> Một đặc điểm cơ bản khác của ES modules, vốn giúp ích cho phụ thuộc vòng, là ý tưởng rằng các module được import thực chất là các read-only live binding đến các giá trị được xuất ra của chúng.

Let's clarify what this means with a simple example:

> Hãy làm rõ điều này nghĩa là gì với một ví dụ đơn giản:

```js
// counter.js
export let count = 0
export function increment () {
  count++
}
```

This module exports two values: a simple integer counter called `count` and an `increment` function that increases the counter by one.

> Module này xuất ra hai giá trị: một bộ đếm số nguyên đơn giản tên là `count` và một hàm `increment` làm tăng bộ đếm lên một.

Let's now write some code that uses this module:

> Bây giờ hãy viết một số code dùng module này:

```js
// main.js
import { count, increment } from './counter.js'
console.log(count) // prints 0
increment()
console.log(count) // prints 1
count++ // TypeError: Assignment to constant variable!
```

What we can see in this code is that we can read the value of `count` at any time and change it using the `increment()` function, but as soon as we try to mutate the `count` variable directly, we get an error as if we were trying to mutate a `const` binding.

> Những gì chúng ta có thể thấy trong code này là chúng ta có thể đọc giá trị của `count` vào bất kỳ lúc nào và thay đổi nó bằng hàm `increment()`, nhưng ngay khi chúng ta cố biến đổi (mutate) biến `count` một cách trực tiếp, chúng ta nhận được một lỗi như thể chúng ta đang cố biến đổi một binding `const`.

This proves that when an entity is imported in the scope, the binding to its original value cannot be changed (read-only binding) unless the bound value changes within the scope of the original module itself (live binding), which is outside the direct control of the consumer code.

> Điều này chứng minh rằng khi một thực thể được import vào phạm vi, binding đến giá trị gốc của nó không thể bị thay đổi (read-only binding) trừ khi giá trị được bind thay đổi trong phạm vi của chính module gốc (live binding), điều nằm ngoài tầm kiểm soát trực tiếp của code tiêu thụ.

This approach is fundamentally different from CommonJS. In fact, in CommonJS, the entire `exports` object is copied (shallow copy) when required from a module. This means that, if the value of primitive variables like numbers or string is changed at a later time, the requiring module won't be able to see those changes.

> Cách tiếp cận này khác biệt cơ bản so với CommonJS. Thực tế, trong CommonJS, toàn bộ đối tượng `exports` được sao chép (shallow copy) khi được require từ một module. Điều này có nghĩa là, nếu giá trị của các biến nguyên thủy như số hay chuỗi bị thay đổi vào một thời điểm muộn hơn, thì module đang require sẽ không thể thấy được những thay đổi đó.

#### Circular dependency resolution — Phân giải phụ thuộc vòng (Circular dependency resolution)

Now to close the circle, let's reimplement the circular dependency example we saw in the CommonJS modules section using the ESM syntax:

> Bây giờ để khép lại vòng tròn, hãy hiện thực lại ví dụ phụ thuộc vòng mà chúng ta đã thấy trong phần module CommonJS bằng cú pháp ESM:

![](images/ch02-79_1.png)

Figure 2.3: An example scenario with circular dependencies

> Hình 2.3: Một kịch bản ví dụ với phụ thuộc vòng

Let's have a look at the modules `a.js` and `b.js` first:

> Hãy xem các module `a.js` và `b.js` trước:

```js
// a.js
import * as bModule from './b.js'
export let loaded = false
export const b = bModule
loaded = true
// b.js
import * as aModule from './a.js'
export let loaded = false
export const a = aModule
loaded = true
```

And now let's see how to import those two modules in our `main.js` file (the entry point):

> Và bây giờ hãy xem cách import hai module này trong file `main.js` của chúng ta (điểm vào):

```js
// main.js
import * as a from './a.js'
import * as b from './b.js'
console.log('a ->', a)
console.log('b ->', b)
```

Note that this time we are not using `JSON.stringify` because that will fail with a `TypeError: Converting circular structure to JSON`, since there's an actual circular reference between `a.js` and `b.js`.

> Lưu ý rằng lần này chúng ta không dùng `JSON.stringify` bởi vì điều đó sẽ thất bại với lỗi `TypeError: Converting circular structure to JSON`, vì có một tham chiếu vòng thực sự giữa `a.js` và `b.js`.

When we run `main.js`, we will see the following output:

> Khi chúng ta chạy `main.js`, chúng ta sẽ thấy kết quả sau:

```js
a -> <ref *1> [Module] {
  b: [Module] { a: [Circular *1], loaded: true },
  loaded: true
}
b -> <ref *1> [Module] {
  a: [Module] { b: [Circular *1], loaded: true },
  loaded: true
}
```

The interesting bit here is that the modules `a.js` and `b.js` have a complete picture of each other, unlike what would happen with CommonJS, where they would only hold partial information of each other. We can see that because all the `loaded` values are set to `true`. Also, `b` within `a` is an actual reference to the same `b` instance available in the current scope, and the same goes for `a` within `b`. That's the reason why we cannot use `JSON.stringify()` to serialize these modules. Finally, if we swap the order of the imports for the modules `a.js` and `b.js`, the final outcome does not change, which is another important difference in comparison with how CommonJS works

> Điểm thú vị ở đây là các module `a.js` và `b.js` có một bức tranh đầy đủ về nhau, không giống như những gì sẽ xảy ra với CommonJS, nơi chúng sẽ chỉ giữ thông tin không đầy đủ về nhau. Chúng ta có thể thấy điều đó vì tất cả các giá trị `loaded` đều được đặt thành `true`. Ngoài ra, `b` bên trong `a` là một tham chiếu thực sự đến cùng instance `b` có sẵn trong phạm vi hiện tại, và điều tương tự cũng đúng với `a` bên trong `b`. Đó là lý do tại sao chúng ta không thể dùng `JSON.stringify()` để tuần tự hóa các module này. Cuối cùng, nếu chúng ta hoán đổi thứ tự của các import cho các module `a.js` và `b.js`, kết quả cuối cùng không thay đổi, đây là một khác biệt quan trọng khác so với cách CommonJS hoạt động

It's worth spending some more time observing what happens in the three phases of the module resolution (parsing, instantiation, and evaluation) for this specific example.

> Đáng để dành thêm chút thời gian quan sát những gì xảy ra trong ba pha của việc phân giải module (parsing, instantiation và evaluation) cho ví dụ cụ thể này.

##### Phase 1: Parsing — Pha 1: Parsing

During the parsing phase, the code is explored starting from the entry point (`main.js`). The interpreter looks only for `import` statements to find all the necessary modules and to load the source code from the module files. The dependency graph is explored in a depth-first fashion, and every module is visited only once. This way the interpreter builds a view of the dependencies that looks like a tree structure, as shown in Figure 2.4:

> Trong pha parsing, code được khám phá bắt đầu từ điểm vào (`main.js`). Trình thông dịch chỉ tìm các câu lệnh `import` để tìm tất cả các module cần thiết và nạp mã nguồn từ các file module. Đồ thị phụ thuộc được khám phá theo kiểu duyệt theo chiều sâu, và mỗi module chỉ được thăm một lần. Theo cách này, trình thông dịch xây dựng một góc nhìn về các phụ thuộc trông giống như một cấu trúc cây, như được thể hiện trong Hình 2.4:

![](images/ch02-81_1.png)

Figure 2.4: Parsing of cyclic dependencies with ESM

> Hình 2.4: Parsing các phụ thuộc vòng với ESM

Given the example in Figure 2.4, let's discuss the various steps of the parsing phase:

> Với ví dụ trong Hình 2.4, hãy bàn về các bước khác nhau của pha parsing:

1. From `main.js`, the first import found leads us straight into `a.js`.
2. In `a.js` we find an import pointing to `b.js`.
3. In `b.js`, we also have an import back to `a.js` (our cycle), but since `a.js` has already been visited, this path is not explored again.
4. At this point, the exploration starts to wind back: `b.js` doesn't have other imports, so we go back to `a.js`; `a.js` doesn't have other `import` statements so we go back to `main.js`. Here we find another import pointing to `b.js`, but again this module has been explored already, so this path is ignored.

> 1. Từ `main.js`, import đầu tiên được tìm thấy dẫn chúng ta thẳng vào `a.js`.
> 2. Trong `a.js` chúng ta tìm thấy một import trỏ đến `b.js`.
> 3. Trong `b.js`, chúng ta cũng có một import trở lại `a.js` (vòng lặp của chúng ta), nhưng vì `a.js` đã được thăm rồi, đường đi này không được khám phá lại.
> 4. Tại thời điểm này, việc khám phá bắt đầu lùi lại: `b.js` không có import nào khác, nên chúng ta quay lại `a.js`; `a.js` không có câu lệnh `import` nào khác nên chúng ta quay lại `main.js`. Ở đây chúng ta tìm thấy một import khác trỏ đến `b.js`, nhưng một lần nữa module này đã được khám phá rồi, nên đường đi này bị bỏ qua.

At this point, our depth-first visit of the dependency graph has been completed and we have a linear view of the modules, as shown in Figure 2.5:

> Tại thời điểm này, việc duyệt theo chiều sâu đồ thị phụ thuộc của chúng ta đã hoàn tất và chúng ta có một góc nhìn tuyến tính về các module, như được thể hiện trong Hình 2.5:

![](images/ch02-82_1.png)

Figure 2.5: A linear view of the module graph where cycles have been removed

> Hình 2.5: Một góc nhìn tuyến tính về đồ thị module nơi các vòng đã được loại bỏ

This particular view is quite simple. In more realistic scenarios with a lot more modules, the view will look more like a tree structure.

> Góc nhìn cụ thể này khá đơn giản. Trong các kịch bản thực tế hơn với nhiều module hơn rất nhiều, góc nhìn sẽ trông giống một cấu trúc cây hơn.

##### Phase 2: Instantiation — Pha 2: Instantiation

In the instantiation phase, the interpreter walks the tree view obtained from the previous phase from the bottom to the top. For every module, the interpreter will look for all the exported properties first and build out a map of the exported names in memory:

> Trong pha instantiation, trình thông dịch đi qua góc nhìn dạng cây thu được từ pha trước từ dưới lên trên. Đối với mỗi module, trình thông dịch sẽ tìm tất cả các thuộc tính được xuất ra trước và xây dựng một bản đồ (map) các tên được xuất ra trong bộ nhớ:

![](images/ch02-82_2.png)

Figure 2.6: A visual representation of the instantiation phase Figure 2.6 describes the order in which every module is instantiated:

> Hình 2.6: Một biểu diễn trực quan về pha instantiation. Hình 2.6 mô tả thứ tự mà mỗi module được instantiate:

1. The interpreter starts from `b.js` and discovers that the module exports `loaded` and `a`.
2. Then, the interpreter moves to `a.js`, which exports `loaded` and `b`.
3. Finally, it moves to `main.js`, which does not export any functionality.
4. Note that, in this phase, the exports map keeps track of the exported names only; their associated values are considered uninitialized for now.

> 1. Trình thông dịch bắt đầu từ `b.js` và phát hiện rằng module xuất ra `loaded` và `a`.
> 2. Sau đó, trình thông dịch chuyển sang `a.js`, vốn xuất ra `loaded` và `b`.
> 3. Cuối cùng, nó chuyển sang `main.js`, vốn không xuất ra bất kỳ chức năng nào.
> 4. Lưu ý rằng, trong pha này, bản đồ exports chỉ theo dõi các tên được xuất ra; các giá trị liên quan của chúng hiện tại được coi là chưa khởi tạo.

After this sequence of steps, the interpreter will do another pass to link the exported names to the modules importing them, as shown in Figure 2.7:

> Sau chuỗi các bước này, trình thông dịch sẽ thực hiện thêm một lượt nữa để liên kết các tên được xuất ra với các module import chúng, như được thể hiện trong Hình 2.7:

![](images/ch02-83_1.png)

Figure 2.7: Linking exports with imports across modules

> Hình 2.7: Liên kết exports với imports giữa các module

We can describe what we see in Figure 2.7 through the following steps:

> Chúng ta có thể mô tả những gì chúng ta thấy trong Hình 2.7 thông qua các bước sau:

1. Module `b.js` will link the exports from `a.js`, referring to them as `aModule`.
2. In turn, `a.js` will link to all the exports from `b.js`, referring to them as `bModule`.
3. Finally, `main.js` will import all the exports in `b.js`, referring to them as `b`; similarly, it will import everything from `a.js`, referring to them as `a`.
4. Again, it's important to note that all the values are still uninitialized. In this phase, we are only linking references to values that will be available at the end of the next phase.

> 1. Module `b.js` sẽ liên kết các exports từ `a.js`, tham chiếu đến chúng dưới dạng `aModule`.
> 2. Đến lượt mình, `a.js` sẽ liên kết đến tất cả các exports từ `b.js`, tham chiếu đến chúng dưới dạng `bModule`.
> 3. Cuối cùng, `main.js` sẽ import tất cả các exports trong `b.js`, tham chiếu đến chúng dưới dạng `b`; tương tự, nó sẽ import mọi thứ từ `a.js`, tham chiếu đến chúng dưới dạng `a`.
> 4. Một lần nữa, điều quan trọng cần lưu ý là tất cả các giá trị vẫn chưa được khởi tạo. Trong pha này, chúng ta chỉ đang liên kết các tham chiếu đến các giá trị sẽ có sẵn vào cuối pha tiếp theo.

##### Phase 3: Evaluation — Pha 3: Evaluation

The last step is the evaluation phase. In this phase, all the code in every file is finally executed. The execution order is again bottom-up respecting the post-order depthfirst visit of our original dependency graph. With this approach, `main.js` is the last file to be executed. This way, we can be sure that all the exported values have been initialized before we start executing our main business logic:

> Bước cuối cùng là pha evaluation. Trong pha này, tất cả code trong mọi file cuối cùng được thực thi. Thứ tự thực thi một lần nữa là từ dưới lên trên, tôn trọng việc duyệt theo chiều sâu hậu thứ tự (post-order depth-first) đồ thị phụ thuộc gốc của chúng ta. Với cách tiếp cận này, `main.js` là file cuối cùng được thực thi. Theo cách này, chúng ta có thể chắc chắn rằng tất cả các giá trị được xuất ra đã được khởi tạo trước khi chúng ta bắt đầu thực thi logic nghiệp vụ chính của mình:

![](images/ch02-84_1.png)

Figure 2.8: A visual representation of the evaluation phase

> Hình 2.8: Một biểu diễn trực quan về pha evaluation

Following along from the diagram in Figure 2.8, this is what happens:

> Theo dõi từ sơ đồ trong Hình 2.8, đây là những gì xảy ra:

1. The execution starts from `b.js` and the first line to be evaluated initializes the `loaded` export to `false` for the module.
2. Similarly, here the exported property `a` gets evaluated. This time, it will be evaluated to a reference to the module object representing module `a.js`.
3. The value of the `loaded` property gets changed to `true`. At this point, we have fully evaluated the state of the exports for module `b.js`.
4. Now the execution moves to `a.js`. Again, we start by setting `loaded` to `false`.
5. At this point, the `b` export is evaluated to a reference to module `b.js`.
6. Finally, the `loaded` property is changed to `true`. Now we have finally evaluated all the exports for `a.js` as well.

> 1. Việc thực thi bắt đầu từ `b.js` và dòng đầu tiên được đánh giá khởi tạo export `loaded` thành `false` cho module.
> 2. Tương tự, ở đây thuộc tính được xuất ra `a` được đánh giá. Lần này, nó sẽ được đánh giá thành một tham chiếu đến đối tượng module đại diện cho module `a.js`.
> 3. Giá trị của thuộc tính `loaded` được đổi thành `true`. Tại thời điểm này, chúng ta đã đánh giá hoàn toàn trạng thái của các exports cho module `b.js`.
> 4. Bây giờ việc thực thi chuyển sang `a.js`. Một lần nữa, chúng ta bắt đầu bằng việc đặt `loaded` thành `false`.
> 5. Tại thời điểm này, export `b` được đánh giá thành một tham chiếu đến module `b.js`.
> 6. Cuối cùng, thuộc tính `loaded` được đổi thành `true`. Bây giờ chúng ta cuối cùng đã đánh giá tất cả các exports cho `a.js` nữa.

After all these steps, the code in `main.js` can be executed, and at this point, all the exported properties are fully evaluated. Since imported modules are tracked as references, we can be sure every module has an up-to-date picture of the other modules, even in the presence of circular dependencies.

> Sau tất cả các bước này, code trong `main.js` có thể được thực thi, và tại thời điểm này, tất cả các thuộc tính được xuất ra đã được đánh giá hoàn toàn. Vì các module được import được theo dõi dưới dạng các tham chiếu, chúng ta có thể chắc chắn rằng mỗi module có một bức tranh cập nhật về các module khác, ngay cả khi có sự hiện diện của phụ thuộc vòng.

### Modifying other modules — Sửa đổi các module khác

We saw that entities imported through ES modules are read-only live bindings, and therefore we cannot reassign them from an external module.

> Chúng ta đã thấy rằng các thực thể được import thông qua ES modules là các read-only live binding, và do đó chúng ta không thể gán lại chúng từ một module bên ngoài.

There's a caveat, though. It is true that we can't change the bindings of the default export or named exports of an existing module from another module, but, if one of these bindings is an object, we can still mutate the object itself by reassigning some of the object properties.

> Tuy nhiên, có một điều cần lưu ý. Đúng là chúng ta không thể thay đổi các binding của default export hoặc named exports của một module hiện có từ một module khác, nhưng, nếu một trong các binding này là một đối tượng, chúng ta vẫn có thể biến đổi (mutate) chính đối tượng đó bằng cách gán lại một số thuộc tính của đối tượng.

This caveat can give us enough freedom to alter the behavior of other modules. To demonstrate this idea, let's write a module that can alter the behavior of the core `fs` module so that it prevents the module from accessing the filesystem and returns mocked data instead. This kind of module is something that could be useful while writing tests for a component that relies on the filesystem:

> Điều cần lưu ý này có thể cho chúng ta đủ tự do để thay đổi hành vi của các module khác. Để minh họa ý tưởng này, hãy viết một module có thể thay đổi hành vi của module lõi `fs` để nó ngăn module truy cập hệ thống tập tin và trả về dữ liệu giả lập (mocked) thay thế. Loại module này là một thứ có thể hữu ích khi viết các bài kiểm thử cho một thành phần dựa vào hệ thống tập tin:

```js
// mock-read-file.js
import fs from 'fs'                                        // (1)
const originalReadFile = fs.readFile                       // (2)
let mockedResponse = null
function mockedReadFile (path, cb) {                       // (3)
  setImmediate(() => {
    cb(null, mockedResponse)
  })
}
export function mockEnable (respondWith) {                 // (4)
  mockedResponse = respondWith
  fs.readFile = mockedReadFile
}
export function mockDisable () {                           // (5)
  fs.readFile = originalReadFile
}
```

Let's review the preceding code:

> Hãy xem lại đoạn code trên:

1. The first thing we do is import the default export of the `fs` module. We will get back to this in a second, for now, just keep in mind that the default export of the `fs` module is an object that contains a collection of functions that allows us to interact with the filesystem.
2. We want to replace the `readFile()` function with a mock implementation. Before doing that, we save a reference to the original implementation. We also declare a `mockedResponse` value that we will be using later.
3. The function `mockedReadFile()` is the actual mocked implementation that we want to use to replace the original implementation. This function invokes the callback with the current value of `mockedResponse`. Note that this is a simplified implementation; the real function accepts an optional `options` argument before the callback argument and is able to handle different types of encoding.
4. The exported `mockEnable()` function can be used to activate the mocked functionality. The original implementation will be swapped with the mocked one. The mocked implementation will return the same value passed here through the `respondWith` argument.
5. Finally, the exported `mockDisable()` function can be used to restore the original implementation of the `fs.readFile()` function.

> 1. Điều đầu tiên chúng ta làm là import default export của module `fs`. Chúng ta sẽ quay lại điều này sau một giây, hiện tại, chỉ cần nhớ rằng default export của module `fs` là một đối tượng chứa một tập các hàm cho phép chúng ta tương tác với hệ thống tập tin.
> 2. Chúng ta muốn thay thế hàm `readFile()` bằng một hiện thực giả lập. Trước khi làm điều đó, chúng ta lưu một tham chiếu đến hiện thực gốc. Chúng ta cũng khai báo một giá trị `mockedResponse` mà chúng ta sẽ dùng sau.
> 3. Hàm `mockedReadFile()` là hiện thực giả lập thực tế mà chúng ta muốn dùng để thay thế hiện thực gốc. Hàm này gọi callback với giá trị hiện tại của `mockedResponse`. Lưu ý rằng đây là một hiện thực đơn giản hóa; hàm thực sự nhận một đối số `options` tùy chọn trước đối số callback và có thể xử lý các loại mã hóa (encoding) khác nhau.
> 4. Hàm được xuất ra `mockEnable()` có thể được dùng để kích hoạt chức năng giả lập. Hiện thực gốc sẽ được hoán đổi bằng hiện thực giả lập. Hiện thực giả lập sẽ trả về cùng giá trị được truyền vào đây thông qua đối số `respondWith`.
> 5. Cuối cùng, hàm được xuất ra `mockDisable()` có thể được dùng để khôi phục hiện thực gốc của hàm `fs.readFile()`.

Now let's see a simple example that uses this module:

> Bây giờ hãy xem một ví dụ đơn giản dùng module này:

```js
// main.js
import fs from 'fs'                                          // (1)
import { mockEnable, mockDisable } from './mock-read-file.js'
mockEnable(Buffer.from('Hello World'))                       // (2)
fs.readFile('fake-path', (err, data) => {                    // (3)
  if (err) {
    console.error(err)
    process.exit(1)
  }
  console.log(data.toString()) // 'Hello World'
})
mockDisable()
```

Let's discuss step by step what happens in this example:

> Hãy bàn từng bước những gì xảy ra trong ví dụ này:

1. The first thing that we do is import the default export of the `fs` module. Again, note that we are importing specifically the default export exactly as we did in our `mock-read-file.js` module, but more on this later.
2. Here we enable the mock functionality. We want, for every file read, to simulate that the file contains the string "Hello World."
3. Finally, we read a file using a fake path. This code will print "Hello World" as it will be using the mocked version of the `readFile()` function. Note that, after calling this function, we restore the original implementation by calling `mockDisable()`.

> 1. Điều đầu tiên chúng ta làm là import default export của module `fs`. Một lần nữa, lưu ý rằng chúng ta đang import cụ thể default export đúng như cách chúng ta đã làm trong module `mock-read-file.js` của mình, nhưng sẽ nói thêm về điều này sau.
> 2. Ở đây chúng ta kích hoạt chức năng giả lập. Chúng ta muốn, đối với mỗi lần đọc file, mô phỏng rằng file chứa chuỗi "Hello World."
> 3. Cuối cùng, chúng ta đọc một file bằng một đường dẫn giả. Code này sẽ in ra "Hello World" vì nó sẽ dùng phiên bản giả lập của hàm `readFile()`. Lưu ý rằng, sau khi gọi hàm này, chúng ta khôi phục hiện thực gốc bằng cách gọi `mockDisable()`.

This approach works, but it is very fragile. In fact, there are a number of ways in which this may not work.

> Cách tiếp cận này hoạt động, nhưng nó rất mong manh. Thực tế, có một số cách mà điều này có thể không hoạt động.

On the `mock-read-file.js` side, we could have tried the two following imports for the `fs` module:

> Về phía `mock-read-file.js`, chúng ta có thể đã thử hai cách import sau cho module `fs`:

```js
import * as fs from 'fs' // then use fs.readFile
```

or

> hoặc

```js
import { readFile } from 'fs'
```

Both of them are valid imports because the `fs` module exports all the filesystem functions as named exports (other than a default export which is an object with the same collection of functions as attributes).

> Cả hai đều là các import hợp lệ vì module `fs` xuất ra tất cả các hàm hệ thống tập tin dưới dạng named exports (ngoài một default export vốn là một đối tượng có cùng tập các hàm dưới dạng các thuộc tính).

There are certain issues with the preceding two `import` statements:

> Có một số vấn đề nhất định với hai câu lệnh `import` trên:

- We would get a read-only live binding into the `readFile()` function, and therefore, we would be unable to mutate it from an external module. If we try these approaches, we will get an error when trying to reassign `readFile()`.
- Another issue is on the consumer side within our `main.js`, where we could use these two alternative import styles as well. In this case, we won't end up using the mocked functionality, and therefore the code will trigger an error while trying to read a nonexistent file.

> - Chúng ta sẽ nhận được một read-only live binding vào hàm `readFile()`, và do đó, chúng ta sẽ không thể biến đổi nó từ một module bên ngoài. Nếu chúng ta thử các cách tiếp cận này, chúng ta sẽ nhận được một lỗi khi cố gán lại `readFile()`.
> - Một vấn đề khác là ở phía bên tiêu thụ trong `main.js` của chúng ta, nơi chúng ta cũng có thể dùng hai kiểu import thay thế này. Trong trường hợp này, chúng ta sẽ không dùng đến chức năng giả lập, và do đó code sẽ kích hoạt một lỗi khi cố đọc một file không tồn tại.

The reason why using one of the two `import` statements mentioned above would not work is because our mocking utility is altering only the copy of the `readFile()` function that is registered inside the object exported as the default export, but not the one available as a named export at the top level of the module. This particular example shows us how monkey patching could be much more complicated and unreliable in the context of ESM. For this reason, testing frameworks such as Jest (`nodejsdp.link/jest`) provide special functionalities to be able to mock ES modules more reliably (`nodejsdp.link/jest-mock`).

> Lý do tại sao việc dùng một trong hai câu lệnh `import` được đề cập ở trên sẽ không hoạt động là vì tiện ích giả lập của chúng ta chỉ đang thay đổi bản sao của hàm `readFile()` được đăng ký bên trong đối tượng được xuất ra dưới dạng default export, chứ không phải bản có sẵn dưới dạng một named export ở cấp cao nhất của module. Ví dụ cụ thể này cho chúng ta thấy monkey patching có thể phức tạp và kém tin cậy hơn nhiều như thế nào trong bối cảnh ESM. Vì lý do này, các framework kiểm thử như Jest (`nodejsdp.link/jest`) cung cấp các chức năng đặc biệt để có thể giả lập các ES module một cách đáng tin cậy hơn (`nodejsdp.link/jest-mock`).

Another approach that can be used to mock modules is to rely on the hooks available in a special Node.js core module called `module` (`nodejsdp.link/module-doc`). One simple library that takes advantage of this module is `mocku` (`nodejsdp.link/mocku`). Check out its source code if you are curious.

> Một cách tiếp cận khác có thể được dùng để giả lập các module là dựa vào các hook có sẵn trong một module lõi đặc biệt của Node.js gọi là `module` (`nodejsdp.link/module-doc`). Một thư viện đơn giản tận dụng module này là `mocku` (`nodejsdp.link/mocku`). Hãy xem mã nguồn của nó nếu bạn tò mò.

We could also use the `syncBuiltinESMExports()` function from the `module` package. When this function is invoked, the value of the properties in the default exports object gets mapped again into the equivalent named exports, effectively allowing us to propagate any external change applied to the module functionality even to named exports:

> Chúng ta cũng có thể dùng hàm `syncBuiltinESMExports()` từ gói `module`. Khi hàm này được gọi, giá trị của các thuộc tính trong đối tượng default exports được ánh xạ lại vào các named exports tương đương, qua đó hiệu quả cho phép chúng ta lan truyền bất kỳ thay đổi bên ngoài nào được áp dụng cho chức năng module ngay cả đến các named exports:

```js
import fs, { readFileSync } from 'fs'
import { syncBuiltinESMExports } from 'module'
fs.readFileSync = () => Buffer.from('Hello, ESM')
syncBuiltinESMExports()
console.log(fs.readFileSync === readFileSync) // true
```

We could use this to make our small filesystem mocking utility a little bit more flexible by invoking the `syncBuiltinESMExports()` function after we enable the mock or after we restore the original functionality.

> Chúng ta có thể dùng điều này để làm cho tiện ích giả lập hệ thống tập tin nhỏ của mình linh hoạt hơn một chút bằng cách gọi hàm `syncBuiltinESMExports()` sau khi chúng ta kích hoạt giả lập hoặc sau khi chúng ta khôi phục chức năng gốc.

Note that `syncBuiltinESMExports()` works only for built-in Node.js modules like the `fs` module in our example.

> Lưu ý rằng `syncBuiltinESMExports()` chỉ hoạt động cho các module dựng sẵn của Node.js như module `fs` trong ví dụ của chúng ta.

This concludes our exploration of ESM. At this point, we should be able to appreciate how ESM works, how it loads modules, and how it deals with cyclic dependencies. To close this chapter, we are now ready to discuss some key differences and some interesting interoperability techniques between CommonJS and ECMAScript modules.

> Điều này kết thúc việc khám phá ESM của chúng ta. Tại thời điểm này, chúng ta hẳn đã có thể nhận thức được ESM hoạt động như thế nào, cách nó nạp module, và cách nó xử lý phụ thuộc vòng. Để khép lại chương này, bây giờ chúng ta đã sẵn sàng để bàn về một số khác biệt then chốt và một số kỹ thuật tương tác thú vị giữa CommonJS và ECMAScript modules.

## ESM and CommonJS differences and interoperability — Sự khác biệt và khả năng tương tác giữa ESM và CommonJS

We already mentioned several important differences between ESM and CommonJS, such as having to explicitly specify file extensions in imports with ESM, while file extensions are totally optional with the CommonJS `require` function.

> Chúng ta đã đề cập đến một số khác biệt quan trọng giữa ESM và CommonJS, chẳng hạn như việc phải chỉ định tường minh phần mở rộng file trong các import với ESM, trong khi phần mở rộng file là hoàn toàn tùy chọn với hàm `require` của CommonJS.

Let's close this chapter by discussing some other important differences between ESM and CommonJS and how the two module systems can work together when necessary.

> Hãy khép lại chương này bằng cách bàn về một số khác biệt quan trọng khác giữa ESM và CommonJS và cách hai hệ thống module có thể làm việc cùng nhau khi cần thiết.

### ESM runs in strict mode — ESM chạy ở chế độ strict

ES modules run implicitly in strict mode. This means that we don't have to explicitly add the `"use strict"` statements at the beginning of every file. Strict mode cannot be disabled; therefore, we cannot use undeclared variables or the `with` statement or have other features that are only available in non-strict mode, but this is definitely a good thing, as strict mode is a safer execution mode.

> ES modules chạy ngầm định ở chế độ strict. Điều này có nghĩa là chúng ta không phải thêm tường minh các câu lệnh `"use strict"` vào đầu mỗi file. Chế độ strict không thể bị tắt; do đó, chúng ta không thể dùng các biến chưa khai báo hoặc câu lệnh `with` hoặc có các tính năng khác chỉ có sẵn ở chế độ non-strict, nhưng đây chắc chắn là một điều tốt, vì chế độ strict là một chế độ thực thi an toàn hơn.

If you are curious to find out more about the differences between the two modes, you can check out a very detailed article on MDN Web Docs (`https://nodejsdp.link/strict-mode`).

> Nếu bạn tò mò muốn tìm hiểu thêm về sự khác biệt giữa hai chế độ, bạn có thể xem một bài viết rất chi tiết trên MDN Web Docs (`https://nodejsdp.link/strict-mode`).

### Missing references in ESM — Các tham chiếu bị thiếu trong ESM

In ESM, some important CommonJS references are not defined. These include `require`, `exports`, `module.exports`, `__filename`, and `__dirname`. If we try to use any of them within an ES module, since it also runs in strict mode, we will get a `ReferenceError`:

> Trong ESM, một số tham chiếu quan trọng của CommonJS không được định nghĩa. Những tham chiếu này bao gồm `require`, `exports`, `module.exports`, `__filename` và `__dirname`. Nếu chúng ta thử dùng bất kỳ tham chiếu nào trong số chúng bên trong một ES module, vì nó cũng chạy ở chế độ strict, chúng ta sẽ nhận được một `ReferenceError`:

```js
console.log(exports) // ReferenceError: exports is not defined
console.log(module) // ReferenceError: module is not defined
console.log(__filename) // ReferenceError: __filename is not defined
console.log(__dirname) // ReferenceError: __dirname is not defined
```

We already discussed at length the meaning of `exports` and `module` in CommonJS; `__filename` and `__dirname` represent the absolute path to the current module file and the absolute path to its parent folder. Those special variables can be very useful when we need to build a path relative to the current file. In ESM, it is possible to get a reference to the current file URL by using the special object `import.meta`. Specifically, `import.meta.url` is a reference to the current module file in a format similar to `file:///path/to/current_module.js`. This value can be used to reconstruct `__filename` and `__dirname` in the form of absolute paths:

> Chúng ta đã bàn rất kỹ về ý nghĩa của `exports` và `module` trong CommonJS; `__filename` và `__dirname` đại diện cho đường dẫn tuyệt đối đến file module hiện tại và đường dẫn tuyệt đối đến thư mục cha của nó. Những biến đặc biệt này có thể rất hữu ích khi chúng ta cần dựng một đường dẫn tương đối với file hiện tại. Trong ESM, có thể lấy một tham chiếu đến URL của file hiện tại bằng cách dùng đối tượng đặc biệt `import.meta`. Cụ thể, `import.meta.url` là một tham chiếu đến file module hiện tại theo định dạng tương tự như `file:///path/to/current_module.js`. Giá trị này có thể được dùng để tái dựng `__filename` và `__dirname` dưới dạng các đường dẫn tuyệt đối:

```js
import { fileURLToPath } from 'url'
import { dirname } from 'path'
const __filename = fileURLToPath(import.meta.url)
const __dirname = dirname(__filename)
```

It is also possible to recreate the `require()` function as follows:

> Cũng có thể tái tạo hàm `require()` như sau:

```js
import { createRequire } from 'module'
const require = createRequire(import.meta.url)
```

Now we can use `require()` to import functionality coming from CommonJS modules in the context of ES modules.

> Bây giờ chúng ta có thể dùng `require()` để import chức năng đến từ các module CommonJS trong bối cảnh của các ES module.

Another interesting difference is the behavior of the `this` keyword.

> Một khác biệt thú vị khác là hành vi của từ khóa `this`.

In the global scope of an ES module, `this` is `undefined`, while in CommonJS, `this` is a reference to `exports`:

> Trong phạm vi toàn cục của một ES module, `this` là `undefined`, trong khi trong CommonJS, `this` là một tham chiếu đến `exports`:

```js
// this.js - ESM
console.log(this) // undefined
// this.cjs – CommonJS
console.log(this === exports) // true
```

### Interoperability — Khả năng tương tác (Interoperability)

We discussed in the previous section how to import CommonJS modules in ESM by using the `module.createRequire` function. It is also possible to import CommonJS modules from ESM by using the standard `import` syntax. This is only limited to default exports, though:

> Chúng ta đã bàn trong phần trước về cách import các module CommonJS trong ESM bằng cách dùng hàm `module.createRequire`. Cũng có thể import các module CommonJS từ ESM bằng cách dùng cú pháp `import` chuẩn. Tuy nhiên, điều này chỉ giới hạn ở các default export:

```js
import packageMain from 'commonjs-package' // Works
import { method } from 'commonjs-package'  // Errors
```

Unfortunately, it is not possible to import ES modules from CommonJS modules. Also, ESM cannot import JSON files directly as modules, a feature that is used quite frequently with CommonJS. The following `import` statement will fail:

> Đáng tiếc, không thể import các ES module từ các module CommonJS. Ngoài ra, ESM không thể import các file JSON trực tiếp dưới dạng module, một tính năng được dùng khá thường xuyên với CommonJS. Câu lệnh `import` sau sẽ thất bại:

```js
import data from './data.json'
```

It will produce a `TypeError` (`Unknown file extension: .json`).

> Nó sẽ tạo ra một `TypeError` (`Unknown file extension: .json`).

To overcome this limitation, we can use again the `module.createRequire` utility:

> Để vượt qua giới hạn này, chúng ta có thể dùng lại tiện ích `module.createRequire`:

```js
import { createRequire } from 'module'
const require = createRequire(import.meta.url)
const data = require('./data.json')
console.log(data)
```

There is ongoing work to support JSON modules natively even in ESM, so we may not need to rely on `createRequire()` in the near future for this functionality.

> Có công việc đang được tiến hành để hỗ trợ các JSON module một cách nguyên bản ngay cả trong ESM, nên chúng ta có thể không cần phải dựa vào `createRequire()` trong tương lai gần cho chức năng này.
