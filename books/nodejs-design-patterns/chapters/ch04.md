# Chapter 4: Asynchronous Control Flow Patterns with Callbacks — Chương 4: Các mẫu điều khiển luồng bất đồng bộ với Callback

Moving from a synchronous programming style to a platform such as Node.js, where continuation-passing style (CPS) and asynchronous APIs are the norm, can be frustrating. Asynchronous code can make it hard to predict the order in which statements are executed. Simple problems such as iterating over a set of files, executing tasks in sequence, or waiting for a set of operations to complete require the developer to take on new approaches and techniques just to avoid ending up writing inefficient and unreadable code. When using callbacks to deal with asynchronous **control flow**, the most common mistake is to fall into the trap of **callback** hell and see the code growing horizontally, rather than vertically, with a nesting that makes even simple routines hard to read and maintain.

> Chuyển từ phong cách lập trình đồng bộ sang một nền tảng như Node.js, nơi continuation-passing style (CPS) và các API bất đồng bộ là chuẩn mực, có thể khiến bạn nản lòng. Code bất đồng bộ có thể làm cho việc dự đoán thứ tự thực thi các câu lệnh trở nên khó khăn. Những bài toán đơn giản như duyệt qua một tập hợp các tệp, thực thi các tác vụ tuần tự, hay chờ một tập hợp các thao tác hoàn tất đều buộc lập trình viên phải tiếp nhận những hướng tiếp cận và kỹ thuật mới chỉ để tránh kết cục là viết ra code kém hiệu quả và khó đọc. Khi dùng callback để xử lý luồng điều khiển bất đồng bộ, sai lầm phổ biến nhất là rơi vào cái bẫy callback hell và chứng kiến code phình to theo chiều ngang thay vì chiều dọc, với mức độ lồng nhau khiến cho ngay cả những thủ tục đơn giản cũng trở nên khó đọc và khó bảo trì.

In this chapter, you will see how it's actually possible to tame callbacks and write clean, manageable asynchronous code by using some discipline and with the aid of some patterns. Knowing how to properly deal with callbacks will pave the way for adopting modern approaches such as promises and **async**/**await**.

> Trong chương này, bạn sẽ thấy thực ra hoàn toàn có thể thuần hóa callback và viết code bất đồng bộ sạch sẽ, dễ quản lý bằng cách áp dụng chút kỷ luật và sự hỗ trợ của một vài mẫu thiết kế. Biết cách xử lý callback đúng đắn sẽ mở đường cho việc tiếp nhận các hướng tiếp cận hiện đại như promise và async/await.

## The difficulties of asynchronous programming — Những khó khăn của lập trình bất đồng bộ

Losing control of asynchronous code in JavaScript is undoubtedly easy. Closures and in-place definitions of anonymous functions allow for a smooth programming experience that doesn't require the developer to jump to other points in the codebase. This is perfectly in line with the KISS principle (Keep It Simple, Stupid); it's simple, it keeps the code flowing, and we get it working in less time. Unfortunately, sacrificing qualities such as modularity, reusability, and maintainability will, sooner or later, lead to the uncontrolled proliferation of callback nesting, functions growing in size, and poor code organization. Most of the time, creating callbacks as in-place functions is not strictly required, so it's more a matter of discipline than a problem related to asynchronous programming. Recognizing that our code is becoming unwieldy or, even better, knowing in advance that it might become unwieldy and then acting accordingly with the most adequate solution, is what differentiates a novice from an expert.

> Đánh mất quyền kiểm soát code bất đồng bộ trong JavaScript chắc chắn là chuyện dễ xảy ra. Closure và việc định nghĩa tại chỗ các hàm vô danh mang lại trải nghiệm lập trình mượt mà, không buộc lập trình viên phải nhảy đến những điểm khác trong codebase. Điều này hoàn toàn phù hợp với nguyên tắc KISS (Keep It Simple, Stupid); nó đơn giản, giữ cho code trôi chảy, và ta hoàn thành việc trong thời gian ngắn hơn. Tiếc thay, việc hy sinh những phẩm chất như tính module hóa, khả năng tái sử dụng và khả năng bảo trì, sớm hay muộn, sẽ dẫn đến sự sinh sôi mất kiểm soát của các lớp callback lồng nhau, các hàm phình to về kích thước, và tổ chức code kém. Phần lớn thời gian, việc tạo callback dưới dạng hàm tại chỗ không thực sự bắt buộc, nên đây là vấn đề về kỷ luật hơn là vấn đề liên quan đến lập trình bất đồng bộ. Nhận ra rằng code của ta đang trở nên khó kiểm soát hoặc, tốt hơn nữa, biết trước rằng nó có thể trở nên khó kiểm soát rồi hành động phù hợp với giải pháp thích đáng nhất, chính là điều phân biệt người mới với chuyên gia.

To explain this problem, we will create a little web spider, a command-line application that takes in a web URL as input and downloads its contents locally into a file. In the code presented in this chapter, we are going to use a couple of npm dependencies:

- `mkdirp`: A small utility to create directories recursively (`nodejsdp.link/` `mkdirp`)

> - `superagent`: Một thư viện giúp tinh giản các lời gọi HTTP (`nodejsdp.link/superagent`)
> - `mkdirp`: Một tiện ích nhỏ để tạo thư mục theo kiểu đệ quy (`nodejsdp.link/` `mkdirp`)

Also, we will often refer to a local **module** named `./utils.js`, which contains some helpers that we will be using in our application. We will omit the contents of this file for brevity, but you can find the full implementation, along with a `**package**.json` file containing the full list of dependencies, in the official repository at `nodejsdp.link/` `repo`.

> Ngoài ra, chúng ta sẽ thường xuyên tham chiếu đến một module cục bộ tên là `./utils.js`, chứa một số hàm trợ giúp mà ta sẽ dùng trong ứng dụng. Chúng tôi sẽ lược bỏ nội dung của tệp này cho ngắn gọn, nhưng bạn có thể tìm thấy bản hiện thực đầy đủ, cùng với một tệp `package.json` chứa danh sách đầy đủ các phụ thuộc, trong kho chính thức tại `nodejsdp.link/` `repo`.

The core functionality of our application is contained inside a module named `spider.js`. Let's see how it looks. To start with, let's load all the dependencies that we are going to use:

> Chức năng cốt lõi của ứng dụng nằm trong một module tên là `spider.js`. Hãy xem nó trông như thế nào. Để bắt đầu, hãy nạp tất cả các phụ thuộc mà ta sẽ dùng:

```js
import fs from 'fs'
import path from 'path'
import superagent from 'superagent'
import mkdirp from 'mkdirp'
import { urlToFilename } from './utils.js'
```

Next, let's create a new function named `spider()`, which takes in the URL to download and a callback function that will be invoked when the download process completes:

> Tiếp theo, hãy tạo một hàm mới tên là `spider()`, nhận vào URL cần tải và một hàm callback sẽ được gọi khi quá trình tải hoàn tất:

```js
export function spider (url, cb) {
  const filename = urlToFilename(url)
  fs.access(filename, err => {                                // (1)
    if (err && err.code === 'ENOENT') {
      console.log(`Downloading ${url} into ${filename}`)
      superagent.get(url).end((err, res) => {                 // (2)
        if (err) {
          cb(err)
        } else {
          mkdirp(path.dirname(filename), err => {             // (3)
            if (err) {
              cb(err)
            } else {
              fs.writeFile(filename, res.text, err => {       // (4)
                if (err) {
                  cb(err)
                } else {
                  cb(null, filename, true)
                }
              })
            }
          })
        }
      })
    } else {
      cb(null, filename, false)
    }
  })
}
```

There is a lot going on here, so let's discuss in more detail what happens in every step:

> Có khá nhiều thứ diễn ra ở đây, vậy hãy thảo luận chi tiết hơn về điều xảy ra ở từng bước:

1. The code checks whether the URL was already downloaded by verifying that the corresponding file was not already created. If `err` is defined and has type `ENOENT`, then the file does not exist and it's safe to create it:

> 1. Code kiểm tra xem URL đã được tải về hay chưa bằng cách xác minh rằng tệp tương ứng chưa được tạo. Nếu `err` được định nghĩa và có kiểu `ENOENT`, thì tệp không tồn tại và việc tạo nó là an toàn:

```js
fs.access(filename, err => ...
```

2. If the file is not found, the URL is downloaded using the following line of code:

> 2. Nếu không tìm thấy tệp, URL được tải về bằng dòng code sau:

```js
superagent.get(url).end((err, res) => ...
```

3. Then, we make sure that the directory that will contain the file exists:

> 3. Sau đó, ta đảm bảo rằng thư mục sẽ chứa tệp đã tồn tại:

```js
mkdirp(path.dirname(filename), err => ...
```

4. Finally, we write the body of the HTTP response to the filesystem:

> 4. Cuối cùng, ta ghi phần thân của HTTP response vào hệ thống tệp:

```js
fs.writeFile(filename, res.text, err => ...
```

To complete our web spider application, we just need to invoke the `spider()` function by providing a URL as an input (in our case, we read it from the command-line arguments). The `spider()` function is exported from the file we defined previously. Let's now create a new file called `spider-cli.js` that can be directly invoked from the command line:

> Để hoàn thiện ứng dụng web spider, ta chỉ cần gọi hàm `spider()` bằng cách cung cấp một URL làm đầu vào (trong trường hợp của ta, ta đọc nó từ tham số dòng lệnh). Hàm `spider()` được export ra từ tệp mà ta đã định nghĩa trước đó. Bây giờ hãy tạo một tệp mới tên là `spider-cli.js` có thể được gọi trực tiếp từ dòng lệnh:

### Callback hell — Callback hell

Looking at the `spider()` function we defined earlier, you will likely notice that even though the algorithm we implemented is really straightforward, the resulting code has several levels of indentation and is very hard to read. Implementing a similar function with a direct style blocking **API** would be straightforward, and most likely, the code would be much more readable. However, using asynchronous CPS is another story, and making bad use of in-place callback definitions can lead to incredibly bad code.

> Nhìn vào hàm `spider()` mà ta đã định nghĩa trước đó, bạn có lẽ sẽ nhận thấy rằng mặc dù thuật toán ta hiện thực thực sự rất đơn giản, code thu được lại có nhiều mức thụt lề và rất khó đọc. Hiện thực một hàm tương tự bằng một API chặn theo phong cách trực tiếp sẽ đơn giản, và rất có thể code sẽ dễ đọc hơn nhiều. Tuy nhiên, dùng CPS bất đồng bộ lại là một câu chuyện khác, và việc lạm dụng việc định nghĩa callback tại chỗ có thể dẫn đến code tệ đến khó tin.

The situation where the abundance of closures and in-place callback definitions transforms the code into an unreadable and unmanageable blob is known as callback hell. It's one of the most widely recognized and severe anti-patterns in Node.js and JavaScript in general. The typical structure of code affected by this problem looks as follows:

> Tình huống mà sự lan tràn của các closure và việc định nghĩa callback tại chỗ biến code thành một khối khó đọc và không thể quản lý được gọi là callback hell. Đây là một trong những antipattern được công nhận rộng rãi và nghiêm trọng nhất trong Node.js và JavaScript nói chung. Cấu trúc điển hình của code mắc phải vấn đề này trông như sau:

```js
asyncFoo(err => {
  asyncBar(err => {
    asyncFooBar(err => {
      //...
    })
  })
})
```

You can see how code written in this way assumes the shape of a pyramid due to deep nesting, and that's why it is also colloquially known as the pyramid of doom.

> Bạn có thể thấy code viết theo cách này mang hình dạng của một kim tự tháp do lồng nhau quá sâu, và đó là lý do tại sao nó cũng được gọi một cách bông đùa là pyramid of doom (kim tự tháp diệt vong).

The most evident problem with code such as the preceding snippet is its poor readability. Due to the nesting being so deep, it's almost impossible to keep track of where a function ends and where another one begins. Another issue is caused by the overlapping of the variable names used in each **scope**. Often, we have to use similar or even identical names to describe the content of a variable. The best example is the error argument received by each callback. Some people often try to use variations of the same name to differentiate the object in each scope, for example, `err`, `error`, `err1`, `err2`, and so on. Others prefer to just shadow the variable defined in the upper scope by always using the same name, for example, `err`. Both alternatives are far from perfect, and cause confusion and increase the probability of introducing defects.

> Vấn đề rõ rệt nhất với code như đoạn trên là khả năng đọc kém. Do mức lồng nhau quá sâu, gần như không thể theo dõi được một hàm kết thúc ở đâu và một hàm khác bắt đầu ở đâu. Một vấn đề khác xuất phát từ sự chồng chéo của các tên biến được dùng trong mỗi phạm vi. Thường thì ta phải dùng những tên tương tự hoặc thậm chí giống hệt nhau để mô tả nội dung của một biến. Ví dụ điển hình nhất là tham số lỗi mà mỗi callback nhận được. Một số người thường cố dùng các biến thể của cùng một tên để phân biệt đối tượng trong từng phạm vi, ví dụ, `err`, `error`, `err1`, `err2`, v.v. Những người khác lại thích che (shadow) biến đã định nghĩa ở phạm vi trên bằng cách luôn dùng cùng một tên, ví dụ, `err`. Cả hai lựa chọn đều khác xa hoàn hảo, và gây nhầm lẫn cũng như làm tăng khả năng đưa vào lỗi.

Also, we have to keep in mind that closures come at a small price in terms of performance and memory consumption. In addition, they can create memory leaks that are not very easy to identify. In fact, we shouldn't forget that any context referenced by an active **closure** is retained from garbage collection.

> Ngoài ra, ta phải nhớ rằng closure có một cái giá nhỏ về mặt hiệu năng và mức tiêu thụ bộ nhớ. Thêm vào đó, chúng có thể tạo ra rò rỉ bộ nhớ không dễ nhận diện. Thực vậy, ta không nên quên rằng bất kỳ ngữ cảnh nào được tham chiếu bởi một closure đang hoạt động đều được giữ lại khỏi việc thu gom rác (garbage collection).

For a great introduction to how closures work in V8, you can refer to the following blog post by Vyacheslav Egorov, a software engineer at Google working on V8, which you can read at `nodejsdp.link/v8-closures`.

> Để có một phần giới thiệu tuyệt vời về cách closure hoạt động trong V8, bạn có thể tham khảo bài blog sau của Vyacheslav Egorov, một kỹ sư phần mềm tại Google làm việc trên V8, mà bạn có thể đọc tại `nodejsdp.link/v8-closures`.

If you look at our `spider()` function, you will notice that it clearly represents a callback hell situation and has all the problems just described. That's exactly what we are going to fix with the patterns and techniques that are covered in the following sections of this chapter.

> Nếu bạn nhìn vào hàm `spider()` của ta, bạn sẽ nhận thấy nó rõ ràng đại diện cho một tình huống callback hell và có tất cả các vấn đề vừa được mô tả. Đó chính xác là điều mà ta sẽ khắc phục bằng các mẫu và kỹ thuật được trình bày trong các phần tiếp theo của chương này.

## Callback best practices and control flow patterns — Best practice về callback và các mẫu điều khiển luồng

Now that you have met your first example of callback hell, you know what you should definitely avoid; however, that's not the only concern when writing asynchronous code. In fact, there are several situations where controlling the flow of a set of asynchronous tasks requires the use of specific patterns and techniques, especially if we are only using plain JavaScript without the aid of any external library. For example, iterating over a collection by applying an asynchronous operation in sequence is not as easy as invoking `forEach()` over an array; it actually requires a technique similar to recursion.

> Giờ đây khi bạn đã gặp ví dụ đầu tiên về callback hell, bạn biết điều mình chắc chắn nên tránh; tuy nhiên, đó không phải là mối lo duy nhất khi viết code bất đồng bộ. Thực vậy, có vài tình huống mà việc kiểm soát luồng của một tập hợp các tác vụ bất đồng bộ đòi hỏi việc dùng những mẫu và kỹ thuật cụ thể, đặc biệt nếu ta chỉ dùng JavaScript thuần mà không có sự hỗ trợ của bất kỳ thư viện ngoài nào. Ví dụ, duyệt qua một tập hợp bằng cách áp dụng một thao tác bất đồng bộ một cách tuần tự không dễ như gọi `forEach()` trên một mảng; nó thực sự đòi hỏi một kỹ thuật tương tự đệ quy.

In this section, you will learn not only about how to avoid callback hell, but also how to implement some of the most common control flow patterns, using only simple and plain JavaScript.

> Trong phần này, bạn sẽ học không chỉ về cách tránh callback hell, mà còn cả cách hiện thực một số mẫu điều khiển luồng phổ biến nhất, chỉ dùng JavaScript đơn giản và thuần túy.

### Callback discipline — Kỷ luật callback

When writing asynchronous code, the first rule to keep in mind is to not abuse in-place function definitions when defining callbacks. It can be tempting to do so, because it does not require any additional thinking for problems such as modularization and reusability; however, you have seen how this can have more disadvantages than advantages. Most of the time, fixing the callback hell problem does not require any libraries, fancy techniques, or changes of paradigm; you just need some common sense.

> Khi viết code bất đồng bộ, quy tắc đầu tiên cần ghi nhớ là không lạm dụng việc định nghĩa hàm tại chỗ khi định nghĩa callback. Việc này có thể rất hấp dẫn, bởi nó không đòi hỏi suy nghĩ thêm cho những vấn đề như module hóa và khả năng tái sử dụng; tuy nhiên, bạn đã thấy điều này có thể có nhiều bất lợi hơn lợi ích. Phần lớn thời gian, khắc phục vấn đề callback hell không đòi hỏi bất kỳ thư viện, kỹ thuật cầu kỳ, hay thay đổi mô thức nào; bạn chỉ cần một chút lẽ thường.

These are some basic principles that can help us keep the nesting level low and improve the organization of our code in general:

> Đây là một số nguyên tắc cơ bản có thể giúp ta giữ mức lồng nhau thấp và cải thiện việc tổ chức code nói chung:

- Exit as soon as possible. Use `return`, `continue`, or `break`, depending on the context, to immediately exit the current statement instead of writing (and nesting) complete `if...else` statements. This will help to keep our code shallow.
- Create named functions for callbacks, keeping them out of closures and passing intermediate results as arguments. Naming our functions will also make them look better in stack traces.
- Modularize the code. Split the code into smaller, reusable functions whenever possible.

> - Thoát ra càng sớm càng tốt. Dùng `return`, `continue`, hoặc `break`, tùy ngữ cảnh, để thoát ngay lập tức khỏi câu lệnh hiện tại thay vì viết (và lồng) trọn vẹn các câu lệnh `if...else`. Điều này sẽ giúp giữ cho code của ta nông.
> - Tạo các hàm có tên cho callback, giữ chúng nằm ngoài closure và truyền các kết quả trung gian dưới dạng tham số. Việc đặt tên cho hàm của ta cũng sẽ làm chúng trông đẹp hơn trong stack trace.
> - Module hóa code. Tách code thành các hàm nhỏ hơn, có thể tái sử dụng bất cứ khi nào có thể.

Now, let's put these principles into practice.

> Bây giờ, hãy đưa các nguyên tắc này vào thực hành.

### Applying the callback discipline — Áp dụng kỷ luật callback

To demonstrate the power of the ideas mentioned in the previous section, let's apply them to fix the callback hell in our web spider application.

> Để minh họa sức mạnh của những ý tưởng được nhắc đến trong phần trước, hãy áp dụng chúng để khắc phục callback hell trong ứng dụng web spider của ta.

For the first step, we can **refactor** our error-checking pattern by removing the `else` statement. This is made possible by returning from the function immediately after we receive an error. So, instead of having code such as the following:

> Ở bước đầu tiên, ta có thể tái cấu trúc mẫu kiểm tra lỗi bằng cách loại bỏ câu lệnh `else`. Điều này khả thi nhờ việc trả về (return) khỏi hàm ngay sau khi ta nhận được một lỗi. Vậy, thay vì có code như sau:

```js
if (err) {
  cb(err)
} else {
  // code to execute when there are no errors
}
```

We can improve the organization of our code by writing the following instead:

> Ta có thể cải thiện việc tổ chức code bằng cách viết như sau thay vào đó:

```js
if (err) {
  return cb(err)
}
// code to execute when there are no errors
```

This is often referred to as the early return principle. With this simple trick, we immediately have a reduction in the nesting level of our functions. It is easy and doesn't require any complex refactoring.

> Điều này thường được gọi là nguyên tắc early return (trả về sớm). Với mánh đơn giản này, ta lập tức giảm được mức lồng nhau của các hàm. Nó dễ và không đòi hỏi bất kỳ việc tái cấu trúc phức tạp nào.

A common mistake when executing the optimization just described is forgetting to terminate the function after the callback is invoked. For an error-handling scenario, the following code is a typical source of defects:

> Một sai lầm phổ biến khi thực hiện sự tối ưu vừa mô tả là quên kết thúc hàm sau khi callback được gọi. Đối với tình huống xử lý lỗi, đoạn code sau là một nguồn gây lỗi điển hình:

```js
if (err) {
  callback(err)
}
// code to execute when there are no errors.
```

We should never forget that the execution of our function will continue even after we invoke the callback. It is then important to insert a `return` instruction to block the execution of the rest of the function. Also, note that it doesn't really matter what value is returned by the function; the real result (or error) is produced asynchronously and passed to the callback. The return value of the asynchronous function is usually ignored. This property allows us to write shortcuts such as the following:

> Ta không bao giờ nên quên rằng việc thực thi hàm của ta sẽ tiếp tục ngay cả sau khi ta gọi callback. Do đó, quan trọng là phải chèn một chỉ thị `return` để chặn việc thực thi phần còn lại của hàm. Ngoài ra, hãy lưu ý rằng giá trị mà hàm trả về thực ra không quan trọng; kết quả thực sự (hoặc lỗi) được tạo ra một cách bất đồng bộ và truyền vào callback. Giá trị trả về của hàm bất đồng bộ thường bị bỏ qua. Tính chất này cho phép ta viết các lối tắt như sau:

```js
return callback(...)
```

Otherwise, we'd have to write slightly more verbose code, such as the following:

> Ngược lại, ta sẽ phải viết code hơi dài dòng hơn, như sau:

```js
callback(...)
return
```

As a second optimization for our `spider()` function, we can try to identify reusable pieces of code. For example, the functionality that writes a given string to a file can be easily factored out into a separate function, as follows:

> Là một tối ưu thứ hai cho hàm `spider()`, ta có thể thử nhận diện những đoạn code có thể tái sử dụng. Ví dụ, chức năng ghi một chuỗi cho trước vào tệp có thể dễ dàng được tách ra thành một hàm riêng, như sau:

```js
function saveFile (filename, contents, cb) {
  mkdirp(path.dirname(filename), err => {
    if (err) {
      return cb(err)
    }
    fs.writeFile(filename, contents, cb)
  })
}
```

Following the same principle, we can create a generic function named `download()` that takes a URL and a filename as input, and downloads the URL into the given file. Internally, we can use the `saveFile()` function we created earlier:

> Theo cùng nguyên tắc, ta có thể tạo một hàm tổng quát tên là `download()` nhận vào một URL và một tên tệp làm đầu vào, rồi tải URL về tệp đã cho. Bên trong, ta có thể dùng hàm `saveFile()` mà ta đã tạo trước đó:

```js
function download (url, filename, cb) {
  console.log(`Downloading ${url}`)
  superagent.get(url).end((err, res) => {
    if (err) {
      return cb(err)
    }
    saveFile(filename, res.text, err => {
      if (err) {
        return cb(err)
      }
      console.log(`Downloaded and saved: ${url}`)
      cb(null, res.text)
    })
  })
}
```

For the last step, we modify the `spider()` function, which, thanks to our changes, will now look like the following:

> Ở bước cuối cùng, ta sửa hàm `spider()`, mà nhờ những thay đổi của ta, giờ sẽ trông như sau:

```js
export function spider (url, cb) {
  const filename = urlToFilename(url)
  fs.access(filename, err => {
    if (!err || err.code !== 'ENOENT') {   // (1)
      return cb(null, filename, false)
    }
    download(url, filename, err => {
      if (err) {
        return cb(err)
      }
      cb(null, filename, true)
    })
  })
}
```

The functionality and the interface of the `spider()` function remained exactly the same; what changed was the way the code was organized. One important detail to notice (1) is that we inverted the check for the file's existence so that we could apply the early return principle discussed previously.

> Chức năng và giao diện của hàm `spider()` vẫn giữ y nguyên; điều thay đổi là cách code được tổ chức. Một chi tiết quan trọng cần lưu ý (1) là ta đã đảo ngược phép kiểm tra sự tồn tại của tệp để có thể áp dụng nguyên tắc early return đã thảo luận trước đó.

By applying the early return principle and the other callback discipline principles, we were able to drastically reduce the nesting of our code and, at the same time, increase its reusability and testability. In fact, we could think about exporting both `saveFile()` and `download()` so that we could reuse them in other modules. This would also allow us to test their functionality as independent units.

> Bằng cách áp dụng nguyên tắc early return và các nguyên tắc kỷ luật callback khác, ta đã có thể giảm đáng kể mức lồng nhau của code và, đồng thời, tăng khả năng tái sử dụng và khả năng kiểm thử của nó. Thực vậy, ta có thể nghĩ đến việc export cả `saveFile()` và `download()` để có thể tái sử dụng chúng trong các module khác. Điều này cũng sẽ cho phép ta kiểm thử chức năng của chúng như những đơn vị độc lập.

The refactoring we carried out in this section clearly demonstrates that most of the time, all we need is some discipline to make sure we do not abuse closures and anonymous functions. It works brilliantly, requires minimal effort, and it doesn't require external libraries.

> Việc tái cấu trúc mà ta thực hiện trong phần này cho thấy rõ rằng phần lớn thời gian, tất cả những gì ta cần là một chút kỷ luật để đảm bảo ta không lạm dụng closure và hàm vô danh. Nó hoạt động xuất sắc, đòi hỏi nỗ lực tối thiểu, và không cần đến thư viện ngoài.

Now that you know how to write clean asynchronous code using callbacks, we are ready to explore some of the most common asynchronous patterns, such as sequential and parallel execution.

> Giờ đây khi bạn đã biết cách viết code bất đồng bộ sạch sẽ bằng callback, ta đã sẵn sàng khám phá một số mẫu bất đồng bộ phổ biến nhất, như thực thi tuần tự và song song.

### Sequential execution — Thực thi tuần tự

In this section, we will look at asynchronous control flow patterns and start by analyzing the sequential execution flow.

> Trong phần này, ta sẽ xem xét các mẫu điều khiển luồng bất đồng bộ và bắt đầu bằng việc phân tích luồng thực thi tuần tự.

Executing a set of tasks in sequence means running them one at a time, one after the other. The order of execution matters and must be preserved, because the result of a task in the list may affect the execution of the next. Figure 4.1 illustrates this concept:

> Thực thi một tập hợp các tác vụ tuần tự nghĩa là chạy chúng từng cái một, cái này sau cái kia. Thứ tự thực thi rất quan trọng và phải được bảo toàn, bởi kết quả của một tác vụ trong danh sách có thể ảnh hưởng đến việc thực thi tác vụ kế tiếp. Hình 4.1 minh họa khái niệm này:

![](images/ch04-127_1.png)

Figure 4.1: An example of sequential execution flow with three tasks

> Hình 4.1: Một ví dụ về luồng thực thi tuần tự với ba tác vụ

There are different variations of this flow:

> Có những biến thể khác nhau của luồng này:

- Executing a set of known tasks in sequence, without propagating data across them.
- Using the output of a task as the input for the next (also known as chain, pipeline, or waterfall).
- Iterating over a collection while running an asynchronous task on each element, one after the other.

> - Thực thi một tập hợp các tác vụ đã biết một cách tuần tự, không truyền dữ liệu giữa chúng.
> - Dùng đầu ra của một tác vụ làm đầu vào cho tác vụ kế tiếp (còn gọi là chain, pipeline, hoặc waterfall).
> - Duyệt qua một tập hợp trong khi chạy một tác vụ bất đồng bộ trên mỗi phần tử, cái này sau cái kia.

Sequential execution, despite being trivial when implemented using a direct style blocking API, is usually the main cause of the callback hell problem when using asynchronous CPS.

> Thực thi tuần tự, dù tầm thường khi hiện thực bằng API chặn theo phong cách trực tiếp, thường lại là nguyên nhân chính của vấn đề callback hell khi dùng CPS bất đồng bộ.

#### Executing a known set of tasks in sequence — Thực thi một tập hợp tác vụ đã biết một cách tuần tự

We already looked at a sequential execution flow while implementing the `spider()` function in the previous section. By applying some simple rules, we were able to organize a set of known tasks in a sequential execution flow. Taking that code as a guideline, we can now generalize the solution with the following pattern:

> Ta đã từng xem xét một luồng thực thi tuần tự khi hiện thực hàm `spider()` trong phần trước. Bằng cách áp dụng một số quy tắc đơn giản, ta đã có thể tổ chức một tập hợp các tác vụ đã biết theo một luồng thực thi tuần tự. Lấy code đó làm kim chỉ nam, giờ ta có thể tổng quát hóa giải pháp bằng mẫu sau:

```js
function task1 (cb) {
  asyncOperation(() => {
    task2(cb)
  })
}
function task2 (cb) {
  asyncOperation(() => {
    task3(cb)
  })
}
function task3 (cb) {
  asyncOperation(() => {
    cb() // finally executes the callback
  })
}
task1(() => {
  // executed when task1, task2 and task3 are completed
  console.log('tasks 1, 2 and 3 executed')
})
```

The preceding pattern shows how each task invokes the next upon completion of a generic asynchronous operation. The pattern puts the emphasis on the modularization of tasks, showing how closures are not always necessary to handle asynchronous code.

> Mẫu trên cho thấy cách mỗi tác vụ gọi tác vụ kế tiếp khi một thao tác bất đồng bộ tổng quát hoàn tất. Mẫu này nhấn mạnh vào việc module hóa các tác vụ, cho thấy rằng closure không phải lúc nào cũng cần thiết để xử lý code bất đồng bộ.

#### Sequential iteration — Lặp tuần tự

The pattern described in the previous section works perfectly if we know in advance what and how many tasks are to be executed. This allows us to hardcode the invocation of the next task in the sequence, but what happens if we want to execute an asynchronous operation for each item in a collection? In cases such as this, we can't hardcode the task sequence anymore; instead, we have to build it dynamically.

> Mẫu được mô tả trong phần trước hoạt động hoàn hảo nếu ta biết trước những tác vụ nào và bao nhiêu tác vụ sẽ được thực thi. Điều này cho phép ta hardcode lời gọi tác vụ kế tiếp trong chuỗi, nhưng điều gì xảy ra nếu ta muốn thực thi một thao tác bất đồng bộ cho từng phần tử trong một tập hợp? Trong những trường hợp như vậy, ta không thể hardcode chuỗi tác vụ được nữa; thay vào đó, ta phải xây dựng nó một cách động.

##### Web spider version 2 — Web spider phiên bản 2

To show an example of sequential iteration, let's introduce a new feature to the web spider application. We now want to download all the links contained in a web page recursively. To do that, we are going to extract all the links from the page and then trigger our web spider on each recursively and in sequence.

> Để cho thấy một ví dụ về lặp tuần tự, hãy giới thiệu một tính năng mới cho ứng dụng web spider. Giờ ta muốn tải về một cách đệ quy tất cả các liên kết chứa trong một trang web. Để làm điều đó, ta sẽ trích xuất tất cả các liên kết từ trang rồi kích hoạt web spider trên mỗi liên kết một cách đệ quy và tuần tự.

The first step is modifying our `spider()` function so that it triggers a recursive download of all the links of a page by using a function named `spiderLinks()`, which we are going to create shortly.

> Bước đầu tiên là sửa hàm `spider()` để nó kích hoạt việc tải về đệ quy tất cả các liên kết của một trang bằng cách dùng một hàm tên là `spiderLinks()`, mà ta sẽ tạo ngay sau đây.

Also, instead of checking whether the file already exists, we will try to read it and start spidering its links. This way, we will be able to resume interrupted downloads. As a final change, we need to make sure we propagate a new parameter, `nesting`, which will help us to limit the recursion depth. The code is as follows:

> Ngoài ra, thay vì kiểm tra xem tệp đã tồn tại hay chưa, ta sẽ thử đọc nó và bắt đầu spider các liên kết của nó. Bằng cách này, ta sẽ có thể tiếp tục các lượt tải về bị gián đoạn. Là thay đổi cuối cùng, ta cần đảm bảo lan truyền một tham số mới, `nesting`, sẽ giúp ta giới hạn độ sâu đệ quy. Code như sau:

```js
export function spider (url, nesting, cb) {
  const filename = urlToFilename(url)
  fs.readFile(filename, 'utf8', (err, fileContent) => {
    if (err) {
      if (err.code !== 'ENOENT') {
        return cb(err)
      }
      // The file doesn't exist, so let's download it
      return download(url, filename, (err, requestContent) => {
        if (err) {
          return cb(err)
        }
        spiderLinks(url, requestContent, nesting, cb)
      })
    }
    // The file already exists, let's process the links
    spiderLinks(url, fileContent, nesting, cb)
  })
}
```

In the next section, we will explore how the `spiderLinks()` function can be implemented.

> Trong phần tiếp theo, ta sẽ khám phá cách hàm `spiderLinks()` có thể được hiện thực.

##### Sequential crawling of links — Thu thập (crawl) các liên kết một cách tuần tự

Now, we can create the core of this new version of our web spider application, the `spiderLinks()` function, which downloads all the links of an HTML page using a sequential asynchronous iteration algorithm. Pay attention to the way we are going to define that in the following code block:

> Giờ đây, ta có thể tạo phần cốt lõi của phiên bản mới này của ứng dụng web spider, hàm `spiderLinks()`, hàm này tải về tất cả các liên kết của một trang HTML bằng một thuật toán lặp bất đồng bộ tuần tự. Hãy chú ý cách ta sẽ định nghĩa điều đó trong khối code sau:

```js
function spiderLinks (currentUrl, body, nesting, cb) {
  if (nesting === 0) {
    // Remember Zalgo from chapter 3?
    return process.nextTick(cb)
  }
  const links = getPageLinks(currentUrl, body)            // (1)
  if (links.length === 0) {
    return process.nextTick(cb)
  }
  function iterate (index) {                              // (2)
    if (index === links.length) {
      return cb()
    }
    spider(links[index], nesting - 1, function (err) {    // (3)
      if (err) {
        return cb(err)
      }
      iterate(index + 1)
    })
  }
  iterate(0)                                              // (4)
}
```

The important steps to understand from this new function are as follows:

> Những bước quan trọng cần hiểu từ hàm mới này như sau:

1. We obtain the list of all the links contained in the page using the `getPageLinks()` function. This function returns only the links pointing to an internal destination (the same hostname).
2. We iterate over the links using a local function called `iterate()`, which takes the `index` of the next link to analyze. In this function, the first thing we do is check whether the `index` is equal to the length of the `links` array, in which case we immediately invoke the `cb()` function, as it means we have processed all the items.
3. At this point, everything should be ready for processing the link. We invoke the `spider()` function by decreasing the nesting level and invoking the next step of the iteration when the operation completes.
4. As the last step in the `spiderLinks()` function, we bootstrap the iteration by invoking `iterate(0)`.

> 1. Ta lấy danh sách tất cả các liên kết chứa trong trang bằng hàm `getPageLinks()`. Hàm này chỉ trả về các liên kết trỏ đến một đích nội bộ (cùng hostname).
> 2. Ta duyệt qua các liên kết bằng một hàm cục bộ tên là `iterate()`, hàm này nhận vào `index` của liên kết kế tiếp cần phân tích. Trong hàm này, điều đầu tiên ta làm là kiểm tra xem `index` có bằng độ dài của mảng `links` hay không, trong trường hợp đó ta lập tức gọi hàm `cb()`, vì điều đó nghĩa là ta đã xử lý xong tất cả các phần tử.
> 3. Tại điểm này, mọi thứ nên đã sẵn sàng để xử lý liên kết. Ta gọi hàm `spider()` bằng cách giảm mức nesting và gọi bước kế tiếp của vòng lặp khi thao tác hoàn tất.
> 4. Là bước cuối cùng trong hàm `spiderLinks()`, ta khởi động vòng lặp bằng cách gọi `iterate(0)`.

The algorithm that was just presented allows us to iterate over an array by executing an asynchronous operation in sequence, which in our case is the `spider()` function.

> Thuật toán vừa được trình bày cho phép ta duyệt qua một mảng bằng cách thực thi một thao tác bất đồng bộ một cách tuần tự, mà trong trường hợp của ta là hàm `spider()`.

Finally, we can change our `spider-cli.js` a bit so that we can specify the nesting level as an additional command-line interface (CLI) argument:

> Cuối cùng, ta có thể thay đổi `spider-cli.js` một chút để có thể chỉ định mức nesting làm một tham số dòng lệnh (CLI) bổ sung:

```js
import { spider } from './spider.js'
const url = process.argv[2]
const nesting = Number.parseInt(process.argv[3], 10) || 1
spider(url, nesting, err => {
  if (err) {
    console.error(err)
    process.exit(1)
  }
  console.log('Download complete')
})
```

To interrupt the process, which can take a while if there are many links, remember that we can always use Ctrl + C. If we then decide to resume it, we can do so by launching the spider application and providing the same URL we used for the first run. For example, do not set a high nesting level or leave the spider running for more than a few seconds. It is not polite to overload a server with thousands of requests. In some circumstances, this can also be considered illegal. Spider responsibly!

> Giờ ta có thể thử phiên bản mới này của ứng dụng spider và xem nó tải về tất cả các liên kết của một trang web một cách đệ quy, cái này sau cái kia. Để gián đoạn quá trình, vốn có thể mất một lúc nếu có nhiều liên kết, hãy nhớ rằng ta luôn có thể dùng Ctrl + C. Nếu sau đó ta quyết định tiếp tục, ta có thể làm vậy bằng cách khởi chạy ứng dụng spider và cung cấp cùng URL mà ta đã dùng cho lần chạy đầu tiên. Giờ đây khi ứng dụng web spider của ta có khả năng kích hoạt việc tải về cả một website, hãy cân nhắc dùng nó một cách cẩn thận. Ví dụ, đừng đặt mức nesting cao hay để spider chạy quá vài giây. Việc nhồi nhét máy chủ bằng hàng nghìn yêu cầu là không lịch sự. Trong một số hoàn cảnh, điều này còn có thể bị xem là bất hợp pháp. Hãy spider một cách có trách nhiệm!

##### The pattern — Mẫu

The code of the `spiderLinks()` function from the previous section is a clear example of how it's possible to iterate over a collection while applying an asynchronous operation. You may also notice that it's a pattern that can be adapted to any other situation where we need to iterate asynchronously over the elements of a collection or, in general, over a list of tasks. This pattern can be generalized as follows:

> Code của hàm `spiderLinks()` ở phần trước là một ví dụ rõ ràng về cách có thể duyệt qua một tập hợp trong khi áp dụng một thao tác bất đồng bộ. Bạn cũng có thể nhận thấy đó là một mẫu có thể được điều chỉnh cho bất kỳ tình huống nào khác mà ta cần duyệt bất đồng bộ qua các phần tử của một tập hợp hoặc, nói chung, qua một danh sách các tác vụ. Mẫu này có thể được tổng quát hóa như sau:

```js
function iterate (index) {
  if (index === tasks.length) {
    return finish()
  }
  const task = tasks[index]
  task(() => iterate(index + 1))
}
function finish () {
  // iteration completed
}
iterate(0)
```

It's important to notice that these types of algorithms become really recursive if `task()` is a synchronous operation. In such a case, the stack will not unwind at every cycle and there might be a risk of hitting the maximum call stack size limit.

> Quan trọng là phải lưu ý rằng những loại thuật toán này trở nên thực sự đệ quy nếu `task()` là một thao tác đồng bộ. Trong trường hợp đó, stack sẽ không được gỡ ra ở mỗi chu kỳ và có thể có nguy cơ chạm tới giới hạn kích thước call stack tối đa.

The pattern that was just presented is very powerful and can be extended or adapted to address several common needs. Just to mention some examples:

> Mẫu vừa được trình bày rất mạnh mẽ và có thể được mở rộng hoặc điều chỉnh để giải quyết vài nhu cầu phổ biến. Chỉ xin nêu một vài ví dụ:

- We can map the values of an array asynchronously.
- We can pass the results of an operation to the next one in the iteration to implement an asynchronous version of the `reduce` algorithm.
- We can quit the loop prematurely if a particular condition is met (asynchronous implementation of the `Array.some()` helper).
- We can even iterate over an infinite number of elements.

> - Ta có thể map các giá trị của một mảng một cách bất đồng bộ.
> - Ta có thể truyền kết quả của một thao tác sang thao tác kế tiếp trong vòng lặp để hiện thực một phiên bản bất đồng bộ của thuật toán `reduce`.
> - Ta có thể thoát khỏi vòng lặp sớm nếu một điều kiện cụ thể được thỏa mãn (bản hiện thực bất đồng bộ của hàm trợ giúp `Array.some()`).
> - Ta thậm chí có thể duyệt qua một số phần tử vô hạn.

We could also choose to generalize the solution even further by wrapping it in a function with a signature such as the following:

> Ta cũng có thể chọn tổng quát hóa giải pháp thêm nữa bằng cách bọc nó trong một hàm với chữ ký như sau:

```js
iterateSeries(collection, iteratorCallback, finalCallback)
```

Here, `collection` is the actual dataset you want to iterate over, `iteratorCallback` is the function to execute over every item, and `finalCallback` is the function that gets executed when all the items are processed or in case of an error. The implementation of this helper function is left to you as an exercise.

> Ở đây, `collection` là tập dữ liệu thực sự bạn muốn duyệt qua, `iteratorCallback` là hàm cần thực thi trên mỗi phần tử, và `finalCallback` là hàm được thực thi khi tất cả các phần tử đã được xử lý hoặc trong trường hợp có lỗi. Việc hiện thực hàm trợ giúp này được để lại cho bạn như một bài tập.

The Sequential **Iterator** pattern

> Mẫu Sequential Iterator

Execute a list of tasks in sequence by creating a function named `iterator`, which invokes the next available task in the collection and makes sure to invoke the next step of the iteration when the current task completes.

> Thực thi một danh sách các tác vụ tuần tự bằng cách tạo một hàm tên là `iterator`, hàm này gọi tác vụ khả dụng kế tiếp trong tập hợp và đảm bảo gọi bước kế tiếp của vòng lặp khi tác vụ hiện tại hoàn tất.

In the next section, we will explore the parallel execution pattern, which is more convenient when the order of the various tasks is not important.

> Trong phần tiếp theo, ta sẽ khám phá mẫu thực thi song song, thuận tiện hơn khi thứ tự của các tác vụ khác nhau không quan trọng.

### Parallel execution — Thực thi song song

There are some situations where the order of execution of a set of asynchronous tasks is not important, and all we want is to be notified when all those running tasks are completed. Such situations are better handled using a parallel execution flow, as shown in Figure 4.2:

> Có một số tình huống mà thứ tự thực thi của một tập hợp các tác vụ bất đồng bộ không quan trọng, và tất cả những gì ta muốn là được thông báo khi tất cả các tác vụ đang chạy đó hoàn tất. Những tình huống như vậy được xử lý tốt hơn bằng một luồng thực thi song song, như minh họa ở Hình 4.2:

![](images/ch04-133_1.png)

Figure 4.2: An example of parallel execution with three tasks This may sound strange if you consider that Node.js is **single-threaded**, but if you remember what we discussed in Chapter 1, The Node.js Platform, you'll realize that even though we have just one **thread**, we can still achieve **concurrency**, thanks to the non-blocking nature of Node.js. In fact, the word parallel is used improperly in this case, as it does not mean that the tasks run simultaneously, but rather that their execution is carried out by an underlying, non-blocking API and interleaved by the **event loop**.

> Hình 4.2: Một ví dụ về thực thi song song với ba tác vụ. Điều này nghe có vẻ lạ nếu bạn xét rằng Node.js là đơn luồng (single-threaded), nhưng nếu bạn nhớ những gì ta đã thảo luận ở Chương 1, Nền tảng Node.js, bạn sẽ nhận ra rằng dù ta chỉ có một luồng, ta vẫn có thể đạt được tính đồng thời (concurrency), nhờ bản chất không chặn của Node.js. Thực vậy, từ "song song" được dùng không thật chính xác trong trường hợp này, vì nó không có nghĩa là các tác vụ chạy đồng thời, mà đúng hơn là việc thực thi chúng được tiến hành bởi một API không chặn ngầm bên dưới và được xen kẽ bởi vòng lặp sự kiện (event loop).

As you know, a task gives control back to the event loop when it requests a new asynchronous operation, allowing the event loop to execute another task. The proper word to use for this kind of flow is concurrency, but we will still use parallel for simplicity.

> Như bạn đã biết, một tác vụ trả quyền điều khiển lại cho vòng lặp sự kiện khi nó yêu cầu một thao tác bất đồng bộ mới, cho phép vòng lặp sự kiện thực thi một tác vụ khác. Từ đúng để dùng cho loại luồng này là tính đồng thời (concurrency), nhưng ta vẫn sẽ dùng "song song" cho đơn giản.

The following diagram shows how two asynchronous tasks can run in parallel in a Node.js program:

> Sơ đồ sau cho thấy cách hai tác vụ bất đồng bộ có thể chạy song song trong một chương trình Node.js:

![](images/ch04-134_1.jpg)

Figure 4.3: An example of how asynchronous tasks run in parallel

> Hình 4.3: Một ví dụ về cách các tác vụ bất đồng bộ chạy song song

In Figure 4.3, we have a Main function that executes two asynchronous tasks:

> Trong Hình 4.3, ta có một hàm Main thực thi hai tác vụ bất đồng bộ:

1. The Main function triggers the execution of Task 1 and Task 2. As they trigger an asynchronous operation, they immediately return control back to the Main function, which then returns it to the event loop.
2. When the asynchronous operation of Task 1 is completed, the event loop gives control to it. When Task 1 completes its internal synchronous processing as well, it notifies the Main function.
3. When the asynchronous operation triggered by Task 2 is complete, the event loop invokes its callback, giving control back to Task 2. At the end of Task 2, the Main function is notified once more. At this point, the Main function knows that both Task 1 and Task 2 are complete, so it can continue its execution or return the results of the operations to another callback.

> 1. Hàm Main kích hoạt việc thực thi Task 1 và Task 2. Vì chúng kích hoạt một thao tác bất đồng bộ, chúng lập tức trả quyền điều khiển lại cho hàm Main, hàm này sau đó trả nó lại cho vòng lặp sự kiện.
> 2. Khi thao tác bất đồng bộ của Task 1 hoàn tất, vòng lặp sự kiện trao quyền điều khiển cho nó. Khi Task 1 cũng hoàn tất việc xử lý đồng bộ nội bộ của mình, nó thông báo cho hàm Main.
> 3. Khi thao tác bất đồng bộ được kích hoạt bởi Task 2 hoàn tất, vòng lặp sự kiện gọi callback của nó, trả quyền điều khiển lại cho Task 2. Ở cuối Task 2, hàm Main lại được thông báo một lần nữa. Tại điểm này, hàm Main biết rằng cả Task 1 và Task 2 đều đã hoàn tất, nên nó có thể tiếp tục việc thực thi của mình hoặc trả kết quả của các thao tác cho một callback khác.

In short, this means that in Node.js, we can only execute asynchronous operations in parallel, because their concurrency is handled internally by the non-blocking APIs. In Node.js, synchronous (blocking) operations can't run concurrently unless their execution is interleaved with an asynchronous operation, or interleaved with `setTimeout()` or `setImmediate()`. You will see this in more detail in Chapter 11, Advanced Recipes.

> Tóm lại, điều này có nghĩa là trong Node.js, ta chỉ có thể thực thi song song các thao tác bất đồng bộ, bởi tính đồng thời của chúng được xử lý nội bộ bởi các API không chặn. Trong Node.js, các thao tác đồng bộ (chặn) không thể chạy đồng thời trừ khi việc thực thi của chúng được xen kẽ với một thao tác bất đồng bộ, hoặc xen kẽ với `setTimeout()` hay `setImmediate()`. Bạn sẽ thấy điều này chi tiết hơn ở Chương 11, Các công thức nâng cao.

#### Web spider version 3 — Web spider phiên bản 3

Our web spider application seems like a perfect candidate to apply the concept of parallel execution. So far, our application is executing the recursive download of the linked pages in a sequential fashion. We can easily improve the performance of this process by downloading all the linked pages in parallel.

> Ứng dụng web spider của ta có vẻ là một ứng viên hoàn hảo để áp dụng khái niệm thực thi song song. Cho đến nay, ứng dụng của ta đang thực hiện việc tải về đệ quy các trang được liên kết theo kiểu tuần tự. Ta có thể dễ dàng cải thiện hiệu năng của quá trình này bằng cách tải về tất cả các trang được liên kết song song.

To do that, we just need to modify the `spiderLinks()` function to make sure we spawn all the `spider()` tasks at once, and then invoke the final callback only when all of them have completed their execution. So, let's modify our `spiderLinks()` function as follows:

> Để làm điều đó, ta chỉ cần sửa hàm `spiderLinks()` để đảm bảo ta sinh ra tất cả các tác vụ `spider()` cùng một lúc, rồi chỉ gọi callback cuối cùng khi tất cả chúng đã hoàn tất việc thực thi. Vậy, hãy sửa hàm `spiderLinks()` của ta như sau:

```js
function spiderLinks (currentUrl, body, nesting, cb) {
  if (nesting === 0) {
    return process.nextTick(cb)
  }
  const links = getPageLinks(currentUrl, body)
  if (links.length === 0) {
    return process.nextTick(cb)
  }
  let completed = 0
  let hasErrors = false
  function done (err) {
    if (err) {
      hasErrors = true
      return cb(err)
    }
    if (++completed === links.length && !hasErrors) {
      return cb()
    }
  }
  links.forEach(link => spider(link, nesting - 1, done))
}
```

Let's discuss what we changed. As mentioned earlier, the `spider()` tasks are now started all at once. This is possible by simply iterating over the `links` array and starting each task without waiting for the previous one to finish:

> Hãy thảo luận về điều ta đã thay đổi. Như đã nhắc đến trước đó, các tác vụ `spider()` giờ được khởi động cùng một lúc. Điều này khả thi bằng cách đơn giản duyệt qua mảng `links` và khởi động mỗi tác vụ mà không chờ tác vụ trước hoàn tất:

```js
links.forEach(link => spider(link, nesting - 1, done))
```

Then, the trick to make our application wait for all the tasks to complete is to provide the `spider()` function with a special callback, which we call `done()`. The `done()` function increases a counter when a `spider` task completes. When the number of completed downloads reaches the size of the `links` array, the final callback is invoked:

> Sau đó, mánh để khiến ứng dụng của ta chờ tất cả các tác vụ hoàn tất là cung cấp cho hàm `spider()` một callback đặc biệt, mà ta gọi là `done()`. Hàm `done()` tăng một bộ đếm khi một tác vụ `spider` hoàn tất. Khi số lượt tải về đã hoàn tất đạt tới kích thước của mảng `links`, callback cuối cùng được gọi:

```js
function done (err) {
  if (err) {
    hasErrors = true
    return cb(err)
  }
  if (++completed === links.length && !hasErrors) {
    return cb()
  }
}
```

The `hasErrors` variable is necessary because if one parallel task fails, we want to immediately call the callback with the given error. Also, we need to make sure that other parallel tasks that might still be running won't invoke the callback again.

> Biến `hasErrors` là cần thiết bởi nếu một tác vụ song song thất bại, ta muốn gọi callback ngay lập tức với lỗi đã cho. Ngoài ra, ta cần đảm bảo rằng các tác vụ song song khác có thể vẫn đang chạy sẽ không gọi callback thêm lần nữa.

With these changes in place, if we now try to run our spider against a web page, we will notice a huge improvement in the speed of the overall process, as every download will be carried out in parallel, without waiting for the previous link to be processed.

> Với những thay đổi này được áp dụng, nếu giờ ta thử chạy spider trên một trang web, ta sẽ nhận thấy sự cải thiện to lớn về tốc độ của toàn bộ quá trình, vì mỗi lượt tải về sẽ được tiến hành song song, không chờ liên kết trước được xử lý.

#### The pattern — Mẫu

Finally, we can extract our nice little pattern for the parallel execution flow. Let's represent a generic version of the pattern with the following code:

> Cuối cùng, ta có thể trích ra mẫu nhỏ gọn gàng của mình cho luồng thực thi song song. Hãy biểu diễn một phiên bản tổng quát của mẫu bằng code sau:

```js
const tasks = [ /* ... */ ]
let completed = 0
tasks.forEach(task => {
  task(() => {
    if (++completed === tasks.length) {
      finish()
    }
  })
})
function finish () {
  // all the tasks completed
}
```

With small modifications, we can adapt the pattern to accumulate the results of each task into a collection, to filter or map the elements of an array, or to invoke the `finish()` callback as soon as one or a given number of tasks complete (this last situation in particular is called competitive race).

> Với những sửa đổi nhỏ, ta có thể điều chỉnh mẫu để tích lũy kết quả của mỗi tác vụ vào một tập hợp, để lọc hoặc map các phần tử của một mảng, hoặc để gọi callback `finish()` ngay khi một hoặc một số tác vụ nhất định hoàn tất (tình huống cuối cùng này nói riêng được gọi là competitive race).

The Unlimited Parallel Execution pattern

> Mẫu Unlimited Parallel Execution

Run a set of asynchronous tasks in parallel by launching them all at once, and then wait for all of them to complete by counting the number of times their callbacks are invoked.

> Chạy một tập hợp các tác vụ bất đồng bộ song song bằng cách khởi chạy tất cả chúng cùng một lúc, rồi chờ tất cả chúng hoàn tất bằng cách đếm số lần callback của chúng được gọi.

When we have multiple tasks running in parallel, we might have race conditions, that is, contention to access external resources (for example, files or records in a database). In the next section, we will talk about race conditions in Node.js and explore some techniques to identify and address them.

> Khi ta có nhiều tác vụ chạy song song, ta có thể gặp race condition, tức là sự tranh chấp để truy cập các tài nguyên bên ngoài (ví dụ, các tệp hoặc bản ghi trong một cơ sở dữ liệu). Trong phần tiếp theo, ta sẽ nói về race condition trong Node.js và khám phá một số kỹ thuật để nhận diện và xử lý chúng.

#### Fixing race conditions with concurrent tasks — Khắc phục race condition với các tác vụ đồng thời

Running a set of tasks in parallel can cause issues when using **blocking I**/O in combination with multiple threads. However, you have just seen that, in Node.js, this is a totally different story. Running multiple asynchronous tasks in parallel is, in fact, straightforward and cheap in terms of resources. This is one of the most important strengths of Node.js, because it makes parallelization a common practice rather than a complex technique to only use when strictly necessary.

> Chạy một tập hợp các tác vụ song song có thể gây ra vấn đề khi dùng I/O chặn (blocking I/O) kết hợp với nhiều luồng. Tuy nhiên, bạn vừa thấy rằng, trong Node.js, đây là một câu chuyện hoàn toàn khác. Chạy nhiều tác vụ bất đồng bộ song song thực ra rất đơn giản và rẻ về mặt tài nguyên. Đây là một trong những thế mạnh quan trọng nhất của Node.js, bởi nó khiến việc song song hóa trở thành một thực hành thường ngày hơn là một kỹ thuật phức tạp chỉ dùng khi thực sự cần thiết.

Another important characteristic of the concurrency model of Node.js is the way we deal with task synchronization and race conditions. In multithreaded programming, this is usually done using constructs such as locks, mutexes, semaphores, and monitors, and it can be one of the most complex aspects of parallelization, and has a considerable impact on performance. In Node.js, we usually don't need a fancy synchronization mechanism, as everything runs on a single thread. However, this doesn't mean that we can't have race conditions; on the contrary, they can be quite common. The root of the problem is the delay between the invocation of an asynchronous operation and the notification of its result.

> Một đặc trưng quan trọng khác của mô hình tính đồng thời của Node.js là cách ta xử lý sự đồng bộ hóa tác vụ và race condition. Trong lập trình đa luồng, việc này thường được làm bằng các kiến trúc như lock, mutex, semaphore, và monitor, và nó có thể là một trong những khía cạnh phức tạp nhất của việc song song hóa, đồng thời có tác động đáng kể đến hiệu năng. Trong Node.js, ta thường không cần một cơ chế đồng bộ hóa cầu kỳ, vì mọi thứ chạy trên một luồng đơn. Tuy nhiên, điều này không có nghĩa là ta không thể có race condition; trái lại, chúng có thể khá phổ biến. Gốc rễ của vấn đề là độ trễ giữa lúc gọi một thao tác bất đồng bộ và lúc thông báo kết quả của nó.

To see a concrete example, we will refer again to our web spider application, and in particular, the last version we created, which actually contains a **race condition** (can you spot it?). The problem we are talking about lies in the `spider()` function, where we check whether a file already exists before we start to download the corresponding URL:

> Để thấy một ví dụ cụ thể, ta sẽ lại tham chiếu đến ứng dụng web spider của mình, và cụ thể là phiên bản cuối cùng ta đã tạo, mà thực ra chứa một race condition (bạn có nhận ra nó không?). Vấn đề ta đang nói đến nằm trong hàm `spider()`, nơi ta kiểm tra xem một tệp đã tồn tại hay chưa trước khi ta bắt đầu tải về URL tương ứng:

    if (err) {

Nếu bạn đang xây dựng một spider có thể phải tải về hàng trăm nghìn trang web, việc xóa `url` đã tải về khỏi tập một khi tệp tương ứng được tải về sẽ giúp bạn giữ cho số phần tử (cardinality) của tập, và do đó mức tiêu thụ bộ nhớ, khỏi tăng vô hạn.

Race conditions can cause many problems, even if we are in a single-threaded environment. In some circumstances, they can lead to data corruption and are usually very hard to debug because of their ephemeral nature. So, it's always good practice to double-check for these types of situations when running tasks in parallel.

> Race condition có thể gây ra nhiều vấn đề, ngay cả khi ta ở trong một môi trường đơn luồng. Trong một số hoàn cảnh, chúng có thể dẫn đến hỏng dữ liệu và thường rất khó debug do bản chất phù du của chúng. Vậy nên, luôn là thực hành tốt khi kiểm tra kỹ những loại tình huống này khi chạy các tác vụ song song.

Also, running an arbitrary number of parallel tasks can be a dangerous practice. In the next section, you will discover why it can be a problem and how to keep the number of parallel tasks under control.

> Ngoài ra, chạy một số lượng tùy ý các tác vụ song song có thể là một thực hành nguy hiểm. Trong phần tiếp theo, bạn sẽ khám phá tại sao nó có thể là một vấn đề và làm thế nào để giữ số lượng tác vụ song song trong tầm kiểm soát.

### Limited parallel execution — Thực thi song song có giới hạn

Spawning parallel tasks without control can often lead to excessive load. Imagine having thousands of files to read, URLs to access, or database queries to run in parallel. A common problem in such situations is running out of resources. The most common example is when an application tries to open too many files at once, utilizing all the file descriptors available to the process. A server that spawns unbounded parallel tasks to handle a user **request** could be exploited with a denial-of-service (DoS) attack. That is when a malicious actor can forge one or more requests to push the server to consume all the available resources and become unresponsive. Limiting the number of parallel tasks is, in general, a good practice that helps with building resilient applications.

> Sinh ra các tác vụ song song không kiểm soát thường có thể dẫn đến tải quá mức. Hãy tưởng tượng có hàng nghìn tệp cần đọc, URL cần truy cập, hoặc truy vấn cơ sở dữ liệu cần chạy song song. Một vấn đề phổ biến trong những tình huống như vậy là cạn kiệt tài nguyên. Ví dụ phổ biến nhất là khi một ứng dụng cố mở quá nhiều tệp cùng một lúc, dùng hết tất cả các file descriptor khả dụng cho tiến trình. Một máy chủ sinh ra các tác vụ song song không giới hạn để xử lý một yêu cầu của người dùng có thể bị khai thác bằng một cuộc tấn công từ chối dịch vụ (DoS). Đó là khi một tác nhân xấu có thể giả mạo một hoặc nhiều yêu cầu để đẩy máy chủ tới mức tiêu thụ hết tất cả các tài nguyên khả dụng và trở nên mất phản hồi. Giới hạn số lượng tác vụ song song, nói chung, là một thực hành tốt giúp xây dựng các ứng dụng có khả năng chống chịu.

Version 3 of our web spider does not limit the number of parallel tasks and therefore, it is susceptible to crashing in a number of cases. For instance, if we try to run it against a significantly big website, we might see it running for a few seconds and then failing with the error code `ECONNREFUSED`. When we are downloading too many pages concurrently from a web server, the server might decide to start rejecting new connections from the same IP. In this case, our spider would just crash and we would be forced to relaunch the process if we wanted to continue crawling the website. We could just handle `ECONNREFUSED` to stop the process from crashing, but we would still be risking allocating too many parallel tasks and might run into other issues.

> Phiên bản 3 của web spider của ta không giới hạn số lượng tác vụ song song và do đó, nó dễ bị crash trong một số trường hợp. Chẳng hạn, nếu ta thử chạy nó trên một website lớn đáng kể, ta có thể thấy nó chạy được vài giây rồi thất bại với mã lỗi `ECONNREFUSED`. Khi ta đang tải về quá nhiều trang đồng thời từ một máy chủ web, máy chủ có thể quyết định bắt đầu từ chối các kết nối mới từ cùng một IP. Trong trường hợp này, spider của ta sẽ chỉ crash và ta buộc phải khởi chạy lại tiến trình nếu ta muốn tiếp tục thu thập website. Ta có thể chỉ cần xử lý `ECONNREFUSED` để ngăn tiến trình khỏi crash, nhưng ta vẫn sẽ mạo hiểm cấp phát quá nhiều tác vụ song song và có thể gặp các vấn đề khác.

In this section, you will see how we can make our spider more resilient by keeping the concurrency limited.

> Trong phần này, bạn sẽ thấy cách ta có thể làm cho spider của mình có khả năng chống chịu tốt hơn bằng cách giữ tính đồng thời ở mức giới hạn.

The following diagram shows a situation where we have five tasks that run in parallel with a concurrency limit of two:

> Sơ đồ sau cho thấy một tình huống mà ta có năm tác vụ chạy song song với giới hạn đồng thời là hai:

![](images/ch04-140_1.jpg)

Figure 4.5: An example of how concurrency can be limited to a maximum of two parallel tasks

> Hình 4.5: Một ví dụ về cách tính đồng thời có thể được giới hạn ở mức tối đa hai tác vụ song song

From Figure 4.5, it should be clear how our algorithm works:

> Từ Hình 4.5, hẳn đã rõ thuật toán của ta hoạt động như thế nào:

1. Initially, we spawn as many tasks as we can without exceeding the concurrency limit.
2. Then, every time a task is completed, we spawn one or more tasks until we reach the limit again.

> 1. Ban đầu, ta sinh ra càng nhiều tác vụ càng tốt mà không vượt quá giới hạn đồng thời.
> 2. Sau đó, mỗi khi một tác vụ hoàn tất, ta sinh ra một hoặc nhiều tác vụ cho đến khi ta lại đạt tới giới hạn.

In the next section, we will explore a possible implementation of the limited parallel execution pattern.

> Trong phần tiếp theo, ta sẽ khám phá một bản hiện thực khả dĩ của mẫu thực thi song song có giới hạn.

#### Limiting concurrency — Giới hạn tính đồng thời

We will now look at a pattern that will execute a set of given tasks in parallel with limited concurrency:

> Giờ ta sẽ xem một mẫu thực thi một tập hợp các tác vụ cho trước song song với tính đồng thời giới hạn:

```js
const tasks = [
  // ...
]
const concurrency = 2
let running = 0
let completed = 0
let index = 0
function next () {                                          // (1)
  while (running < concurrency && index < tasks.length) {
    const task = tasks[index++]
    task(() => {                                            // (2)
      if (++completed === tasks.length) {
        return finish()
      }
      running--
      next()
    })
    running++
  }
}
next()
function finish() {
  // all tasks finished
}
```

This algorithm can be considered a mixture of sequential execution and parallel execution. In fact, you might notice similarities with both patterns:

> Thuật toán này có thể được xem là sự pha trộn giữa thực thi tuần tự và thực thi song song. Thực vậy, bạn có thể nhận thấy những điểm tương đồng với cả hai mẫu:

1. We have an iterator function, which we call `next()`, and then an inner loop that spawns as many tasks as possible in parallel while staying within the concurrency limit.
2. The next important part is the callback we pass to each task, which checks whether we completed all the tasks in the list. If there are still tasks to run, it invokes `next()` to spawn another set of tasks.

> 1. Ta có một hàm iterator, mà ta gọi là `next()`, rồi một vòng lặp bên trong sinh ra càng nhiều tác vụ càng tốt song song trong khi vẫn nằm trong giới hạn đồng thời.
> 2. Phần quan trọng kế tiếp là callback ta truyền cho mỗi tác vụ, callback này kiểm tra xem ta đã hoàn tất tất cả các tác vụ trong danh sách hay chưa. Nếu vẫn còn tác vụ cần chạy, nó gọi `next()` để sinh ra một tập tác vụ khác.

Pretty simple, isn't it?

> Khá đơn giản, phải không?

#### Globally limiting concurrency — Giới hạn tính đồng thời ở phạm vi toàn cục

Our web spider application is perfect for applying what we just learned about limiting the concurrency of a set of tasks. In fact, to avoid the situation in which we have thousands of links being crawled at the same time, we can enforce a limit on the concurrency of this process by adding some predictability regarding the number of concurrent downloads.

> Ứng dụng web spider của ta là hoàn hảo để áp dụng những gì ta vừa học về giới hạn tính đồng thời của một tập hợp các tác vụ. Thực vậy, để tránh tình huống mà ta có hàng nghìn liên kết bị thu thập cùng một lúc, ta có thể áp đặt một giới hạn lên tính đồng thời của quá trình này bằng cách thêm chút khả năng dự đoán về số lượt tải về đồng thời.

We could apply this implementation of the limited concurrency pattern to our `spiderLinks()` function, but by doing that, we would only be limiting the concurrency of tasks spawned from the links found within a given page. If we chose, for example, a concurrency of two, we would have, at most, two links downloaded in parallel for each page. However, as we can download multiple links at once, each page would then spawn another two downloads, causing the grand total of download operations to grow exponentially anyway.

> Ta có thể áp dụng bản hiện thực này của mẫu đồng thời giới hạn cho hàm `spiderLinks()`, nhưng làm vậy, ta chỉ giới hạn tính đồng thời của các tác vụ được sinh ra từ các liên kết tìm thấy trong một trang cho trước. Nếu ta chọn, ví dụ, một mức đồng thời là hai, ta sẽ có nhiều nhất hai liên kết được tải về song song cho mỗi trang. Tuy nhiên, vì ta có thể tải về nhiều liên kết cùng một lúc, mỗi trang khi đó sẽ sinh ra hai lượt tải về khác, khiến tổng số thao tác tải về dù sao vẫn tăng theo cấp số mũ.

In general, this implementation of the limited concurrency pattern works very well when we have a predetermined set of tasks to execute, or when the set of tasks grows linearly over time. When, instead, a task can spawn two or more tasks directly, as happens with our web spider, this implementation is not suitable for limiting the global concurrency.

> Nói chung, bản hiện thực này của mẫu đồng thời giới hạn hoạt động rất tốt khi ta có một tập hợp các tác vụ định trước để thực thi, hoặc khi tập hợp các tác vụ tăng tuyến tính theo thời gian. Ngược lại, khi một tác vụ có thể trực tiếp sinh ra hai hoặc nhiều tác vụ, như xảy ra với web spider của ta, bản hiện thực này không thích hợp để giới hạn tính đồng thời toàn cục.

##### Queues to the rescue — Hàng đợi đến giải cứu

What we really want, then, is to limit the global number of download operations we can have running in parallel. We could slightly modify the pattern shown in the previous section, but this is left as an exercise for you. Instead, let's discuss another mechanism that makes use of queues to limit the concurrency of multiple tasks. Let's see how this works.

> Vậy điều ta thực sự muốn là giới hạn tổng số thao tác tải về toàn cục mà ta có thể có đang chạy song song. Ta có thể sửa đổi đôi chút mẫu được trình bày ở phần trước, nhưng điều này được để lại như một bài tập cho bạn. Thay vào đó, hãy thảo luận một cơ chế khác dùng hàng đợi để giới hạn tính đồng thời của nhiều tác vụ. Hãy xem cách này hoạt động.

We are now going to implement a simple class named `TaskQueue`, which will combine a queue with the algorithm that was presented while discussing limited concurrency. Let's create a new module named `taskQueue.js`:

> Giờ ta sẽ hiện thực một lớp đơn giản tên là `TaskQueue`, lớp này sẽ kết hợp một hàng đợi với thuật toán được trình bày khi thảo luận về đồng thời giới hạn. Hãy tạo một module mới tên là `taskQueue.js`:

```js
export class TaskQueue {
  constructor (concurrency) {
    this.concurrency = concurrency
    this.running = 0
    this.queue = []
  }
  pushTask (task) {
    this.queue.push(task)
    process.nextTick(this.next.bind(this))
    return this
  }
  next () {
    while (this.running < this.concurrency && this.queue.length) {
      const task = this.queue.shift()
      task(() => {
        this.running--
        process.nextTick(this.next.bind(this))
      })
      this.running++
    }
  }
}
```

The constructor of this class takes, as input, only the concurrency limit, but besides that, it initializes the instance variables `running` and `queue`. The former variable is a counter used to keep track of all the running tasks, while the latter is the array that will be used as a queue to store the pending tasks.

> Hàm khởi tạo (constructor) của lớp này chỉ nhận vào một đầu vào duy nhất là giới hạn đồng thời, nhưng ngoài đó, nó khởi tạo các biến thực thể `running` và `queue`. Biến đầu là một bộ đếm dùng để theo dõi tất cả các tác vụ đang chạy, còn biến sau là mảng sẽ được dùng làm hàng đợi để lưu các tác vụ đang chờ.

The `pushTask()` method simply adds a new task to the queue and then bootstraps the execution of the worker by asynchronously invoking `this.next()`. Note that we have to use `bind` because otherwise, the `next` function will lose its context when invoked by `process.nextTick`.

> Phương thức `pushTask()` đơn giản thêm một tác vụ mới vào hàng đợi rồi khởi động việc thực thi worker bằng cách gọi `this.next()` một cách bất đồng bộ. Lưu ý rằng ta phải dùng `bind` bởi nếu không, hàm `next` sẽ mất ngữ cảnh của nó khi được `process.nextTick` gọi.

The `next()` method spawns a set of tasks from the queue, ensuring that it does not exceed the concurrency limit.

> Phương thức `next()` sinh ra một tập hợp các tác vụ từ hàng đợi, đảm bảo rằng nó không vượt quá giới hạn đồng thời.

You may notice that this method has some similarities with the pattern presented at the beginning of the Limiting concurrency section. It essentially starts as many tasks from the queue as possible, without exceeding the concurrency limit. When each task is complete, it updates the count of running tasks and then starts another round of tasks by asynchronously invoking `next()` again. The interesting property of the `TaskQueue` class is that it allows us to dynamically add new items to the queue. The other advantage is that, now, we have a central entity responsible for the limitation of the concurrency of our tasks, which can be shared across all the instances of a function's execution. In our case, it's the `spider()` function, as you will see in a moment.

> Bạn có thể nhận thấy phương thức này có một số điểm tương đồng với mẫu được trình bày ở đầu phần Giới hạn tính đồng thời. Về cơ bản nó khởi động càng nhiều tác vụ từ hàng đợi càng tốt, mà không vượt quá giới hạn đồng thời. Khi mỗi tác vụ hoàn tất, nó cập nhật số đếm các tác vụ đang chạy rồi khởi động một vòng tác vụ khác bằng cách gọi `next()` lại một lần nữa một cách bất đồng bộ. Tính chất thú vị của lớp `TaskQueue` là nó cho phép ta thêm các mục mới vào hàng đợi một cách động. Lợi thế khác là, giờ đây, ta có một thực thể trung tâm chịu trách nhiệm về việc giới hạn tính đồng thời của các tác vụ, có thể được chia sẻ xuyên suốt tất cả các lần thực thi của một hàm. Trong trường hợp của ta, đó là hàm `spider()`, như bạn sẽ thấy trong chốc lát.

##### Refining the TaskQueue — Tinh chỉnh TaskQueue

The previous implementation of `TaskQueue` is sufficient to demonstrate the queue pattern, but in order to be used in real-life projects, it needs a couple of extra features. For instance, how can we tell when one of the tasks has failed? How do we know whether all the work in the queue has been completed?

> Bản hiện thực trước của `TaskQueue` đủ để minh họa mẫu hàng đợi, nhưng để được dùng trong các dự án thực tế, nó cần một vài tính năng bổ sung. Chẳng hạn, làm thế nào ta có thể biết khi một trong các tác vụ đã thất bại? Làm thế nào ta biết liệu tất cả công việc trong hàng đợi đã hoàn tất hay chưa?

Let's bring back some of the concepts we discussed in Chapter 3, Callbacks and Events, and let's turn the `TaskQueue` into an `EventEmitter` so that we can emit events to propagate task failures and to inform any **observer** when the queue is empty.

> Hãy nhắc lại một số khái niệm ta đã thảo luận ở Chương 3, Callback và Sự kiện, rồi biến `TaskQueue` thành một `EventEmitter` để ta có thể phát ra các sự kiện nhằm lan truyền các thất bại của tác vụ và để thông báo cho bất kỳ observer nào khi hàng đợi trống.

The first change we have to make is to import the `EventEmitter` class and let our `TaskQueue` extend it:

> Thay đổi đầu tiên ta phải thực hiện là import lớp `EventEmitter` và cho `TaskQueue` của ta kế thừa nó:

```js
import { EventEmitter } from 'events'
export class TaskQueue extends EventEmitter {
  constructor (concurrency) {
    super()
    // ...
  }
  // ...
}
```

At this point, we can use `this.emit` to fire events from within the `TaskQueue` `next()` method:

> Tại điểm này, ta có thể dùng `this.emit` để kích hoạt các sự kiện từ bên trong phương thức `next()` của `TaskQueue`:

```js
next () {
  if (this.running === 0 && this.queue.length === 0) {         // (1)
    return this.emit('empty')
  }
  while (this.running < this.concurrency && this.queue.length) {
    const task = this.queue.shift()
    task((err) => {                                            // (2)
      if (err) {
        this.emit('error', err)
      }
      this.running--
      process.nextTick(this.next.bind(this))
    })
    this.running++
  }
}
```

Comparing this implementation with the previous one, there are two additions here:

> So sánh bản hiện thực này với bản trước, có hai điểm bổ sung ở đây:

- Every time the `next()` function is called, we check that no task is running and whether the queue is empty. In such a case, it means that the queue has been drained and we can fire the `empty` event.
- The completion callback of every task can now be invoked by passing an error. We check whether an error is actually passed, indicating that the task has failed, and in that case, we propagate such an error with an `error` event.

> - Mỗi khi hàm `next()` được gọi, ta kiểm tra rằng không có tác vụ nào đang chạy và liệu hàng đợi có trống hay không. Trong trường hợp đó, điều đó nghĩa là hàng đợi đã được rút cạn và ta có thể kích hoạt sự kiện `empty`.
> - Callback hoàn tất của mỗi tác vụ giờ có thể được gọi bằng cách truyền một lỗi. Ta kiểm tra xem một lỗi có thực sự được truyền hay không, cho thấy rằng tác vụ đã thất bại, và trong trường hợp đó, ta lan truyền lỗi đó bằng một sự kiện `error`.

Notice that in case of an error, we are deliberately keeping the queue running. We are not stopping other tasks in progress, nor removing any pending tasks. This is quite common with queue-based systems. Errors are expected to happen and rather than letting the system crash on these occasions, it is generally better to identify errors and to think about retry or recovery strategies. We will discuss these concepts a bit more in Chapter 13, Messaging and Integration Patterns.

> Lưu ý rằng trong trường hợp có lỗi, ta cố ý giữ cho hàng đợi tiếp tục chạy. Ta không dừng các tác vụ khác đang tiến hành, cũng không xóa bất kỳ tác vụ đang chờ nào. Điều này khá phổ biến với các hệ thống dựa trên hàng đợi. Lỗi được kỳ vọng sẽ xảy ra và thay vì để hệ thống crash trong những dịp này, nói chung tốt hơn là nhận diện lỗi và nghĩ về các chiến lược thử lại hoặc khôi phục. Ta sẽ thảo luận những khái niệm này thêm một chút ở Chương 13, Các mẫu Nhắn tin và Tích hợp.

##### Web spider version 4 — Web spider phiên bản 4

Now that we have our generic queue to execute tasks in a limited parallel flow, let's use it straightaway to refactor our web spider application.

> Giờ đây khi ta có hàng đợi tổng quát của mình để thực thi các tác vụ trong một luồng song song giới hạn, hãy dùng nó ngay để tái cấu trúc ứng dụng web spider của ta.

We are going to use an instance of `TaskQueue` as a work backlog; every URL that we want to crawl needs to be appended to the queue as a task. The starting URL will be added as the first task, then every other URL discovered during the crawling process will be added as well. The queue will manage all the scheduling for us, making sure that the number of tasks in progress (that is, the number of pages being downloaded or read from the filesystem) at any given time is never greater than the concurrency limit configured for the given `TaskQueue` instance.

> Ta sẽ dùng một thực thể của `TaskQueue` làm một danh sách công việc tồn đọng (work backlog); mỗi URL mà ta muốn thu thập cần được đẩy vào hàng đợi như một tác vụ. URL khởi đầu sẽ được thêm vào làm tác vụ đầu tiên, rồi mọi URL khác được phát hiện trong quá trình thu thập cũng sẽ được thêm vào. Hàng đợi sẽ quản lý toàn bộ việc lên lịch cho ta, đảm bảo rằng số tác vụ đang tiến hành (tức là số trang đang được tải về hoặc đọc từ hệ thống tệp) tại bất kỳ thời điểm cho trước nào không bao giờ lớn hơn giới hạn đồng thời được cấu hình cho thực thể `TaskQueue` đã cho.

We have already defined the logic to crawl a given URL inside our `spider()` function. We can consider this to be our generic crawling task. For more clarity, it's best to rename this function `spiderTask`:

> Ta đã định nghĩa logic để thu thập một URL cho trước bên trong hàm `spider()` của ta. Ta có thể xem đây là tác vụ thu thập tổng quát của mình. Cho rõ ràng hơn, tốt nhất là đổi tên hàm này thành `spiderTask`:

```js
function spiderTask (url, nesting, queue, cb) {             // (1)
  const filename = urlToFilename(url)
  fs.readFile(filename, 'utf8', (err, fileContent) => {
    if (err) {
      if (err.code !== 'ENOENT') {
        return cb(err)
      }
      return download(url, filename, (err, requestContent) => {
        if (err) {
          return cb(err)
        }
        spiderLinks(url, requestContent, nesting, queue)    // (2)
        return cb()
      })
    }
    spiderLinks(url, fileContent, nesting, queue)           // (3)
    return cb()
  })
}
```

Other than renaming the function, you might have noticed that we applied some other small changes:

> Ngoài việc đổi tên hàm, bạn có thể đã nhận thấy ta áp dụng một số thay đổi nhỏ khác:

- The function signature now accepts a new parameter called `queue`. This is an instance of `TaskQueue` that we need to carry over to be able to append new tasks when necessary.
- The function responsible for adding new links to crawl is `spiderLinks`, so we need to make sure that we pass the queue instance when we call this function after downloading a new page.
- We also need to pass the queue instance to `spiderLinks` when we are invoking that from an already downloaded file.

> - Chữ ký của hàm giờ chấp nhận một tham số mới tên là `queue`. Đây là một thực thể của `TaskQueue` mà ta cần mang theo để có thể đẩy thêm các tác vụ mới khi cần thiết.
> - Hàm chịu trách nhiệm thêm các liên kết mới để thu thập là `spiderLinks`, nên ta cần đảm bảo rằng ta truyền thực thể queue khi ta gọi hàm này sau khi tải về một trang mới.
> - Ta cũng cần truyền thực thể queue cho `spiderLinks` khi ta gọi nó từ một tệp đã được tải về.

Let's revisit the `spiderLinks()` function. This function can now be greatly simplified as it doesn't have to keep track of task completion anymore, as this work has been delegated to the queue. Its job becomes effectively synchronous now; it just needs to invoke the new `spider()` function (which we will define shortly) to push a new task to the queue, one for each discovered link:

> Hãy xem lại hàm `spiderLinks()`. Hàm này giờ có thể được đơn giản hóa rất nhiều vì nó không còn phải theo dõi sự hoàn tất của tác vụ nữa, vì công việc này đã được ủy thác cho hàng đợi. Công việc của nó giờ thực chất trở thành đồng bộ; nó chỉ cần gọi hàm `spider()` mới (mà ta sẽ định nghĩa ngay sau đây) để đẩy một tác vụ mới vào hàng đợi, một tác vụ cho mỗi liên kết được phát hiện:

```js
function spiderLinks (currentUrl, body, nesting, queue) {
  if (nesting === 0) {
    return
  }
  const links = getPageLinks(currentUrl, body)
  if (links.length === 0) {
    return
  }
  links.forEach(link => spider(link, nesting - 1, queue))
}
```

Let's now revisit the `spider()` function, which needs to act as the entry point for the first URL; it will also be used to add every new discovered URL to the `queue`:

> Giờ hãy xem lại hàm `spider()`, hàm này cần đóng vai trò là điểm vào (entry point) cho URL đầu tiên; nó cũng sẽ được dùng để thêm mọi URL mới được phát hiện vào `queue`:

```js
const spidering = new Set()                               // (1)
export function spider (url, nesting, queue) {
  if (spidering.has(url)) {
    return
  }
  spidering.add(url)
  queue.pushTask((done) => {                              // (2)
    spiderTask(url, nesting, queue, done)
  })
}
```

As you can see, this function now has two main responsibilities:

> Như bạn có thể thấy, hàm này giờ có hai trách nhiệm chính:

1. It manages the bookkeeping of the URLs already visited or in progress by using the `spidering` set.
2. It pushes a new task to the `queue`. Once executed, this task will invoke the `spiderTask()` function, effectively starting the crawling of the given URL.

> 1. Nó quản lý việc ghi sổ (bookkeeping) các URL đã được ghé thăm hoặc đang tiến hành bằng cách dùng tập `spidering`.
> 2. Nó đẩy một tác vụ mới vào `queue`. Một khi được thực thi, tác vụ này sẽ gọi hàm `spiderTask()`, qua đó thực sự bắt đầu việc thu thập URL đã cho.

Finally, we can update the `spider-cli.js` script, which allows us to invoke our spider from the command line:

> Cuối cùng, ta có thể cập nhật script `spider-cli.js`, cho phép ta gọi spider của mình từ dòng lệnh:

```js
import { spider } from './spider.js'
import { TaskQueue } from './TaskQueue.js'
const url = process.argv[2]                                    // (1)
const nesting = Number.parseInt(process.argv[3], 10) || 1
const concurrency = Number.parseInt(process.argv[4], 10) || 2
const spiderQueue = new TaskQueue(concurrency)                 // (2)
spiderQueue.on('error', console.error)
spiderQueue.on('empty', () => console.log('Download complete'))
spider(url, nesting, spiderQueue)                              // (3)
```

This script is now composed of three main parts:

> Script này giờ gồm ba phần chính:

1. CLI arguments parsing. Note that the script now accepts a third additional parameter that can be used to customize the concurrency level.
2. A `TaskQueue` object is created and listeners are attached to the `error` and `empty` events. When an error occurs, we simply want to print it. When the queue is empty, that means that we've finished crawling the website.
3. Finally, we start the crawling process by invoking the `spider` function.

> 1. Phân tích các tham số CLI. Lưu ý rằng script giờ chấp nhận một tham số bổ sung thứ ba có thể được dùng để tùy chỉnh mức độ đồng thời.
> 2. Một đối tượng `TaskQueue` được tạo và các listener được gắn vào sự kiện `error` và `empty`. Khi một lỗi xảy ra, ta đơn giản muốn in nó ra. Khi hàng đợi trống, điều đó nghĩa là ta đã thu thập xong website.
> 3. Cuối cùng, ta bắt đầu quá trình thu thập bằng cách gọi hàm `spider`.

After we have applied these changes, we can try to run the spider module again. When we run the following command:

> Sau khi ta đã áp dụng những thay đổi này, ta có thể thử chạy lại module spider. Khi ta chạy lệnh sau:

```js
node spider-cli.js https://loige.co 1 4
```

We should notice that no more than four downloads will be active at the same time.

> Ta sẽ nhận thấy rằng không quá bốn lượt tải về sẽ hoạt động cùng một lúc.

With this final example, we've concluded our exploration of callback-based patterns. In the next section, we will close this chapter by looking at a famous library that provides a production-ready implementation of these patterns and many other asynchronous utilities.

> Với ví dụ cuối cùng này, ta đã kết thúc cuộc khám phá các mẫu dựa trên callback của mình. Trong phần tiếp theo, ta sẽ khép lại chương này bằng cách xem xét một thư viện nổi tiếng cung cấp bản hiện thực sẵn sàng cho production của những mẫu này và nhiều tiện ích bất đồng bộ khác.

## The async library — Thư viện async

If you take a look for a moment at every control flow pattern we have analyzed so far, you will see that they can be used as a base to build reusable and more generic solutions. For example, we could wrap the unlimited parallel execution algorithm into a function that accepts a list of tasks, runs them in parallel, and invokes the given callback when all of them are complete. This way of wrapping control flow algorithms into reusable functions can lead to a more declarative and expressive way of defining asynchronous control flows, and that's exactly what `async` (`nodejsdp.` `link/async`) does.

> Nếu bạn nhìn một lát vào mọi mẫu điều khiển luồng mà ta đã phân tích cho đến nay, bạn sẽ thấy rằng chúng có thể được dùng làm nền tảng để xây dựng những giải pháp có thể tái sử dụng và tổng quát hơn. Ví dụ, ta có thể bọc thuật toán thực thi song song không giới hạn vào một hàm nhận một danh sách các tác vụ, chạy chúng song song, và gọi callback đã cho khi tất cả chúng hoàn tất. Cách bọc các thuật toán điều khiển luồng vào các hàm có thể tái sử dụng này có thể dẫn đến một cách định nghĩa luồng điều khiển bất đồng bộ mang tính khai báo (declarative) và biểu cảm hơn, và đó chính xác là điều mà `async` (`nodejsdp.` `link/async`) làm.

The `async` library (not to be confused with the `async`/`await` keywords, which we will discuss later in this book) is a very popular solution, in Node.js and JavaScript in general, for dealing with asynchronous code. It offers a set of functions that greatly simplify the execution of tasks in different configurations, and it also provides useful helpers for dealing with collections asynchronously. Even though there are several other libraries with a similar goal, `async` is the de facto standard in Node.js due to its historic popularity, especially when using callbacks to define asynchronous tasks.

> Thư viện `async` (đừng nhầm với các từ khóa `async`/`await`, mà ta sẽ thảo luận sau trong cuốn sách này) là một giải pháp rất phổ biến, trong Node.js và JavaScript nói chung, để xử lý code bất đồng bộ. Nó cung cấp một tập hợp các hàm giúp đơn giản hóa rất nhiều việc thực thi các tác vụ trong những cấu hình khác nhau, và nó cũng cung cấp các hàm trợ giúp hữu ích để xử lý các tập hợp một cách bất đồng bộ. Mặc dù có vài thư viện khác với mục tiêu tương tự, `async` là chuẩn mực trên thực tế (de facto) trong Node.js nhờ sự phổ biến mang tính lịch sử của nó, đặc biệt khi dùng callback để định nghĩa các tác vụ bất đồng bộ.

Just to give you an idea of some of the most important capabilities of the async module, here is a sample of the functionalities it exposes:

> Chỉ để cho bạn một ý niệm về một số khả năng quan trọng nhất của module async, đây là một mẫu các chức năng mà nó phơi bày:

- Execute asynchronous functions over a collection of elements (in series or in parallel with limited concurrency).
- Execute a chain of asynchronous functions (waterfall) where the output of every function becomes the input of the next one.
- Offers a queue abstraction functionally equivalent to the one we implemented with our `TaskQueue` utility.
- Provides other interesting asynchronous patterns such as race (executes multiple asynchronous functions in parallel and stops when the first one completes).

> - Thực thi các hàm bất đồng bộ trên một tập hợp các phần tử (theo chuỗi hoặc song song với tính đồng thời giới hạn).
> - Thực thi một chuỗi các hàm bất đồng bộ (waterfall) nơi đầu ra của mỗi hàm trở thành đầu vào của hàm kế tiếp.
> - Cung cấp một sự trừu tượng hàng đợi tương đương về mặt chức năng với cái mà ta đã hiện thực với tiện ích `TaskQueue` của mình.
> - Cung cấp các mẫu bất đồng bộ thú vị khác như race (thực thi nhiều hàm bất đồng bộ song song và dừng khi hàm đầu tiên hoàn tất).

Check out the `async` documentation (`nodejsdp.link/async`) to find out more about the module and to see some examples.

> Hãy xem tài liệu của `async` (`nodejsdp.link/async`) để tìm hiểu thêm về module và xem một số ví dụ.

Once you've understood the fundamentals of the asynchronous patterns described in this chapter, you shouldn't rely on the simplified implementations presented here for your everyday control flow needs. Instead, it's better to adopt a broadly used and battle-tested library like `async` for your production applications, unless your use case is so advanced that you require a custom algorithm.
