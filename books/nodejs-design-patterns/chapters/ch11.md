## Dealing with asynchronously initialized components — Xử lý các thành phần được khởi tạo bất đồng bộ

One of the reasons for the existence of synchronous APIs in the Node.js core modules and many npm packages is because they are handy to use for implementing initialization tasks. For simple programs, using synchronous APIs at initialization time can streamline things a lot and the drawbacks associated with their use remain contained because they are used only once, which is when the program or a particular component is initialized.

> Một trong những lý do tồn tại của các API đồng bộ trong các module lõi của Node.js và nhiều gói npm là vì chúng tiện dùng để hiện thực các tác vụ khởi tạo. Với những chương trình đơn giản, dùng API đồng bộ tại thời điểm khởi tạo có thể giúp mọi thứ gọn gàng hơn rất nhiều, và những nhược điểm khi sử dụng chúng vẫn nằm trong tầm kiểm soát vì chúng chỉ được dùng một lần duy nhất, đó là khi chương trình hoặc một thành phần cụ thể được khởi tạo.

Unfortunately, this is not always possible. A synchronous **API** might not always be available, especially for components using the network during their initialization phase to, for example, perform handshake protocols or to retrieve configuration parameters. This is the case for many database drivers and clients for **middleware** systems such as message queues.

> Tiếc thay, điều này không phải lúc nào cũng khả thi. Một API đồng bộ có thể không phải lúc nào cũng sẵn có, đặc biệt là với các thành phần dùng mạng trong giai đoạn khởi tạo để, ví dụ, thực hiện các giao thức bắt tay (handshake) hoặc lấy về các tham số cấu hình. Đây là trường hợp của nhiều trình điều khiển cơ sở dữ liệu (database driver) và client cho các hệ thống middleware như hàng đợi thông điệp (message queue).

### The issue with asynchronously initialized components — Vấn đề với các thành phần được khởi tạo bất đồng bộ

Let's consider an example where a **module** called `db` is used to interact with a remote database. The `db` module will accept API requests only after the connection and handshake with the database server have been successfully completed. Therefore, no queries or other commands can be sent until the initialization phase is complete. The following is the code for such a sample module (the `db.js` file):

> Hãy xét một ví dụ trong đó một module tên là `db` được dùng để tương tác với một cơ sở dữ liệu từ xa. Module `db` sẽ chỉ chấp nhận các yêu cầu API sau khi kết nối và bắt tay với máy chủ cơ sở dữ liệu đã hoàn tất thành công. Do đó, không thể gửi truy vấn hay lệnh nào khác cho đến khi giai đoạn khởi tạo hoàn thành. Sau đây là mã cho một module mẫu như vậy (tệp `db.js`):

```js
import { EventEmitter } from 'events'
class DB extends EventEmitter {
  connected = false
  connect () {
    // simulate the delay of the connection
    setTimeout(() => {
      this.connected = true
      this.emit('connected')
    }, 500)
  }
  async query (queryString) {
    if (!this.connected) {
      throw new Error('Not connected yet')
    }
    console.log(`Query executed: ${queryString}`)
  }
}
export const db = new DB()
```

This is a typical example of an asynchronously initialized component. Under these assumptions, we usually have two quick and easy solutions to this problem, which we can call local initialization check and delayed startup. Let's analyze them in more detail.

> Đây là một ví dụ điển hình của thành phần được khởi tạo bất đồng bộ. Theo các giả định này, ta thường có hai giải pháp nhanh và dễ cho bài toán này, mà ta có thể gọi là kiểm tra khởi tạo cục bộ (local initialization check) và khởi động trì hoãn (delayed startup). Hãy phân tích chúng chi tiết hơn.

#### Local initialization check — Kiểm tra khởi tạo cục bộ

The first solution makes sure that the module is initialized before any of its APIs are invoked; otherwise, we wait for its initialization. This check has to be done every time we want to invoke an operation on the asynchronous module:

> Giải pháp đầu tiên đảm bảo rằng module đã được khởi tạo trước khi bất kỳ API nào của nó được gọi; nếu chưa, ta đợi quá trình khởi tạo của nó. Việc kiểm tra này phải được thực hiện mỗi khi ta muốn gọi một thao tác trên module bất đồng bộ:

```js
import { once } from 'events'
import { db } from './db.js'
db.connect()
async function updateLastAccess () {
  if (!db.connected) {
    await once(db, 'connected')
  }
  await db.query(`INSERT (${Date.now()}) INTO "LastAccesses"`)
}
updateLastAccess()
setTimeout(() => {
  updateLastAccess()
}, 600)
```

As we already anticipated, any time we want to invoke the `query()` method on the `db` component, we have to check if the module is initialized; otherwise, we wait for its initialization by listening for the `'connected'` event. A variation of this technique performs the check inside the `query()` method itself, which shifts the burden of the boilerplate code from the consumer to the provider of the service.

> Như đã dự đoán trước, mỗi khi ta muốn gọi phương thức `query()` trên thành phần `db`, ta phải kiểm tra xem module đã được khởi tạo chưa; nếu chưa, ta đợi quá trình khởi tạo của nó bằng cách lắng nghe sự kiện `'connected'`. Một biến thể của kỹ thuật này thực hiện việc kiểm tra ngay bên trong chính phương thức `query()`, qua đó dịch chuyển gánh nặng mã soạn sẵn (boilerplate) từ phía bên tiêu thụ sang phía bên cung cấp dịch vụ.

#### Delayed startup — Khởi động trì hoãn

The second quick and dirty solution to the problem of asynchronously initialized components involves delaying the execution of any code relying on the asynchronously initialized component until the component has finished its initialization routine. We can see an example of such a technique in the following code fragment:

> Giải pháp nhanh và thô thứ hai cho bài toán các thành phần được khởi tạo bất đồng bộ liên quan đến việc trì hoãn việc thực thi mọi đoạn mã phụ thuộc vào thành phần được khởi tạo bất đồng bộ cho đến khi thành phần đó hoàn tất quy trình khởi tạo của mình. Ta có thể thấy một ví dụ của kỹ thuật này trong đoạn mã sau:

```js
import { db } from './db.js'
import { once } from 'events'
async function initialize () {
  db.connect()
  await once(db, 'connected')
}
async function updateLastAccess () {
  await db.query(`INSERT (${Date.now()}) INTO "LastAccesses"`)
}
initialize()
  .then(() => {
    updateLastAccess()
    setTimeout(() => {
      updateLastAccess()
    }, 600)
  })
```

As we can see from the preceding code, we first wait for the initialization to complete, and then we proceed with executing any routine that uses the `db` object.

> Như ta thấy từ đoạn mã trên, trước tiên ta đợi quá trình khởi tạo hoàn tất, rồi sau đó mới tiến hành thực thi bất kỳ routine nào sử dụng đối tượng `db`.

The main disadvantage of this technique is that it requires us to know, in advance, which components will make use of the asynchronously initialized component, which makes our code fragile and exposed to mistakes. One solution to this problem is delaying the startup of the entire application until all the asynchronous services are initialized. This has the advantage of being simple and effective; however, it can add a significant delay to the overall startup time of the application and moreover, it won't take into account the case in which the asynchronously initialized component has to be reinitialized.

> Nhược điểm chính của kỹ thuật này là nó đòi hỏi ta phải biết trước những thành phần nào sẽ sử dụng thành phần được khởi tạo bất đồng bộ, điều này khiến mã của ta trở nên mong manh và dễ mắc lỗi. Một giải pháp cho vấn đề này là trì hoãn việc khởi động toàn bộ ứng dụng cho đến khi tất cả các dịch vụ bất đồng bộ đã được khởi tạo. Cách này có lợi thế là đơn giản và hiệu quả; tuy nhiên, nó có thể thêm một độ trễ đáng kể vào tổng thời gian khởi động của ứng dụng, và hơn nữa, nó sẽ không tính đến trường hợp thành phần được khởi tạo bất đồng bộ phải được khởi tạo lại.

As we will see in the next section, there is a third alternative that allows us to transparently and efficiently delay every operation until the asynchronous initialization step has completed.

> Như ta sẽ thấy trong phần tiếp theo, có một lựa chọn thứ ba cho phép ta trì hoãn mọi thao tác một cách minh bạch và hiệu quả cho đến khi bước khởi tạo bất đồng bộ đã hoàn tất.

### Pre-initialization queues — Hàng đợi tiền khởi tạo

Another recipe to make sure that the services of a component are invoked only after the component is initialized involves the use of queues and the Command pattern. The idea is to queue the method invocations (only those requiring the component to be initialized) received while the component is not yet initialized, and then execute them as soon as all the initialization steps have been completed.

> Một công thức khác để đảm bảo các dịch vụ của một thành phần chỉ được gọi sau khi thành phần đã được khởi tạo liên quan đến việc dùng hàng đợi và mẫu Command. Ý tưởng là xếp hàng các lời gọi phương thức (chỉ những lời gọi đòi hỏi thành phần phải được khởi tạo) nhận được trong khi thành phần chưa được khởi tạo, rồi thực thi chúng ngay khi tất cả các bước khởi tạo đã hoàn tất.

Let's see how this technique can be applied to our sample `db` component:

> Hãy xem kỹ thuật này có thể được áp dụng cho thành phần `db` mẫu của ta như thế nào:

```js
import { EventEmitter } from 'events'
class DB extends EventEmitter {
  connected = false
  commandsQueue = []
  async query (queryString) {
    if (!this.connected) {
      console.log(`Request queued: ${queryString}`)
      return new Promise((resolve, reject) => {            // (1)
        const command = () => {
          this.query(queryString)
            .then(resolve, reject)
        }
        this.commandsQueue.push(command)
      })
    }
    console.log(`Query executed: ${queryString}`)
  }
  connect () {
    // simulate the delay of the connection
    setTimeout(() => {
      this.connected = true
      this.emit('connected')
      this.commandsQueue.forEach(command => command())     // (2)
      this.commandsQueue = []
    }, 500)
  }
}
export const db = new DB()
```

As we already mentioned, the technique described here consists of two parts:

> Như đã đề cập, kỹ thuật được mô tả ở đây gồm hai phần:

1. If the component has not been initialized—which, in our case, is when the `connected` property is `false`—we create a command from the parameters received with the current invocation and push it to the `commandsQueue` array. When the command is executed, it will run the original `query()` method again and forward the result to the `**Promise**` we are returning to the caller.
2. When the initialization of the component is completed—which, in our case, means that the connection with the database server is established—we go through the `commandsQueue`, executing all the commands that have been previously queued.

> 1. Nếu thành phần chưa được khởi tạo - trong trường hợp của ta là khi thuộc tính `connected` là `false` - ta tạo một command từ các tham số nhận được với lời gọi hiện tại và đẩy nó vào mảng `commandsQueue`. Khi command được thực thi, nó sẽ chạy lại phương thức `query()` gốc và chuyển tiếp kết quả tới `Promise` mà ta đang trả về cho phía gọi.
> 2. Khi quá trình khởi tạo của thành phần hoàn tất - trong trường hợp của ta nghĩa là kết nối với máy chủ cơ sở dữ liệu đã được thiết lập - ta duyệt qua `commandsQueue`, thực thi tất cả các command đã được xếp hàng trước đó.

With the `DB` class we just implemented, there is no need to check if the component is initialized before invoking its methods. In fact, all the logic is embedded in the component itself and any consumer can just transparently use it without worrying about its initialization status.

> Với lớp `DB` ta vừa hiện thực, không cần phải kiểm tra xem thành phần đã được khởi tạo chưa trước khi gọi các phương thức của nó. Thực tế, toàn bộ logic được nhúng ngay trong chính thành phần và bất kỳ bên tiêu thụ nào cũng có thể dùng nó một cách minh bạch mà không cần bận tâm đến trạng thái khởi tạo của nó.

We can also go a step further and try to reduce the boilerplate of the `DB` class we just created and, at the same time, improve its modularity. We can achieve that by applying the **State** pattern, which we learned about in Chapter 9, Behavioral Design Patterns, with two states:

> Ta cũng có thể tiến thêm một bước nữa và cố gắng giảm bớt mã soạn sẵn của lớp `DB` ta vừa tạo, đồng thời cải thiện tính module hóa của nó. Ta có thể đạt được điều đó bằng cách áp dụng mẫu State, mà ta đã học ở Chương 9, Behavioral Design Patterns, với hai trạng thái:

- The first state implements all the methods that require the component to be initialized, and it's activated only when there is a successful initialization. Each of these methods implements its own business logic without worrying about the initialization status of the `db` component
- The second state is activated before the initialization has completed and it implements the same methods as the first state, but their only role here is to add a new command to the queue using the parameters passed to the invocation.

> - Trạng thái thứ nhất hiện thực tất cả các phương thức đòi hỏi thành phần phải được khởi tạo, và nó chỉ được kích hoạt khi việc khởi tạo thành công. Mỗi phương thức này hiện thực logic nghiệp vụ riêng mà không cần bận tâm đến trạng thái khởi tạo của thành phần `db`
> - Trạng thái thứ hai được kích hoạt trước khi việc khởi tạo hoàn tất và nó hiện thực cùng các phương thức như trạng thái thứ nhất, nhưng vai trò duy nhất của chúng ở đây là thêm một command mới vào hàng đợi dùng các tham số được truyền cho lời gọi.

Let's see how we can apply the structure we just described to our `db` component. First, we create the `InitializedState`, which implements the actual business logic of our component:

> Hãy xem ta có thể áp dụng cấu trúc vừa mô tả vào thành phần `db` của mình như thế nào. Trước tiên, ta tạo `InitializedState`, lớp hiện thực logic nghiệp vụ thực sự của thành phần:

```js
class InitializedState {
  async query (queryString) {
    console.log(`Query executed: ${queryString}`)
  }
}
```

As we can see, the only method that we need to implement in the `InitializedState` class is the `query()` method, which will print a message to the console when it receives a new query. Next, we implement the `QueuingState`, the core of our recipe. This state implements the queuing logic:

> Như ta thấy, phương thức duy nhất ta cần hiện thực trong lớp `InitializedState` là phương thức `query()`, sẽ in một thông báo ra console khi nó nhận được một truy vấn mới. Tiếp theo, ta hiện thực `QueuingState`, phần cốt lõi của công thức. Trạng thái này hiện thực logic xếp hàng:

```js
const METHODS_REQUIRING_CONNECTION = ['query']
const deactivate = Symbol('deactivate')
class QueuingState {
  constructor (db) {
    this.db = db
    this.commandsQueue = []
    METHODS_REQUIRING_CONNECTION.forEach(methodName => {
      this[methodName] = function (...args) {
        console.log('Command queued:', methodName, args)
        return new Promise((resolve, reject) => {
          const command = () => {
            db[methodName](...args)
              .then(resolve, reject)
          }
          this.commandsQueue.push(command)
        })
      }
    })
  }
  [deactivate] () {
    this.commandsQueue.forEach(command => command())
    this.commandsQueue = []
  }
}
```

It's interesting to note how the `QueuingState` is mostly built dynamically at creation time. For each method that requires an active connection, we create a new method for the current instance, which queues a new command representing the function invocation. When the command is executed at a later time, when a connection is established, the result of the invocation of the method on the `db` instance is forwarded to the caller (through the returned promise).

> Thật thú vị khi lưu ý cách `QueuingState` chủ yếu được xây dựng động tại thời điểm tạo. Với mỗi phương thức đòi hỏi một kết nối đang hoạt động, ta tạo một phương thức mới cho thực thể (instance) hiện tại, phương thức này xếp hàng một command mới đại diện cho lời gọi hàm. Khi command được thực thi vào lúc sau, khi một kết nối đã được thiết lập, kết quả của việc gọi phương thức trên thực thể `db` được chuyển tiếp tới phía gọi (thông qua promise được trả về).

The other important part of this state class is `[deactivate]()`. This method is invoked when the state is deactivated (which is when the component is initialized) and it executes all the commands in the queue. Note how we used a `Symbol` to name the method. This will avoid any name clashes in the future if we add more methods to the state (for example, what if we need to decorate a hypothetical `deactivate()` method of the `DB` class?).

> Phần quan trọng khác của lớp trạng thái này là `[deactivate]()`. Phương thức này được gọi khi trạng thái bị vô hiệu hóa (đó là khi thành phần được khởi tạo) và nó thực thi tất cả các command trong hàng đợi. Lưu ý cách ta dùng một `Symbol` để đặt tên cho phương thức. Điều này sẽ tránh mọi xung đột tên trong tương lai nếu ta thêm các phương thức khác vào trạng thái (ví dụ, sẽ ra sao nếu ta cần trang trí (decorate) một phương thức `deactivate()` giả định của lớp `DB`?).

Now, it's time to reimplement the `DB` class using the two states we just described:

> Bây giờ, đã đến lúc hiện thực lại lớp `DB` dùng hai trạng thái ta vừa mô tả:

```js
class DB extends EventEmitter {
  constructor () {
    super()
    this.state = new QueuingState(this)                    // (1)
  }
  async query (queryString) {
    return this.state.query(queryString)                   // (2)
  }
  connect () {
    // simulate the delay of the connection
    setTimeout(() => {
      this.connected = true
      this.emit('connected')
      const oldState = this.state                          // (3)
      this.state = new InitializedState(this)
      oldState[deactivate] && oldState[deactivate]()
    }, 500)
  }
}
export const db = new DB()
```

Let's further analyze the most important parts of the new `DB` class:

> Hãy phân tích kỹ hơn những phần quan trọng nhất của lớp `DB` mới:

1. In the constructor, we initialize the current `state` of the instance. It's going to be the `QueuingState` as the asynchronous initialization of the component hasn't been completed yet.
2. The only method of our class implementing some (stub) business logic is the `query()` method. Here, all we have to do is invoke the homonymous method on the currently active state.
3. Finally, when we establish the connection with the database (initialization complete), we switch the current state to the `InitializedState` and we deactivate the old one. The effect of deactivating the `QueuedState`, as we've seen previously, is that any command that had been queued is now executed.

> 1. Trong hàm khởi tạo, ta khởi tạo `state` hiện tại của thực thể. Nó sẽ là `QueuingState` vì quá trình khởi tạo bất đồng bộ của thành phần chưa hoàn tất.
> 2. Phương thức duy nhất của lớp ta hiện thực một chút logic nghiệp vụ (dạng stub) là phương thức `query()`. Ở đây, tất cả những gì ta phải làm là gọi phương thức cùng tên trên trạng thái đang được kích hoạt.
> 3. Cuối cùng, khi ta thiết lập kết nối với cơ sở dữ liệu (khởi tạo hoàn tất), ta chuyển trạng thái hiện tại sang `InitializedState` và vô hiệu hóa trạng thái cũ. Hệ quả của việc vô hiệu hóa `QueuedState`, như ta đã thấy trước đó, là mọi command đã được xếp hàng giờ đây được thực thi.

We can immediately see how this approach allows us to reduce the boilerplate and, at the same time, create a class that is purely business logic (the `InitializedState`) free from any repetitive initialization check.

> Ta có thể ngay lập tức thấy cách tiếp cận này cho phép ta giảm mã soạn sẵn, đồng thời tạo ra một lớp thuần túy là logic nghiệp vụ (lớp `InitializedState`) không còn bất kỳ kiểm tra khởi tạo lặp đi lặp lại nào.

The approach we've just seen will only work if we can modify the code of our asynchronously initialized component. In all those cases in which we can't make modifications to the component, we will need to create a wrapper or **proxy**, but the technique will be mostly similar to what we've seen here.

> Cách tiếp cận ta vừa thấy chỉ hoạt động nếu ta có thể chỉnh sửa mã của thành phần được khởi tạo bất đồng bộ của mình. Trong tất cả những trường hợp ta không thể thay đổi thành phần, ta sẽ cần tạo một wrapper hoặc proxy, nhưng kỹ thuật sẽ phần lớn tương tự với những gì ta đã thấy ở đây.

### In the wild — Trong thực tế

The pattern we just presented is used by many database drivers and ORM libraries. The most notable is Mongoose (`nodejsdp.link/mongoose`), which is an ORM for MongoDB. With Mongoose, it's not necessary to wait for the database connection to open in order to be able to send queries. This is because each operation is queued and then executed later when the connection with the database is fully established, exactly as we've described in this section. This is clearly a must for any API that wants to provide a good developer experience (DX).

> Mẫu ta vừa trình bày được nhiều trình điều khiển cơ sở dữ liệu và thư viện ORM sử dụng. Nổi bật nhất là Mongoose (`nodejsdp.link/mongoose`), một ORM cho MongoDB. Với Mongoose, không cần phải đợi kết nối cơ sở dữ liệu mở ra để có thể gửi truy vấn. Đó là vì mỗi thao tác được xếp hàng rồi thực thi sau đó khi kết nối với cơ sở dữ liệu đã được thiết lập đầy đủ, đúng như ta đã mô tả trong phần này. Đây rõ ràng là điều bắt buộc với bất kỳ API nào muốn cung cấp trải nghiệm lập trình viên (developer experience - DX) tốt.

Take a look at the code of Mongoose to see how every method in the native driver is proxied to add the pre-initialization queue. This also demonstrates an alternative way of implementing the recipe we presented in this section. You can find the relevant code fragment at `nodejsdp.link/mongoose-init-queue`.

> Hãy xem mã của Mongoose để thấy cách mọi phương thức trong trình điều khiển gốc (native driver) được proxy để thêm hàng đợi tiền khởi tạo. Điều này cũng minh họa một cách thay thế để hiện thực công thức ta đã trình bày trong phần này. Bạn có thể tìm đoạn mã liên quan tại `nodejsdp.link/mongoose-init-queue`.

Similarly, the `pg` **package** (`nodejsdp.link/pg`), which is a client for the PostgreSQL database, leverages pre-initialization queues, but in a slightly different fashion. `pg` queues every query, regardless of the initialization status of the database, and then immediately tries to execute all the commands in the queue. Take a look at the relevant code line at `nodejsdp.link/pg-queue`.

> Tương tự, gói `pg` (`nodejsdp.link/pg`), một client cho cơ sở dữ liệu PostgreSQL, tận dụng hàng đợi tiền khởi tạo, nhưng theo một cách hơi khác. `pg` xếp hàng mọi truy vấn, bất kể trạng thái khởi tạo của cơ sở dữ liệu, rồi ngay lập tức cố gắng thực thi tất cả các command trong hàng đợi. Hãy xem dòng mã liên quan tại `nodejsdp.link/pg-queue`.

## Asynchronous request batching and caching — Gộp lô và lưu đệm các yêu cầu bất đồng bộ

In high-load applications, caching plays a critical role and it's used almost everywhere on the web, from static resources such as web pages, images, and stylesheets, to pure data such as the result of database queries. In this section, we are going to learn how caching applies to asynchronous operations and how a high request throughput can be turned to our advantage.

> Trong các ứng dụng tải cao, caching đóng vai trò then chốt và nó được dùng gần như ở khắp mọi nơi trên web, từ các tài nguyên tĩnh như trang web, hình ảnh và stylesheet, đến dữ liệu thuần túy như kết quả của các truy vấn cơ sở dữ liệu. Trong phần này, ta sẽ học cách caching áp dụng cho các thao tác bất đồng bộ và cách một thông lượng (throughput) yêu cầu cao có thể được biến thành lợi thế của ta.

### What's asynchronous request batching? — Gộp lô yêu cầu bất đồng bộ là gì?

When dealing with asynchronous operations, the most basic level of caching can be achieved by batching together a set of invocations to the same API. The idea is very simple: if we invoke an asynchronous function while there is still another one pending, we can piggyback on the already running operation instead of creating a brand new request. Take a look at the following diagram:

> Khi xử lý các thao tác bất đồng bộ, mức caching cơ bản nhất có thể đạt được bằng cách gộp lô (batching) một tập hợp các lời gọi tới cùng một API lại với nhau. Ý tưởng rất đơn giản: nếu ta gọi một hàm bất đồng bộ trong khi vẫn còn một hàm khác đang chờ xử lý, ta có thể bám theo (piggyback) thao tác đang chạy thay vì tạo một yêu cầu hoàn toàn mới. Hãy xem sơ đồ sau:

![](images/ch11-465_1.png)

Figure 11.1: Two asynchronous requests with no batching

> Hình 11.1: Hai yêu cầu bất đồng bộ không gộp lô

The previous diagram shows two clients invoking the same asynchronous operation with exactly the same input. Of course, the natural way to picture this situation is with the two clients starting two separate operations that will complete at two different moments. Now, consider the following scenario:

> Sơ đồ trên cho thấy hai client gọi cùng một thao tác bất đồng bộ với chính xác cùng một đầu vào. Tất nhiên, cách tự nhiên để hình dung tình huống này là hai client bắt đầu hai thao tác riêng biệt sẽ hoàn tất ở hai thời điểm khác nhau. Bây giờ, hãy xét kịch bản sau:

![](images/ch11-466_1.png)

Figure 11.2: Batching of two asynchronous requests

> Hình 11.2: Gộp lô hai yêu cầu bất đồng bộ

Figure 11.2 shows us how two identical requests—which invoke the same API with the same input—can be batched, or in other words, appended to the same running operation. By doing this, when the operation completes, both clients are notified, even though the **async** operation is actually executed only once. This represents a simple, yet extremely powerful, way to optimize the load of an application while not having to deal with more complex caching mechanisms, which usually require an adequate memory management and invalidation **strategy**.

> Hình 11.2 cho ta thấy cách hai yêu cầu giống hệt nhau - cùng gọi một API với cùng đầu vào - có thể được gộp lô, hay nói cách khác, được gắn vào cùng một thao tác đang chạy. Bằng cách này, khi thao tác hoàn tất, cả hai client đều được thông báo, dù thao tác bất đồng bộ thực ra chỉ được thực thi một lần. Điều này đại diện cho một cách đơn giản nhưng cực kỳ mạnh mẽ để tối ưu tải của một ứng dụng mà không phải đối phó với các cơ chế caching phức tạp hơn, vốn thường đòi hỏi một chiến lược quản lý bộ nhớ và vô hiệu hóa (invalidation) thích hợp.

### Optimal asynchronous request caching — Lưu đệm yêu cầu bất đồng bộ tối ưu

Request batching is less effective if the operation is fast enough or if matching requests are spread across a longer period of time. Also, most of the time, we can safely assume that the result of two identical API invocations will not change so often, so simple request batching will not provide the best performance. In all these circumstances, the best candidate to reduce the load of an application and increase its responsiveness is definitely a more aggressive caching mechanism. The idea is simple: as soon as a request completes, we store its result in the cache, which can be an in-memory variable or an item in a specialized caching server (such as Redis). Hence, the next time the API is invoked, the result can be retrieved immediately from the cache, instead of spawning another request.

> Gộp lô yêu cầu kém hiệu quả hơn nếu thao tác đủ nhanh hoặc nếu các yêu cầu trùng khớp trải dài trên một khoảng thời gian dài hơn. Ngoài ra, hầu hết thời gian, ta có thể giả định an toàn rằng kết quả của hai lời gọi API giống hệt nhau sẽ không thay đổi quá thường xuyên, do đó gộp lô yêu cầu đơn thuần sẽ không cho hiệu năng tốt nhất. Trong tất cả những trường hợp này, ứng cử viên tốt nhất để giảm tải cho ứng dụng và tăng khả năng đáp ứng của nó chắc chắn là một cơ chế caching quyết liệt hơn. Ý tưởng rất đơn giản: ngay khi một yêu cầu hoàn tất, ta lưu kết quả của nó vào cache, có thể là một biến trong bộ nhớ hoặc một mục trong máy chủ caching chuyên dụng (như Redis). Do đó, lần sau khi API được gọi, kết quả có thể được lấy ngay lập tức từ cache, thay vì sinh ra một yêu cầu khác.

The idea of caching should not be new to an experienced developer, but what makes this technique different in asynchronous programming is that it should be combined with request batching to be optimal. The reason for this is because multiple requests might run concurrently while the cache is not set and when those requests complete, the cache would be set multiple times.

> Ý tưởng về caching không nên là điều mới mẻ với một lập trình viên có kinh nghiệm, nhưng điều làm cho kỹ thuật này khác biệt trong lập trình bất đồng bộ là nó nên được kết hợp với gộp lô yêu cầu để đạt tối ưu. Lý do là vì nhiều yêu cầu có thể chạy đồng thời trong khi cache chưa được thiết lập, và khi các yêu cầu đó hoàn tất, cache sẽ bị thiết lập nhiều lần.

Based on these assumptions, we can illustrate the Combined Request Batching and Caching pattern as follows:

> Dựa trên các giả định này, ta có thể minh họa mẫu Gộp lô và Lưu đệm Yêu cầu Kết hợp (Combined Request Batching and Caching) như sau:

![](images/ch11-467_1.png)

Figure 11.3: Combined batching and caching

> Hình 11.3: Gộp lô và lưu đệm kết hợp

The preceding figure shows the two phases of an optimal asynchronous caching algorithm:

> Hình trên cho thấy hai giai đoạn của một thuật toán caching bất đồng bộ tối ưu:

- The first phase is totally identical to the batching pattern. Any request received while the cache is not set will be batched together. When the request completes, the cache is set, once.
- When the cache is finally set, any subsequent request will be served directly from it.

> - Giai đoạn thứ nhất hoàn toàn giống với mẫu gộp lô. Bất kỳ yêu cầu nào nhận được trong khi cache chưa được thiết lập sẽ được gộp lô lại với nhau. Khi yêu cầu hoàn tất, cache được thiết lập một lần.
> - Khi cache cuối cùng đã được thiết lập, bất kỳ yêu cầu nào sau đó sẽ được phục vụ trực tiếp từ nó.

Another crucial detail to consider is the Zalgo anti-pattern (as we saw in Chapter 3, Callbacks and Events). Since we are dealing with asynchronous APIs, we must be sure to always return the cached value asynchronously, even if accessing the cache involves only a synchronous operation, such as in the case in which the cached value is retrieved from an in-memory variable.

> Một chi tiết quan trọng khác cần xem xét là antipattern Zalgo (như ta đã thấy ở Chương 3, Callbacks and Events). Vì ta đang xử lý các API bất đồng bộ, ta phải đảm bảo luôn trả về giá trị đã cache một cách bất đồng bộ, ngay cả khi việc truy cập cache chỉ liên quan đến một thao tác đồng bộ, chẳng hạn như trong trường hợp giá trị đã cache được lấy từ một biến trong bộ nhớ.

### An API server without caching or batching — Một máy chủ API không có caching hay batching

Before we start diving into this new challenge, let's implement a small demo server that we will use as a reference to measure the impact of the various techniques we are going to implement.

> Trước khi bắt đầu đi sâu vào thử thách mới này, hãy hiện thực một máy chủ demo nhỏ mà ta sẽ dùng làm tham chiếu để đo lường tác động của các kỹ thuật khác nhau mà ta sắp hiện thực.

Let's consider an API server that manages the sales of an e-commerce company. In particular, we want to query our server for the sum of all the transactions of a particular type of merchandise. For this purpose, we are going to use a LevelUP database through the `level` npm package (`nodejsdp.link/level`). The data model that we are going to use is a simple list of transactions stored in the `sales` sublevel (a subsection of the database), which is organized in the following format:

> Hãy xét một máy chủ API quản lý doanh số của một công ty thương mại điện tử. Cụ thể, ta muốn truy vấn máy chủ để lấy tổng tất cả các giao dịch của một loại hàng hóa cụ thể. Với mục đích này, ta sẽ dùng một cơ sở dữ liệu LevelUP thông qua gói npm `level` (`nodejsdp.link/level`). Mô hình dữ liệu ta sẽ dùng là một danh sách giao dịch đơn giản được lưu trong sublevel `sales` (một phần con của cơ sở dữ liệu), được tổ chức theo định dạng sau:

```js
transactionId {amount, product}
```

The key is represented by `transactionId` and the value is a JSON object that contains the amount of the sale (`amount`) and the product type (`product`).

> Khóa được biểu diễn bởi `transactionId` và giá trị là một đối tượng JSON chứa số tiền của giao dịch bán (`amount`) và loại sản phẩm (`product`).

The data to process is really basic, so let's implement a simple query over the database that we can use for our experiments. Let's say that we want to get the total amount of sales for a particular product. The routine would look as follows (file `totalSales.js`):

> Dữ liệu cần xử lý thực sự rất cơ bản, nên hãy hiện thực một truy vấn đơn giản trên cơ sở dữ liệu mà ta có thể dùng cho các thí nghiệm của mình. Giả sử ta muốn lấy tổng số tiền bán hàng cho một sản phẩm cụ thể. Routine sẽ trông như sau (tệp `totalSales.js`):

```js
import level from 'level'
import sublevel from 'subleveldown'
const db = level('example-db')
const salesDb = sublevel(db, 'sales', { valueEncoding: 'json' })
export async function totalSales (product) {
  const now = Date.now()
  let sum = 0
  for await (const transaction of salesDb.createValueStream()) {
    if (!product || transaction.product === product) {
      sum += transaction.amount
    }
  }
  console.log(`totalSales() took: ${Date.now() - now}ms`)
  return sum
}
```

The `totalSales()` function iterates over all the transactions of the `sales` sublevel and calculates the sum of the amounts of a particular product. The algorithm is intentionally slow as we want to highlight the effect of batching and caching later on. In a real-world application, we would have used an index to query the transactions by `product` or, even better, we could have used an incremental map/reduce algorithm to continuously calculate the sum for every product

> Hàm `totalSales()` lặp qua tất cả các giao dịch của sublevel `sales` và tính tổng số tiền của một sản phẩm cụ thể. Thuật toán cố ý chậm vì ta muốn làm nổi bật tác động của batching và caching ở phần sau. Trong một ứng dụng thực tế, ta sẽ dùng một chỉ mục (index) để truy vấn các giao dịch theo `product` hoặc, tốt hơn nữa, ta có thể dùng một thuật toán map/reduce tăng dần để liên tục tính tổng cho mỗi sản phẩm

We can now expose the `totalSales()` API through a simple HTTP server (the `server.js` file):

> Giờ ta có thể phơi bày API `totalSales()` thông qua một máy chủ HTTP đơn giản (tệp `server.js`):

```js
import { createServer } from 'http'
import { totalSales } from './totalSales.js'
createServer(async (req, res) => {
  const url = new URL(req.url, 'http://localhost')
  const product = url.searchParams.get('product')
  console.log(`Processing query: ${url.search}`)
  const sum = await totalSales(product)
  res.setHeader('Content-Type', 'application/json')
  res.writeHead(200)
  res.end(JSON.stringify({
    product,
    sum
  }))
}).listen(8000, () => console.log('Server started'))
```

Before we start the server for the first time, we need to populate the database with some sample data. We can do this with the `populateDb.js` script, which can be found in this book's code repository in the folder dedicated to this section. This script creates 100,000 random sales transactions in the database so that our query spends some time crunching data:

> Trước khi khởi động máy chủ lần đầu tiên, ta cần điền vào cơ sở dữ liệu một số dữ liệu mẫu. Ta có thể làm điều này với script `populateDb.js`, có thể tìm thấy trong kho mã của cuốn sách này ở thư mục dành riêng cho phần này. Script này tạo 100.000 giao dịch bán hàng ngẫu nhiên trong cơ sở dữ liệu để truy vấn của ta tốn một chút thời gian để xử lý dữ liệu:

```js
node populateDb.js
```

Okay! Now, everything is ready. Let's start the server:

> Được rồi! Giờ thì mọi thứ đã sẵn sàng. Hãy khởi động máy chủ:

```js
node server.js
```

### Batching and caching with promises — Batching và caching với promise

Promises are a great tool for implementing asynchronous batching and caching of requests. Let's see why.

> Promise là một công cụ tuyệt vời để hiện thực batching và caching yêu cầu bất đồng bộ. Hãy xem tại sao.

If we recall what we learned about promises in Chapter 5, Asynchronous **Control Flow** Patterns with Promises and Async/**Await**, there are two properties that can be exploited to our advantage in this circumstance:

> Nếu ta nhớ lại những gì đã học về promise ở Chương 5, Asynchronous Control Flow Patterns with Promises and Async/Await, có hai thuộc tính có thể được khai thác để có lợi cho ta trong tình huống này:

- Multiple `then()` listeners can be attached to the same promise.
- The `then()` listener is guaranteed to be invoked (only once), and it works even if it's attached after the promise is already resolved. Moreover, `then()` is guaranteed to always be invoked asynchronously.

> - Nhiều listener `then()` có thể được gắn vào cùng một promise.
> - Listener `then()` được đảm bảo sẽ được gọi (chỉ một lần), và nó hoạt động ngay cả khi nó được gắn sau khi promise đã được giải quyết (resolved). Hơn nữa, `then()` được đảm bảo luôn được gọi một cách bất đồng bộ.

In short, the first property is exactly what we need for batching requests, while the second means that a promise is already a cache for the resolved value and offers a natural mechanism for returning a cached value in a consistent, asynchronous way. In other words, this means that batching and caching become extremely simple and concise with promises.

> Tóm lại, thuộc tính thứ nhất chính là điều ta cần để gộp lô yêu cầu, trong khi thuộc tính thứ hai nghĩa là một promise vốn đã là một cache cho giá trị đã resolve và cung cấp một cơ chế tự nhiên để trả về giá trị đã cache một cách nhất quán, bất đồng bộ. Nói cách khác, điều này có nghĩa là batching và caching trở nên cực kỳ đơn giản và ngắn gọn với promise.

#### Batching requests in the total sales web server — Gộp lô yêu cầu trong máy chủ web tổng doanh số

Let's now add a batching layer on top of our `totalSales` API. The pattern we are going to use is very simple: if there is another identical request pending when the API is invoked, we will wait for that request to complete instead of launching a new one. As we will see, this can easily be implemented with promises. In fact, all we have to do is save the promise in a map, associating it to the specified request parameters (the `product` type, in our case) every time we launch a new request. Then, at every subsequent request, we check if there is already a promise for the specified `product` and if there is one, we just return it; otherwise, we launch a new request.

> Giờ hãy thêm một lớp batching lên trên API `totalSales` của ta. Mẫu ta sẽ dùng rất đơn giản: nếu có một yêu cầu giống hệt khác đang chờ xử lý khi API được gọi, ta sẽ đợi yêu cầu đó hoàn tất thay vì khởi chạy một yêu cầu mới. Như ta sẽ thấy, điều này có thể dễ dàng được hiện thực với promise. Thực tế, tất cả những gì ta phải làm là lưu promise vào một map, liên kết nó với các tham số yêu cầu được chỉ định (loại `product`, trong trường hợp của ta) mỗi khi ta khởi chạy một yêu cầu mới. Sau đó, ở mỗi yêu cầu tiếp theo, ta kiểm tra xem đã có một promise cho `product` được chỉ định chưa và nếu có, ta chỉ cần trả về nó; nếu không, ta khởi chạy một yêu cầu mới.

Now, let's see how this translates into code. Let's create a new module named `totalSalesBatch.js`. Here, we're going to implement a batching layer on top of the original `totalSales()` API:

> Giờ hãy xem điều này được dịch thành mã như thế nào. Hãy tạo một module mới tên là `totalSalesBatch.js`. Ở đây, ta sẽ hiện thực một lớp batching lên trên API `totalSales()` gốc:

```js
import { totalSales as totalSalesRaw } from './totalSales.js'
const runningRequests = new Map()
export function totalSales (product) {
  if (runningRequests.has(product)) {                      // (1)
    console.log('Batching')
    return runningRequests.get(product)
  }
  const resultPromise = totalSalesRaw(product)             // (2)
  runningRequests.set(product, resultPromise)
  resultPromise.finally(() => {
    runningRequests.delete(product)
  })
  return resultPromise
}
```

The `totalSales()` function of the `totalSalesBatch` module is a proxy for the original `totalSales()` API, and it works as follows:

> Hàm `totalSales()` của module `totalSalesBatch` là một proxy cho API `totalSales()` gốc, và nó hoạt động như sau:

1. If a promise for the given `product` already exists, we just return it. This is where we piggyback on an already running request.
2. If there is no request running for the given `product`, we execute the original `totalSales()` function and we save the resulting promise into the `runningRequests` map. Next, we make sure to remove the same promise from the `runningRequests` map as soon as the request completes.

> 1. Nếu một promise cho `product` đã cho đã tồn tại, ta chỉ cần trả về nó. Đây là nơi ta bám theo (piggyback) một yêu cầu đang chạy.
> 2. Nếu không có yêu cầu nào đang chạy cho `product` đã cho, ta thực thi hàm `totalSales()` gốc và lưu promise kết quả vào map `runningRequests`. Tiếp theo, ta đảm bảo xóa chính promise đó khỏi map `runningRequests` ngay khi yêu cầu hoàn tất.

The behavior of the new `totalSales()` function is identical to that of the original `totalSales()` API, with the difference that, now, multiple calls to the API using the same input are batched, thus saving us time and resources.

> Hành vi của hàm `totalSales()` mới giống hệt với hành vi của API `totalSales()` gốc, với điểm khác biệt là giờ đây, nhiều lời gọi API dùng cùng một đầu vào được gộp lô, qua đó tiết kiệm cho ta thời gian và tài nguyên.

Curious to know what the performance improvement compared to the raw, non-batched version of the `totalSales()` API is? Let's then replace the `totalSales` module used by the HTTP server with the one we just created (the `app.js` file):

> Tò mò muốn biết mức cải thiện hiệu năng so với phiên bản `totalSales()` API thô, không gộp lô là gì? Vậy thì hãy thay thế module `totalSales` được máy chủ HTTP dùng bằng module ta vừa tạo (tệp `app.js`):

```js

```

If we now try to start the server again and run the load test against it, the first thing we will see is that the requests are returned in batches. This is the effect of the recipe we just implemented, and it's a great practical demonstration of how it works.

> Nếu giờ ta thử khởi động lại máy chủ và chạy bài kiểm tra tải đối với nó, điều đầu tiên ta sẽ thấy là các yêu cầu được trả về theo từng lô. Đây là hệ quả của công thức ta vừa hiện thực, và nó là một minh chứng thực tế tuyệt vời về cách nó hoạt động.

> Kết quả này chứng thực cú hích hiệu năng to lớn mà ta có thể đạt được chỉ bằng cách áp dụng một lớp batching đơn giản, mà không cần toàn bộ sự phức tạp của việc quản lý một cache đầy đủ, và quan trọng hơn, mà không phải lo lắng về các chiến lược vô hiệu hóa.

The Request Batching pattern reaches its best potential in highload applications and with slow APIs. This is because it's exactly in these circumstances that we can batch together a high number of requests.

> Mẫu Gộp lô Yêu cầu (Request Batching) đạt tiềm năng tốt nhất trong các ứng dụng tải cao và với các API chậm. Đó là vì chính trong những hoàn cảnh này, ta có thể gộp lô một số lượng lớn yêu cầu lại với nhau.

Let's now see how we can implement both batching and caching using a slight variation of the technique we've just explored.

> Giờ hãy xem cách ta có thể hiện thực cả batching và caching bằng một biến thể nhỏ của kỹ thuật ta vừa khám phá.

#### Caching requests in the total sales web server — Lưu đệm yêu cầu trong máy chủ web tổng doanh số

Adding a caching layer to our batching API is straightforward, thanks to promises. All we have to do is leave the promise in our request map, even after the request has completed. Let's implement the `totalSalesCache.js` module straightaway:

> Thêm một lớp caching vào API batching của ta rất dễ dàng, nhờ promise. Tất cả những gì ta phải làm là để promise lại trong map yêu cầu của ta, ngay cả sau khi yêu cầu đã hoàn tất. Hãy hiện thực module `totalSalesCache.js` ngay:

```js
import { totalSales as totalSalesRaw } from './totalSales.js'
const CACHE_TTL = 30 * 1000 // 30 seconds TTL
const cache = new Map()
export function totalSales (product) {
  if (cache.has(product)) {
    console.log('Cache hit')
    return cache.get(product)
  }
  const resultPromise = totalSalesRaw(product)
  cache.set(product, resultPromise)
  resultPromise.then(() => {
    setTimeout(() => {
      cache.delete(product)
    }, CACHE_TTL)
  }, err => {
    cache.delete(product)
    throw err
  })
  return resultPromise
}
```

The relevant code that enables caching is highlighted. All we have to do is remove the promise from the cache after a certain time (`CACHE_TTL`) after the request has completed, or immediately if the request has failed. This is a very basic cache invalidation technique, but it works perfectly for our demonstration.

> Mã liên quan kích hoạt caching được tô sáng. Tất cả những gì ta phải làm là xóa promise khỏi cache sau một khoảng thời gian nhất định (`CACHE_TTL`) sau khi yêu cầu đã hoàn tất, hoặc ngay lập tức nếu yêu cầu thất bại. Đây là một kỹ thuật vô hiệu hóa cache rất cơ bản, nhưng nó hoạt động hoàn hảo cho phần minh họa của ta.

Now, we are ready to try the `totalSales()` caching wrapper we just created. To do that, we only need to update the `app.js` module, as follows:

```js

  // ...

With the default test parameters, we should see a 10% reduction in the execution time compared to simple batching. Of course, this is highly dependent on a lot of factors; for example, the number of requests received and the delay between one request and the other. Lợi thế của việc dùng caching so với batching sẽ đáng kể hơn nhiều khi số lượng yêu cầu cao hơn và trải dài trên một khoảng thời gian dài hơn.

#### Notes about implementing caching mechanisms — Ghi chú về việc hiện thực các cơ chế caching

We must remember that in real-life applications, we may want to use more advanced cache invalidation techniques and storage mechanisms. This is necessary for the following reasons:

> Ta phải nhớ rằng trong các ứng dụng thực tế, ta có thể muốn dùng các kỹ thuật vô hiệu hóa cache và cơ chế lưu trữ tiên tiến hơn. Điều này cần thiết vì những lý do sau:

- A large amount of cached values can easily consume a lot of memory. In this case, a least recently used (LRU) or a first in first out (FIFO) policy can be applied to maintain constant memory utilization.
- When the application is distributed across multiple processes, keeping the cache in memory may produce different results across each server instance. If that's undesired for the particular application we are implementing, the solution is to use a shared store for the cache. This is also more performant than a simple in-memory solution as the cache is shared across multiple instances. Popular caching solutions include Redis (`nodejsdp.link/redis`) and Memcached (`nodejsdp.link/memcached`).
- A manual cache invalidation (for example, when the related non-cached value changes), as opposed to a timed expiry, can enable a longer-living cache and at the same time provide more up-to-date data, but, of course, it would be a lot more complex to manage. Let's not forget the famous quote by Phil Karlton (principal engineer at Netscape, Silicon Graphics, and more): "There are only two hard things in Computer Science: cache invalidation and naming things." With this, we conclude this section on request batching and caching. Next, we are going to learn how to tackle a tricky business: canceling asynchronous operations.

> - Một lượng lớn giá trị đã cache có thể dễ dàng tiêu tốn rất nhiều bộ nhớ. Trong trường hợp này, một chính sách ít được dùng gần đây nhất (least recently used - LRU) hoặc vào trước ra trước (first in first out - FIFO) có thể được áp dụng để duy trì mức sử dụng bộ nhớ không đổi.
> - Khi ứng dụng được phân tán trên nhiều tiến trình, việc giữ cache trong bộ nhớ có thể tạo ra các kết quả khác nhau giữa mỗi thực thể máy chủ. Nếu điều đó không mong muốn cho ứng dụng cụ thể ta đang hiện thực, giải pháp là dùng một kho lưu trữ chung (shared store) cho cache. Cách này cũng có hiệu năng cao hơn so với một giải pháp trong bộ nhớ đơn thuần vì cache được chia sẻ giữa nhiều thực thể. Các giải pháp caching phổ biến bao gồm Redis (`nodejsdp.link/redis`) và Memcached (`nodejsdp.link/memcached`).
> - Vô hiệu hóa cache thủ công (ví dụ, khi giá trị không cache liên quan thay đổi), trái ngược với hết hạn theo thời gian, có thể cho phép một cache tồn tại lâu hơn và đồng thời cung cấp dữ liệu cập nhật hơn, nhưng, tất nhiên, nó sẽ phức tạp hơn rất nhiều để quản lý. Đừng quên câu nói nổi tiếng của Phil Karlton (kỹ sư chính tại Netscape, Silicon Graphics, và hơn thế nữa): "Chỉ có hai điều khó trong Khoa học Máy tính: vô hiệu hóa cache và đặt tên cho mọi thứ." Với điều này, ta kết thúc phần về gộp lô và lưu đệm yêu cầu. Tiếp theo, ta sẽ học cách giải quyết một việc khó nhằn: hủy các thao tác bất đồng bộ.

## Canceling asynchronous operations — Hủy các thao tác bất đồng bộ

Being able to stop a long-running operation is particularly useful if the operation has been canceled by the user or if it has become redundant. In multithreaded programming, we can just terminate the **thread**, but on a **single-threaded** platform such as Node.js, things can get a little bit more complicated. In this section, we'll be talking about canceling asynchronous operations and not about canceling promises, which is a different matter altogether. By the way, the Promises/A+ standard doesn't include an API for canceling promises. However, you can use a third-party promise library such as bluebird if you need such a feature (more at `nodejsdp.link/bluebird-cancelation`). Note that canceling a promise doesn't mean that the operation the promise refers to will also be canceled; in fact, bluebird offers an `onCancel` **callback** in the promise constructor, in addition to `resolve` and `reject`, which can be used to cancel the underlying async operation when the promise is canceled. This is actually what this section is about.

> Khả năng dừng một thao tác chạy lâu đặc biệt hữu ích nếu thao tác đã bị người dùng hủy hoặc nếu nó đã trở nên dư thừa. Trong lập trình đa luồng, ta chỉ cần chấm dứt luồng (thread), nhưng trên một nền tảng đơn luồng như Node.js, mọi thứ có thể trở nên phức tạp hơn một chút. Trong phần này, ta sẽ nói về việc hủy các thao tác bất đồng bộ chứ không phải về việc hủy promise, vốn là một vấn đề hoàn toàn khác. Nhân tiện, chuẩn Promises/A+ không bao gồm một API để hủy promise. Tuy nhiên, bạn có thể dùng một thư viện promise của bên thứ ba như bluebird nếu bạn cần tính năng đó (thêm tại `nodejsdp.link/bluebird-cancelation`). Lưu ý rằng việc hủy một promise không có nghĩa là thao tác mà promise đề cập đến cũng sẽ bị hủy; thực tế, bluebird cung cấp một callback `onCancel` trong hàm khởi tạo promise, ngoài `resolve` và `reject`, có thể được dùng để hủy thao tác bất đồng bộ bên dưới khi promise bị hủy. Đây thực ra chính là điều phần này nói đến.

### A basic recipe for creating cancelable functions — Một công thức cơ bản để tạo các hàm có thể hủy

Actually, in asynchronous programming, the basic principle for canceling the execution of a function is very simple: we check if the operation has been canceled after every asynchronous call, and if that's the case, we prematurely quit the operation. Consider, for example, the following code:

> Thực ra, trong lập trình bất đồng bộ, nguyên tắc cơ bản để hủy việc thực thi một hàm rất đơn giản: ta kiểm tra xem thao tác đã bị hủy chưa sau mỗi lời gọi bất đồng bộ, và nếu đúng vậy, ta thoát khỏi thao tác sớm. Hãy xét, ví dụ, đoạn mã sau:

```js
import { asyncRoutine } from './asyncRoutine.js'
import { CancelError } from './cancelError.js'
async function cancelable (cancelObj) {
  const resA = await asyncRoutine('A')
  console.log(resA)
  if (cancelObj.cancelRequested) {
    throw new CancelError()
  }
  const resB = await asyncRoutine('B')
  console.log(resB)
  if (cancelObj.cancelRequested) {
    throw new CancelError()
  }
  const resC = await asyncRoutine('C')
  console.log(resC)
}
```

The `cancelable()` function receives, as input, an object (`cancelObj`) containing a single property called `cancelRequested`. In the function, we check the `cancelRequested` property after every asynchronous call, and if that's `true`, we throw a special `CancelError` exception to interrupt the execution of the function.

> Hàm `cancelable()` nhận vào, làm đầu vào, một đối tượng (`cancelObj`) chứa một thuộc tính duy nhất tên là `cancelRequested`. Trong hàm, ta kiểm tra thuộc tính `cancelRequested` sau mỗi lời gọi bất đồng bộ, và nếu nó là `true`, ta ném một ngoại lệ đặc biệt `CancelError` để ngắt việc thực thi của hàm.

The `asyncRoutine()` function is just a demo function that prints a string to the console and returns another string after 100 ms. You will find its full implementation, along with that of `CancelError`, in the code repository for this book.

> Hàm `asyncRoutine()` chỉ là một hàm demo in một chuỗi ra console và trả về một chuỗi khác sau 100 ms. Bạn sẽ tìm thấy phần hiện thực đầy đủ của nó, cùng với phần hiện thực của `CancelError`, trong kho mã của cuốn sách này.

It's important to note that any code external to the `cancelable()` function will be able to set the `cancelRequested` property only after the `cancelable()` function gives back control to the **event loop**, which is usually when an asynchronous operation is awaited. This is why it's worth checking the `cancelRequested` property only after the completion of an asynchronous operation and not more often.

> Điều quan trọng cần lưu ý là bất kỳ đoạn mã nào bên ngoài hàm `cancelable()` sẽ chỉ có thể thiết lập thuộc tính `cancelRequested` sau khi hàm `cancelable()` trả lại quyền điều khiển cho vòng lặp sự kiện (event loop), điều này thường xảy ra khi một thao tác bất đồng bộ được await. Đây là lý do tại sao đáng để kiểm tra thuộc tính `cancelRequested` chỉ sau khi hoàn tất một thao tác bất đồng bộ chứ không phải thường xuyên hơn.

The following code demonstrates how we can cancel the `cancelable()` function:

> Đoạn mã sau minh họa cách ta có thể hủy hàm `cancelable()`:

```js
const cancelObj = { cancelRequested: false }
cancelable(cancelObj)
  .catch(err => {
    if (err instanceof CancelError) {
      console.log('Function canceled')
    } else {
      console.error(err)
    }
  })
setTimeout(() => {
  cancelObj.cancelRequested = true
}, 100)
```

As we can see, all we have to do to cancel the function is set the `cancelObj.cancelRequested` property to `true`. This will cause the function to stop and throw a `CancelError`.

> Như ta thấy, tất cả những gì ta phải làm để hủy hàm là thiết lập thuộc tính `cancelObj.cancelRequested` thành `true`. Điều này sẽ khiến hàm dừng lại và ném ra một `CancelError`.

### Wrapping asynchronous invocations — Bao bọc các lời gọi bất đồng bộ

Creating and using a basic asynchronous cancelable function is very easy, but there is a lot of boilerplate involved. In fact, it involves so much extra code that it becomes hard to identify the actual business logic of the function.

> Tạo và sử dụng một hàm có thể hủy bất đồng bộ cơ bản rất dễ, nhưng có rất nhiều mã soạn sẵn liên quan. Thực tế, nó liên quan đến nhiều mã thừa đến mức trở nên khó nhận ra logic nghiệp vụ thực sự của hàm.

We can reduce the boilerplate by including the cancelation logic inside a wrapping function, which we can use to invoke asynchronous routines. Such a wrapper would look as follows (the `cancelWrapper.js` file):

> Ta có thể giảm bớt mã soạn sẵn bằng cách đưa logic hủy vào bên trong một hàm bao bọc (wrapping function), mà ta có thể dùng để gọi các routine bất đồng bộ. Một wrapper như vậy sẽ trông như sau (tệp `cancelWrapper.js`):

```js
import { CancelError } from './cancelError.js'
export function createCancelWrapper () {
  let cancelRequested = false
  function cancel () {
    cancelRequested = true
  }
  function cancelWrapper (func, ...args) {
    if (cancelRequested) {
      return Promise.reject(new CancelError())
    }
    return func(...args)
  }
  return { cancelWrapper, cancel }
}
```

Our wrapper is created through a **factory** function called `createCancelWrapper()`. The factory returns two functions: the wrapper function (`cancelWrapper`) and a function to trigger the cancelation of the asynchronous operation (`cancel`). This allows us to create a wrapper function to wrap multiple asynchronous invocations and then use a single `cancel()` function to cancel all of them.

> Wrapper của ta được tạo thông qua một hàm factory tên là `createCancelWrapper()`. Factory này trả về hai hàm: hàm wrapper (`cancelWrapper`) và một hàm để kích hoạt việc hủy thao tác bất đồng bộ (`cancel`). Điều này cho phép ta tạo một hàm wrapper để bao bọc nhiều lời gọi bất đồng bộ rồi dùng một hàm `cancel()` duy nhất để hủy tất cả chúng.

The `cancelWrapper()` function takes, as input, a function to invoke (`func`) and a set of parameters to pass to the function (`args`). The wrapper simply checks if a cancelation has been requested and if positive, it will return a promise rejected with a `CancelError` object as the rejection reason; otherwise, it will invoke `func`.

> Hàm `cancelWrapper()` nhận vào, làm đầu vào, một hàm để gọi (`func`) và một tập hợp các tham số để truyền cho hàm (`args`). Wrapper chỉ đơn giản kiểm tra xem việc hủy có được yêu cầu hay không và nếu có, nó sẽ trả về một promise bị reject với một đối tượng `CancelError` làm lý do reject; nếu không, nó sẽ gọi `func`.

Let's now see how our wrapper factory can greatly improve the readability and modularity of our `cancelable()` function:

> Giờ hãy xem cách factory wrapper của ta có thể cải thiện đáng kể tính dễ đọc và tính module hóa của hàm `cancelable()`:

```js
import { asyncRoutine } from './asyncRoutine.js'
import { createCancelWrapper } from './cancelWrapper.js'
import { CancelError } from './cancelError.js'
async function cancelable (cancelWrapper) {
  const resA = await cancelWrapper(asyncRoutine, 'A')
  console.log(resA)
  const resB = await cancelWrapper(asyncRoutine, 'B')
  console.log(resB)
  const resC = await cancelWrapper(asyncRoutine, 'C')
  console.log(resC)
}
const { cancelWrapper, cancel } = createCancelWrapper()
cancelable(cancelWrapper)
  .catch(err => {
    if (err instanceof CancelError) {
      console.log('Function canceled')
    } else {
      console.error(err)
    }
  })
setTimeout(() => {
  cancel()
}, 100)
```

We can immediately see the benefits of using a wrapper function for implementing our cancelation logic. In fact, the `cancelable()` function is now much more concise and **readable**.

> Ta có thể ngay lập tức thấy lợi ích của việc dùng một hàm wrapper để hiện thực logic hủy của mình. Thực tế, hàm `cancelable()` giờ đây ngắn gọn và dễ đọc hơn nhiều.

### Cancelable async functions with generators — Các hàm bất đồng bộ có thể hủy với generator

The `cancelable` wrapper function we just created is already a big step ahead compared to embedding the cancelation logic directly in our code. However, it's still not ideal for two reasons: it is error prone (what if we forget to wrap one function?) and it still affects the readability of our code, which makes it not ideal for implementing cancelable asynchronous operations that are already large and complex.

> Hàm wrapper `cancelable` ta vừa tạo đã là một bước tiến lớn so với việc nhúng logic hủy trực tiếp vào mã của ta. Tuy nhiên, nó vẫn chưa lý tưởng vì hai lý do: nó dễ gây lỗi (sẽ ra sao nếu ta quên bao bọc một hàm?) và nó vẫn ảnh hưởng đến tính dễ đọc của mã, điều khiến nó không lý tưởng để hiện thực các thao tác bất đồng bộ có thể hủy vốn đã lớn và phức tạp.

An even neater solution involves the use of generators. In Chapter 9, Behavioral Design Patterns, we introduced generators as a means to implement iterators. However, they are a very versatile tool and can be used to implement all sorts of algorithms. In this case, we will be using generators to build a supervisor to control the asynchronous flow of a function. The result will be an asynchronous function that is transparently cancelable, whose behavior resembles an async function in which the `await` instruction is replaced by `yield`. Let's see how we can implement this cancelable function using generators (the `createAsyncCancelable.js` file):

> Một giải pháp gọn gàng hơn nữa liên quan đến việc dùng generator. Ở Chương 9, Behavioral Design Patterns, ta đã giới thiệu generator như một phương tiện để hiện thực iterator. Tuy nhiên, chúng là một công cụ rất đa năng và có thể dùng để hiện thực đủ loại thuật toán. Trong trường hợp này, ta sẽ dùng generator để xây dựng một bộ giám sát (supervisor) nhằm điều khiển luồng bất đồng bộ của một hàm. Kết quả sẽ là một hàm bất đồng bộ có thể hủy một cách minh bạch, có hành vi giống một hàm async trong đó lệnh `await` được thay bằng `yield`. Hãy xem cách ta có thể hiện thực hàm có thể hủy này dùng generator (tệp `createAsyncCancelable.js`):

```js
import { CancelError } from './cancelError.js'
export function createAsyncCancelable (generatorFunction) {   // (1)
  return function asyncCancelable (...args) {
    const generatorObject = generatorFunction(...args)        // (3)
    let cancelRequested = false
    function cancel () {
      cancelRequested = true
    }
    const promise = new Promise((resolve, reject) => {
      async function nextStep (prevResult) {                  // (4)
        if (cancelRequested) {
          return reject(new CancelError())
        }
        if (prevResult.done) {
          return resolve(prevResult.value)
        }
        try {                                                 // (5)
          nextStep(generatorObject.next(await prevResult.value))
        } catch (err) {
          try {                                               // (6)
            nextStep(generatorObject.throw(err))
          } catch (err2) {
            reject(err2)
          }
        }
      }
      nextStep({})
    })
    return { promise, cancel }                                // (2)
  }
}
```

The `createAsyncCancelable()` function may seem complex, so let's analyze it in more detail:

> Hàm `createAsyncCancelable()` có thể trông phức tạp, vậy hãy phân tích nó chi tiết hơn:

1. First, we should note that the `createAsyncCancelable()` function takes, as input, a **generator** function (the supervised function) and returns another function (`asyncCancelable()`) that wraps the generator function with our supervising logic. The `asyncCancelable()` function is what we will use to invoke the asynchronous operation.
2. The `asyncCancelable()` function returns an object with two properties: a. The `promise` property, which contains the promise representing the eventual resolution (or rejection) of the asynchronous operation. b. The `cancel` property, which is a function that can be used to cancel the supervised asynchronous flow.
3. When invoked, the first task of `asyncCancelable()` is to invoke the generator function with the arguments received as input (`args`) and obtain a generator object, which we can use to control the execution flow of the running coroutine.
4. The entire logic of the supervisor is implemented within the `nextStep()` function, which is responsible for iterating over the values yielded by the supervised coroutine (`prevResult`). Those can be actual values or promises. If a cancelation is requested, we throw the usual `CancelError`; otherwise, if the coroutine has been terminated (for example, `prevResult.done` is `true`), we immediately resolve the outer promise and complete the return.
5. The core part of the `nextStep()` function is where we retrieve the next value yielded by the supervised coroutine (which, let's not forget, it's a generator). We `await` on that value so we can make sure we get the actual resolution value in case we are dealing with a promise. This also makes sure that if `prevResult.value` is a promise and it rejects, we end up in the `catch` statement. We can end up in the `catch` statement even if the supervised coroutine actually throws an exception.
6. In the `catch` statement, we throw the caught error inside the coroutine. This is redundant if that error was already thrown by the coroutine, but not if it's the result of a promise rejection. Even if not optimal, this trick can simplify our code a bit for the sake of this demonstration. We invoke `nextStep()` using whatever value is yielded next after throwing it inside the coroutine, but if the result is another exception (for example, the exception is not caught inside the coroutine or another one is thrown), we immediately reject the outer promise and complete the asynchronous operation.

> 1. Trước tiên, ta nên lưu ý rằng hàm `createAsyncCancelable()` nhận vào, làm đầu vào, một hàm generator (hàm được giám sát) và trả về một hàm khác (`asyncCancelable()`) bao bọc hàm generator với logic giám sát của ta. Hàm `asyncCancelable()` là thứ ta sẽ dùng để gọi thao tác bất đồng bộ.
> 2. Hàm `asyncCancelable()` trả về một đối tượng với hai thuộc tính: a. Thuộc tính `promise`, chứa promise đại diện cho việc giải quyết (hoặc reject) cuối cùng của thao tác bất đồng bộ. b. Thuộc tính `cancel`, là một hàm có thể dùng để hủy luồng bất đồng bộ được giám sát.
> 3. Khi được gọi, nhiệm vụ đầu tiên của `asyncCancelable()` là gọi hàm generator với các đối số nhận được làm đầu vào (`args`) và lấy về một đối tượng generator, mà ta có thể dùng để điều khiển luồng thực thi của coroutine đang chạy.
> 4. Toàn bộ logic của bộ giám sát được hiện thực bên trong hàm `nextStep()`, chịu trách nhiệm lặp qua các giá trị được yield bởi coroutine được giám sát (`prevResult`). Đó có thể là các giá trị thực hoặc promise. Nếu một yêu cầu hủy được đưa ra, ta ném `CancelError` quen thuộc; nếu không, nếu coroutine đã kết thúc (ví dụ, `prevResult.done` là `true`), ta ngay lập tức resolve promise bên ngoài và hoàn tất việc trả về.
> 5. Phần cốt lõi của hàm `nextStep()` là nơi ta lấy về giá trị tiếp theo được yield bởi coroutine được giám sát (mà, đừng quên, nó là một generator). Ta `await` trên giá trị đó để có thể đảm bảo ta nhận được giá trị giải quyết thực sự trong trường hợp ta đang xử lý một promise. Điều này cũng đảm bảo rằng nếu `prevResult.value` là một promise và nó reject, ta sẽ rơi vào câu lệnh `catch`. Ta cũng có thể rơi vào câu lệnh `catch` ngay cả khi coroutine được giám sát thực sự ném ra một ngoại lệ.
> 6. Trong câu lệnh `catch`, ta ném lỗi đã bắt được vào bên trong coroutine. Điều này dư thừa nếu lỗi đó đã được ném bởi coroutine, nhưng không dư thừa nếu nó là kết quả của một promise bị reject. Dù không tối ưu, mẹo này có thể đơn giản hóa mã của ta một chút cho mục đích minh họa này. Ta gọi `nextStep()` dùng bất kỳ giá trị nào được yield tiếp theo sau khi ném nó vào bên trong coroutine, nhưng nếu kết quả là một ngoại lệ khác (ví dụ, ngoại lệ không được bắt bên trong coroutine hoặc một ngoại lệ khác được ném ra), ta ngay lập tức reject promise bên ngoài và hoàn tất thao tác bất đồng bộ.

As we saw, there are a lot of moving parts in the `createAsyncCancelable()` function. But we should appreciate the fact that, in just a few lines of code, we were able to create a cancelable function that doesn't require any manual cancelation logic. As we will see now, the results are impressive.

> Như ta đã thấy, có rất nhiều thành phần chuyển động trong hàm `createAsyncCancelable()`. Nhưng ta nên trân trọng thực tế rằng, chỉ trong vài dòng mã, ta đã có thể tạo một hàm có thể hủy không đòi hỏi bất kỳ logic hủy thủ công nào. Như ta sẽ thấy ngay bây giờ, kết quả thật ấn tượng.

Let's rewrite our sample asynchronous cancelable operation using the supervisor we implemented in the `createAsyncCancelable()` function:

> Hãy viết lại thao tác bất đồng bộ có thể hủy mẫu của ta dùng bộ giám sát ta đã hiện thực trong hàm `createAsyncCancelable()`:

```js
import { asyncRoutine } from './asyncRoutine.js'
import { createAsyncCancelable } from './createAsyncCancelable.js'
import { CancelError } from './cancelError.js'
const cancelable = createAsyncCancelable(function * () {
  const resA = yield asyncRoutine('A')
  console.log(resA)
  const resB = yield asyncRoutine('B')
  console.log(resB)
  const resC = yield asyncRoutine('C')
  console.log(resC)
})
const { promise, cancel } = cancelable()
promise.catch(err => {
  if (err instanceof CancelError) {
    console.log('Function canceled')
  } else {
    console.error(err)
  }
})
setTimeout(() => {
  cancel()
}, 100)
```

We can immediately see that the generator wrapped by `createAsyncCancelable()` closely resembles an async function, but we are using `yield` instead of `await`. Also, there is no visible cancelation logic at all. The generator function maintains the excellent properties of async functions (for example, to make asynchronous code look like synchronous code), but unlike the async function and thanks to the supervisor introduced by `createAsyncCancelable()`, it's also possible to cancel the operation. The second interesting aspect is that `createAsyncCancelable()` creates a function (called `cancelable`) that can be invoked like any other function but at the same time returns a promise representing the result of the operation and a function to cancel the operation.

> Ta có thể ngay lập tức thấy rằng generator được bao bọc bởi `createAsyncCancelable()` rất giống một hàm async, nhưng ta đang dùng `yield` thay vì `await`. Ngoài ra, hoàn toàn không có logic hủy nào hiện rõ. Hàm generator duy trì các thuộc tính tuyệt vời của các hàm async (ví dụ, làm cho mã bất đồng bộ trông như mã đồng bộ), nhưng khác với hàm async và nhờ bộ giám sát được giới thiệu bởi `createAsyncCancelable()`, nó cũng có thể hủy thao tác. Khía cạnh thú vị thứ hai là `createAsyncCancelable()` tạo ra một hàm (gọi là `cancelable`) có thể được gọi như bất kỳ hàm nào khác nhưng đồng thời trả về một promise đại diện cho kết quả của thao tác và một hàm để hủy thao tác.

This technique of using generators represents the best option we have to implement cancelable asynchronous operations.

> Kỹ thuật dùng generator này đại diện cho lựa chọn tốt nhất ta có để hiện thực các thao tác bất đồng bộ có thể hủy.

For use in production, most of the time, we can rely on a widely used package from the Node.js ecosystem such as `caf` (the acronym means Cancelable Async Flows), which you can find at `nodejsdp.link/caf`.

> Để dùng trong production, hầu hết thời gian, ta có thể dựa vào một gói được sử dụng rộng rãi từ hệ sinh thái Node.js như `caf` (viết tắt của Cancelable Async Flows), mà bạn có thể tìm thấy tại `nodejsdp.link/caf`.

## Running CPU-bound tasks — Chạy các tác vụ chiếm dụng CPU

The `totalSales()` API that we implemented in the Asynchronous request batching and caching section was (intentionally) expensive in terms of resources and took a few hundred milliseconds to run. Nonetheless, invoking the `totalSales()` function did not affect the ability of the application to process concurrent incoming requests. What we learned about the event loop in Chapter 1, The Node.js Platform, should explain this behavior: invoking an asynchronous operation always causes the stack to unwind back to the event loop, leaving it free to handle other requests.

> API `totalSales()` mà ta đã hiện thực trong phần Gộp lô và lưu đệm yêu cầu bất đồng bộ vốn (cố ý) tốn kém về mặt tài nguyên và mất vài trăm mili giây để chạy. Tuy nhiên, việc gọi hàm `totalSales()` không ảnh hưởng đến khả năng xử lý các yêu cầu đến đồng thời của ứng dụng. Những gì ta đã học về vòng lặp sự kiện ở Chương 1, The Node.js Platform, sẽ giải thích hành vi này: việc gọi một thao tác bất đồng bộ luôn khiến ngăn xếp (stack) cuộn lại về vòng lặp sự kiện, để nó rảnh rỗi xử lý các yêu cầu khác.

But what happens when we run a synchronous task that takes a long time to complete and that never gives back the control to the event loop until it has finished? This kind of task is also known as CPU-bound, because its main characteristic is that it is heavy on CPU utilization rather than being heavy on I/O operations.

> Nhưng điều gì xảy ra khi ta chạy một tác vụ đồng bộ mất nhiều thời gian để hoàn tất và không bao giờ trả lại quyền điều khiển cho vòng lặp sự kiện cho đến khi nó kết thúc? Loại tác vụ này còn được gọi là chiếm dụng CPU (CPU-bound), vì đặc điểm chính của nó là nó nặng về mức sử dụng CPU chứ không phải nặng về các thao tác I/O.

Let's work immediately on an example to see how these types of task behave in Node.js.

> Hãy bắt tay ngay vào một ví dụ để xem các loại tác vụ này hành xử như thế nào trong Node.js.

### Solving the subset sum problem — Giải bài toán tổng tập con

Let's now choose a computationally expensive problem to use as a base for our experiment. A good candidate is the subset sum problem, which decides whether a set (or multiset) of integers contains a non-empty subset that has a sum equal to zero. For example, if we had as input the set [1, 2, –4, 5, –3], the subsets satisfying the problem are [1, 2, –3] and [2, –4, 5, –3]. The simplest algorithm is the one that checks every possible combination of subsets of any size and has a computational cost of O(2n), or in other words, it grows exponentially with the size of the input. This means that a set of 20 integers would require up to 1,048,576 combinations to be checked, which is not bad for testing our assumptions. For our example, we are going to consider the following variation of the subset sum problem: given a set of integers, we want to calculate all the possible combinations whose sum is equal to a given arbitrary integer, not just zero.

> Giờ hãy chọn một bài toán tốn kém về mặt tính toán để dùng làm cơ sở cho thí nghiệm của ta. Một ứng cử viên tốt là bài toán tổng tập con (subset sum problem), quyết định xem một tập hợp (hoặc đa tập - multiset) các số nguyên có chứa một tập con không rỗng có tổng bằng không hay không. Ví dụ, nếu ta có đầu vào là tập hợp [1, 2, –4, 5, –3], các tập con thỏa mãn bài toán là [1, 2, –3] và [2, –4, 5, –3]. Thuật toán đơn giản nhất là thuật toán kiểm tra mọi tổ hợp có thể của các tập con với bất kỳ kích thước nào và có chi phí tính toán O(2n), hay nói cách khác, nó tăng theo cấp số mũ với kích thước của đầu vào. Điều này nghĩa là một tập hợp gồm 20 số nguyên sẽ đòi hỏi kiểm tra tới 1.048.576 tổ hợp, điều không tệ để kiểm chứng các giả định của ta. Cho ví dụ của mình, ta sẽ xét biến thể sau của bài toán tổng tập con: cho một tập hợp các số nguyên, ta muốn tính tất cả các tổ hợp có thể có tổng bằng một số nguyên tùy ý cho trước, chứ không chỉ bằng không.

Now, let's work to implement such an algorithm. First, let's create a new module called `subsetSum.js`. We will start by creating a class called `SubsetSum`:

> Giờ hãy bắt tay vào hiện thực một thuật toán như vậy. Trước tiên, hãy tạo một module mới tên là `subsetSum.js`. Ta sẽ bắt đầu bằng việc tạo một lớp tên là `SubsetSum`:

```js
export class SubsetSum extends EventEmitter {
  constructor (sum, set) {
    super()
    this.sum = sum
    this.set = set
    this.totalSubsets = 0
  }
//...
```

The `SubsetSum` class is extending `EventEmitter`. This allows us to produce an event every time we find a new subset matching the sum received as input. As we will see, this will give us a lot of flexibility.

> Lớp `SubsetSum` mở rộng `EventEmitter`. Điều này cho phép ta phát ra một sự kiện mỗi khi ta tìm thấy một tập con mới khớp với tổng nhận được làm đầu vào. Như ta sẽ thấy, điều này sẽ cho ta rất nhiều sự linh hoạt.

Next, let's see how we can generate all the possible combinations of subsets:

> Tiếp theo, hãy xem cách ta có thể sinh tất cả các tổ hợp có thể của các tập con:

```js
_combine (set, subset) {
  for (let i = 0; i < set.length; i++) {
    const newSubset = subset.concat(set[i])
    this._combine(set.slice(i + 1), newSubset)
    this._processSubset(newSubset)
  }
}
```

We will not go into too much detail about the algorithm, but there are two important things to notice:

> Ta sẽ không đi quá sâu vào chi tiết về thuật toán, nhưng có hai điều quan trọng cần lưu ý:

- The `_combine()` method is completely synchronous. It recursively generates every possible subset without ever giving back control to the event loop.
- Every time a new combination is generated, we provide it to the `_processSubset()` method for further processing.

> - Phương thức `_combine()` hoàn toàn đồng bộ. Nó đệ quy sinh ra mọi tập con có thể mà không bao giờ trả lại quyền điều khiển cho vòng lặp sự kiện.
> - Mỗi khi một tổ hợp mới được sinh ra, ta cung cấp nó cho phương thức `_processSubset()` để xử lý thêm.

The `_processSubset()` method is responsible for verifying that the sum of the elements of the given subset is equal to the number we are looking for:

> Phương thức `_processSubset()` chịu trách nhiệm kiểm chứng rằng tổng các phần tử của tập con đã cho bằng với số ta đang tìm kiếm:

```js
_processSubset (subset) {
  console.log('Subset', ++this.totalSubsets, subset)
  const res = subset.reduce((prev, item) => (prev + item), 0)
  if (res === this.sum) {
    this.emit('match', subset)
  }
}
```

Trivially, the `_processSubset()` method applies a `reduce` operation to the subset in order to calculate the sum of its elements. Then, it emits an event of the type `match` when the resulting sum is equal to the one we are interested in finding (`this.sum`).

> Một cách đơn giản, phương thức `_processSubset()` áp dụng một thao tác `reduce` lên tập con để tính tổng các phần tử của nó. Sau đó, nó phát ra một sự kiện loại `match` khi tổng kết quả bằng với tổng ta muốn tìm (`this.sum`).

Finally, the `start()` method puts all the preceding pieces together:

> Cuối cùng, phương thức `start()` ghép tất cả các phần trên lại với nhau:

```js
start () {
  this._combine(this.set, [])
  this.emit('end')
}
```

The `start()` method triggers the generation of all the combinations by invoking `_combine()`, and lastly, emits an `end` event, signaling that all the combinations were checked and any possible match has already been emitted. This is possible because `_combine()` is synchronous; therefore, the `end` event is emitted as soon as the function returns, which means that all the combinations have been calculated.

> Phương thức `start()` kích hoạt việc sinh tất cả các tổ hợp bằng cách gọi `_combine()`, và cuối cùng, phát ra một sự kiện `end`, báo hiệu rằng tất cả các tổ hợp đã được kiểm tra và bất kỳ kết quả khớp nào có thể đã được phát ra. Điều này khả thi vì `_combine()` là đồng bộ; do đó, sự kiện `end` được phát ra ngay khi hàm trả về, nghĩa là tất cả các tổ hợp đã được tính toán.

Next, we have to expose the algorithm we just created over the network. As always, we can use a simple HTTP server for this task. In particular, we want to create an endpoint that accepts requests in the format `/subsetSum?data=<Array>&sum=<Integ` `er>` that invokes the `SubsetSum` algorithm with the given array of integers and sum to match.

> Tiếp theo, ta phải phơi bày thuật toán ta vừa tạo qua mạng. Như mọi khi, ta có thể dùng một máy chủ HTTP đơn giản cho tác vụ này. Cụ thể, ta muốn tạo một endpoint chấp nhận các yêu cầu theo định dạng `/subsetSum?data=<Array>&sum=<Integ` `er>` gọi thuật toán `SubsetSum` với mảng số nguyên đã cho và tổng cần khớp.

Let's implement this simple server in a module named `index.js`:

> Hãy hiện thực máy chủ đơn giản này trong một module tên là `index.js`:

```js
import { createServer } from 'http'
import { SubsetSum } from './subsetSum.js'
createServer((req, res) => {
  const url = new URL(req.url, 'http://localhost')
  if (url.pathname !== '/subsetSum') {
    res.writeHead(200)
    return res.end('I\'m alive!\n')
  }
  const data = JSON.parse(url.searchParams.get('data'))
  const sum = JSON.parse(url.searchParams.get('sum'))
  res.writeHead(200)
  const subsetSum = new SubsetSum(sum, data)
  subsetSum.on('match', match => {
    res.write(`Match: ${JSON.stringify(match)}\n`)
  })
  subsetSum.on('end', () => res.end())
  subsetSum.start()
}).listen(8000, () => console.log('Server started'))
```

Thanks to the fact that the `SubsetSum` object returns its results using events, we can **stream** the matching subsets as soon as they are generated by the algorithm, in real time. Another detail to mention is that our server responds with the text `I'm` `alive!` every time we hit a URL different than `/subsetSum`. We will use this for checking the responsiveness of our server, as we will see in a moment.

> Nhờ thực tế là đối tượng `SubsetSum` trả về kết quả của nó bằng sự kiện, ta có thể truyền (stream) các tập con khớp ngay khi chúng được sinh ra bởi thuật toán, trong thời gian thực. Một chi tiết khác cần đề cập là máy chủ của ta phản hồi với văn bản `I'm` `alive!` mỗi khi ta truy cập một URL khác với `/subsetSum`. Ta sẽ dùng điều này để kiểm tra khả năng đáp ứng của máy chủ, như ta sẽ thấy trong giây lát.

We are now ready to try our subset sum algorithm. Curious to know how our server will handle it? Let's fire it up, then:

> Giờ ta đã sẵn sàng thử thuật toán tổng tập con của mình. Tò mò muốn biết máy chủ của ta sẽ xử lý nó như thế nào? Vậy thì hãy khởi động nó lên:

```js

```

As soon as the server starts, we are ready to send our first request. Let's try it with a multiset of 17 random numbers, which will result in the generation of 131,071 combinations, a nice amount to keep our server busy for a while:

> Ngay khi máy chủ khởi động, ta đã sẵn sàng gửi yêu cầu đầu tiên. Hãy thử nó với một đa tập gồm 17 số ngẫu nhiên, điều sẽ dẫn đến việc sinh ra 131.071 tổ hợp, một lượng vừa đủ để giữ máy chủ của ta bận rộn một lúc:

```js
curl -G http://localhost:8000/subsetSum --data-urlencode "data=[16,
19,1,1,-16,9,1,-5,-2,17,-15,-97,19,-16,-4,-5,15]" --data-urlencode
"sum=0"
```

After a few seconds, we should see the results coming from the server. But if we try the following command in another terminal while the first request is still running, we will spot a huge problem:

> Sau vài giây, ta sẽ thấy các kết quả đến từ máy chủ. Nhưng nếu ta thử lệnh sau trong một terminal khác trong khi yêu cầu đầu tiên vẫn đang chạy, ta sẽ phát hiện ra một vấn đề lớn:

```js
curl -G http://localhost:8000
```

We will immediately see that this last request hangs until the subset sum algorithm of the first request has finished: the server is unresponsive! This was actually expected. The Node.js event loop runs in a single thread, and if this thread is blocked by a long synchronous computation, it will be unable to execute even a single cycle in order to respond with a simple `I'm alive!` We quickly understand that this behavior does not work for any kind of application meant to process multiple concurrent requests. But don't despair. In Node.js, we can tackle this type of situation in several ways. So, let's analyze the three most popular methods, which are "interleaving with `setImmediate`," "using external processes," and "using worker threads."

> Ta sẽ ngay lập tức thấy rằng yêu cầu cuối này bị treo cho đến khi thuật toán tổng tập con của yêu cầu đầu tiên kết thúc: máy chủ không phản hồi! Điều này thực ra đã được dự đoán. Vòng lặp sự kiện của Node.js chạy trong một luồng đơn, và nếu luồng này bị chặn bởi một phép tính đồng bộ dài, nó sẽ không thể thực thi dù chỉ một chu kỳ để phản hồi với một câu `I'm alive!` đơn giản. Ta nhanh chóng hiểu rằng hành vi này không phù hợp với bất kỳ loại ứng dụng nào nhằm xử lý nhiều yêu cầu đồng thời. Nhưng đừng tuyệt vọng. Trong Node.js, ta có thể giải quyết loại tình huống này theo nhiều cách. Vậy, hãy phân tích ba phương pháp phổ biến nhất, đó là "đan xen với `setImmediate`", "dùng tiến trình bên ngoài", và "dùng worker thread".

### Interleaving with setImmediate — Đan xen với setImmediate

Usually, a CPU-bound algorithm is built upon a set of steps. This can be a set of recursive invocations, a loop, or any variation/combination of these. So, a simple solution to our problem would be to give back the control to the event loop after each of these steps completes (or after a certain number of them). This way, any pending I/O can still be processed by the event loop in those intervals in which the long-running algorithm yields the CPU. A simple way to achieve this is to schedule the next step of the algorithm to run after any pending I/O requests. This sounds like the perfect use case for the `setImmediate()` function (we already introduced this API in Chapter 3, Callbacks and Events).

> Thông thường, một thuật toán chiếm dụng CPU được xây dựng dựa trên một tập hợp các bước. Đây có thể là một tập hợp các lời gọi đệ quy, một vòng lặp, hoặc bất kỳ biến thể/kết hợp nào của những thứ này. Vậy, một giải pháp đơn giản cho bài toán của ta là trả lại quyền điều khiển cho vòng lặp sự kiện sau khi mỗi bước này hoàn tất (hoặc sau một số bước nhất định). Bằng cách này, bất kỳ I/O đang chờ nào vẫn có thể được vòng lặp sự kiện xử lý trong những khoảng thời gian mà thuật toán chạy lâu nhường lại CPU. Một cách đơn giản để đạt được điều này là lên lịch chạy bước tiếp theo của thuật toán sau bất kỳ yêu cầu I/O đang chờ nào. Điều này nghe có vẻ như là trường hợp dùng hoàn hảo cho hàm `setImmediate()` (ta đã giới thiệu API này ở Chương 3, Callbacks and Events).

#### Interleaving the steps of the subset sum algorithm — Đan xen các bước của thuật toán tổng tập con

Let's now see how this technique applies to our subset sum algorithm. All we have to do is slightly modify the `subsetSum.js` module. For convenience, we are going to create a new module called `subsetSumDefer.js`, taking the code of the original `subsetSum` class as a starting point.

> Giờ hãy xem kỹ thuật này áp dụng cho thuật toán tổng tập con của ta như thế nào. Tất cả những gì ta phải làm là chỉnh sửa nhẹ module `subsetSum.js`. Cho tiện, ta sẽ tạo một module mới tên là `subsetSumDefer.js`, lấy mã của lớp `subsetSum` gốc làm điểm khởi đầu.

The first change we are going to make is to add a new method called `_combineInterleaved()`, which is the core of the technique we are implementing:

> Thay đổi đầu tiên ta sẽ thực hiện là thêm một phương thức mới tên là `_combineInterleaved()`, phần cốt lõi của kỹ thuật ta đang hiện thực:

```js
_combineInterleaved (set, subset) {
  this.runningCombine++
  setImmediate(() => {
    this._combine(set, subset)
    if (--this.runningCombine === 0) {
      this.emit('end')
    }
  })
}
```

As we can see, all we had to do is defer the invocation of the original (synchronous) `_combine()` method with `setImmediate()`. However, now, it becomes more difficult to know when the function has finished generating all the combinations, because the algorithm is not synchronous anymore. To fix this, we have to keep track of all the running instances of the `_combine()` method using a pattern very similar to the asynchronous **parallel execution** flow that we saw in Chapter 4, Asynchronous Control Flow Patterns with Callbacks. When all the instances of the `_combine()` method have finished running, we can emit the `end` event, notifying any listener that the process has completed.

> Như ta thấy, tất cả những gì ta phải làm là trì hoãn việc gọi phương thức `_combine()` (đồng bộ) gốc bằng `setImmediate()`. Tuy nhiên, giờ đây, việc biết khi nào hàm đã hoàn tất việc sinh tất cả các tổ hợp trở nên khó hơn, vì thuật toán không còn đồng bộ nữa. Để khắc phục điều này, ta phải theo dõi tất cả các thực thể đang chạy của phương thức `_combine()` dùng một mẫu rất giống với luồng thực thi song song bất đồng bộ mà ta đã thấy ở Chương 4, Asynchronous Control Flow Patterns with Callbacks. Khi tất cả các thực thể của phương thức `_combine()` đã chạy xong, ta có thể phát ra sự kiện `end`, thông báo cho bất kỳ listener nào rằng tiến trình đã hoàn tất.

To finish refactoring the subset sum algorithm, we need to make a couple more tweaks. First, we need to replace the recursive step in the `_combine()` method with its deferred counterpart:

> Để hoàn tất việc tái cấu trúc thuật toán tổng tập con, ta cần thực hiện thêm một vài điều chỉnh. Trước tiên, ta cần thay thế bước đệ quy trong phương thức `_combine()` bằng phiên bản trì hoãn của nó:

```js
_combine (set, subset) {
  for (let i = 0; i < set.length; i++) {
    const newSubset = subset.concat(set[i])
    this._combineInterleaved(set.slice(i + 1), newSubset)
    this._processSubset(newSubset)
  }
}
```

With the preceding change, we make sure that each step of the algorithm will be queued in the event loop using `setImmediate()` and, therefore, executed after any pending I/O request instead of being run synchronously.

> Với thay đổi trên, ta đảm bảo rằng mỗi bước của thuật toán sẽ được xếp hàng trong vòng lặp sự kiện dùng `setImmediate()` và, do đó, được thực thi sau bất kỳ yêu cầu I/O đang chờ nào thay vì được chạy đồng bộ.

The other small tweak is in the `start()` method:

> Điều chỉnh nhỏ khác là trong phương thức `start()`:

```js
start () {
  this.runningCombine = 0
  this._combineInterleaved(this.set, [])
}
```

In the preceding code, we initialized the number of running instances of the `_combine()` method to `0`. We also replaced the call to `_combine()` with a call to `_combineInterleaved()` and removed the emission of the `end` event because this is now handled asynchronously in `_combineInterleaved()`.

> Trong đoạn mã trên, ta khởi tạo số lượng thực thể đang chạy của phương thức `_combine()` thành `0`. Ta cũng thay thế lời gọi tới `_combine()` bằng một lời gọi tới `_combineInterleaved()` và xóa việc phát ra sự kiện `end` vì điều này giờ được xử lý bất đồng bộ trong `_combineInterleaved()`.

With this last change, our subset sum algorithm should now be able to run its CPUbound code in steps interleaved by intervals, where the event loop can run and process any other pending request.

> Với thay đổi cuối này, thuật toán tổng tập con của ta giờ sẽ có thể chạy mã chiếm dụng CPU của nó theo các bước được đan xen bởi các khoảng thời gian, nơi vòng lặp sự kiện có thể chạy và xử lý bất kỳ yêu cầu đang chờ nào khác.

The last missing bit is updating the `index.js` module so that it can use the new version of the `SubsetSum` API. This is actually a trivial change:

> Phần cuối còn thiếu là cập nhật module `index.js` để nó có thể dùng phiên bản mới của API `SubsetSum`. Đây thực ra là một thay đổi tầm thường:

```js

  // ...
```

#### Considerations on the interleaving approach — Những cân nhắc về cách tiếp cận đan xen

As we saw, running a CPU-bound task while preserving the responsiveness of an application is not that complicated; it just requires the use of `setImmediate()` to schedule the next step of an algorithm to run after any pending I/O. However, this is not the best recipe in terms of efficiency. In fact, deferring a task introduces a small overhead that, multiplied by all the steps that an algorithm has to run, can have a significant impact on the overall running time. This is usually the last thing we want when running a CPU-bound task. A possible solution to mitigate this problem would be using `setImmediate()` only after a certain number of steps—instead of using it at every single step—but still, this would not solve the root of the problem.

> Như ta đã thấy, chạy một tác vụ chiếm dụng CPU trong khi vẫn giữ khả năng đáp ứng của một ứng dụng không quá phức tạp; nó chỉ đòi hỏi dùng `setImmediate()` để lên lịch chạy bước tiếp theo của thuật toán sau bất kỳ I/O đang chờ nào. Tuy nhiên, đây không phải là công thức tốt nhất về mặt hiệu quả. Thực tế, việc trì hoãn một tác vụ tạo ra một chi phí phụ nhỏ mà, khi nhân với tất cả các bước mà một thuật toán phải chạy, có thể có tác động đáng kể lên tổng thời gian chạy. Đây thường là điều cuối cùng ta muốn khi chạy một tác vụ chiếm dụng CPU. Một giải pháp khả thi để giảm nhẹ vấn đề này là chỉ dùng `setImmediate()` sau một số bước nhất định - thay vì dùng nó ở mỗi bước đơn lẻ - nhưng dù vậy, điều này sẽ không giải quyết được gốc rễ của vấn đề.

Also, this technique doesn't work very well if each step takes a long time to run. In this case, in fact, the event loop would lose responsiveness, and the whole application would start lagging, which is undesirable in a production environment.

> Ngoài ra, kỹ thuật này không hoạt động tốt lắm nếu mỗi bước mất nhiều thời gian để chạy. Trong trường hợp này, thực tế, vòng lặp sự kiện sẽ mất khả năng đáp ứng, và toàn bộ ứng dụng sẽ bắt đầu bị lag, điều không mong muốn trong môi trường production.

Bear in mind that this does not mean that the technique we have just seen should be avoided at all costs. In certain situations in which the synchronous task is executed sporadically and doesn't take too long to run, using `setImmediate()` to interleave its execution is sometimes the simplest and most effective way to avoid blocking the event loop. Note that `process.nextTick()` cannot be used to interleave a long-running task. As we saw in Chapter 3, Callbacks and Events, `nextTick()` schedules a task to run before any pending I/O, and this will cause I/O starvation in case of repeated calls. You can verify this yourself by replacing `setImmediate()` with `process.nextTick()` in the previous example.

> Hãy nhớ rằng điều này không có nghĩa là kỹ thuật ta vừa thấy nên bị tránh bằng mọi giá. Trong một số tình huống nhất định khi tác vụ đồng bộ được thực thi không thường xuyên và không mất quá nhiều thời gian để chạy, dùng `setImmediate()` để đan xen việc thực thi của nó đôi khi là cách đơn giản và hiệu quả nhất để tránh chặn vòng lặp sự kiện. Lưu ý rằng `process.nextTick()` không thể dùng để đan xen một tác vụ chạy lâu. Như ta đã thấy ở Chương 3, Callbacks and Events, `nextTick()` lên lịch chạy một tác vụ trước bất kỳ I/O đang chờ nào, và điều này sẽ gây ra tình trạng đói I/O (I/O starvation) trong trường hợp gọi lặp lại. Bạn có thể tự kiểm chứng điều này bằng cách thay thế `setImmediate()` bằng `process.nextTick()` trong ví dụ trước.

### Using external processes — Dùng tiến trình bên ngoài

Deferring the steps of an algorithm is not the only option we have for running CPU-bound tasks. Another pattern for preventing the event loop from blocking is using child processes. We already know that Node.js gives its best when running I/O-intensive applications such as web servers, which allows us to optimize resource utilization thanks to its asynchronous architecture. So, the best way we have to maintain the responsiveness of an application is to not run expensive CPU-bound tasks in the context of the main application and, instead, use separate processes. This has three main advantages:

> Trì hoãn các bước của một thuật toán không phải là lựa chọn duy nhất ta có để chạy các tác vụ chiếm dụng CPU. Một mẫu khác để ngăn vòng lặp sự kiện bị chặn là dùng các tiến trình con (child process). Ta đã biết rằng Node.js cho ra điều tốt nhất của nó khi chạy các ứng dụng nặng I/O như máy chủ web, điều cho phép ta tối ưu mức sử dụng tài nguyên nhờ kiến trúc bất đồng bộ của nó. Vậy, cách tốt nhất ta có để duy trì khả năng đáp ứng của một ứng dụng là không chạy các tác vụ chiếm dụng CPU tốn kém trong ngữ cảnh của ứng dụng chính mà, thay vào đó, dùng các tiến trình riêng biệt. Điều này có ba lợi thế chính:

- The synchronous task can run at full speed, without the need to interleave the steps of its execution.
- Working with processes in Node.js is simple, probably easier than modifying an algorithm to use `setImmediate()`, and allows us to easily use multiple processors without the need to scale the main application itself.
- If we really need maximum performance, the external process could be created in lower-level languages, such as good old C or more modern compiled languages like Go or Rust. Always use the best tool for the job!

> - Tác vụ đồng bộ có thể chạy với tốc độ tối đa, mà không cần phải đan xen các bước thực thi của nó.
> - Làm việc với các tiến trình trong Node.js rất đơn giản, có lẽ dễ hơn việc sửa đổi một thuật toán để dùng `setImmediate()`, và cho phép ta dễ dàng dùng nhiều bộ xử lý mà không cần mở rộng quy mô chính ứng dụng chính.
> - Nếu ta thực sự cần hiệu năng tối đa, tiến trình bên ngoài có thể được tạo bằng các ngôn ngữ cấp thấp hơn, như C cũ kỹ hoặc các ngôn ngữ biên dịch hiện đại hơn như Go hay Rust. Luôn dùng công cụ tốt nhất cho công việc!

Node.js has an ample toolbelt of APIs for interacting with external processes. We can find all we need in the `child_process` module. Moreover, when the external process is just another Node.js program, connecting it to the main application is extremely easy and allows seamless communication with the local application. This magic happens thanks to the `child_process.fork()` function, which creates a new child Node.js process and also automatically creates a communication channel with it, allowing us to exchange information using an interface very similar to the `EventEmitter`. Let's see how this works by refactoring our subset sum server again.

> Node.js có một bộ công cụ API phong phú để tương tác với các tiến trình bên ngoài. Ta có thể tìm thấy tất cả những gì ta cần trong module `child_process`. Hơn nữa, khi tiến trình bên ngoài chỉ là một chương trình Node.js khác, việc kết nối nó với ứng dụng chính cực kỳ dễ dàng và cho phép giao tiếp liền mạch với ứng dụng cục bộ. Điều kỳ diệu này xảy ra nhờ hàm `child_process.fork()`, vốn tạo ra một tiến trình con Node.js mới và cũng tự động tạo một kênh giao tiếp với nó, cho phép ta trao đổi thông tin dùng một giao diện rất giống `EventEmitter`. Hãy xem cách điều này hoạt động bằng cách tái cấu trúc lại máy chủ tổng tập con của ta một lần nữa.

#### Delegating the subset sum task to an external process — Ủy thác tác vụ tổng tập con cho một tiến trình bên ngoài

The goal of refactoring the `SubsetSum` task is to create a separate **child process** responsible for handling the synchronous processing, leaving the event loop of the main server free to handle requests coming from the network. This is the recipe we are going to follow to make this possible:

> Mục tiêu của việc tái cấu trúc tác vụ `SubsetSum` là tạo một tiến trình con riêng biệt chịu trách nhiệm xử lý phần xử lý đồng bộ, để vòng lặp sự kiện của máy chủ chính rảnh rỗi xử lý các yêu cầu đến từ mạng. Đây là công thức ta sẽ làm theo để biến điều này thành khả thi:

1. We will create a new module named `processPool.js` that will allow us to create a pool of running processes. Starting a new process is expensive and requires time, so keeping them constantly running and ready to handle requests allows us to save time and CPU cycles. Also, the pool will help us limit the number of processes running at the same time to prevent exposing the application to denial-of-service (DoS) attacks.
2. Next, we will create a module called `subsetSumFork.js` responsible for abstracting a `SubsetSum` task running in a child process. Its role will be communicating with the child process and forwarding the results of the task as if they were coming from the current application.
3. Finally, we need a worker (our child process), a new Node.js program with the only goal of running the subset sum algorithm and forwarding its results to the parent process.

> 1. Ta sẽ tạo một module mới tên là `processPool.js` cho phép ta tạo một pool các tiến trình đang chạy. Khởi động một tiến trình mới tốn kém và đòi hỏi thời gian, nên giữ chúng liên tục chạy và sẵn sàng xử lý yêu cầu cho phép ta tiết kiệm thời gian và chu kỳ CPU. Ngoài ra, pool sẽ giúp ta giới hạn số lượng tiến trình chạy cùng lúc để ngăn ứng dụng bị phơi bày trước các cuộc tấn công từ chối dịch vụ (denial-of-service - DoS).
> 2. Tiếp theo, ta sẽ tạo một module tên là `subsetSumFork.js` chịu trách nhiệm trừu tượng hóa một tác vụ `SubsetSum` chạy trong một tiến trình con. Vai trò của nó sẽ là giao tiếp với tiến trình con và chuyển tiếp các kết quả của tác vụ như thể chúng đến từ ứng dụng hiện tại.
> 3. Cuối cùng, ta cần một worker (tiến trình con của ta), một chương trình Node.js mới với mục tiêu duy nhất là chạy thuật toán tổng tập con và chuyển tiếp kết quả của nó cho tiến trình cha.

The purpose of a DoS attack is to make a machine unavailable to its users. This is usually achieved by exhausting the capacity of such a machine by exploiting a vulnerability or massively overloading it with requests (DDoS – distributed DoS).

> Mục đích của một cuộc tấn công DoS là làm cho một máy không khả dụng đối với người dùng của nó. Điều này thường đạt được bằng cách làm cạn kiệt năng lực của máy đó bằng cách khai thác một lỗ hổng hoặc làm quá tải nó hàng loạt với các yêu cầu (DDoS - DoS phân tán).

##### Implementing a process pool — Hiện thực một process pool

Let's start by building the `processPool.js` module piece by piece:

> Hãy bắt đầu xây dựng module `processPool.js` từng phần một:

```js
import { fork } from 'child_process'
export class ProcessPool {
  constructor (file, poolMax) {
    this.file = file
    this.poolMax = poolMax
    this.pool = []
    this.active = []
    this.waiting = []
  }
//...
```

In the first part of the module, we import the `fork()` function from the `child_process` module, which we will use to create new processes. Then, we define the `ProcessPool` constructor, which accepts a `file` parameter representing the Node.js program to run, and the maximum number of running instances in the pool (`poolMax`). We then define three instance variables:

> Trong phần đầu của module, ta import hàm `fork()` từ module `child_process`, mà ta sẽ dùng để tạo các tiến trình mới. Sau đó, ta định nghĩa hàm khởi tạo `ProcessPool`, nhận một tham số `file` đại diện cho chương trình Node.js để chạy, và số lượng tối đa các thực thể đang chạy trong pool (`poolMax`). Sau đó ta định nghĩa ba biến thực thể:

- `pool` is the set of running processes ready to be used.
- `active` contains the list of the processes currently being used.
- `waiting` contains a queue of callbacks for all those requests that could not be fulfilled immediately because of the lack of an available process.

> - `pool` là tập hợp các tiến trình đang chạy sẵn sàng để dùng.
> - `active` chứa danh sách các tiến trình hiện đang được dùng.
> - `waiting` chứa một hàng đợi các callback cho tất cả những yêu cầu không thể được đáp ứng ngay lập tức do thiếu một tiến trình khả dụng.

The next piece of the `ProcessPool` class is the `acquire()` method, which is responsible for eventually returning a process ready to be used, when one becomes available:

> Phần tiếp theo của lớp `ProcessPool` là phương thức `acquire()`, chịu trách nhiệm cuối cùng trả về một tiến trình sẵn sàng để dùng, khi có một tiến trình trở nên khả dụng:

```js
acquire () {
  return new Promise((resolve, reject) => {
    let worker
    if (this.pool.length > 0) {                            // (1)
      worker = this.pool.pop()
      this.active.push(worker)
      return resolve(worker)
    }
    if (this.active.length >= this.poolMax) {              // (2)
      return this.waiting.push({ resolve, reject })
    }
    worker = fork(this.file)                               // (3)
    worker.once('message', message => {
      if (message === 'ready') {
        this.active.push(worker)
        return resolve(worker)
      }
      worker.kill()
      reject(new Error('Improper process start'))
    })
    worker.once('exit', code => {
      console.log(`Worker exited with code ${code}`)
      this.active = this.active.filter(w => worker !== w)
      this.pool = this.pool.filter(w => worker !== w)
    })
  })
}
```

The logic of `acquire()` is very simple and is explained as follows:

> Logic của `acquire()` rất đơn giản và được giải thích như sau:

1. If we have a process in the `pool` ready to be used, we simply move it to the `active` list and then use it to fulfill the outer promise with `resolve()`.
2. If there are no available processes in the `pool` and we have already reached the maximum number of running processes, we have to wait for one to be available. We achieve this by queuing the `resolve()` and `reject()` callbacks of the outer promise, for later use.
3. If we haven't reached the maximum number of running processes yet, we create a new one using `child_process.fork()`. Then, we wait for the `ready` message coming from the newly launched process, which indicates that the process has started and is ready to accept new jobs. This message-based channel is automatically provided with all processes started with `child_` `process.fork()`. The last method of the `ProcessPool` class is `release()`, whose purpose is to put a process back into the `pool` once we are done with it:

> 1. Nếu ta có một tiến trình trong `pool` sẵn sàng để dùng, ta chỉ cần chuyển nó sang danh sách `active` rồi dùng nó để đáp ứng promise bên ngoài bằng `resolve()`.
> 2. Nếu không có tiến trình khả dụng nào trong `pool` và ta đã đạt số lượng tối đa các tiến trình đang chạy, ta phải đợi một tiến trình trở nên khả dụng. Ta đạt được điều này bằng cách xếp hàng các callback `resolve()` và `reject()` của promise bên ngoài, để dùng sau.
> 3. Nếu ta chưa đạt số lượng tối đa các tiến trình đang chạy, ta tạo một tiến trình mới dùng `child_process.fork()`. Sau đó, ta đợi thông điệp `ready` đến từ tiến trình vừa khởi chạy, báo hiệu rằng tiến trình đã khởi động và sẵn sàng chấp nhận các công việc mới. Kênh dựa trên thông điệp này được cung cấp tự động với tất cả các tiến trình được khởi động bằng `child_` `process.fork()`. Phương thức cuối cùng của lớp `ProcessPool` là `release()`, mục đích của nó là đưa một tiến trình trở lại `pool` khi ta đã dùng xong nó:

```js
release (worker) {
  if (this.waiting.length > 0) {                           // (1)
    const { resolve } = this.waiting.shift()
    return resolve(worker)
  }
  this.active = this.active.filter(w => worker !== w)      // (2)
  this.pool.push(worker)
}
```

This is how the `release()` method works:

> Đây là cách phương thức `release()` hoạt động:

1. If there is a request in the `waiting` list, we simply reassign the `worker` we are releasing by passing it to the `resolve()` callback at the head of the `waiting` queue.
2. Otherwise, we remove the worker that we are releasing from the `active` list and put it back into the `pool`.

> 1. Nếu có một yêu cầu trong danh sách `waiting`, ta chỉ cần tái phân công `worker` ta đang giải phóng bằng cách truyền nó cho callback `resolve()` ở đầu hàng đợi `waiting`.
> 2. Nếu không, ta xóa worker mà ta đang giải phóng khỏi danh sách `active` và đưa nó trở lại `pool`.

As we can see, the processes are never stopped but just reassigned, allowing us to save time by not restarting them at each request. However, it's important to observe that this might not always be the best choice, and this greatly depends on the requirements of your application. Other possible tweaks for reducing long-term memory usage and adding resilience to our process pool are:

> Như ta thấy, các tiến trình không bao giờ bị dừng mà chỉ được tái phân công, cho phép ta tiết kiệm thời gian bằng cách không khởi động lại chúng ở mỗi yêu cầu. Tuy nhiên, điều quan trọng cần lưu ý là điều này có thể không phải lúc nào cũng là lựa chọn tốt nhất, và điều này phụ thuộc rất nhiều vào các yêu cầu của ứng dụng của bạn. Các điều chỉnh khả thi khác để giảm mức sử dụng bộ nhớ dài hạn và thêm tính bền bỉ cho process pool của ta là:

- Terminate idle processes to free memory after a certain time of inactivity.
- Add a mechanism to kill non-responsive processes or restart those that have crashed.

> - Chấm dứt các tiến trình nhàn rỗi để giải phóng bộ nhớ sau một khoảng thời gian không hoạt động nhất định.
> - Thêm một cơ chế để giết các tiến trình không phản hồi hoặc khởi động lại những tiến trình đã sập.

In this example, we will keep the implementation of our process pool simple as the details we could add are really endless.

> Trong ví dụ này, ta sẽ giữ phần hiện thực process pool của mình đơn giản vì các chi tiết ta có thể thêm vào thực sự là vô tận.

##### Communicating with a child process — Giao tiếp với một tiến trình con

Now that our `ProcessPool` class is ready, we can use it to implement the `SubsetSumFork` class, whose role is to communicate with the worker and forward the results it produces. As we already mentioned, starting a process with `child_process.fork()` also gives us a simple message-based communication channel, so let's see how this works by implementing the `subsetSumFork.js` module:

> Giờ khi lớp `ProcessPool` của ta đã sẵn sàng, ta có thể dùng nó để hiện thực lớp `SubsetSumFork`, vai trò của nó là giao tiếp với worker và chuyển tiếp các kết quả mà nó tạo ra. Như đã đề cập, khởi động một tiến trình bằng `child_process.fork()` cũng cho ta một kênh giao tiếp dựa trên thông điệp đơn giản, vậy hãy xem cách điều này hoạt động bằng cách hiện thực module `subsetSumFork.js`:

```js
import { EventEmitter } from 'events'
import { dirname, join } from 'path'
import { fileURLToPath } from 'url'
import { ProcessPool } from './processPool.js'
const __dirname = dirname(fileURLToPath(import.meta.url))
const workerFile = join(__dirname,
  'workers', 'subsetSumProcessWorker.js')
const workers = new ProcessPool(workerFile, 2)
export class SubsetSum extends EventEmitter {
  constructor (sum, set) {
    super()
    this.sum = sum
    this.set = set
  }
  async start () {
    const worker = await workers.acquire()                 // (1)
    worker.send({ sum: this.sum, set: this.set })
    const onMessage = msg => {
      if (msg.event === 'end') {                           // (3)
        worker.removeListener('message', onMessage)
        workers.release(worker)
      }
      this.emit(msg.event, msg.data)                       // (4)
    }
    worker.on('message', onMessage)                        // (2)
  }
}
```

The first thing to note is that we created a new `ProcessPool` object using the file `./workers/subsetSumProcessWorker.js` as the child worker. We also set the maximum capacity of the pool to `2`.

> Điều đầu tiên cần lưu ý là ta đã tạo một đối tượng `ProcessPool` mới dùng tệp `./workers/subsetSumProcessWorker.js` làm worker con. Ta cũng đặt năng lực tối đa của pool là `2`.

Another point worth mentioning is that we tried to maintain the same public API of the original `SubsetSum` class. In fact, `SubsetSumFork` is an `EventEmitter` whose constructor accepts `sum` and `set`, while the `start()` method triggers the execution of the algorithm, which, this time, runs on a separate process. This is what happens when the `start()` method is invoked:

> Một điểm khác đáng đề cập là ta đã cố gắng duy trì cùng API công khai của lớp `SubsetSum` gốc. Thực tế, `SubsetSumFork` là một `EventEmitter` có hàm khởi tạo nhận `sum` và `set`, trong khi phương thức `start()` kích hoạt việc thực thi thuật toán, lần này chạy trên một tiến trình riêng biệt. Đây là những gì xảy ra khi phương thức `start()` được gọi:

1. We try to acquire a new child process from the pool. When the operation completes, we immediately use the `worker` handle to send a message to the child process with the data of the job to run. The `send()` API is provided automatically by Node.js to all processes started with `child_process.fork()`. This is essentially the communication channel that we were talking about.
2. We then start listening for any message sent by the worker process using the `on()` method to attach a new listener (this is also a part of the communication channel provided by all processes started with `child_process.fork()`).
3. In the `onMessage` listener, we first check if we received an `end` event, which means that the `SubsetSum` task has finished, in which case we remove the `onMessage` listener and release the `worker`, putting it back into the pool.
4. The worker produces messages in the format `{event, data}`, allowing us to seamlessly forward (re-emit) any event produced by the child process.

> 1. Ta cố gắng lấy (acquire) một tiến trình con mới từ pool. Khi thao tác hoàn tất, ta ngay lập tức dùng handle `worker` để gửi một thông điệp tới tiến trình con với dữ liệu của công việc cần chạy. API `send()` được Node.js cung cấp tự động cho tất cả các tiến trình được khởi động bằng `child_process.fork()`. Đây thực chất là kênh giao tiếp mà ta đã nói đến.
> 2. Sau đó ta bắt đầu lắng nghe bất kỳ thông điệp nào được gửi bởi tiến trình worker dùng phương thức `on()` để gắn một listener mới (đây cũng là một phần của kênh giao tiếp được cung cấp bởi tất cả các tiến trình được khởi động bằng `child_process.fork()`).
> 3. Trong listener `onMessage`, trước tiên ta kiểm tra xem ta có nhận được một sự kiện `end` không, nghĩa là tác vụ `SubsetSum` đã hoàn tất, trong trường hợp đó ta xóa listener `onMessage` và giải phóng `worker`, đưa nó trở lại pool.
> 4. Worker tạo ra các thông điệp theo định dạng `{event, data}`, cho phép ta chuyển tiếp (phát lại) một cách liền mạch bất kỳ sự kiện nào được tạo ra bởi tiến trình con.

That's it for the `SubsetSumFork` wrapper. Let's now implement the worker (our child process). It is good to know that the `send()` method available on a child process instance can also be used to propagate a socket handle from the main application to a child process (look at the documentation at `nodejsdp.link/childprocess-send`). This is actually the technique used by the `**cluster**` module to distribute the load of an HTTP server across multiple processes. We will see this in more detail in the next chapter.

> Vậy là xong với wrapper `SubsetSumFork`. Giờ hãy hiện thực worker (tiến trình con của ta). Cũng nên biết rằng phương thức `send()` có sẵn trên một thực thể tiến trình con cũng có thể được dùng để truyền một handle socket từ ứng dụng chính tới một tiến trình con (hãy xem tài liệu tại `nodejsdp.link/childprocess-send`). Đây thực ra là kỹ thuật được module `cluster` dùng để phân phối tải của một máy chủ HTTP trên nhiều tiến trình. Ta sẽ xem điều này chi tiết hơn ở chương tiếp theo.

##### Implementing the worker — Hiện thực worker

Let's now create the `workers/subsetSumProcessWorker.js` module, our worker process:

> Giờ hãy tạo module `workers/subsetSumProcessWorker.js`, tiến trình worker của ta:

```js
import { SubsetSum } from '../subsetSum.js'
process.on('message', msg => {                             // (1)
  const subsetSum = new SubsetSum(msg.sum, msg.set)
  subsetSum.on('match', data => {                          // (2)
    process.send({ event: 'match', data: data })
  })
  subsetSum.on('end', data => {
    process.send({ event: 'end', data: data })
  })
  subsetSum.start()
})
process.send('ready')
```

We can immediately see that we are reusing the original (and synchronous) `SubsetSum` as it is. Now that we are in a separate process, we don't have to worry about blocking the event loop anymore; all the HTTP requests will continue to be handled by the event loop of the main application without disruptions.

> Ta có thể ngay lập tức thấy rằng ta đang tái sử dụng `SubsetSum` gốc (và đồng bộ) y nguyên. Giờ khi ta ở trong một tiến trình riêng biệt, ta không phải lo lắng về việc chặn vòng lặp sự kiện nữa; tất cả các yêu cầu HTTP sẽ tiếp tục được xử lý bởi vòng lặp sự kiện của ứng dụng chính mà không bị gián đoạn.

When the worker is started as a child process, this is what happens:

> Khi worker được khởi động như một tiến trình con, đây là những gì xảy ra:

1. It immediately starts listening for messages coming from the parent process. This can be easily done with the `process.on()` function (a part of the communication API provided when the process is started with `child_process.fork()`).

> 1. Nó ngay lập tức bắt đầu lắng nghe các thông điệp đến từ tiến trình cha. Điều này có thể dễ dàng thực hiện với hàm `process.on()` (một phần của API giao tiếp được cung cấp khi tiến trình được khởi động bằng `child_process.fork()`).

The only message we expect from the parent process is the one providing the input to a new `SubsetSum` task. As soon as such a message is received, we create a new instance of the `SubsetSum` class and register the listeners for the `match` and `end` events. Lastly, we start the computation with `subsetSum.start()`.

> Thông điệp duy nhất ta mong đợi từ tiến trình cha là thông điệp cung cấp đầu vào cho một tác vụ `SubsetSum` mới. Ngay khi một thông điệp như vậy được nhận, ta tạo một thực thể mới của lớp `SubsetSum` và đăng ký các listener cho các sự kiện `match` và `end`. Cuối cùng, ta bắt đầu phép tính với `subsetSum.start()`.

2. Every time an event is received from the running algorithm, we wrap it in an object having the format `{event, data}` and send it to the parent process. These messages are then handled in the `subsetSumFork.js` module, as we have seen in the previous section.

> 2. Mỗi khi một sự kiện được nhận từ thuật toán đang chạy, ta bọc nó trong một đối tượng có định dạng `{event, data}` và gửi nó tới tiến trình cha. Các thông điệp này sau đó được xử lý trong module `subsetSumFork.js`, như ta đã thấy trong phần trước.

As we can see, we just had to wrap the algorithm we already built, without modifying its internals. This clearly shows that any portion of an application can be easily put in an external process by simply using the technique we have just seen.

> Như ta thấy, ta chỉ phải bọc thuật toán mà ta đã xây dựng, mà không cần sửa đổi phần bên trong của nó. Điều này cho thấy rõ ràng rằng bất kỳ phần nào của một ứng dụng có thể dễ dàng được đưa vào một tiến trình bên ngoài chỉ bằng cách dùng kỹ thuật ta vừa thấy.

When the child process is not a Node.js program, the simple communication channel we just described (`on()`, `send()`) is not available. In these situations, we can still establish an interface with the child process by implementing our own protocol on top of the standard input and standard output streams, which are exposed to the parent process. To find out more about all the capabilities of the `child_process` API, you can refer to the official Node.js documentation at `nodejsdp.link/child_process`.

> Khi tiến trình con không phải là một chương trình Node.js, kênh giao tiếp đơn giản ta vừa mô tả (`on()`, `send()`) không có sẵn. Trong những tình huống này, ta vẫn có thể thiết lập một giao diện với tiến trình con bằng cách hiện thực giao thức riêng của ta trên nền các luồng đầu vào chuẩn (standard input) và đầu ra chuẩn (standard output), vốn được phơi bày cho tiến trình cha. Để tìm hiểu thêm về tất cả các năng lực của API `child_process`, bạn có thể tham khảo tài liệu Node.js chính thức tại `nodejsdp.link/child_process`.

// import { SubsetSum } from './subsetSum.js'
// import { SubsetSum } from './subsetSumDefer.js'

We can now start the server again and try to send a sample request:

As for the interleaving approach that we saw previously, with this new version of the `SubsetSum` module, the event loop is not blocked while running the CPU-bound task. This can be confirmed by sending another concurrent request, as follows:

As we saw, the multi-process approach has many advantages compared to the interleaving approach. First, it doesn't introduce any computational penalty when running the algorithm. Second, it can take full advantage of a multi-processor machine.

> Như ta đã thấy, cách tiếp cận đa tiến trình có nhiều lợi thế so với cách tiếp cận đan xen. Thứ nhất, nó không đưa ra bất kỳ hình phạt tính toán nào khi chạy thuật toán. Thứ hai, nó có thể tận dụng tối đa một máy đa bộ xử lý.

> Giờ hãy xem một cách tiếp cận thay thế dùng luồng (thread) thay vì tiến trình.

### Using worker threads — Dùng worker thread

Since Node 10.5.0, we have a new mechanism for running CPU-intensive algorithms outside of the main event loop called worker threads. Worker threads can be seen as a lightweight alternative to `child_process.fork()` with some extra goodies. Compared to processes, worker threads have a smaller memory footprint and a faster startup time since they run within the main process but inside different threads.

> Kể từ Node 10.5.0, ta có một cơ chế mới để chạy các thuật toán nặng CPU bên ngoài vòng lặp sự kiện chính gọi là worker thread. Worker thread có thể được xem như một lựa chọn thay thế nhẹ cho `child_process.fork()` với một số tính năng bổ sung. So với các tiến trình, worker thread có dung lượng bộ nhớ nhỏ hơn và thời gian khởi động nhanh hơn vì chúng chạy bên trong tiến trình chính nhưng trong các luồng khác nhau.

Even though worker threads are based on real threads, they don't allow the deep synchronization and sharing capabilities supported by other languages such as Java or Python. This is because JavaScript is a single-threaded language and it doesn't have any built-in mechanism to synchronize access to variables from multiple threads. JavaScript with threads simply wouldn't be JavaScript. The solution to bring all the advantages of threads within Node.js without actually changing the language is worker threads. Worker threads are essentially threads that, by default, don't share anything with the main application thread; they run within their own V8 instance, with an independent Node.js **runtime** and event loop. Communication with the main thread is possible thanks to message-based communication channels, the transfer of `ArrayBuffer` objects, and the use of `SharedArrayBuffer` objects whose synchronization is managed by the user (usually with the help of `Atomics`).

> Mặc dù worker thread dựa trên các luồng thực, chúng không cho phép việc đồng bộ hóa sâu và các năng lực chia sẻ được hỗ trợ bởi các ngôn ngữ khác như Java hay Python. Đó là vì JavaScript là một ngôn ngữ đơn luồng và nó không có bất kỳ cơ chế tích hợp sẵn nào để đồng bộ hóa việc truy cập các biến từ nhiều luồng. JavaScript với các luồng đơn giản là sẽ không còn là JavaScript. Giải pháp để mang tất cả các lợi thế của luồng vào trong Node.js mà không thực sự thay đổi ngôn ngữ là worker thread. Worker thread về cơ bản là các luồng mà, theo mặc định, không chia sẻ bất cứ thứ gì với luồng ứng dụng chính; chúng chạy bên trong thực thể V8 riêng của mình, với một runtime Node.js và vòng lặp sự kiện độc lập. Giao tiếp với luồng chính khả thi nhờ các kênh giao tiếp dựa trên thông điệp, việc chuyển các đối tượng `ArrayBuffer`, và việc dùng các đối tượng `SharedArrayBuffer` mà việc đồng bộ hóa được quản lý bởi người dùng (thường với sự trợ giúp của `Atomics`).

You can read more about `SharedArrayBuffer` and `Atomics` in the following article: `nodejsdp.link/shared-array-**buffer**`. Even though the article focuses on Web Workers, a lot of concepts are similar to Node.js's worker threads.

> Bạn có thể đọc thêm về `SharedArrayBuffer` và `Atomics` trong bài viết sau: `nodejsdp.link/shared-array-buffer`. Mặc dù bài viết tập trung vào Web Workers, rất nhiều khái niệm tương tự với worker thread của Node.js.

This extensive level of isolation of worker threads from the main thread preserves the integrity of the language. At the same time, the basic communication facilities and data-sharing capabilities are more than enough for 99% of use cases.

> Mức độ cô lập rộng rãi này của worker thread khỏi luồng chính bảo toàn tính toàn vẹn của ngôn ngữ. Đồng thời, các phương tiện giao tiếp cơ bản và các năng lực chia sẻ dữ liệu là quá đủ cho 99% các trường hợp sử dụng.

Now, let's use worker threads in our `SubsetSum` example.

> Giờ hãy dùng worker thread trong ví dụ `SubsetSum` của ta.

#### Running the subset sum task in a worker thread — Chạy tác vụ tổng tập con trong một worker thread

The worker threads API has a lot in common with that of `ChildProcess`, so the changes to our code will be minimal.

> API worker thread có rất nhiều điểm chung với API của `ChildProcess`, nên các thay đổi với mã của ta sẽ là tối thiểu.

First, we need to create a new class called `ThreadPool`, which is our `ProcessPool` adapted to operate with worker threads instead of processes. The following code shows the differences between the new `ThreadPool` class and the `ProcessPool` class. There are only a few differences in the `acquire()` method, which are highlighted; the rest of the code is identical:

> Trước tiên, ta cần tạo một lớp mới tên là `ThreadPool`, đó là `ProcessPool` của ta được điều chỉnh để hoạt động với worker thread thay vì tiến trình. Đoạn mã sau cho thấy các điểm khác biệt giữa lớp `ThreadPool` mới và lớp `ProcessPool`. Chỉ có một vài điểm khác biệt trong phương thức `acquire()`, được tô sáng; phần còn lại của mã thì giống hệt:

```js
import { Worker } from 'worker_threads'
export class ThreadPool {
  // ...
  acquire () {
    return new Promise((resolve, reject) => {
      let worker
      if (this.pool.length > 0) {
        worker = this.pool.pop()
        this.active.push(worker)
        return resolve(worker)
      }
      if (this.active.length >= this.poolMax) {
        return this.waiting.push({ resolve, reject })
      }
      worker = new Worker(this.file)
      worker.once('online', () => {
        this.active.push(worker)
        resolve(worker)
      })
      worker.once('exit', code => {
        console.log(`Worker exited with code ${code}`)
        this.active = this.active.filter(w => worker !== w)
        this.pool = this.pool.filter(w => worker !== w)
      })
    })
  }
  //...
}
```

Next, we need to adapt the worker and place it in a new file called `subsetSumThreadWorker.js`. The main difference from our old worker is that instead of using `process.send()` and `process.on()`, we'll have to use `parentPort.postMessage()` and `parentPort.on()`:

> Tiếp theo, ta cần điều chỉnh worker và đặt nó trong một tệp mới tên là `subsetSumThreadWorker.js`. Điểm khác biệt chính so với worker cũ của ta là thay vì dùng `process.send()` và `process.on()`, ta sẽ phải dùng `parentPort.postMessage()` và `parentPort.on()`:

```js
import { parentPort } from 'worker_threads'
import { SubsetSum } from '../subsetSum.js'
parentPort.on('message', msg => {
  const subsetSum = new SubsetSum(msg.sum, msg.set)
  subsetSum.on('match', data => {
    parentPort.postMessage({ event: 'match', data: data })
  })
  subsetSum.on('end', data => {
    parentPort.postMessage({ event: 'end', data: data })
  })
  subsetSum.start()
})
```

Similarly, the module `subsetSumThreads.js` is essentially the same as the `subsetSumFork.js` module except for a couple of lines of code, which are highlighted in the following code fragment:

> Tương tự, module `subsetSumThreads.js` về cơ bản giống với module `subsetSumFork.js` ngoại trừ một vài dòng mã, được tô sáng trong đoạn mã sau:

```js
import { EventEmitter } from 'events'
import { dirname, join } from 'path'
import { fileURLToPath } from 'url'
import { ThreadPool } from './threadPool.js'
const __dirname = dirname(fileURLToPath(import.meta.url))
const workerFile = join(__dirname,
  'workers', 'subsetSumThreadWorker.js')
const workers = new ThreadPool(workerFile, 2)
export class SubsetSum extends EventEmitter {
  constructor (sum, set) {
    super()
    this.sum = sum
    this.set = set
  }
  async start () {
    const worker = await workers.acquire()
    worker.postMessage({ sum: this.sum, set: this.set })
    const onMessage = msg => {
      if (msg.event === 'end') {
        worker.removeListener('message', onMessage)
        workers.release(worker)
      }
      this.emit(msg.event, msg.data)
    }
    worker.on('message', onMessage)
  }
}
```

As we've seen, adapting an existing application to use worker threads instead of forked processes is a trivial operation. This is because the API of the two components are very similar, but also because a **worker thread** has a lot in common with a fullfledged Node.js process. Finally, we need to update the `index.js` module so that it can use the new `subsetSumThreads.js` module, as we've seen for the other implementations of the algorithm:

> Như ta đã thấy, điều chỉnh một ứng dụng hiện có để dùng worker thread thay vì các tiến trình được fork là một thao tác tầm thường. Đó là vì API của hai thành phần rất giống nhau, nhưng cũng vì một worker thread có rất nhiều điểm chung với một tiến trình Node.js đầy đủ. ### Running CPU-bound tasks in production — Chạy các tác vụ chiếm dụng CPU trong production

The examples we've seen so far should give you an idea of the tools at our disposal for running CPU-intensive operations in Node.js. However, components such as process pools and thread pools are complex pieces of machinery that require proper mechanisms to deal with timeouts, errors, and other types of failures, which, for brevity, we left out from our implementation. Therefore, unless you have special requirements, it's better to rely on more battle-tested libraries for production use. Two of those libraries are `workerpool` (`nodejsdp.link/workerpool`) and `piscina` (`nodejsdp.link/piscina`), which are based on the same concepts we've seen in this section. They allow us to coordinate the execution of CPU-intensive tasks using external processes or worker threads.

> Các ví dụ ta đã thấy cho đến nay sẽ cho bạn một ý niệm về các công cụ ta có sẵn để chạy các thao tác nặng CPU trong Node.js. Tuy nhiên, các thành phần như process pool và thread pool là những bộ máy phức tạp đòi hỏi các cơ chế thích hợp để xử lý timeout, lỗi, và các loại thất bại khác, mà, cho ngắn gọn, ta đã bỏ qua trong phần hiện thực của mình. Do đó, trừ khi bạn có các yêu cầu đặc biệt, tốt hơn là dựa vào các thư viện đã được kiểm chứng kỹ hơn để dùng trong production. Hai trong số các thư viện đó là `workerpool` (`nodejsdp.link/workerpool`) và `piscina` (`nodejsdp.link/piscina`), vốn dựa trên cùng các khái niệm ta đã thấy trong phần này. Chúng cho phép ta điều phối việc thực thi các tác vụ nặng CPU dùng các tiến trình bên ngoài hoặc worker thread.

One last observation is that we must consider that if we have particularly complex algorithms to run or if the number of CPU-bound tasks exceeds the capacity of a single node, we may have to think about scaling out the computation across multiple nodes. This is a completely different problem and we'll learn more about this in the next two chapters.

> Một quan sát cuối cùng là ta phải xem xét rằng nếu ta có các thuật toán đặc biệt phức tạp để chạy hoặc nếu số lượng tác vụ chiếm dụng CPU vượt quá năng lực của một node đơn lẻ, ta có thể phải nghĩ đến việc mở rộng ra (scaling out) phép tính trên nhiều node. Đây là một bài toán hoàn toàn khác và ta sẽ tìm hiểu thêm về điều này trong hai chương tiếp theo.
