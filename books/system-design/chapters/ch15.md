Other than clarifying requirements, it is important to understand non-functional requirements:

> Ngoài việc làm rõ yêu cầu, hiểu các yêu cầu phi chức năng cũng rất quan trọng:

> - Khả năng mở rộng (scalability). Hệ thống phải xử lý được lưu lượng truy cập lớn.
> - Tính sẵn sàng cao. Người dùng vẫn phải dùng được hệ thống khi một số máy chủ ngừng hoạt động, chạy chậm, hoặc gặp lỗi mạng ngoài dự kiến.

A region is a geographic area where Amazon web services (AWS) have data centers. As shown in Figure 15-6, data can be replicated on the same-region (left side) and cross-region (right side). Redundant files are stored in multiple regions to guard against data loss and ensure availability. A bucket is like a folder in file systems.

After putting files in S3, you can finally have a good night's sleep without worrying about data losses. To stop similar problems from happening in the future, you decide to do further research on areas you can improve. Here are a few areas you find:

> Sau khi đưa tệp vào S3, cuối cùng bạn cũng có thể ngủ ngon mà không phải lo mất dữ liệu. Để ngăn các vấn đề tương tự tái diễn trong tương lai, bạn quyết định nghiên cứu thêm những mảng có thể cải thiện. Dưới đây là vài mảng bạn tìm ra:

- **Load balancer**: Add a load balancer to distribute network traffic. A load balancer ensures evenly distributed traffic, and if a web server goes down, it will redistribute the traffic.
- Web servers: After a load balancer is added, more web servers can be added/removed easily, depending on the traffic load.
- Metadata database: Move the database out of the server to avoid **single point of failure**. In the meantime, set up data replication and sharding to meet the availability and scalability requirements.
- File storage: Amazon S3 is used for file storage. To ensure availability and durability, files are replicated in two separate geographical regions.

> - Bộ cân bằng tải (load balancer): Thêm một bộ cân bằng tải để phân phối lưu lượng mạng. Bộ cân bằng tải bảo đảm lưu lượng được phân phối đều, và nếu một máy chủ web ngừng hoạt động, nó sẽ phân phối lại lưu lượng.
> - Máy chủ web: Sau khi thêm bộ cân bằng tải, có thể dễ dàng thêm/bớt máy chủ web tùy theo tải lưu lượng.
> - Cơ sở dữ liệu metadata: Tách cơ sở dữ liệu ra khỏi máy chủ để tránh điểm hỏng đơn (single point of failure). Đồng thời, thiết lập sao chép và phân mảnh dữ liệu để đáp ứng yêu cầu về tính sẵn sàng và khả năng mở rộng.
> - Lưu trữ tệp: Amazon S3 được dùng để lưu trữ tệp. Để bảo đảm tính sẵn sàng và độ bền, tệp được sao chép ở hai vùng địa lý riêng biệt.

## Sync conflicts — Xung đột đồng bộ

For a large storage system like Google Drive, sync conflicts happen from time to time. When two users modify the same file or folder at the same time, a conflict happens. How can we resolve the conflict? Here is our strategy: the first version that gets processed wins, and the version that gets processed later receives a conflict. Figure 15-8 shows an example of a sync conflict.

> Với một hệ thống lưu trữ lớn như Google Drive, xung đột đồng bộ thỉnh thoảng vẫn xảy ra. Khi hai người dùng cùng lúc sửa cùng một tệp hoặc thư mục, xung đột xảy ra. Ta giải quyết xung đột thế nào? Đây là chiến lược của ta: phiên bản được xử lý trước sẽ thắng, còn phiên bản được xử lý sau sẽ nhận về một xung đột. Hình 15-8 cho thấy một ví dụ về xung đột đồng bộ.

![](images/ch15-252_1.jpg)

In Figure 15-8, user 1 and user 2 tries to update the same file at the same time, but user 1’s file is processed by our system first. User 1’s update operation goes through, but, user 2 gets a sync conflict. How can we resolve the conflict for user 2? Our system presents both copies of the same file: user 2’s local copy and the latest version from the server (Figure 15-9). User 2 has the option to merge both files or override one version with the other.

## Block servers — Máy chủ khối

For large files that are updated regularly, sending the whole file on each update consumes a lot of bandwidth. Two optimizations are proposed to minimize the amount of network traffic being transmitted:

> Với những tệp lớn được cập nhật thường xuyên, gửi toàn bộ tệp trong mỗi lần cập nhật tiêu tốn rất nhiều băng thông. Hai tối ưu được đề xuất để giảm thiểu lưu lượng mạng phải truyền:

- **Delta sync**. When a file is modified, only modified blocks are synced instead of the whole file using a sync algorithm [7] [8].
- **Compression**. Applying compression on blocks can significantly reduce the data size. Thus, blocks are compressed using compression algorithms depending on file types. For example, gzip and bzip2 are used to compress text files. Different compression algorithms are needed to compress images and videos.

> - Đồng bộ phần thay đổi (delta sync). Khi một tệp bị sửa đổi, chỉ các khối bị sửa được đồng bộ thay vì toàn bộ tệp, bằng một thuật toán đồng bộ [7] [8].
> - Nén (compression). Áp dụng nén lên các khối có thể giảm đáng kể kích thước dữ liệu. Do đó, các khối được nén bằng thuật toán nén tùy theo loại tệp. Ví dụ, gzip và bzip2 được dùng để nén tệp văn bản. Cần các thuật toán nén khác để nén ảnh và video.

In our system, block servers do the heavy lifting work for uploading files. Block servers process files passed from clients by splitting a file into blocks, compressing each block, and encrypting them. Instead of uploading the whole file to the storage system, only modified blocks are transferred.

> Trong hệ thống của ta, máy chủ khối gánh phần việc nặng nhọc của quá trình tải tệp lên. Máy chủ khối xử lý các tệp do client chuyển tới bằng cách tách tệp thành các khối, nén từng khối, và mã hóa chúng. Thay vì tải toàn bộ tệp lên hệ thống lưu trữ, chỉ các khối bị sửa đổi mới được truyền đi.

## High consistency requirement — Yêu cầu nhất quán cao

Our system requires strong **consistency** by default. It is unacceptable for a file to be shown differently by different clients at the same time. The system needs to provide strong consistency for metadata cache and database layers.

> Hệ thống của ta mặc định yêu cầu tính nhất quán mạnh (strong consistency). Việc cùng một tệp được hiển thị khác nhau trên các client khác nhau tại cùng thời điểm là không thể chấp nhận. Hệ thống cần bảo đảm nhất quán mạnh cho tầng bộ nhớ đệm metadata và tầng cơ sở dữ liệu.

Memory caches adopt an **eventual consistency** model by default, which means different replicas might have different data. To achieve strong consistency, we must ensure the following:

> Bộ nhớ đệm trong bộ nhớ (memory cache) mặc định dùng mô hình nhất quán cuối cùng (eventual consistency), nghĩa là các bản sao (replica) khác nhau có thể chứa dữ liệu khác nhau. Để đạt được nhất quán mạnh, ta phải bảo đảm những điều sau:

- Data in cache replicas and the **master** is consistent.
- Invalidate caches on database write to ensure cache and database hold the same value. Achieving strong consistency in a relational database is easy because it maintains the ACID (Atomicity, Consistency, Isolation, Durability) properties [9]. However, NoSQL databases do not support ACID properties by default. ACID properties must be programmatically incorporated in synchronization logic. In our design, we choose relational databases because the ACID is natively supported.

> - Dữ liệu trong các bản sao của cache và master (bản chính) phải nhất quán.
> - Vô hiệu hóa cache khi ghi vào cơ sở dữ liệu để bảo đảm cache và cơ sở dữ liệu giữ cùng một giá trị. Đạt nhất quán mạnh trong cơ sở dữ liệu quan hệ khá dễ vì nó duy trì các thuộc tính ACID (Atomicity, Consistency, Isolation, Durability — nguyên tử, nhất quán, cô lập, bền vững) [9]. Tuy nhiên, cơ sở dữ liệu NoSQL mặc định không hỗ trợ các thuộc tính ACID. Các thuộc tính ACID phải được tích hợp vào logic đồng bộ hóa bằng lập trình. Trong thiết kế này, ta chọn cơ sở dữ liệu quan hệ vì ACID được hỗ trợ sẵn.

## Download flow — Luồng tải xuống

Download flow is triggered when a file is added or edited elsewhere. How does a client know if a file is added or edited by another client? There are two ways a client can know:

> Luồng tải xuống được kích hoạt khi một tệp được thêm hoặc chỉnh sửa ở nơi khác. Làm sao một client biết có tệp vừa được client khác thêm hoặc chỉnh sửa? Có hai cách để client biết điều đó:

- If client A is online while a file is changed by another client, notification service will inform client A that changes are made somewhere so it needs to pull the latest data.

> - Nếu client A đang trực tuyến khi một tệp bị client khác thay đổi, dịch vụ thông báo sẽ báo cho client A rằng có thay đổi ở đâu đó nên nó cần kéo về dữ liệu mới nhất.

- If client A is offline while a file is changed by another client, data will be saved to the cache. When the offline client is online again, it pulls the latest changes.

> - Nếu client A đang ngoại tuyến khi một tệp bị client khác thay đổi, dữ liệu sẽ được lưu vào bộ nhớ đệm. Khi client ngoại tuyến đó trực tuyến trở lại, nó kéo về các thay đổi mới nhất.

## Notification service — Dịch vụ thông báo

To maintain file consistency, any mutation of a file performed locally needs to be informed to other clients to reduce conflicts. Notification service is built to serve this purpose. At the high-level, notification service allows data to be transferred to clients as events happen. Here are a few options:

> Để duy trì tính nhất quán của tệp, mọi thay đổi trên tệp được thực hiện cục bộ cần được thông báo cho các client khác nhằm giảm xung đột. Dịch vụ thông báo được xây dựng cho mục đích này. Ở mức tổng quan, dịch vụ thông báo cho phép dữ liệu được truyền tới các client khi sự kiện xảy ra. Dưới đây là vài lựa chọn:

- **Long polling**. Dropbox uses long polling [10].
- **WebSocket**. WebSocket provides a persistent connection between the client and the server. Communication is bi-directional.

> - Long polling (kỹ thuật giữ một yêu cầu mở trong thời gian dài để chờ phản hồi từ máy chủ). Dropbox dùng long polling [10].
> - WebSocket. WebSocket cung cấp một kết nối bền vững giữa client và máy chủ. Giao tiếp là hai chiều.

Even though both options work well, we opt for long polling for the following two reasons:

> Dù cả hai lựa chọn đều hoạt động tốt, ta chọn long polling vì hai lý do sau:

- Communication for notification service is not bi-directional. The server sends information about file changes to the client, but not vice versa.
- WebSocket is suited for real-time bi-directional communication such as a chat app. For Google Drive, notifications are sent infrequently with no burst of data.

> - Giao tiếp của dịch vụ thông báo không phải hai chiều. Máy chủ gửi thông tin về các thay đổi tệp cho client, chứ không có chiều ngược lại.
> - WebSocket phù hợp với giao tiếp hai chiều thời gian thực như ứng dụng chat. Với Google Drive, thông báo được gửi không thường xuyên và không có đợt dữ liệu dồn dập.

With long polling, each client establishes a long poll connection to the notification service. If changes to a file are detected, the client will close the long poll connection. Closing the connection means a client must connect to the metadata server to download the latest changes. After a response is received or connection timeout is reached, a client immediately sends a new request to keep the connection open.

> Với long polling, mỗi client thiết lập một kết nối long poll tới dịch vụ thông báo. Nếu phát hiện thay đổi trên một tệp, client sẽ đóng kết nối long poll đó. Đóng kết nối nghĩa là client phải kết nối tới máy chủ metadata để tải về các thay đổi mới nhất. Sau khi nhận được phản hồi hoặc hết thời gian chờ kết nối, client lập tức gửi một yêu cầu mới để duy trì kết nối mở.

## Save storage space — Tiết kiệm không gian lưu trữ

To support file version history and ensure reliability, multiple versions of the same file are stored across multiple data centers. Storage space can be filled up quickly with frequent backups of all file revisions. Three techniques are proposed to reduce storage costs:

> Để hỗ trợ lịch sử phiên bản tệp và bảo đảm độ tin cậy, nhiều phiên bản của cùng một tệp được lưu trên nhiều trung tâm dữ liệu. Không gian lưu trữ có thể đầy lên rất nhanh nếu sao lưu thường xuyên mọi bản sửa đổi của tệp. Ba kỹ thuật được đề xuất để giảm chi phí lưu trữ:

- De-duplicate data blocks. Eliminating redundant blocks at the account level is an easy way to save space. Two blocks are identical if they have the same hash value.
- Adopt an intelligent data backup strategy. Two optimization strategies can be applied:
- Set a limit: We can set a limit for the number of versions to store. If the limit is reached, the oldest version will be replaced with the new version.
- Keep valuable versions only: Some files might be edited frequently. For example, saving every edited version for a heavily modified document could mean the file is saved over 1000 times within a short period. To avoid unnecessary copies, we could limit the number of saved versions. We give more weight to recent versions. Experimentation is helpful to figure out the optimal number of versions to save.
- Moving infrequently used data to cold storage. Cold data is the data that has not been active for months or years. Cold storage like Amazon S3 glacier [11] is much cheaper than S3.

> - Khử trùng lặp (de-duplicate) các khối dữ liệu. Loại bỏ các khối dư thừa ở cấp tài khoản là cách dễ dàng để tiết kiệm không gian. Hai khối là giống hệt nhau nếu chúng có cùng giá trị băm.
> - Áp dụng chiến lược sao lưu dữ liệu thông minh. Có thể áp dụng hai chiến lược tối ưu:
> - Đặt giới hạn: Ta có thể đặt giới hạn cho số phiên bản được lưu. Khi chạm giới hạn, phiên bản cũ nhất sẽ bị thay bằng phiên bản mới.
> - Chỉ giữ các phiên bản có giá trị: Một số tệp có thể bị chỉnh sửa thường xuyên. Ví dụ, nếu lưu mọi phiên bản chỉnh sửa của một tài liệu bị sửa đổi dày đặc, tệp đó có thể được lưu hơn 1000 lần trong một khoảng thời gian ngắn. Để tránh các bản sao không cần thiết, ta có thể giới hạn số phiên bản được lưu. Ta dành trọng số cao hơn cho các phiên bản gần đây. Thử nghiệm thực tế sẽ giúp tìm ra số phiên bản tối ưu cần lưu.
> - Chuyển dữ liệu ít dùng sang kho lưu trữ lạnh (cold storage). Dữ liệu lạnh là dữ liệu không hoạt động trong nhiều tháng hoặc nhiều năm. Kho lưu trữ lạnh như Amazon S3 glacier [11] rẻ hơn nhiều so với S3.

## Failure Handling — Xử lý sự cố

Failures can occur in a large-scale system and we must adopt design strategies to address these failures. Your interviewer might be interested in hearing about how you handle the following system failures:

> Sự cố có thể xảy ra trong một hệ thống quy mô lớn và ta phải áp dụng các chiến lược thiết kế để xử lý chúng. Người phỏng vấn có thể muốn nghe cách bạn xử lý các sự cố hệ thống sau:

- Load balancer failure: If a load balancer fails, the secondary would become active and pick up the traffic. Load balancers usually monitor each other using a **heartbeat**, a periodic signal sent between load balancers. A load balancer is considered as failed if it has not sent a heartbeat for some time.
- Block server failure: If a block server fails, other servers pick up unfinished or pending jobs.
- Cloud storage failure: S3 buckets are replicated multiple times in different regions. If files are not available in one region, they can be fetched from different regions.
- API server failure: It is a **stateless** service. If an API server fails, the traffic is redirected to other API servers by a load balancer.
- Metadata cache failure: Metadata cache servers are replicated multiple times. If one node goes down, you can still access other nodes to fetch data. We will bring up a new cache server to replace the failed one.
- Metadata DB failure.
- Master down: If the master is down, promote one of the slaves to act as a new master and bring up a new **slave** node.
- Slave down: If a slave is down, you can use another slave for read operations and bring another database server to replace the failed one.
- Notification service failure: Every online user keeps a long poll connection with the notification server. Thus, each notification server is connected with many users. According to the Dropbox talk in 2012 [6], over 1 million connections are open per machine. If a server goes down, all the long poll connections are lost so clients must reconnect to a different server. Even though one server can keep many open connections, it cannot reconnect all the lost connections at once. Reconnecting with all the lost clients is a relatively slow process.
- Offline backup queue failure: Queues are replicated multiple times. If one queue fails, consumers of the queue may need to re-subscribe to the backup queue.

> - Sự cố bộ cân bằng tải: Nếu một bộ cân bằng tải gặp sự cố, bộ thứ cấp sẽ được kích hoạt và tiếp nhận lưu lượng. Các bộ cân bằng tải thường giám sát lẫn nhau bằng nhịp tim (heartbeat), một tín hiệu định kỳ được gửi giữa các bộ cân bằng tải. Một bộ cân bằng tải bị coi là hỏng nếu nó không gửi nhịp tim trong một khoảng thời gian.
> - Sự cố máy chủ khối: Nếu một máy chủ khối gặp sự cố, các máy chủ khác sẽ tiếp nhận những công việc chưa hoàn thành hoặc đang chờ.
> - Sự cố kho lưu trữ đám mây: Các bucket S3 được sao chép nhiều lần ở các vùng khác nhau. Nếu tệp không khả dụng ở một vùng, có thể lấy chúng từ các vùng khác.
> - Sự cố máy chủ API: Đây là dịch vụ không trạng thái. Nếu một máy chủ API gặp sự cố, lưu lượng được bộ cân bằng tải chuyển hướng sang các máy chủ API khác.
> - Sự cố bộ nhớ đệm metadata: Các máy chủ đệm metadata được sao chép nhiều lần. Nếu một nút ngừng hoạt động, bạn vẫn có thể truy cập các nút khác để lấy dữ liệu. Ta sẽ đưa một máy chủ đệm mới lên thay thế máy bị hỏng.
> - Sự cố DB metadata.
> - Master ngừng hoạt động: Nếu master ngừng hoạt động, thăng cấp một trong các slave (máy tớ) lên làm master mới và đưa thêm một nút slave mới vào.
> - Slave ngừng hoạt động: Nếu một slave ngừng hoạt động, bạn có thể dùng slave khác cho các thao tác đọc và đưa một máy chủ cơ sở dữ liệu khác lên thay thế máy bị hỏng.
> - Sự cố dịch vụ thông báo: Mỗi người dùng trực tuyến giữ một kết nối long poll với máy chủ thông báo. Do đó, mỗi máy chủ thông báo kết nối với rất nhiều người dùng. Theo bài nói của Dropbox năm 2012 [6], mỗi máy mở hơn 1 triệu kết nối. Nếu một máy chủ ngừng hoạt động, tất cả các kết nối long poll bị mất nên các client phải kết nối lại tới một máy chủ khác. Dù một máy chủ có thể giữ nhiều kết nối mở, nó không thể kết nối lại tất cả các kết nối đã mất cùng một lúc. Kết nối lại với toàn bộ các client đã mất kết nối là một quá trình tương đối chậm.
> - Sự cố hàng đợi sao lưu ngoại tuyến: Hàng đợi được sao chép nhiều lần. Nếu một hàng đợi hỏng, các bên tiêu thụ (consumer) của hàng đợi đó có thể phải đăng ký lại với hàng đợi dự phòng.

## Step 4 - Wrap up — Bước 4 - Tổng kết

In this chapter, we proposed a system design to support Google Drive. The combination of strong consistency, low network bandwidth and fast sync make the design interesting. Our design contains two flows: manage file metadata and file sync. Notification service is another important component of the system. It uses long polling to keep clients up to date with file changes.

> Trong chương này, ta đã đề xuất một thiết kế hệ thống cho Google Drive. Sự kết hợp giữa tính nhất quán mạnh, băng thông mạng thấp và đồng bộ nhanh khiến thiết kế trở nên thú vị. Thiết kế của ta gồm hai luồng: quản lý metadata của tệp và đồng bộ tệp. Dịch vụ thông báo là một thành phần quan trọng khác của hệ thống. Nó dùng long polling để giúp client luôn cập nhật kịp thời các thay đổi của tệp.

Like any **system design interview** questions, there is no perfect solution. Every company has its unique constraints and you must design a system to fit those constraints. Knowing the tradeoffs of your design and technology choices are important. If there are a few minutes left, you can talk about different design choices.

> Như mọi câu hỏi phỏng vấn thiết kế hệ thống, không có lời giải hoàn hảo. Mỗi công ty có những ràng buộc riêng và bạn phải thiết kế một hệ thống phù hợp với các ràng buộc đó. Hiểu rõ các đánh đổi (trade-off) trong thiết kế và lựa chọn công nghệ của mình là điều quan trọng. Nếu còn vài phút, bạn có thể nói về các lựa chọn thiết kế khác nhau.

For example, we can upload files directly to cloud storage from the client instead of going through block servers. The advantage of this approach is that it makes file upload faster because a file only needs to be transferred once to the cloud storage. In our design, a file is transferred to block servers first, and then to the cloud storage. However, the new approach has a few drawbacks:

> Ví dụ, ta có thể tải tệp trực tiếp từ client lên kho lưu trữ đám mây thay vì đi qua máy chủ khối. Ưu điểm của cách này là tải tệp nhanh hơn vì tệp chỉ cần truyền một lần tới kho lưu trữ đám mây. Trong thiết kế của ta, tệp được truyền tới máy chủ khối trước, rồi mới tới kho lưu trữ đám mây. Tuy nhiên, cách tiếp cận mới có vài nhược điểm:

- First, the same chunking, compression, and encryption logic must be implemented on different platforms (iOS, Android, Web). It is error-prone and requires a lot of engineering effort. In our design, all those logics are implemented in a centralized place: block servers.
- Second, as a client can easily be hacked or manipulated, implementing encrypting logic on the client side is not ideal.

> - Thứ nhất, cùng một logic cắt khối, nén và mã hóa phải được hiện thực trên các nền tảng khác nhau (iOS, Android, Web). Việc này dễ sinh lỗi và đòi hỏi rất nhiều công sức kỹ thuật. Trong thiết kế của ta, toàn bộ logic đó được hiện thực tại một nơi tập trung: máy chủ khối.
> - Thứ hai, vì client dễ bị hack hoặc thao túng, hiện thực logic mã hóa ở phía client là không lý tưởng.

Another interesting evolution of the system is moving online/offline logic to a separate service. Let us call it presence service. By moving presence service out of notification servers, online/offline functionality can easily be integrated by other services.

> Một hướng tiến hóa thú vị khác của hệ thống là chuyển logic trực tuyến/ngoại tuyến sang một dịch vụ riêng. Ta gọi nó là dịch vụ hiện diện (presence service). Khi tách dịch vụ presence ra khỏi các máy chủ thông báo, chức năng trực tuyến/ngoại tuyến có thể dễ dàng được các dịch vụ khác tích hợp.
