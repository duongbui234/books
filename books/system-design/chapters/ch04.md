# CHAPTER 4: DESIGN A RATE LIMITER — Thiết kế bộ giới hạn tốc độ

In a network system, a **rate limiter** is used to control the rate of traffic sent by a client or a service. In the HTTP world, a rate limiter limits the number of client requests allowed to be sent over a specified period. If the API request count exceeds the threshold defined by the rate limiter, all the excess calls are blocked. Here are a few examples:

> Trong một hệ thống mạng, bộ giới hạn tốc độ (rate limiter) được dùng để kiểm soát tốc độ lưu lượng do một client hay một dịch vụ gửi đi. Trong thế giới HTTP, bộ giới hạn tốc độ giới hạn số yêu cầu mà client được phép gửi trong một khoảng thời gian xác định. Nếu số yêu cầu API vượt ngưỡng do bộ giới hạn tốc độ định ra, toàn bộ các lời gọi vượt mức sẽ bị chặn. Dưới đây là vài ví dụ:

- A user can write no more than 2 posts per second.
- You can create a maximum of 10 accounts per day from the same IP address.
- You can claim rewards no more than 5 times per week from the same device.

> - Một người dùng chỉ được viết tối đa 2 bài đăng mỗi giây.
> - Bạn chỉ được tạo tối đa 10 tài khoản mỗi ngày từ cùng một địa chỉ IP.
> - Bạn chỉ được nhận thưởng tối đa 5 lần mỗi tuần từ cùng một thiết bị.

In this chapter, you are asked to design a rate limiter. Before starting the design, we first look at the benefits of using an API rate limiter:

> Trong chương này, bạn được yêu cầu thiết kế một bộ giới hạn tốc độ. Trước khi bắt tay vào thiết kế, ta hãy điểm qua những lợi ích của việc dùng bộ giới hạn tốc độ cho API:

- Prevent resource starvation caused by Denial of Service (DoS) attack [1]. Almost all APIs published by large tech companies enforce some form of **rate limiting**. For example, Twitter limits the number of tweets to 300 per 3 hours [2]. Google docs APIs have the following default limit: 300 per user per 60 seconds for read requests [3]. A rate limiter prevents DoS attacks, either intentional or unintentional, by blocking the excess calls.
- Reduce cost. Limiting excess requests means fewer servers and allocating more resources to high priority APIs. Rate limiting is extremely important for companies that use paid third party APIs. For example, you are charged on a per-call basis for the following external APIs: check credit, make a payment, retrieve health records, etc. Limiting the number of calls is essential to reduce costs.
- Prevent servers from being overloaded. To reduce server load, a rate limiter is used to filter out excess requests caused by bots or users’ misbehavior.

> - Ngăn cạn kiệt tài nguyên do tấn công từ chối dịch vụ (Denial of Service - DoS) [1]. Hầu như mọi API do các công ty công nghệ lớn công bố đều áp dụng một hình thức giới hạn tốc độ nào đó. Ví dụ, Twitter giới hạn số tweet ở mức 300 mỗi 3 giờ [2]. API của Google Docs có giới hạn mặc định sau: 300 yêu cầu đọc mỗi người dùng mỗi 60 giây [3]. Bộ giới hạn tốc độ ngăn chặn các cuộc tấn công DoS, dù cố ý hay vô tình, bằng cách chặn những lời gọi vượt mức.
> - Giảm chi phí. Hạn chế các yêu cầu vượt mức nghĩa là cần ít máy chủ hơn và dành được nhiều tài nguyên hơn cho các API có độ ưu tiên cao. Giới hạn tốc độ cực kỳ quan trọng với những công ty sử dụng API trả phí của bên thứ ba. Ví dụ, bạn bị tính phí theo từng lời gọi đối với các API bên ngoài sau: kiểm tra tín dụng, thực hiện thanh toán, truy xuất hồ sơ sức khỏe, v.v. Hạn chế số lời gọi là điều thiết yếu để giảm chi phí.
> - Ngăn máy chủ bị quá tải. Để giảm tải cho máy chủ, bộ giới hạn tốc độ được dùng để lọc bỏ các yêu cầu vượt mức do bot hoặc do hành vi không đúng mực của người dùng gây ra.

## Where to put the rate limiter? — Đặt bộ giới hạn tốc độ ở đâu?

Intuitively, you can implement a rate limiter at either the client or server-side.

> Một cách trực giác, bạn có thể hiện thực bộ giới hạn tốc độ ở phía client hoặc phía máy chủ.

- Client-side implementation. Generally speaking, client is an unreliable place to enforce rate limiting because client requests can easily be forged by malicious actors. Moreover, we might not have control over the client implementation.
- Server-side implementation. Figure 4-1 shows a rate limiter that is placed on the server-side.

> - Hiện thực phía client. Nói chung, client là nơi không đáng tin cậy để thực thi giới hạn tốc độ vì yêu cầu từ client có thể dễ dàng bị kẻ xấu giả mạo. Hơn nữa, ta có thể không kiểm soát được phần hiện thực của client.
> - Hiện thực phía máy chủ. Hình 4-1 minh họa một bộ giới hạn tốc độ được đặt ở phía máy chủ.

![](images/ch04-53_1.jpg)

Besides the client and server-side implementations, there is an alternative way. Instead of putting a rate limiter at the API servers, we create a rate limiter middleware, which throttles requests to your APIs as shown in Figure 4-2.

> Bên cạnh cách hiện thực phía client và phía máy chủ, còn có một cách khác. Thay vì đặt bộ giới hạn tốc độ tại các máy chủ API, ta tạo một middleware giới hạn tốc độ, có nhiệm vụ tiết lưu các yêu cầu gửi tới API của bạn như trong Hình 4-2.

![](images/ch04-53_2.jpg)

Let us use an example in Figure 4-3 to illustrate how rate limiting works in this design. Assume our API allows 2 requests per second, and a client sends 3 requests to the server within a second. The first two requests are routed to API servers. However, the rate limiter middleware throttles the third request and returns a HTTP status code 429. The HTTP 429 response status code indicates a user has sent too many requests.

> Ta hãy dùng ví dụ trong Hình 4-3 để minh họa cách giới hạn tốc độ hoạt động trong thiết kế này. Giả sử API của ta cho phép 2 yêu cầu mỗi giây, và một client gửi 3 yêu cầu tới máy chủ trong vòng một giây. Hai yêu cầu đầu được định tuyến tới các máy chủ API. Tuy nhiên, middleware giới hạn tốc độ tiết lưu yêu cầu thứ ba và trả về mã trạng thái HTTP 429. Mã trạng thái phản hồi HTTP 429 cho biết người dùng đã gửi quá nhiều yêu cầu.

![](images/ch04-54_1.jpg)

Cloud microservices [4] have become widely popular and rate limiting is usually implemented within a component called **API gateway**. API gateway is a fully managed service that supports rate limiting, SSL termination, authentication, IP whitelisting, servicing static content, etc. For now, we only need to know that the API gateway is a middleware that supports rate limiting.

> Microservice trên nền đám mây [4] đã trở nên phổ biến rộng rãi và việc giới hạn tốc độ thường được hiện thực bên trong một thành phần gọi là API gateway. API gateway là một dịch vụ được quản lý hoàn toàn, hỗ trợ giới hạn tốc độ, kết thúc SSL (SSL termination), xác thực, danh sách trắng IP (IP whitelisting), phục vụ nội dung tĩnh, v.v. Hiện tại, ta chỉ cần biết rằng API gateway là một middleware hỗ trợ giới hạn tốc độ.

While designing a rate limiter, an important question to ask ourselves is: where should the rater limiter be implemented, on the server-side or in a gateway? There is no absolute answer. It depends on your company’s current technology stack, engineering resources, priorities, goals, etc. Here are a few general guidelines:

> Khi thiết kế bộ giới hạn tốc độ, một câu hỏi quan trọng ta cần tự hỏi là: bộ giới hạn tốc độ nên được hiện thực ở đâu, phía máy chủ hay trong gateway? Không có câu trả lời tuyệt đối. Điều này phụ thuộc vào nền tảng công nghệ (technology stack) hiện tại của công ty bạn, nguồn lực kỹ thuật, các ưu tiên, mục tiêu, v.v. Dưới đây là vài chỉ dẫn chung:

- Evaluate your current technology stack, such as programming language, cache service, etc. Make sure your current programming language is efficient to implement rate limiting on the server-side.
- Identify the rate limiting algorithm that fits your business needs. When you implement everything on the server-side, you have full control of the algorithm. However, your choice might be limited if you use a third-party gateway.
- If you have already used **microservice** architecture and included an API gateway in the design to perform authentication, IP whitelisting, etc., you may add a rate limiter to the API gateway.
- Building your own rate limiting service takes time. If you do not have enough engineering resources to implement a rate limiter, a commercial API gateway is a better option.

> - Đánh giá nền tảng công nghệ hiện tại, chẳng hạn ngôn ngữ lập trình, dịch vụ đệm, v.v. Hãy chắc chắn rằng ngôn ngữ lập trình hiện tại của bạn đủ hiệu quả để hiện thực giới hạn tốc độ ở phía máy chủ.
> - Xác định thuật toán giới hạn tốc độ phù hợp với nhu cầu nghiệp vụ của bạn. Khi hiện thực mọi thứ ở phía máy chủ, bạn có toàn quyền kiểm soát thuật toán. Tuy nhiên, lựa chọn của bạn có thể bị giới hạn nếu dùng gateway của bên thứ ba.
> - Nếu bạn đã dùng kiến trúc microservice và đã đưa API gateway vào thiết kế để thực hiện xác thực, lập danh sách trắng IP, v.v., bạn có thể bổ sung bộ giới hạn tốc độ vào API gateway.
> - Tự xây dựng dịch vụ giới hạn tốc độ riêng cần thời gian. Nếu bạn không có đủ nguồn lực kỹ thuật để hiện thực bộ giới hạn tốc độ, một API gateway thương mại là lựa chọn tốt hơn.

## Algorithms for rate limiting — Các thuật toán giới hạn tốc độ

Rate limiting can be implemented using different algorithms, and each of them has distinct pros and cons. Even though this chapter does not focus on algorithms, understanding them at high-level helps to choose the right algorithm or combination of algorithms to fit our use cases. Here is a list of popular algorithms:

> Giới hạn tốc độ có thể được hiện thực bằng nhiều thuật toán khác nhau, và mỗi thuật toán có những ưu, nhược điểm riêng biệt. Dù chương này không tập trung vào thuật toán, việc hiểu chúng ở mức tổng quan giúp ta chọn được thuật toán hay tổ hợp thuật toán phù hợp với tình huống sử dụng của mình. Dưới đây là danh sách các thuật toán phổ biến:

- **Token bucket**
- **Leaking bucket**
- **Fixed window counter**
- **Sliding window** log
- Sliding window counter

> - Thùng token (token bucket)
> - Thùng rò (leaking bucket)
> - Bộ đếm cửa sổ cố định (fixed window counter)
> - Log cửa sổ trượt (sliding window log)
> - Bộ đếm cửa sổ trượt (sliding window counter)

Token bucket algorithm

> Thuật toán thùng token

The token bucket algorithm is widely used for rate limiting. It is simple, well understood and commonly used by internet companies. Both Amazon [5] and Stripe [6] use this algorithm to throttle their API requests.

> Thuật toán thùng token được sử dụng rộng rãi cho việc giới hạn tốc độ. Nó đơn giản, dễ hiểu và được các công ty internet dùng phổ biến. Cả Amazon [5] lẫn Stripe [6] đều dùng thuật toán này để tiết lưu các yêu cầu API của họ.

The token bucket algorithm work as follows:

> Thuật toán thùng token hoạt động như sau:

- A token bucket is a container that has pre-defined capacity. Tokens are put in the bucket at preset rates periodically. Once the bucket is full, no more tokens are added. As shown in Figure 4-4, the token bucket capacity is 4. The refiller puts 2 tokens into the bucket every second. Once the bucket is full, extra tokens will overflow.

> - Thùng token là một vật chứa có dung lượng định trước. Token được bỏ vào thùng định kỳ theo tốc độ đặt sẵn. Khi thùng đã đầy, không token nào được thêm vào nữa. Như trong Hình 4-4, dung lượng thùng token là 4. Bộ nạp (refiller) bỏ 2 token vào thùng mỗi giây. Khi thùng đã đầy, các token thừa sẽ tràn ra ngoài.

![](images/ch04-55_1.jpg)

- Each request consumes one token. When a request arrives, we check if there are enough tokens in the bucket. Figure 4-5 explains how it works.
- If there are enough tokens, we take one token out for each request, and the request goes through.
- If there are not enough tokens, the request is dropped.

> - Mỗi yêu cầu tiêu thụ một token. Khi một yêu cầu đến, ta kiểm tra xem trong thùng có đủ token hay không. Hình 4-5 giải thích cách nó hoạt động.
> - Nếu có đủ token, ta lấy ra một token cho mỗi yêu cầu, và yêu cầu được cho qua.
> - Nếu không đủ token, yêu cầu bị loại bỏ.

![](images/ch04-56_1.jpg)

Figure 4-6 illustrates how token consumption, refill and rate limiting logic work. In this , example, the token bucket size is 4, and the refill rate is 4 per 1 minute.

> Hình 4-6 minh họa cách logic tiêu thụ token, nạp lại token và giới hạn tốc độ hoạt động. Trong ví dụ này, kích thước thùng token là 4, và tốc độ nạp lại là 4 token mỗi 1 phút.

![](images/ch04-57_1.jpg)

The token bucket algorithm takes two parameters:

> Thuật toán thùng token nhận hai tham số:

- Bucket size: the maximum number of tokens allowed in the bucket
- Refill rate: number of tokens put into the bucket every second How many buckets do we need? This varies, and it depends on the rate-limiting rules. Here are a few examples.
- It is usually necessary to have different buckets for different API endpoints. For instance, if a user is allowed to make 1 post per second, add 150 friends per day, and like 5 posts per second, 3 buckets are required for each user.
- If we need to throttle requests based on IP addresses, each IP address requires a bucket.
- If the system allows a maximum of 10,000 requests per second, it makes sense to have a global bucket shared by all requests.

> - Kích thước thùng (bucket size): số token tối đa được phép có trong thùng
> - Tốc độ nạp lại (refill rate): số token được bỏ vào thùng mỗi giây. Vậy ta cần bao nhiêu thùng? Điều này thay đổi tùy trường hợp, và nó phụ thuộc vào các quy tắc giới hạn tốc độ. Dưới đây là vài ví dụ.
> - Thường thì cần các thùng khác nhau cho các điểm cuối (endpoint) API khác nhau. Chẳng hạn, nếu một người dùng được phép đăng 1 bài mỗi giây, thêm 150 bạn bè mỗi ngày, và thích 5 bài mỗi giây, thì cần 3 thùng cho mỗi người dùng.
> - Nếu cần tiết lưu yêu cầu theo địa chỉ IP, mỗi địa chỉ IP cần một thùng.
> - Nếu hệ thống cho phép tối đa 10.000 yêu cầu mỗi giây, việc có một thùng toàn cục dùng chung cho mọi yêu cầu là hợp lý.

Pros:

> Ưu điểm:

- The algorithm is easy to implement.
- Memory efficient.
- Token bucket allows a burst of traffic for short periods. A request can go through as long as there are tokens left. Cons:
- Two parameters in the algorithm are bucket size and token refill rate. However, it might be challenging to tune them properly.

> - Thuật toán dễ hiện thực.
> - Tiết kiệm bộ nhớ.
> - Thùng token cho phép lưu lượng bùng nổ (burst) trong thời gian ngắn. Một yêu cầu có thể đi qua miễn là vẫn còn token. Nhược điểm:
> - Hai tham số của thuật toán là kích thước thùng và tốc độ nạp token. Tuy nhiên, việc tinh chỉnh chúng cho phù hợp có thể là một thách thức.

Leaking bucket algorithm

> Thuật toán thùng rò

The leaking bucket algorithm is similar to the token bucket except that requests are processed at a fixed rate. It is usually implemented with a first-in-first-out (FIFO) queue. The algorithm works as follows:

> Thuật toán thùng rò tương tự thùng token, ngoại trừ việc các yêu cầu được xử lý ở tốc độ cố định. Nó thường được hiện thực bằng một hàng đợi vào-trước-ra-trước (FIFO). Thuật toán hoạt động như sau:

- When a request arrives, the system checks if the queue is full. If it is not full, the request is added to the queue.
- Otherwise, the request is dropped.
- Requests are pulled from the queue and processed at regular intervals.

> - Khi một yêu cầu đến, hệ thống kiểm tra xem hàng đợi đã đầy chưa. Nếu chưa đầy, yêu cầu được thêm vào hàng đợi.
> - Ngược lại, yêu cầu bị loại bỏ.
> - Các yêu cầu được lấy ra khỏi hàng đợi và xử lý theo những khoảng thời gian đều đặn.

Figure 4-7 explains how the algorithm works.

> Hình 4-7 giải thích cách thuật toán hoạt động.

![](images/ch04-58_1.jpg)

Leaking bucket algorithm takes the following two parameters:

> Thuật toán thùng rò nhận hai tham số sau:

- Bucket size: it is equal to the queue size. The queue holds the requests to be processed at a fixed rate.
- Outflow rate: it defines how many requests can be processed at a fixed rate, usually in seconds.

> - Kích thước thùng: bằng với kích thước hàng đợi. Hàng đợi giữ các yêu cầu sẽ được xử lý ở tốc độ cố định.
> - Tốc độ chảy ra (outflow rate): xác định bao nhiêu yêu cầu có thể được xử lý ở một tốc độ cố định, thường tính theo giây.

Shopify, an ecommerce company, uses leaky buckets for rate-limiting [7].

> Shopify, một công ty thương mại điện tử, dùng thùng rò để giới hạn tốc độ [7].

Pros:

> Ưu điểm:

- Memory efficient given the limited queue size.
- Requests are processed at a fixed rate therefore it is suitable for use cases that a stable outflow rate is needed.

> - Tiết kiệm bộ nhớ nhờ kích thước hàng đợi có giới hạn.
> - Các yêu cầu được xử lý ở tốc độ cố định, do đó phù hợp với những tình huống sử dụng cần tốc độ chảy ra ổn định.

Cons:

> Nhược điểm:

- A burst of traffic fills up the queue with old requests, and if they are not processed in time, recent requests will be rate limited.
- There are two parameters in the algorithm. It might not be easy to tune them properly.

> - Một đợt bùng nổ lưu lượng làm hàng đợi đầy ắp các yêu cầu cũ, và nếu chúng không được xử lý kịp thời, các yêu cầu mới đến sẽ bị giới hạn tốc độ.
> - Thuật toán có hai tham số. Việc tinh chỉnh chúng cho phù hợp có thể không dễ.

Fixed window counter algorithm

> Thuật toán bộ đếm cửa sổ cố định

Fixed window counter algorithm works as follows:

> Thuật toán bộ đếm cửa sổ cố định hoạt động như sau:

- The algorithm divides the timeline into fix-sized time windows and assign a counter for each window.
- Each request increments the counter by one.
- Once the counter reaches the pre-defined threshold, new requests are dropped until a new time window starts.

> - Thuật toán chia trục thời gian thành các cửa sổ thời gian có kích thước cố định và gán một bộ đếm cho mỗi cửa sổ.
> - Mỗi yêu cầu làm bộ đếm tăng thêm một.
> - Khi bộ đếm chạm ngưỡng định trước, các yêu cầu mới bị loại bỏ cho tới khi một cửa sổ thời gian mới bắt đầu.

Let us use a concrete example to see how it works. In Figure 4-8, the time unit is 1 second and the system allows a maximum of 3 requests per second. In each second window, if more than 3 requests are received, extra requests are dropped as shown in Figure 4-8.

> Ta hãy dùng một ví dụ cụ thể để xem nó hoạt động ra sao. Trong Hình 4-8, đơn vị thời gian là 1 giây và hệ thống cho phép tối đa 3 yêu cầu mỗi giây. Trong mỗi cửa sổ một giây, nếu nhận được hơn 3 yêu cầu, các yêu cầu thừa sẽ bị loại bỏ như minh họa trong Hình 4-8.

![](images/ch04-59_1.jpg)

A major problem with this algorithm is that a burst of traffic at the edges of time windows could cause more requests than allowed quota to go through. Consider the following case:

> Một vấn đề lớn của thuật toán này là một đợt bùng nổ lưu lượng ở rìa các cửa sổ thời gian có thể khiến số yêu cầu đi qua nhiều hơn hạn mức cho phép. Xét trường hợp sau:

![](images/ch04-59_2.jpg)

In Figure 4-9, the system allows a maximum of 5 requests per minute, and the available quota resets at the human-friendly round minute. As seen, there are five requests between 2:00:00 and 2:01:00 and five more requests between 2:01:00 and 2:02:00. For the one-minute window between 2:00:30 and 2:01:30, 10 requests go through. That is twice as many as allowed requests.

> Trong Hình 4-9, hệ thống cho phép tối đa 5 yêu cầu mỗi phút, và hạn mức khả dụng được đặt lại tại mốc phút tròn thân thiện với con người. Như có thể thấy, có năm yêu cầu trong khoảng từ 2:00:00 đến 2:01:00 và thêm năm yêu cầu nữa từ 2:01:00 đến 2:02:00. Với cửa sổ một phút từ 2:00:30 đến 2:01:30, có 10 yêu cầu đi qua. Con số đó gấp đôi số yêu cầu được phép.

Pros:

> Ưu điểm:

- Memory efficient.

> - Tiết kiệm bộ nhớ.

- Easy to understand.
- Resetting available quota at the end of a unit time window fits certain use cases.

> - Dễ hiểu.
> - Việc đặt lại hạn mức khả dụng vào cuối mỗi đơn vị cửa sổ thời gian phù hợp với một số tình huống sử dụng nhất định.

Cons:

> Nhược điểm:

- Spike in traffic at the edges of a window could cause more requests than the allowed quota to go through.

> - Lưu lượng tăng vọt ở rìa cửa sổ có thể khiến số yêu cầu đi qua nhiều hơn hạn mức cho phép.

Sliding window log algorithm As discussed previously, the fixed window counter algorithm has a major issue: it allows more requests to go through at the edges of a window. The sliding window log algorithm fixes the issue. It works as follows:

> Thuật toán log cửa sổ trượt. Như đã thảo luận ở trên, thuật toán bộ đếm cửa sổ cố định có một vấn đề lớn: nó cho phép nhiều yêu cầu hơn đi qua ở rìa cửa sổ. Thuật toán log cửa sổ trượt khắc phục vấn đề này. Nó hoạt động như sau:

- The algorithm keeps track of request timestamps. Timestamp data is usually kept in cache, such as sorted sets of Redis [8].
- When a new request comes in, remove all the outdated timestamps. Outdated timestamps are defined as those older than the start of the current time window.
- Add timestamp of the new request to the log.
- If the log size is the same or lower than the allowed count, a request is accepted. Otherwise, it is rejected.

> - Thuật toán theo dõi dấu thời gian (timestamp) của các yêu cầu. Dữ liệu dấu thời gian thường được giữ trong bộ nhớ đệm, chẳng hạn tập có thứ tự (sorted set) của Redis [8].
> - Khi một yêu cầu mới đến, loại bỏ mọi dấu thời gian đã lỗi thời. Dấu thời gian lỗi thời được định nghĩa là những dấu thời gian cũ hơn thời điểm bắt đầu của cửa sổ thời gian hiện tại.
> - Thêm dấu thời gian của yêu cầu mới vào log.
> - Nếu kích thước log bằng hoặc nhỏ hơn số lượng cho phép, yêu cầu được chấp nhận. Ngược lại, nó bị từ chối.

We explain the algorithm with an example as revealed in Figure 4-10.

> Ta giải thích thuật toán bằng một ví dụ như trong Hình 4-10.

![](images/ch04-60_1.jpg)

In this example, the rate limiter allows 2 requests per minute. Usually, Linux timestamps are stored in the log. However, human-readable representation of time is used in our example for better readability.

> Trong ví dụ này, bộ giới hạn tốc độ cho phép 2 yêu cầu mỗi phút. Thông thường, dấu thời gian kiểu Linux được lưu trong log. Tuy nhiên, ví dụ của ta dùng cách biểu diễn thời gian dễ đọc với con người để tiện theo dõi.

- The log is empty when a new request arrives at 1:00:01. Thus, the request is allowed.

> - Log rỗng khi một yêu cầu mới đến lúc 1:00:01. Vì vậy, yêu cầu được chấp nhận.

- A new request arrives at 1:00:30, the timestamp 1:00:30 is inserted into the log. After the insertion, the log size is 2, not larger than the allowed count. Thus, the request is allowed.
- A new request arrives at 1:00:50, and the timestamp is inserted into the log. After the insertion, the log size is 3, larger than the allowed size 2. Therefore, this request is rejected even though the timestamp remains in the log.
- A new request arrives at 1:01:40. Requests in the range [1:00:40,1:01:40) are within the latest time frame, but requests sent before 1:00:40 are outdated. Two outdated timestamps, 1:00:01 and 1:00:30, are removed from the log. After the remove operation, the log size becomes 2; therefore, the request is accepted.

> - Một yêu cầu mới đến lúc 1:00:30, dấu thời gian 1:00:30 được chèn vào log. Sau khi chèn, kích thước log là 2, không lớn hơn số lượng cho phép. Vì vậy, yêu cầu được chấp nhận.
> - Một yêu cầu mới đến lúc 1:00:50, và dấu thời gian được chèn vào log. Sau khi chèn, kích thước log là 3, lớn hơn kích thước cho phép là 2. Do đó, yêu cầu này bị từ chối mặc dù dấu thời gian vẫn nằm lại trong log.
> - Một yêu cầu mới đến lúc 1:01:40. Các yêu cầu trong khoảng [1:00:40,1:01:40) nằm trong khung thời gian mới nhất, nhưng các yêu cầu gửi trước 1:00:40 đã lỗi thời. Hai dấu thời gian lỗi thời, 1:00:01 và 1:00:30, bị xóa khỏi log. Sau thao tác xóa, kích thước log còn 2; do đó, yêu cầu được chấp nhận.

Pros:

> Ưu điểm:

- Rate limiting implemented by this algorithm is very accurate. In any rolling window, requests will not exceed the rate limit.

> - Giới hạn tốc độ được hiện thực bằng thuật toán này rất chính xác. Trong bất kỳ cửa sổ trượt nào, số yêu cầu cũng không vượt quá giới hạn tốc độ.

Cons:

> Nhược điểm:

- The algorithm consumes a lot of memory because even if a request is rejected, its timestamp might still be stored in memory.

> - Thuật toán tiêu tốn rất nhiều bộ nhớ vì ngay cả khi một yêu cầu bị từ chối, dấu thời gian của nó có thể vẫn được lưu trong bộ nhớ.

Sliding window counter algorithm

> Thuật toán bộ đếm cửa sổ trượt

The sliding window counter algorithm is a hybrid approach that combines the fixed window counter and sliding window log. The algorithm can be implemented by two different approaches. We will explain one implementation in this section and provide reference for the other implementation at the end of the section. Figure 4-11 illustrates how this algorithm works.

> Thuật toán bộ đếm cửa sổ trượt là một cách tiếp cận lai, kết hợp bộ đếm cửa sổ cố định và log cửa sổ trượt. Thuật toán có thể được hiện thực theo hai cách khác nhau. Ta sẽ giải thích một cách hiện thực trong mục này và cung cấp tài liệu tham khảo cho cách còn lại ở cuối mục. Hình 4-11 minh họa cách thuật toán này hoạt động.

![](images/ch04-61_1.jpg)

Assume the rate limiter allows a maximum of 7 requests per minute, and there are 5 requests in the previous minute and 3 in the current minute. For a new request that arrives at a 30% position in the current minute, the number of requests in the rolling window is calculated using the following formula:

> Giả sử bộ giới hạn tốc độ cho phép tối đa 7 yêu cầu mỗi phút, và có 5 yêu cầu trong phút trước cùng 3 yêu cầu trong phút hiện tại. Với một yêu cầu mới đến tại vị trí 30% của phút hiện tại, số yêu cầu trong cửa sổ trượt được tính theo công thức sau:

- Requests in current window requests in the previous window overlap percentage of + * the rolling window and previous window
- Using this formula, we get 3 + 5 * 0.7% = 6.5 request. Depending on the use case, the number can either be rounded up or down. In our example, it is rounded down to 6.

> - Số yêu cầu trong cửa sổ hiện tại + số yêu cầu trong cửa sổ trước * phần trăm chồng lấp giữa cửa sổ trượt và cửa sổ trước
> - Theo công thức này, ta được 3 + 5 * 0,7 = 6,5 yêu cầu. Tùy tình huống sử dụng, con số này có thể được làm tròn lên hoặc xuống. Trong ví dụ của ta, nó được làm tròn xuống thành 6.

Since the rate limiter allows a maximum of 7 requests per minute, the current request can go through. However, the limit will be reached after receiving one more request.

> Vì bộ giới hạn tốc độ cho phép tối đa 7 yêu cầu mỗi phút, yêu cầu hiện tại được cho qua. Tuy nhiên, chỉ cần nhận thêm một yêu cầu nữa là sẽ chạm mức giới hạn.

Due to the space limitation, we will not discuss the other implementation here. Interested readers should refer to the reference material [9]. This algorithm is not perfect. It has pros and cons.

> Do giới hạn về dung lượng sách, ta sẽ không thảo luận cách hiện thực còn lại ở đây. Bạn đọc quan tâm có thể xem tài liệu tham khảo [9]. Thuật toán này không hoàn hảo. Nó có những ưu và nhược điểm.

Pros

> Ưu điểm

- It smooths out spikes in traffic because the rate is based on the average rate of the previous window.
- Memory efficient.

> - Nó làm mượt các đợt tăng vọt lưu lượng vì tốc độ được tính dựa trên tốc độ trung bình của cửa sổ trước.
> - Tiết kiệm bộ nhớ.

Cons

> Nhược điểm

- It only works for not-so-strict look back window. It is an approximation of the actual rate because it assumes requests in the previous window are evenly distributed. However, this problem may not be as bad as it seems. According to experiments done by Cloudflare [10], only 0.003% of requests are wrongly allowed or rate limited among 400 million requests.

> - Nó chỉ phù hợp khi cửa sổ nhìn lại (look back window) không quá khắt khe. Đây là một phép xấp xỉ của tốc độ thực vì nó giả định các yêu cầu trong cửa sổ trước phân bố đều. Tuy nhiên, vấn đề này có thể không tệ như ta tưởng. Theo các thí nghiệm do Cloudflare thực hiện [10], chỉ 0,003% yêu cầu bị cho qua nhầm hoặc bị giới hạn tốc độ nhầm trong số 400 triệu yêu cầu.

## High-level architecture — Kiến trúc tổng quan

The basic idea of rate limiting algorithms is simple. At the high-level, we need a counter to keep track of how many requests are sent from the same user, IP address, etc. If the counter is larger than the limit, the request is disallowed.

> Ý tưởng cơ bản của các thuật toán giới hạn tốc độ rất đơn giản. Ở mức tổng quan, ta cần một bộ đếm để theo dõi xem có bao nhiêu yêu cầu được gửi từ cùng một người dùng, cùng một địa chỉ IP, v.v. Nếu bộ đếm lớn hơn giới hạn, yêu cầu sẽ không được phép.

Where shall we store counters? Using the database is not a good idea due to slowness of disk access. In-memory cache is chosen because it is fast and supports time-based expiration strategy. For instance, Redis [11] is a popular option to implement rate limiting. It is an in-memory store that offers two commands: INCR and EXPIRE.

> Ta nên lưu các bộ đếm ở đâu? Dùng cơ sở dữ liệu không phải là ý hay do truy cập đĩa chậm. Bộ nhớ đệm trong RAM (in-memory cache) được chọn vì nó nhanh và hỗ trợ chiến lược hết hạn theo thời gian. Chẳng hạn, Redis [11] là một lựa chọn phổ biến để hiện thực giới hạn tốc độ. Nó là một kho lưu trữ trong bộ nhớ cung cấp hai lệnh: INCR và EXPIRE.

- INCR: It increases the stored counter by 1.
- EXPIRE: It sets a timeout for the counter. If the timeout expires, the counter is automatically deleted.

> - INCR: tăng bộ đếm đang lưu thêm 1.
> - EXPIRE: đặt thời gian hết hạn cho bộ đếm. Nếu quá thời gian này, bộ đếm tự động bị xóa.

Figure 4-12 shows the high-level architecture for rate limiting, and this works as follows:

> Hình 4-12 thể hiện kiến trúc tổng quan cho việc giới hạn tốc độ, hoạt động như sau:

![](images/ch04-62_1.jpg)

- The client sends a request to rate limiting middleware.
- Rate limiting middleware fetches the counter from the corresponding bucket in Redis and checks if the limit is reached or not.
- If the limit is reached, the request is rejected.
- If the limit is not reached, the request is sent to API servers. Meanwhile, the system increments the counter and saves it back to Redis.

> - Client gửi một yêu cầu tới middleware giới hạn tốc độ.
> - Middleware giới hạn tốc độ lấy bộ đếm từ thùng tương ứng trong Redis và kiểm tra xem đã chạm giới hạn hay chưa.
> - Nếu đã chạm giới hạn, yêu cầu bị từ chối.
> - Nếu chưa chạm giới hạn, yêu cầu được gửi tới các máy chủ API. Đồng thời, hệ thống tăng bộ đếm và lưu nó trở lại Redis.

> domain : messaging descriptors : -key : message_type Value : marketing rate_limit : unit : day requests_per_unit : 5

In the above example, the system is configured to allow a maximum of 5 marketing messages per day. Here is another example:

> Trong ví dụ trên, hệ thống được cấu hình để cho phép tối đa 5 thông điệp marketing mỗi ngày. Đây là một ví dụ khác:

domain : auth descriptors : -key : auth_type Value : login rate_limit : unit : minute requests_per_unit : 5

> domain : auth descriptors : -key : auth_type Value : login rate_limit : unit : minute requests_per_unit : 5

This rule shows that clients are not allowed to login more than 5 times in 1 minute. Rules are generally written in configuration files and saved on disk.

> Quy tắc này cho thấy client không được phép đăng nhập quá 5 lần trong 1 phút. Các quy tắc thường được viết trong file cấu hình và lưu trên đĩa.

## Exceeding the rate limit — Vượt quá giới hạn tốc độ

In case a request is rate limited, APIs return a HTTP response code 429 (too many requests) to the client. Depending on the use cases, we may enqueue the rate-limited requests to be processed later. For example, if some orders are rate limited due to system overload, we may keep those orders to be processed later.

> Trong trường hợp một yêu cầu bị giới hạn tốc độ, API trả về cho client mã phản hồi HTTP 429 (quá nhiều yêu cầu). Tùy tình huống sử dụng, ta có thể đưa các yêu cầu bị giới hạn tốc độ vào hàng đợi để xử lý sau. Ví dụ, nếu một số đơn hàng bị giới hạn tốc độ do hệ thống quá tải, ta có thể giữ các đơn hàng đó lại để xử lý sau.

Rate limiter headers

> Các header của bộ giới hạn tốc độ

How does a client know whether it is being throttled? And how does a client know the number of allowed remaining requests before being throttled? The answer lies in HTTP response headers. The rate limiter returns the following HTTP headers to clients:

> Làm sao một client biết mình đang bị tiết lưu? Và làm sao client biết số yêu cầu còn lại được phép trước khi bị tiết lưu? Câu trả lời nằm ở các header phản hồi HTTP. Bộ giới hạn tốc độ trả về cho client các header HTTP sau:

X-Ratelimit-Remaining : The remaining number of allowed requests within the window.

> X-Ratelimit-Remaining: Số yêu cầu còn lại được phép trong cửa sổ hiện tại.

X-Ratelimit-Limit: It indicates how many calls the client can make per time window.

> X-Ratelimit-Limit: Cho biết client có thể thực hiện bao nhiêu lời gọi trong mỗi cửa sổ thời gian.

X-Ratelimit-Retry-After: The number of seconds to wait until you can make a request again without being throttled.

> X-Ratelimit-Retry-After: Số giây cần chờ cho tới khi bạn có thể gửi yêu cầu trở lại mà không bị tiết lưu.

When a user has sent too many requests, a 429 too many requests error and X-Ratelimit- Retry-After header are returned to the client.

> Khi một người dùng đã gửi quá nhiều yêu cầu, lỗi 429 too many requests cùng header X-Ratelimit-Retry-After được trả về cho client.

## Detailed design — Thiết kế chi tiết

Figure 4-13 presents a detailed design of the system.

> Hình 4-13 trình bày thiết kế chi tiết của hệ thống.

![](images/ch04-65_1.jpg)

- Rules are stored on the disk. Workers frequently pull rules from the disk and store them in the cache.
- When a client sends a request to the server, the request is sent to the rate limiter middleware first.
- Rate limiter middleware loads rules from the cache. It fetches counters and last request timestamp from Redis cache. Based on the response, the rate limiter decides:
- if the request is not rate limited, it is forwarded to API servers.
- if the request is rate limited, the rate limiter returns 429 too many requests error to the client. In the meantime, the request is either dropped or forwarded to the queue.

> - Các quy tắc được lưu trên đĩa. Các worker thường xuyên kéo quy tắc từ đĩa về và lưu vào bộ nhớ đệm.
> - Khi một client gửi yêu cầu tới máy chủ, yêu cầu được gửi tới middleware giới hạn tốc độ trước tiên.
> - Middleware giới hạn tốc độ nạp các quy tắc từ bộ nhớ đệm. Nó lấy bộ đếm và dấu thời gian của yêu cầu gần nhất từ bộ nhớ đệm Redis. Dựa trên kết quả trả về, bộ giới hạn tốc độ quyết định:
> - nếu yêu cầu không bị giới hạn tốc độ, nó được chuyển tiếp tới các máy chủ API.
> - nếu yêu cầu bị giới hạn tốc độ, bộ giới hạn tốc độ trả về lỗi 429 too many requests cho client. Đồng thời, yêu cầu hoặc bị loại bỏ hoặc được chuyển tiếp vào hàng đợi.

## Rate limiter in a distributed environment — Bộ giới hạn tốc độ trong môi trường phân tán

Building a rate limiter that works in a single server environment is not difficult. However, **scaling** the system to support multiple servers and concurrent threads is a different story. There are two challenges:

> Xây dựng một bộ giới hạn tốc độ hoạt động trong môi trường một máy chủ thì không khó. Tuy nhiên, mở rộng hệ thống để hỗ trợ nhiều máy chủ và các luồng đồng thời lại là chuyện khác. Có hai thách thức:

- Race condition
- Synchronization issue

> - Tranh đua (race condition)
> - Vấn đề đồng bộ hóa

Race condition

> Tranh đua

As discussed earlier, rate limiter works as follows at the high-level:

> Như đã thảo luận ở trên, ở mức tổng quan bộ giới hạn tốc độ hoạt động như sau:

- Read the counter value from Redis.
- Check if ( counter + 1 )  exceeds the threshold.
- If not, increment the counter value by 1 in Redis.

> - Đọc giá trị bộ đếm từ Redis.
> - Kiểm tra xem (giá trị bộ đếm + 1) có vượt ngưỡng hay không.
> - Nếu không, tăng giá trị bộ đếm thêm 1 trong Redis.

Race conditions can happen in a highly concurrent environment as shown in Figure 4-14.

> Tranh đua có thể xảy ra trong môi trường có độ đồng thời cao như minh họa trong Hình 4-14.

![](images/ch04-66_1.jpg)

Assume the counter value in Redis is 3. If two requests concurrently read the counter value before either of them writes the value back, each will increment the counter by one and write it back without checking the other thread. Both requests (threads) believe they have the correct counter value 4. However, the correct counter value should be 5.

> Giả sử giá trị bộ đếm trong Redis là 3. Nếu hai yêu cầu đồng thời đọc giá trị bộ đếm trước khi một trong hai kịp ghi giá trị trở lại, mỗi yêu cầu sẽ tăng bộ đếm thêm một và ghi lại mà không kiểm tra luồng còn lại. Cả hai yêu cầu (luồng) đều tin rằng mình đang có giá trị bộ đếm đúng là 4. Tuy nhiên, giá trị bộ đếm đúng phải là 5.

Locks are the most obvious solution for solving race condition. However, locks will significantly slow down the system. Two strategies are commonly used to solve the problem: Lua script [13] and sorted sets data structure in Redis [8]. For readers interested in these strategies, refer to the corresponding reference materials [8] [13].

> Khóa (lock) là giải pháp hiển nhiên nhất để xử lý tranh đua. Tuy nhiên, khóa sẽ làm hệ thống chậm đi đáng kể. Hai chiến lược thường được dùng để giải quyết vấn đề này: Lua script [13] và cấu trúc dữ liệu tập có thứ tự (sorted set) trong Redis [8]. Bạn đọc quan tâm tới các chiến lược này có thể xem các tài liệu tham khảo tương ứng [8] [13].

Synchronization issue Synchronization is another important factor to consider in a distributed environment. To support millions of users, one rate limiter server might not be enough to handle the traffic. When multiple rate limiter servers are used, synchronization is required. For example, on the left side of Figure 4-15, client 1 sends requests to rate limiter 1, and client 2 sends requests to rate limiter 2. As the **web tier** is **stateless**, clients can send requests to a different rate limiter as shown on the right side of Figure 4-15. If no synchronization happens, rate limiter 1 does not contain any data about client 2. Thus, the rate limiter cannot work properly.

> Vấn đề đồng bộ hóa. Đồng bộ hóa là một yếu tố quan trọng khác cần cân nhắc trong môi trường phân tán. Để hỗ trợ hàng triệu người dùng, một máy chủ giới hạn tốc độ có thể không đủ để xử lý lưu lượng. Khi dùng nhiều máy chủ giới hạn tốc độ, việc đồng bộ hóa là bắt buộc. Ví dụ, ở phía bên trái Hình 4-15, client 1 gửi yêu cầu tới bộ giới hạn tốc độ 1, còn client 2 gửi yêu cầu tới bộ giới hạn tốc độ 2. Vì tầng web là không trạng thái, client có thể gửi yêu cầu tới một bộ giới hạn tốc độ khác như ở phía bên phải Hình 4-15. Nếu không có đồng bộ hóa, bộ giới hạn tốc độ 1 không chứa bất kỳ dữ liệu nào về client 2. Do đó, bộ giới hạn tốc độ không thể hoạt động đúng.

![](images/ch04-67_1.jpg)

One possible solution is to use sticky sessions that allow a client to send traffic to the same rate limiter. This solution is not advisable because it is neither scalable nor flexible. A better approach is to use centralized data stores like Redis. The design is shown in Figure 4-16.

> Một giải pháp khả dĩ là dùng phiên dính (sticky session) cho phép một client gửi lưu lượng tới cùng một bộ giới hạn tốc độ. Giải pháp này không được khuyến khích vì nó vừa không có khả năng mở rộng vừa thiếu linh hoạt. Cách tiếp cận tốt hơn là dùng kho dữ liệu tập trung như Redis. Thiết kế được thể hiện trong Hình 4-16.

![](images/ch04-67_2.jpg)

## Performance optimization — Tối ưu hiệu năng

Performance optimization is a common topic in system design interviews. We will cover two areas to improve.

> Tối ưu hiệu năng là một chủ đề thường gặp trong các buổi phỏng vấn thiết kế hệ thống. Ta sẽ đề cập hai điểm cần cải thiện.

First, multi-**data center** setup is crucial for a rate limiter because latency is high for users located far away from the data center. Most cloud service providers build many edge server locations around the world. For example, as of 5/20 2020, Cloudflare has 194 geographically distributed edge servers [14]. Traffic is automatically routed to the closest edge server to reduce latency.

> Thứ nhất, thiết lập đa trung tâm dữ liệu là điều then chốt với bộ giới hạn tốc độ, vì độ trễ sẽ cao đối với những người dùng ở xa trung tâm dữ liệu. Hầu hết các nhà cung cấp dịch vụ đám mây xây dựng nhiều vị trí máy chủ biên (edge server) khắp thế giới. Ví dụ, tính đến ngày 20/5/2020, Cloudflare có 194 máy chủ biên phân tán theo địa lý [14]. Lưu lượng được tự động định tuyến tới máy chủ biên gần nhất để giảm độ trễ.

![](images/ch04-68_1.jpg)

Second, synchronize data with an **eventual consistency** model. If you are unclear about the eventual **consistency** model, refer to the “Consistency” section in “Chapter 6: Design a **Key-value Store**.”

> Thứ hai, đồng bộ dữ liệu theo mô hình nhất quán cuối cùng (eventual consistency). Nếu bạn chưa rõ về mô hình nhất quán cuối cùng, hãy xem mục "Tính nhất quán" trong "Chương 6: Thiết kế kho khóa-giá trị".

## Monitoring — Giám sát

After the rate limiter is put in place, it is important to gather analytics data to check whether the rate limiter is effective. Primarily, we want to make sure:

> Sau khi bộ giới hạn tốc độ được đưa vào vận hành, điều quan trọng là thu thập dữ liệu phân tích để kiểm tra xem bộ giới hạn tốc độ có hiệu quả hay không. Chủ yếu, ta muốn bảo đảm:

- The rate limiting algorithm is effective.
- The rate limiting rules are effective.

> - Thuật toán giới hạn tốc độ hiệu quả.
> - Các quy tắc giới hạn tốc độ hiệu quả.

For example, if rate limiting rules are too strict, many valid requests are dropped. In this case, we want to relax the rules a little bit. In another example, we notice our rate limiter becomes ineffective when there is a sudden increase in traffic like flash sales. In this scenario, we may replace the algorithm to support burst traffic. Token bucket is a good fit here.

> Ví dụ, nếu quy tắc giới hạn tốc độ quá khắt khe, nhiều yêu cầu hợp lệ sẽ bị loại bỏ. Trong trường hợp này, ta muốn nới lỏng quy tắc một chút. Một ví dụ khác, ta nhận thấy bộ giới hạn tốc độ trở nên kém hiệu quả khi lưu lượng tăng đột ngột như trong các đợt flash sale. Trong kịch bản này, ta có thể thay thuật toán để hỗ trợ lưu lượng bùng nổ. Thùng token là lựa chọn phù hợp ở đây.

- Hard vs soft rate limiting.
- Hard: The number of requests cannot exceed the threshold.
- Soft:  Requests can exceed the threshold for a short period.
- Rate limiting at different levels. In this chapter, we only talked about rate limiting at the application level (HTTP: layer 7). It is possible to apply rate limiting at other layers. For example, you can apply rate limiting by IP addresses using Iptables [15] (IP: layer 3). Note: The Open Systems Interconnection model (OSI model) has 7 layers [16]:  Layer 1: Physical layer, Layer 2: Data link layer, Layer 3: Network layer, Layer 4: Transport layer, Layer 5: Session layer, Layer 6: Presentation layer, Layer 7: Application layer.
- Avoid being rate limited. Design your client with best practices:
- Use client cache to avoid making frequent API calls.
- Understand the limit and do not send too many requests in a short time frame.
- Include code to catch exceptions or errors so your client can gracefully recover from exceptions.
- Add sufficient back off time to retry logic.
