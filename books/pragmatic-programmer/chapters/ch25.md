This can never happen…

### Use Assertions to Prevent the Impossible — Dùng assertion để chặn những điều bất khả

Whenever you find yourself thinking “but of course that could never happen,” add code to check it. The easiest way to do this is with assertions. In many language implementations, you’ll find [31] some form of that checks a Boolean condition. These assert checks can be invaluable. If a parameter or a result should never be , then check for it explicitly: null

> Mỗi khi bạn bắt gặp mình đang nghĩ “nhưng tất nhiên chuyện đó chẳng bao giờ xảy ra nổi,” hãy thêm mã để kiểm tra nó. Cách dễ nhất là dùng các khẳng định (assertion). Trong nhiều ngôn ngữ, bạn sẽ tìm thấy [31] một dạng nào đó kiểm tra một điều kiện Boolean. Những lệnh assert này có thể vô giá. Nếu một tham số hay một kết quả lẽ ra không bao giờ được là , thì hãy kiểm tra nó một cách tường minh: null

assert (result != null );

> assert (result != null );

> Trong cài đặt Java, bạn có thể (và nên) thêm một chuỗi mô tả:

result != && result.size() > 0 : "Empty result from XYZ" ; assert null Assertions are also useful checks on an algorithm’s operation. Maybe you’ve written a clever sort algorithm, named . my_sort Check that it works:

> result != && result.size() > 0 : "Empty result from XYZ" ; assert null Assertion cũng là những phép kiểm tra hữu ích cho hoạt động của một thuật toán. Có thể bạn đã viết một thuật toán sắp xếp rất khéo, đặt tên là . my_sort Hãy kiểm tra xem nó có chạy đúng không:

> books = my_sort(find( "scifi" )) assert(is_sorted?(books))

Don’t use assertions in place of real error handling. Assertions check for things that should never happen: you don’t want to be writing code such as the following:

> Đừng dùng assertion để thay cho việc xử lý lỗi thực sự. Assertion kiểm tra những thứ lẽ ra không bao giờ được xảy ra: bạn không muốn viết mã kiểu như sau đây:

puts( "Enter 'Y' or 'N': " ) ans = gets[0] # Grab first character of response assert((ch == 'Y' ) || (ch == 'N' )) # Very bad idea!

> puts( "Enter 'Y' or 'N': " ) ans = gets[0] # Lấy ký tự đầu tiên của phản hồi assert((ch == 'Y' ) || (ch == 'N' )) # Ý tưởng cực tệ!

And just because most implementations will terminate the assert **process** when an **assertion** fails, there’s no reason why versions you write should. If you need to free resources, catch the assertion’s **exception** or trap the exit, and run your own error handler. Just make sure the code you execute in those dying milliseconds doesn’t rely on the information that triggered the assertion failure in the first place.

> Và chỉ vì hầu hết các cài đặt sẽ kết thúc tiến trình khi một assertion thất bại, không có nghĩa là phiên bản bạn viết cũng phải làm vậy. Nếu cần giải phóng tài nguyên, hãy bắt ngoại lệ của assertion hoặc chặn lệnh thoát, rồi chạy trình xử lý lỗi của riêng bạn. Chỉ cần chắc chắn rằng đoạn mã bạn cho chạy trong những mili giây hấp hối ấy không phụ thuộc vào chính cái thông tin đã làm assertion thất bại ngay từ đầu.

### Assertions And Side Effects — Assertion và tác dụng phụ

It’s embarrassing when the code we add to detect errors actually ends up creating new errors. This can happen with assertions if evaluating the condition has side effects. For example, it would be a bad idea to code something such as

> Thật ngượng khi mã ta thêm vào để phát hiện lỗi lại rốt cuộc đẻ ra lỗi mới. Điều này có thể xảy ra với assertion nếu việc đánh giá điều kiện gây ra tác dụng phụ. Ví dụ, sẽ là một ý tồi nếu viết một thứ kiểu như

while (iter.hasMoreElements()) { assert (iter.nextElement() != null ); Object obj = iter.nextElement(); // .... }

> while (iter.hasMoreElements()) { assert (iter.nextElement() != null ); Object obj = iter.nextElement(); // .... }

The call in the assertion has the side effect of .nextElement() moving the iterator past the element being fetched, and so the loop will process only half the elements in the collection. It would be better to write

> Lời gọi .nextElement() trong assertion có tác dụng phụ là đẩy con trỏ duyệt vượt qua phần tử đang được lấy, và vì thế vòng lặp sẽ chỉ xử lý một nửa số phần tử trong tập hợp. Tốt hơn nên viết

(iter.hasMoreElements()) { while Object obj = iter.nextElement(); (obj != ); assert null // .... }

> (iter.hasMoreElements()) { while Object obj = iter.nextElement(); (obj != ); assert null // .... }

[32] This problem is a kind of Heisenbug —**debugging** that changes the behavior of the system being debugged.

> [32] Vấn đề này là một dạng Heisenbug — việc gỡ lỗi (debugging) lại làm thay đổi hành vi của chính hệ thống đang được gỡ lỗi.

(We also believe that nowadays, when most languages have decent support for iterating functions over collections, this kind of explicit loop is unnecessary and bad form.)

> (Chúng tôi cũng tin rằng ngày nay, khi hầu hết các ngôn ngữ đều hỗ trợ tử tế việc duyệt hàm trên các tập hợp, kiểu vòng lặp tường minh này là không cần thiết và là phong cách dở.)

### Leave Assertions Turned On — Cứ để assertion luôn bật

There is a common misunderstanding about assertions. It goes something like this:

> Có một hiểu lầm phổ biến về assertion. Nó đại loại thế này:

Assertions add some overhead to code. Because they check for things that should never happen, they’ll get triggered only by a bug in the code. Once the code has been tested and shipped, they are no longer needed, and should be turned off to make the code run faster. Assertions are a debugging facility.

> Assertion thêm chút chi phí cho mã. Vì chúng kiểm tra những thứ lẽ ra không bao giờ xảy ra, nên chúng chỉ bị kích hoạt bởi một lỗi (bug) trong mã. Một khi mã đã được kiểm thử và giao đi, chúng không còn cần nữa, và nên tắt đi để mã chạy nhanh hơn. Assertion là một công cụ gỡ lỗi.

There are two patently wrong assumptions here. First, they assume that testing finds all the bugs. In reality, for any

> Ở đây có hai giả định sai bét. Thứ nhất, chúng giả định rằng kiểm thử tìm ra mọi lỗi. Trên thực tế, với bất kỳ

complex program you are unlikely to test even a minuscule percentage of the permutations your code will be put through. Second, the optimists are forgetting that your program runs in a dangerous world. During testing, rats probably won’t gnaw through a communications cable, someone playing a game won’t exhaust memory, and log files won’t fill the storage partition. These things might happen when your program runs in a production environment. Your first line of defense is checking for any possible error, and your second is using assertions to try to detect those you’ve missed.

> chương trình phức tạp nào, bạn khó lòng kiểm thử nổi dù chỉ một phần trăm cực nhỏ các hoán vị mà mã của bạn sẽ phải trải qua. Thứ hai, những người lạc quan đang quên rằng chương trình của bạn chạy trong một thế giới đầy hiểm họa. Khi kiểm thử, có lẽ chuột sẽ không gặm đứt cáp truyền thông, ai đó chơi game sẽ không làm cạn bộ nhớ, và file log sẽ không lấp đầy phân vùng lưu trữ. Những chuyện này có thể xảy ra khi chương trình của bạn chạy trong môi trường sản xuất (production). Tuyến phòng thủ đầu tiên của bạn là kiểm tra mọi lỗi có thể, và tuyến thứ hai là dùng assertion để cố phát hiện những lỗi bạn đã bỏ sót.

Turning off assertions when you deliver a program to production is like crossing a high wire without a net because you once made it across in practice. There’s dramatic value, but it’s hard to get life insurance.

> Tắt assertion khi giao chương trình lên production cũng giống như đi trên dây cao mà không có lưới chỉ vì có lần bạn đã qua trót lọt lúc tập luyện. Nó có giá trị kịch tính đấy, nhưng khó mà mua được bảo hiểm nhân thọ.

Even if you do have performance issues, turn off only those assertions that really hit you. The sort example above may be a critical part of your application, and may need to be fast. Adding the check means another pass through the data, which might be unacceptable. Make that particular check optional, but leave the rest in.

> Kể cả khi bạn thực sự gặp vấn đề về hiệu năng, cũng chỉ tắt những assertion thực sự đè nặng lên bạn thôi. Ví dụ sắp xếp ở trên có thể là một phần then chốt của ứng dụng, và có thể cần phải nhanh. Thêm phép kiểm tra nghĩa là thêm một lượt duyệt qua dữ liệu, điều này có thể không chấp nhận được. Hãy làm cho riêng phép kiểm tra đó thành tùy chọn, nhưng giữ những cái còn lại.

Use Assertions in Production, Win Big Money

> Dùng assertion trong production, thắng đậm

A former neighbor of Andy’s headed up a small startup company that made network devices. One of their secrets to success was the decision to leave assertions in place in production releases. These assertions were well crafted to report all the pertinent data leading to the failure, and presented via a nice-looking UI to the end user. This level of feedback, from real users under actual conditions, allowed the developers to plug the holes and fix these obscure, hard-to-reproduce bugs, resulting in remarkably stable, bullet-proof software.

> Một người hàng xóm cũ của Andy điều hành một công ty khởi nghiệp nhỏ chuyên làm thiết bị mạng. Một trong những bí quyết thành công của họ là quyết định giữ nguyên các assertion trong các bản phát hành production. Những assertion này được làm rất khéo để báo cáo mọi dữ liệu liên quan dẫn tới sự cố, và trình bày qua một giao diện người dùng đẹp mắt cho người dùng cuối. Mức độ phản hồi này, từ người dùng thật trong điều kiện thật, cho phép các lập trình viên bịt kín các lỗ hổng và sửa những lỗi tối nghĩa, khó tái hiện đó, kết quả là phần mềm ổn định và chống đạn một cách đáng kinh ngạc.

This small, unknown company had such a solid product, it was soon acquired for hundreds of millions of dollars.

> Công ty nhỏ vô danh này có một sản phẩm vững chắc đến mức chẳng bao lâu nó được mua lại với giá hàng trăm triệu đô la.
