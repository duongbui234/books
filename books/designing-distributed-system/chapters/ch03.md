# CHAPTER 3 Ambassadors — CHAPTER 3 Ambassador (Đại sứ)

The previous chapter introduced the **sidecar** pattern, where one **container** augments a pre-existing container to add functionality. This chapter introduces the **ambassador** pattern, where an ambassador container brokers interactions between the application container and the rest of the world. As with other single-**node** patterns, the two containers are tightly linked in a symbiotic pairing that is scheduled to a single machine. A canonical diagram of this pattern is shown in Figure 3-1 .

> Chương trước đã giới thiệu mẫu sidecar, trong đó một container bổ sung cho một container có sẵn để thêm chức năng. Chương này giới thiệu mẫu ambassador, trong đó một container ambassador làm trung gian cho các tương tác giữa container ứng dụng và phần còn lại của thế giới. Cũng như những mẫu đơn nút (single-node) khác, hai container được liên kết chặt chẽ trong một cặp cộng sinh được lập lịch chạy trên cùng một máy. Sơ đồ tiêu biểu của mẫu này được trình bày trong Hình 3-1.

![](images/ch03-35_1.png)

Figure 3-1. Generic ambassador pattern

> Hình 3-1. Mẫu ambassador tổng quát

The value of the ambassador pattern is twofold. First, as with the other single-node patterns, there is inherent value in building modular, reusable containers. The separation of concerns makes the containers easier to build and maintain. Likewise, the ambassador container can be reused with a number of different application containers. This reuse speeds up application development because the container’s code can be reused in a number of places. Additionally the implementation is both more consistent and of a higher quality because it is built once and used in many different contexts.

> Giá trị của mẫu ambassador có hai mặt. Thứ nhất, cũng như các mẫu đơn nút khác, bản thân việc xây dựng những container mô-đun, có thể tái sử dụng đã mang lại giá trị. Việc tách bạch các mối quan tâm khiến container dễ xây dựng và dễ bảo trì hơn. Tương tự, container ambassador có thể được tái sử dụng với nhiều container ứng dụng khác nhau. Sự tái sử dụng này giúp đẩy nhanh quá trình phát triển ứng dụng vì mã của container có thể được dùng lại ở nhiều nơi. Ngoài ra, phần triển khai cũng nhất quán hơn và chất lượng cao hơn vì nó được xây dựng một lần và dùng trong nhiều bối cảnh khác nhau.

The rest of this chapter provides a number of examples of using the ambassador pattern to implement a series of real-world applications.

> Phần còn lại của chương này cung cấp một số ví dụ về việc dùng mẫu ambassador để hiện thực một loạt ứng dụng thực tế.

## Using an Ambassador to Shard a Service — Dùng Ambassador để Phân mảnh một Dịch vụ

Sometimes the data that you want to store in a storage layer becomes too big for a single machine to handle. In such situations, you need to **shard** your storage layer. Sharding splits up the layer into multiple disjoint pieces, each hosted by a separate machine. This chapter focuses on a **single-node pattern** for adapting an existing **service** to talk to a **sharded service** that exists somewhere in the world. It does not discuss how the sharded service came to exist. Sharding and a multi-node sharded service **design pattern** are discussed in great detail in Chapter 6 . A diagram of a sharded service is shown in Figure 3-2 .

> Đôi khi dữ liệu mà bạn muốn lưu trong tầng lưu trữ trở nên quá lớn để một máy đơn lẻ có thể xử lý. Trong những tình huống như vậy, bạn cần phân mảnh (shard) tầng lưu trữ của mình. Việc phân mảnh chia tầng này thành nhiều phần rời nhau, mỗi phần do một máy riêng biệt lưu giữ. Chương này tập trung vào một mẫu đơn nút (single-node) để thích ứng một dịch vụ có sẵn sao cho nó nói chuyện được với một dịch vụ phân mảnh (sharded) tồn tại đâu đó ngoài kia. Nó không bàn về việc dịch vụ phân mảnh đó ra đời như thế nào. Phân mảnh và mẫu thiết kế dịch vụ phân mảnh đa nút (multi-node) được bàn rất chi tiết trong Chương 6. Sơ đồ một dịch vụ phân mảnh được trình bày trong Hình 3-2.

![](images/ch03-36_1.png)

Figure 3-2. A generic sharded service

> Hình 3-2. Một dịch vụ phân mảnh tổng quát

When deploying a sharded service, one question that arises is how to integrate it with the frontend or middleware code that stores data. Clearly there needs to be logic that routes a particular request to a particular shard, but often it is difficult to retrofit such a sharded client into existing source code that expects to connect to a single storage backend. Additionally, sharded services make it difficult to share **configuration** between development environments (where there is often only a single storage shard) and production environments (where there are often many storage shards).

> Khi triển khai một dịch vụ phân mảnh, một câu hỏi nảy sinh là làm thế nào để tích hợp nó với mã frontend hoặc middleware vốn lưu dữ liệu. Rõ ràng cần có logic định tuyến một yêu cầu cụ thể tới một mảnh (shard) cụ thể, nhưng thường rất khó để gắn thêm một client phân mảnh như vậy vào mã nguồn có sẵn vốn chỉ kỳ vọng kết nối tới một backend lưu trữ duy nhất. Ngoài ra, các dịch vụ phân mảnh khiến việc chia sẻ cấu hình giữa môi trường phát triển (nơi thường chỉ có một mảnh lưu trữ duy nhất) và môi trường sản xuất (nơi thường có nhiều mảnh lưu trữ) trở nên khó khăn.

Once approach is to build all of the sharding logic into the sharded service itself. In this approach, the sharded service also has a **stateless** **load balancer** that directs traffic to the appropriate shard. Effectively, this load balancer is a distributed ambassador as a service. This makes a client-side ambassador unnecessary at the expense of a more complicated **deployment** for the sharded service. The alternative is to integrate a single-node ambassador on the client side to route traffic to the appropriate shard. This makes deploying the client somewhat more complicated but simplifies the deployment of the sharded service. As is always the case with trade-offs, it is up to the particulars of your specific application to determine which approach makes the most sense. Some factors to consider include where team lines fall in your architecture, as well as where you are writing code versus simply deploying off-the-shelf software. Ultimately, either choice is valid. The following section describes how to use the single-node ambassador pattern for client-side sharding.

> Một cách tiếp cận là xây dựng toàn bộ logic phân mảnh vào ngay bên trong bản thân dịch vụ phân mảnh. Trong cách này, dịch vụ phân mảnh cũng có một bộ cân bằng tải không trạng thái (stateless) chuyển hướng lưu lượng tới mảnh phù hợp. Trên thực tế, bộ cân bằng tải này chính là một ambassador phân tán dưới dạng dịch vụ. Điều này khiến không cần ambassador phía client, đổi lại là việc triển khai dịch vụ phân mảnh phức tạp hơn. Phương án thay thế là tích hợp một ambassador đơn nút ở phía client để định tuyến lưu lượng tới mảnh phù hợp. Cách này khiến việc triển khai client phức tạp hơn đôi chút nhưng đơn giản hóa việc triển khai dịch vụ phân mảnh. Như thường thấy với các đánh đổi, việc xác định cách tiếp cận nào hợp lý nhất tùy thuộc vào đặc thù riêng của ứng dụng của bạn. Một số yếu tố cần cân nhắc bao gồm các ranh giới giữa các nhóm (team) trong kiến trúc của bạn nằm ở đâu, cũng như chỗ nào bạn tự viết mã so với chỗ nào bạn chỉ đơn thuần triển khai phần mềm có sẵn. Rốt cuộc, cả hai lựa chọn đều hợp lệ. Phần tiếp theo mô tả cách dùng mẫu ambassador đơn nút để phân mảnh phía client.

When adapting an existing application to a sharded backend, you can introduce an ambassador container that contains all of the logic needed to route requests to the appropriate storage shard. Thus, your frontend or middleware application only connects to what appears to be a single storage backend running on localhost. However, this server is in fact actually a sharding ambassador **proxy** , which receives all of the requests from your application code, sends a request to the appropriate storage shard, and then returns the result to your application. This use of an ambassador is diagrammed in Figure 3-3 .

> Khi thích ứng một ứng dụng có sẵn với một backend phân mảnh, bạn có thể đưa vào một container ambassador chứa toàn bộ logic cần thiết để định tuyến yêu cầu tới mảnh lưu trữ phù hợp. Nhờ đó, ứng dụng frontend hoặc middleware của bạn chỉ kết nối tới thứ trông như một backend lưu trữ duy nhất chạy trên localhost. Tuy nhiên, máy chủ này trên thực tế lại là một proxy ambassador phân mảnh (sharding ambassador proxy), thứ nhận tất cả yêu cầu từ mã ứng dụng của bạn, gửi yêu cầu tới mảnh lưu trữ phù hợp, rồi trả kết quả về cho ứng dụng. Việc sử dụng ambassador này được minh họa trong Hình 3-3.

The net result of applying the ambassador pattern to sharded services is a separation of concerns between the application container, which simply knows it needs to talk to a storage service and discovers that service on localhost, and the sharding ambassador proxy, which only contains the code necessary to perform appropriate sharding. As with all good single-node patterns, this ambassador can be reused between many different applications. Or, as we’ll see in the following hands-on example, an off-the shelf **open source** implementation can be used for the ambassador, speeding up the development of the overall **distributed system**.

> Kết quả cuối cùng của việc áp dụng mẫu ambassador cho các dịch vụ phân mảnh là sự tách bạch các mối quan tâm giữa container ứng dụng, vốn chỉ biết rằng nó cần nói chuyện với một dịch vụ lưu trữ và phát hiện dịch vụ đó trên localhost, với proxy ambassador phân mảnh, vốn chỉ chứa mã cần thiết để thực hiện việc phân mảnh phù hợp. Như với mọi mẫu đơn nút tốt, ambassador này có thể được tái sử dụng giữa nhiều ứng dụng khác nhau. Hoặc, như chúng ta sẽ thấy trong ví dụ thực hành tiếp theo, một phần triển khai mã nguồn mở có sẵn có thể được dùng làm ambassador, giúp đẩy nhanh quá trình phát triển toàn bộ hệ phân tán.

## Hands On: Implementing a Sharded Redis — Thực hành: Hiện thực một Redis Phân mảnh

**Redis** is a fast **key-value store** that can be used as a cache or for more persistent storage. In this example, we’ll be using it as a cache. We’ll begin by deploying a sharded Redis service to a **Kubernetes** cluster. We’ll use the **StatefulSet** **API** object to deploy it, since it will give us unique DNS names for each shard that we can use when configuring the proxy.

> Redis là một kho key-value nhanh, có thể dùng làm cache hoặc để lưu trữ bền vững hơn. Trong ví dụ này, chúng ta sẽ dùng nó làm cache. Chúng ta sẽ bắt đầu bằng việc triển khai một dịch vụ Redis phân mảnh lên một cụm Kubernetes. Chúng ta sẽ dùng đối tượng API StatefulSet để triển khai nó, vì nó sẽ cho chúng ta những tên DNS duy nhất cho từng mảnh mà ta có thể dùng khi cấu hình proxy.

The StatefulSet for Redis looks like this:

> StatefulSet cho Redis trông như sau:

```
apiVersion: apps/v1beta1
kind: StatefulSet
metadata:
  name: sharded-redis
spec:
  serviceName:"redis"
  replicas: 3
  template:
    metadata:
      labels:
        app: redis
    spec:
      terminationGracePeriodSeconds: 10
      containers:
      - name: redis
        image: redis
        ports:
        - containerPort: 6379
          name: redis
```

Save this to a file named redis-shards.yaml and you can deploy this with **kubectl** create -f redis-shards.yaml . This will create three containers running redis. You can see these by running kubectl get pods ; you should see sharded-redis-

Of course, just running the replicas isn’t sufficient; we also need names by which we can refer to the replicas. In this case, we’ll use a Kubernetes Service , which will create DNS names for the replicas we have created. The Service looks like this:

> Tất nhiên, chỉ chạy các bản sao thôi thì chưa đủ; chúng ta cũng cần những tên để tham chiếu tới các bản sao đó. Trong trường hợp này, chúng ta sẽ dùng một Service của Kubernetes, thứ sẽ tạo các tên DNS cho những bản sao mà ta đã tạo. Service trông như sau:

```
apiVersion: v1
kind: Service
metadata:
  name: redis
  labels:
    app: redis
spec:
  ports:
  - port: 6379
    name: redis
  clusterIP: None
  selector:
    app: redis
```

Save this to a file named redis-service.yaml and deploy with kubectl create -f redis-service.yaml . You should now have DNS entries for sharded-redis-0.redis , sharded-redis-1.redis , etc. We can use these names to configure **twemproxy** . twemproxy is a lightweight, highly performant proxy for **memcached** and Redis , which was originally developed by Twitter and is open source and available on GitHub . We can configure twemproxy to point to the replicas we created by using the following configuration:

> Lưu nội dung này vào một tệp tên là redis-service.yaml và triển khai bằng kubectl create -f redis-service.yaml. Bây giờ bạn sẽ có các mục DNS cho sharded-redis-0.redis, sharded-redis-1.redis, v.v. Chúng ta có thể dùng những tên này để cấu hình twemproxy. twemproxy là một proxy nhẹ, hiệu năng cao cho memcached và Redis, ban đầu được Twitter phát triển và là mã nguồn mở, có sẵn trên GitHub. Chúng ta có thể cấu hình twemproxy trỏ tới những bản sao đã tạo bằng cách dùng cấu hình sau:

```
redis:
  listen: 127.0.0.1:6379
  hash: fnv1a_64
  distribution: ketama
  auto_eject_hosts: true
  redis: true
  timeout: 400
  server_retry_timeout: 2000
  server_failure_limit: 1
  servers:
   - sharded-redis-0.redis:6379:1
   - sharded-redis-1.redis:6379:1
   - sharded-redis-2.redis:6379:1
```

In this config, you can see that we are serving the Redis protocol on localhost:6379 so that the application container can access the ambassador. We will deploy this into our ambassador **pod** using a Kubernetes **ConfigMap** object that we can create with:

> Trong cấu hình này, bạn có thể thấy chúng ta đang phục vụ giao thức Redis trên localhost:6379 để container ứng dụng có thể truy cập ambassador. Chúng ta sẽ triển khai nó vào pod ambassador của mình bằng một đối tượng ConfigMap của Kubernetes mà ta có thể tạo bằng:

```
kubectl create configmap --from-file=nutcracker.yaml
```

Finally, all of the preparations are done and we can deploy our ambasssador example. We define a pod that looks like:

> Cuối cùng, mọi sự chuẩn bị đã xong và chúng ta có thể triển khai ví dụ ambassador của mình. Chúng ta định nghĩa một pod trông như sau:

```
apiVersion: v1
kind: Pod
metadata:
  name: ambassador-example
spec:
  containers:
# This is where the application container would go, for example
# - name: nginx
#   image: nginx
# This is the ambassador container
    - name: twemproxy
      image: ganomede/twemproxy
      command:
      - nutcracker
      - -c
      - /etc/config/nutcracker.yaml
      - -v
      - 7
      - -s
      - 6222
```

This pod defines the ambassador; then the specific user’s application container can be injected to complete the pod.

> Pod này định nghĩa ambassador; sau đó container ứng dụng cụ thể của người dùng có thể được tiêm vào để hoàn thiện pod.

## Using an Ambassador for Service Brokering — Dùng Ambassador để Làm Trung gian Dịch vụ

When trying to render an application **portable** across multiple environments (e.g., public cloud, physical datacenter, or private cloud), one of the primary challenges is **service discovery** and configuration. To understand what this means, imagine a frontend that relies on a MySQL database to store its data. In the public cloud, this MySQL service might be provided as software-as-a-service (SaaS), whereas in a private cloud it might be necessary to dynamically spin up a new virtual machine or container running MySQL.

> Khi cố gắng làm cho một ứng dụng có thể di động (portable) qua nhiều môi trường (ví dụ: public cloud, trung tâm dữ liệu vật lý, hoặc private cloud), một trong những thách thức chính là phát hiện dịch vụ (service discovery) và cấu hình. Để hiểu điều này nghĩa là gì, hãy hình dung một frontend dựa vào một cơ sở dữ liệu MySQL để lưu dữ liệu của nó. Trên public cloud, dịch vụ MySQL này có thể được cung cấp dưới dạng phần mềm như một dịch vụ (software-as-a-service, SaaS), trong khi trên private cloud có thể cần khởi tạo động một máy ảo hoặc container mới chạy MySQL.

Consequently, building a portable application requires that the application know how to **introspect** its environment and find the appropriate MySQL service to connect to. This process is called service discovery , and the system that performs this discovery and linking is commonly called a **service broker** . As with previous examples, the ambassador pattern enables a system to separate the logic of the application container from the logic of the service broker ambassador. The application simply always connects to an instance of the service (e.g., MySQL) running on localhost. It is the responsibility of the service broker ambassador to introspect its environment and broker the appropriate connection. This process is shown in Figure 3-3 .

> Do đó, việc xây dựng một ứng dụng di động đòi hỏi ứng dụng biết cách xem xét nội tại (introspect) môi trường của nó và tìm ra dịch vụ MySQL phù hợp để kết nối. Quá trình này gọi là phát hiện dịch vụ (service discovery), và hệ thống thực hiện việc phát hiện và liên kết này thường được gọi là bộ trung gian dịch vụ (service broker). Như trong các ví dụ trước, mẫu ambassador cho phép một hệ thống tách logic của container ứng dụng khỏi logic của ambassador trung gian dịch vụ. Ứng dụng đơn giản luôn kết nối tới một thực thể của dịch vụ (ví dụ: MySQL) chạy trên localhost. Trách nhiệm của ambassador trung gian dịch vụ là xem xét nội tại môi trường của nó và làm trung gian cho kết nối phù hợp. Quá trình này được minh họa trong Hình 3-3.

## Using an Ambassador to Do Experimentation or Request Splitting — Dùng Ambassador để Thử nghiệm hoặc Tách Yêu cầu

A final example application of the ambassador pattern is to perform experimentation or other forms of **request splitting**. In many production systems, it is advantageous to be able to perform request splitting, where some fraction of all requests are not serviced by the main production service but rather are redirected to a different implementation of the service. Most often, this is used to perform experiments with new, beta versions of the service to determine if the new version of the software is reliable or comparable in performance to the currently deployed version.

> Một ví dụ ứng dụng cuối cùng của mẫu ambassador là thực hiện thử nghiệm hoặc các hình thức tách yêu cầu (request splitting) khác. Trong nhiều hệ thống sản xuất, việc có khả năng tách yêu cầu là một lợi thế, theo đó một phần nhỏ trong tất cả các yêu cầu không được phục vụ bởi dịch vụ sản xuất chính mà thay vào đó được chuyển hướng tới một phần triển khai khác của dịch vụ. Thông thường nhất, điều này được dùng để thực hiện các thử nghiệm với những phiên bản beta mới của dịch vụ nhằm xác định xem phiên bản phần mềm mới có đáng tin cậy hay có hiệu năng tương đương với phiên bản đang triển khai hiện tại hay không.

Additionally, request splitting is sometimes used to **tee** or split traffic such that all traffic goes to both the production system as well as a newer, undeployed version. The responses from the production system are returned to the user, while the responses from the tee-d service are ignored. Most often, this form of request splitting is used to simulate production load on the new version of the service without risking impact to existing production users.

> Ngoài ra, tách yêu cầu đôi khi được dùng để rẽ nhánh (tee) hoặc tách lưu lượng sao cho toàn bộ lưu lượng đi tới cả hệ thống sản xuất lẫn một phiên bản mới hơn, chưa triển khai. Các phản hồi từ hệ thống sản xuất được trả về cho người dùng, trong khi các phản hồi từ dịch vụ được rẽ nhánh (tee) thì bị bỏ qua. Thông thường nhất, hình thức tách yêu cầu này được dùng để mô phỏng tải sản xuất trên phiên bản mới của dịch vụ mà không gây rủi ro ảnh hưởng tới những người dùng sản xuất hiện hữu.

Given the previous examples, it is straightforward to see how a request-splitting ambassador can interact with an application container to implement request splitting. As before, the application container simply connects to the service on localhost, while the ambassador container receives the requests, proxies responses to both the production and experimental systems, and then returns the production responses back as if it had performed the work itself.

> Với các ví dụ trước, có thể dễ dàng thấy được làm thế nào một ambassador tách yêu cầu tương tác với một container ứng dụng để hiện thực việc tách yêu cầu. Như trước, container ứng dụng đơn giản kết nối tới dịch vụ trên localhost, trong khi container ambassador nhận các yêu cầu, làm proxy cho các phản hồi tới cả hệ thống sản xuất lẫn hệ thống thử nghiệm, rồi trả các phản hồi sản xuất về như thể chính nó đã thực hiện công việc.

This separation of concerns keeps the code in each container slim and focused, and the modular factoring of the application ensures that the request-splitting ambassador can be reused for a variety of different applications and settings.

> Sự tách bạch các mối quan tâm này giữ cho mã trong mỗi container gọn nhẹ và tập trung, và việc phân tách mô-đun của ứng dụng đảm bảo rằng ambassador tách yêu cầu có thể được tái sử dụng cho nhiều ứng dụng và bối cảnh khác nhau.

## Hands On: Implementing 10% Experiments — Thực hành: Hiện thực Thử nghiệm 10%

To implement our request-splitting experiment, we’re going to use the **nginx** web server. Nginx is a powerful, richly featured open source server. To configure nginx as the ambassador, we’ll use the following configuration (note that this is for HTTP but it could easily be adapted for HTTPS as well).

> Để hiện thực thử nghiệm tách yêu cầu của mình, chúng ta sẽ dùng máy chủ web nginx. Nginx là một máy chủ mã nguồn mở mạnh mẽ, giàu tính năng. Để cấu hình nginx làm ambassador, chúng ta sẽ dùng cấu hình sau (lưu ý rằng đây là cho HTTP nhưng nó cũng có thể dễ dàng được thích ứng cho HTTPS).

```
worker_processes  5;
error_log  error.log;
pid        nginx.pid;
worker_rlimit_nofile 8192;
events {
  worker_connections  1024;
}
http {
    upstream backend {
        ip_hash;
        server web weight=9;
        server experiment;
    }
    server {
        listen localhost:80;
        location / {
            proxy_pass http://backend;
        }
    }
}
```

![](images/ch03-41_1.png)

As with the previous discussion of sharded services, it’s also possible to deploy the experiment framework as a separate **microservice** in front of your application instead of integrating it as a part of your client pods. Of course, by doing this you are introducing another service that needs to be maintained, scaled, monitored, etc. If experimentation is likely to be a longstanding component in your architecture, this might be worthwhile. If it is used more occasionally, then a client-side ambassador might make more sense.

> Cũng như phần thảo luận trước về các dịch vụ phân mảnh, ta cũng có thể triển khai khung thử nghiệm như một microservice riêng biệt đặt trước ứng dụng của bạn thay vì tích hợp nó như một phần của các pod client. Tất nhiên, khi làm vậy bạn đang đưa vào thêm một dịch vụ cần được bảo trì, mở rộng quy mô, giám sát, v.v. Nếu việc thử nghiệm có khả năng trở thành một thành phần lâu dài trong kiến trúc của bạn, thì điều này có thể đáng làm. Nếu nó chỉ được dùng thỉnh thoảng, thì một ambassador phía client có thể hợp lý hơn.

You’ll note that I’m using **IP hashing** in this configuration. This is important because it ensures that the user doesn’t flip-flop back and forth between the experiment and the main site. This assures that every user has a consistent experience with the application.

> Bạn sẽ nhận thấy rằng tôi đang dùng băm theo IP (IP hashing) trong cấu hình này. Điều này quan trọng vì nó đảm bảo người dùng không bị nhảy qua nhảy lại giữa thử nghiệm và trang chính. Việc này đảm bảo mỗi người dùng có một trải nghiệm nhất quán với ứng dụng.

The weight parameter is used to send 90% of the traffic to the main existing application, while 10% of the traffic is redirected to the experiment.

> Tham số weight được dùng để gửi 90% lưu lượng tới ứng dụng chính hiện hữu, trong khi 10% lưu lượng được chuyển hướng tới thử nghiệm.

As with other examples, we’ll deploy this configuration as a ConfigMap object in Kubernetes:

> Như với các ví dụ khác, chúng ta sẽ triển khai cấu hình này như một đối tượng ConfigMap trong Kubernetes:

```
kubectl create configmaps --from-file=nginx.conf
```

Of course, this assumes that you have both a web and experiment service defined. If you don’t, you need to create them now before you try to create the ambassador container, since nginx doesn’t like to start if it can’t find the services it is proxying to. Here are some example service configs:

> Tất nhiên, điều này giả định rằng bạn đã định nghĩa cả dịch vụ web và experiment. Nếu chưa, bạn cần tạo chúng ngay bây giờ trước khi cố tạo container ambassador, vì nginx không thích khởi động nếu nó không tìm thấy các dịch vụ mà nó làm proxy tới. Dưới đây là một số ví dụ cấu hình dịch vụ:

```
# This is the 'experiment' service
apiVersion: v1
kind: Service
metadata:
  name: experiment
  labels:
    app: experiment
spec:
  ports:
  - port: 80
    name: web
  selector:
# Change this selector to match your application's labels
    app: experiment
---
# This is the 'prod' service
apiVersion: v1
kind: Service
metadata:
  name: web
  labels:
    app: web
spec:
  ports:
  - port: 80
    name: web
  selector:
# Change this selector to match your application's labels
    app: web
```

And then we will deploy nginx itself as the ambassador container within a pod:

> Và sau đó chúng ta sẽ triển khai chính nginx như container ambassador bên trong một pod:

```
apiVersion: v1
kind: Pod
```

```
metadata:
  name: experiment-example
spec:
  containers:
# This is where the application container would go, for example
# - name: some-name
#   image: some-image
# This is the ambassador container
    - name: nginx
      image: nginx
      volumeMounts:
      - name: config-volume
        mountPath: /etc/nginx
  volumes:
    - name: config-volume
      configMap:
        name: experiment-config
```

You can add a second (or third, or fourth) container to the pod to take advantage of the ambassador.

> Bạn có thể thêm một container thứ hai (hoặc thứ ba, hoặc thứ tư) vào pod để tận dụng ambassador.
